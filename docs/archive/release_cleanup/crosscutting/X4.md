# X4 — Public-API ergonomics (the shop window)

Scope swept: the public C++ headers (`include/steppe/*.hpp`), the CLI surface (`src/app/cli_parse.cpp`, `main.cpp`, the `cmd_*.cpp` error strings, `result_emit.cpp`), and the Python facade (`bindings/steppe/__init__.py` + the compiled-extension docstrings in `bindings/module.cpp`). Compared names / error messages / docstrings ACROSS the three boundaries.

Overall this is a strong, deliberate shop window: the headers are uniformly documented, CUDA-free by contract, the result types are consistent (parallel-array tables + a per-call `status` value + a `precision_tag`), the CLI error messages are uniformly `steppe <subcommand>: <reason>` and actionable, and the Python facade has docstrings, `__all__`, type hints, and `__repr__` on every result class. The findings below are the small set of real inconsistencies / stale text — none are HIGH.

## Findings

- [X4][MED] bindings/module.cpp:1176 — the `run_qpdstat` compiled-extension docstring ends "The normalized-D magnitude needs a genotype prefix (Part B, **not yet implemented**)." Part B IS implemented and shipped: `run_dstat` is defined directly below (module.cpp:1178) and the Python facade exposes it as `steppe.dstat` (`__init__.py:818`). A user reading `help(steppe._core.run_qpdstat)` is told a shipped feature does not exist. Suggested: replace with "use `steppe.dstat(prefix, ...)` / the CLI `qpdstat --prefix` for the genotype-path normalized-D (Part B)."

- [X4][MED] precision-token vocabulary is INCONSISTENT across the three boundaries. CLI (`config_builder.cpp:270-280`) accepts `emu40 | emu32 | fp64 | tf32` (`emu` = emu40); Python `extract_f2`/`qpfstats` (`module.cpp:806-814`, `__init__.py:436-438`) accept `fp64 | native | emulated_fp64 | emu | tf32` — it has `native` and `emulated_fp64` (which the CLI does NOT), and lacks `emu40`/`emu32` so the 32-bit mantissa is UNSELECTABLE from Python; the emitted `precision` column/field uses a THIRD spelling `emu | tf32 | fp64` (`result_emit.cpp:129-131`, `module.cpp:853`) while the `_core` enum mapper returns `emulated_fp64` (`module.cpp:163-165`). So "emulated FP64" is spelled `emu40`, `emulated_fp64`, and `emu` depending on which door you came through. Suggested: pick one accepted-token set + one emitted token (e.g. accept `emu40|emu32|fp64|tf32` everywhere, emit `emu40`), and have Python accept the same tokens the CLI does (add `emu32`, keep `native` as a documented alias).

- [X4][LOW] result_emit.cpp:298 (CSV) and :339 (JSON) — the `qpadm-rotate` per-model table emits `r.est_rank` under a column/field NAMED `"f4rank"`, whereas the single-model `qpadm` emitter puts the real `r.f4rank` under `"f4rank"` (result_emit.cpp:122/196) AND `est_rank` under a separate `"est_rank"` column. So the same `f4rank` column name carries a different quantity between the `qpadm` and `qpadm-rotate` outputs of the same tool. The Python `_search_dataframe` (`__init__.py:1003`) reads `r.f4rank` for its `f4rank` column, so the Python rotation table and the CLI rotation table disagree on what `f4rank` means. Suggested: in the rotation emitters emit `r.f4rank` (matching the single-model emitter + the Python path), or rename the rotation column to `est_rank`.

- [X4][LOW] bindings/steppe/__init__.py:855-879 — `dates()` returns a bare `dict` (and is type-hint-less: no `-> ...`), while EVERY other public entry returns a typed, `__repr__`-bearing result class (`QpAdmResult`, `F4Result`, `QpGraphResult`, ...). It is the one ergonomic outlier in the facade — no `.table`/attribute accessors, no repr, the caller index-keys a dict. The C++ side already has a clean `DatesResult` value type (`dates.hpp:78`). Suggested: wrap it in a small `DatesResult` Python class (mirroring the others) and add to `__all__`, or at minimum a `-> dict` hint + a documented key list as a `TypedDict`.

- [X4][LOW] src/app/cli_parse.hpp:9-12 and cli_parse.cpp:3,356 — stale "M(cli-0) SCAFFOLD" / "qpwave remains a scaffold no-op" comments. The header docstring still says "the fit subcommands ... print a 'not yet implemented' notice and exit cleanly", but cli_parse.cpp:456-463 fully wires `qpwave` to `run_qpwave_command` and the cpp's own line 13 says "qpadm, qpwave, qpadm-rotate, and extract-f2 are ALL wired." A reviewer opening the CLI parser header first is told the opposite of the truth. (Comment-only; not user-facing at runtime, hence LOW.) Suggested: drop the scaffold framing from cli_parse.hpp:9-12 and the "qpwave ... remains a scaffold no-op" at cli_parse.cpp:356.

- [X4][LOW] CLI/Python surface ASYMMETRY for `qpdstat`: the CLI `qpdstat` is one subcommand with two paths (`--f2-dir` → f4 Part A; `--prefix` → genotype normalized-D Part B) (cli_parse.cpp:500-531), but the Python facade splits these into two differently-named functions — `qpdstat(f2, quartets)` (Part A) and `dstat(prefix, quartets)` (Part B) (`__init__.py:786`, `:818`). A user moving between the CLI and Python has to learn that the single CLI `qpdstat` maps to two Python names, and that the Python `dstat` name has no CLI counterpart. The split is defensible (different inputs: handle vs prefix), and the docstrings cross-reference each other (`__init__.py:802`), so this is documented, not broken. Suggested: keep, but note the mapping prominently (e.g. a one-line "CLI equivalent: `qpdstat --prefix`" in the `dstat` docstring).

- [X4][LOW] naming asymmetry: standalone-stat Python functions take their tuples as a POSITIONAL second arg (`f4(f2, quartets)`, `f3(f2, triples)`, `f4ratio(f2, tuples)`, `qpdstat(f2, quartets)`; `__init__.py:760,882,910,786`) while `qpadm` / `qpwave` / `qpadm_search` take ALL inputs as keyword-only (`*, target=, left=, right=`; `__init__.py:499,527,940`). The mix is reasonable (the fit functions have many named knobs; the stat functions have one list), and `as_dataframe` is keyword-only on the stat path — but a user cannot uniformly predict positional-vs-keyword across the surface. Suggested: optional — document the convention ("the one bulk input is positional; everything else is keyword-only") once at module level.

## The overall recommendation

The shop window is clean and senior-grade: consistent result-type shape, uniform `steppe <cmd>: <reason>` CLI errors, uniform GPU-only "no CUDA device" message (module.cpp:105/701/735/838/920), every header CUDA-free with a precise banner, every Python result class with `__all__`/hints/`__repr__`. The only items that would embarrass at a hiring bar are the two MED stale/contradictory user-facing strings (the "Part B not yet implemented" docstring at module.cpp:1176, and the three-way precision-token vocabulary split) — fix those two first. The LOW items are polish: align the rotation `f4rank` column, give `dates()` a typed result + hint, and scrub the stale "scaffold" comments in the CLI parser. There is no awkward or unprofessional naming, no leaked internal type, and no missing/cryptic error path.
