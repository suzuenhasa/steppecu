# X6 — Simplicity / over-engineering

Scope: cross-file sweep of `src/ include/ bindings/` (docs excluded). The headline up
front: the codebase is, on balance, NOT over-engineered for its problem. The bulk of
what *looks* heavy — a 50-virtual `ComputeBackend`, opaque PIMPL device handles, the
`*_core.{hpp,cpp}` splits, the tier/streaming/multi-GPU machinery — is genuinely-earned
GPU-shape and host-testability complexity (the CUDA-free `core`↔`device` seam, GPU-free
unit tests against a fake backend, parity-by-construction oracle/device twins). I did
NOT flag those. The findings below are the residue: a few seams/knobs built ahead of (or
past) the code that would consume them.

## Findings

- [X6][MED] include/steppe/config.hpp:353,357,362 (+ kDefaultSearchStreams :183) — Three
  `DeviceConfig` knobs are dead configuration: `stream_count`, `search_streams`, and
  `use_mem_pool` are declared, heavily documented, defaulted, and asserted in
  `tests/unit/test_config.cpp`, but NEVER read by any production path. Grep across
  `src/ bindings/` (minus config.hpp) returns 0 hits for each: `stream_count` is only in
  test_config + comments; `search_streams`/`kDefaultSearchStreams` only in config.hpp +
  test_config; `use_mem_pool` only in config.hpp + test_config. `config_builder.cpp`,
  `resources.cpp`, and `cuda_backend.cu` consume none of them (e.g. the actual allocator
  is `cudaMallocAsync` unconditionally; `use_mem_pool` never gates it). These are config
  knobs nothing uses — the test that "exercises" them only checks the defaults equal the
  literals. Suggested: drop the three fields (and `kDefaultSearchStreams`) until a real
  consumer lands, or, if they are forward-reserved for the parked multi-GPU/S8 work, mark
  them as such in one line rather than a paragraph implying they are live. (Contrast
  `deterministic` (54 hits), `force_tier` (18), `enable_peer_access` (10) — those ARE
  consumed.)

- [X6][MED] src/device/backend.hpp:1682 `rank_test` — a vestigial virtual carrying two
  full implementations and zero callers. The base throws; it is OVERRIDDEN in BOTH
  backends (cuda_backend.cu:4355, cpu_backend.cpp:1558), yet no code in `src/` ever calls
  `rank_test(` (verified by grep — only the decl + the two overrides match). The actual
  S5/S6 path in `qpadm_fit.cpp` uses `rank_sweep` + `gls_weights` (qpadm_fit.cpp:88,164);
  the `rank_sweep` docstring (backend.hpp:1694) describes `rank_test` as the conceptual
  "seed" but the seam itself is never invoked. It is dead weight that two backends must
  keep in sync. Suggested: delete the `rank_test` virtual and its two overrides (its math
  lives inside the `rank_sweep`/`gls_weights` paths already), or if kept as the documented
  single-rank entry, give it the one caller that justifies a public virtual.

- [X6][LOW] src/device/backend.hpp:1085 `set_solve_precision` — a per-stage solve-precision
  promotion seam with no production consumer. Its own docstring concedes
  "EXPLORATORY/measurement use today … the S8 milestone wires the validated per-stage
  policy." The only caller is the parity TEST (test_qpadm_parity.cu:833,836); the CUDA
  override (cuda_backend.cu:2600) exists but is reached only from that probe. This is a
  generalization built ahead of the policy that would use it (premature abstraction). It is
  LOW because it is a no-op default + a single small override (cheap to carry) and there is
  a concrete ROADMAP §6 plan to wire it. Suggested: leave it, but treat it as the canonical
  example of "seam landed before its consumer" — do not add more such promotion knobs
  speculatively before S8 actually needs them.

- [X6][LOW] src/device/backend.hpp:446 `enum class TileEncoding { Identity }` (+ the device
  twin `TransposeEncoding` at transpose_canonical_kernel.cuh:44) — a single-enumerator enum
  with `switch (encoding) { case Identity: default: … }` blocks in three places
  (backend.hpp:962, cuda_backend.cu:1337, transpose_canonical_kernel.cu:64) and a comment
  "later formats add enumerators." Every real reader is `Identity` (read_canonical_tile.cpp:37
  and all the io-leaf call sites pass Identity). This is premature generalization: a 1-value
  enum, its mirror twin, and three switch statements model a remap that does not yet exist.
  LOW because it is a deliberately-documented format-extension seam (EIGENSTRAT/PLINK are
  named as the future second value) and the cost is tiny. Suggested: acceptable as-is given
  the explicit roadmap; if a second format does not land soon, collapse the enum + the three
  switches to a plain identity copy and reintroduce the enum when the second encoding is real.

- [X6][LOW] cross-file — the ploidy-detection scan loop exists in four near-identical copies:
  `io::detect_sample_ploidy` (ploidy_detect.cpp), the backend.hpp:863 base default
  `detect_sample_ploidy_device` (host loop), the CUDA `detect_ploidy_kernel.cu:45`, and a
  CpuBackend re-port (cpu_backend.cpp:108). Each is annotated "LITERAL port … bit-parity by
  construction." This is the parity-oracle tension, not accidental duplication: the device
  kernel and the host oracle are *required* to be independent re-derivations so the bit-parity
  gate is meaningful, and all four share the same `core::genotype_code`/`kHeterozygousGenotypeCode`
  primitives (so they cannot silently drift on the *math*, only on the *loop shape*). LOW /
  borderline-not-a-finding. Suggested: no change — but note it as the one place where the
  "literal port for parity" doctrine produces the most copies; if a fifth port is ever
  proposed, prefer sharing a single `STEPPE_HD` scan primitive (the f2_estimator.hpp pattern)
  over a fourth hand-copy.

## The canonical form / recommendation

Overall recommendation: this is a senior-grade, deliberately-layered codebase, and the
right X6 verdict is "clean, with a short dead-seam list" — not "over-engineered." The
heavy machinery earns its keep: the 50-virtual fat interface is the price of the
compiler-enforced CUDA-free `core` boundary; the opaque PIMPL handles (DevicePartial /
DeviceF2Blocks / DeviceDecodeResult) are exactly how that boundary stays CUDA-free; the
`F2Src` template (5 instantiations across f3/qpadm/qpwave/qpgraph/model_search) is the
*simpler* form (it dedups the orchestration body across the host-tensor-oracle and
device-resident paths rather than copy-pasting it); the `*_core.{hpp,cpp}` splits exist
specifically to make the orchestration host-unit-testable against a fake backend with no
GPU. None of that is the "too-clever indirection / abstraction with one caller" X6 is
hunting for.

The actionable residue is small and shares one theme — seams built ahead of, or kept
past, their consumers:
  1. delete or explicitly-reserve the three dead `DeviceConfig` knobs (MED, the clearest
     "config knob nothing uses" hit);
  2. delete the `rank_test` virtual + its two overrides (MED, a dead virtual with two
     live implementations to maintain);
  3. accept `set_solve_precision`, the `TileEncoding` enum, and the ploidy-port copies as
     documented forward-seams, but treat them as the ceiling — resist adding more
     speculative promotion knobs / format enumerators / hand-ported loops before a real
     consumer exists.
