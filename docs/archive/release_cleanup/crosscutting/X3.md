# X3 — Architecture / structure / navigability

Scope: whole-codebase structural review of `src/ include/ bindings/` (docs/ excluded).
Layering model under test is architecture.md §4: include/steppe (public) -> src/core
(host-pure) -> src/device (backend.hpp seam + cpu/cuda) ; src/io = isolated CUDA-free
leaf ; bindings/ = nanobind facade.

Headline: the layering is real and compiler-enforced — io is a clean CUDA-free leaf, core
is host-pure, CUDA stays PRIVATE to steppe_device, public headers leak no internals. The
structural debt is concentrated in a handful of GOD-FILES (cuda_backend.cu 5648 LOC, a
single 5400-line class), a very FAT backend seam (~50 virtuals), and two naming/placement
smells (the `qpadm/` dir is a grab-bag; access/extract targets compile sources that live in
src/app).

## Findings

- [X3][HIGH] src/device/cuda/cuda_backend.cu:241-5648 — GOD-FILE. The entire `CudaBackend final` class is ONE class body spanning 241..5648 (~5400 LOC), implementing ~30 `ComputeBackend` overrides plus a swarm of private helpers (`stream_f2_blocks_impl` @844, `large_svd_V`, `fit_one_bucket`, `fill_rankdrop` @204) in a single translation unit (file total 5648 LOC, the largest in the tree by 2.5x). Individual private method spans exceed 400 LOC (decode/transpose region 818..1237 ≈ 419 lines; an SVD-path region near 4358 ≈ 305 lines). Kernels are correctly factored OUT (zero `__global__` defined here — they live in `*_kernel.cu`), so the bloat is pure host-side orchestration glue. A new engineer cannot navigate one 5400-line class. Suggested: split CudaBackend into per-subsystem partial impls grouped by seam concern (f2-blocks / decode+transpose / fit / sweep / jackknife / dates), one .cu (or shared .inc) each, keeping the single class declared via a thin aggregate header.

- [X3][MED] src/device/backend.hpp:667-1842 — FAT INTERFACE. `ComputeBackend` declares ~50 `virtual` methods (grep: 50 `virtual` lines), only one truly pure (`= 0` count is inflated by array inits; the visible pure-virtuals are the core compute set, the rest carry default bodies). One seam now spans f2 (compute_f2/_blocks/_resident/_into/_streamed), decode (decode_af/_compact_*), transpose, f4/f3 assembly (8 overloads), qpgraph fleet, dstat, dates (3), qpfstats smooth (3), 6 jackknife variants, sweep, rank/gls/se, and fit_models_batched. This is a single interface absorbing every device operation across all 9 tools — high coupling, and every CpuBackend/CudaBackend must track all of it (cpu_backend.cpp carries 25 overrides). Suggested: segregate into role interfaces (IF2Backend / IDecodeBackend / IFitBackend / IStatsBackend) the concrete backends multiply-inherit, so a tool only depends on the slice it uses and a new override surface is scoped.

- [X3][MED] src/core/qpadm/ (27 files) — MISNOMER / GRAB-BAG DIRECTORY. The directory named `qpadm/` actually houses qpadm_fit + ranktest + nested_models AND all of qpgraph (qpgraph_model/_fit/_objective/_enumerate/_search), the standalone f4/f3/f4ratio statistics, fstat_sweep, and model_search. Eight distinct subsystems under one tool's name. By contrast genuinely-distinct tools sit in src/core/stats/ (dstat/qpfstats/dates). The `qpadm/` name actively misleads navigation — qpGraph topology search is not qpAdm. Suggested: rename to src/core/fit/ (the GLS/fit-engine family) or split qpgraph_* into src/core/qpgraph/ and f3/f4/f4ratio/fstat_sweep into the existing src/core/fstats/ alongside the f2 machinery.

- [X3][MED] src/access/CMakeLists.txt:31-33, src/extract/CMakeLists.txt:25-27 — TARGET/SOURCE LOCATION SPLIT. steppe::access and steppe::extract are real library targets whose source files physically live in src/app/ (access compiles ../app/f2_dir_io.cpp + ../app/pop_resolver.cpp; extract compiles ../app/extract_f2_core.cpp + ../app/f2_dir_writer.cpp via `${CMAKE_CURRENT_SOURCE_DIR}/../app/...`). So src/access/ and src/extract/ each contain ONLY a CMakeLists.txt, and four src/app/ .cpp files belong to libraries other than the app. The CMake comments justify it (DRY: shared by CLI + bindings), but a reader who opens src/access/ finds an empty-looking dir, and a reader in src/app/ can't tell which files are app vs access vs extract. Suggested: physically move f2_dir_io.cpp/pop_resolver.cpp into src/access/ and extract_f2_core.cpp/f2_dir_writer.cpp into src/extract/ so target boundaries match directory boundaries.

- [X3][LOW] src/device/{cpu/cpu_backend.cpp:66-68, cuda/cuda_backend.cu:91, cuda/qpadm_fit_kernels.cu:24} -> core/qpadm/* — DEVICE INCLUDES THE ORCHESTRATION LAYER'S HEADERS. The device layer (conceptually below core in §4) includes core/qpadm/qpadm_bounds.hpp, qpgraph_model.hpp, qpgraph_objective.hpp. Verified NON-fatal: all consumed functions (qpgraph_score/opt_edge_lengths @ qpgraph_objective.hpp:125,188; fill_pwts_centered @ qpgraph_model.hpp:128) are `inline`, the only .cpp-compiled symbol (parse_qpgraph @ qpgraph_model.cpp:45) is NOT used by device, and steppe_device does NOT link steppe::core (device CMake:63-65 links only api + core_internal). So it's a header-only single-source DRY edge, not a link cycle. But it does mean the device TUs reach UP into core/qpadm/'s namespace for the fleet-objective oracle + bounds constants, blurring the "device sits below core" story. Suggested: relocate the device-shared header-only primitives (qpadm_bounds, the fleet objective body) into core/internal/ (the existing CUDA-free shared-helper home that device already legitimately depends on via core_internal), so the device->core edge points only at core_internal.

- [X3][LOW] bindings/module.cpp:1-1239 — ALL BINDINGS IN ONE TU. The nanobind facade is a single 1239-LOC file holding 16 `m.def`s plus their hand-written marshalling wrappers (run_qpadm_py @366, run_dstat_py @646, run_dates_py @722, run_extract_f2_py @777, run_qpfstats_py @894, the 6 *_to_dict converters @174..433, etc.) and the F2Handle class @1119. It is cohesive (one module) but every tool's numpy<->struct glue lives together; adding a tool grows this one file. Lower severity than the .cu god-file because it is flat glue, not deep logic. Suggested: split per-tool wrapper TUs (bindings/bind_qpadm.cpp, bind_dates.cpp, ...) each contributing to NB_MODULE via small register_*(m) entry points; keep module.cpp as the thin assembler.

- [X3][LOW] src/device/cpu/cpu_backend.cpp:1-2263 — second-largest single file (2263 LOC) carrying all 25 ComputeBackend overrides for the parity oracle. It is the test-only oracle (MEMORY: CPU is test-only) so the bar is lower, but it mirrors the same fat-seam problem as cuda_backend.cu and would shrink naturally if the seam were segregated (see the FAT INTERFACE finding). Suggested: track with the backend.hpp interface-segregation change rather than splitting independently.

## The canonical form / recommendation

What is genuinely good and should be PRESERVED as the structural baseline:
- §4 layering HOLDS where it matters: io is a clean CUDA-free leaf (no core/device includes except the header-only STEPPE_HD macro at filter_decision.hpp:53 / snp_summary_reduce.hpp:37, which carries no link edge); core has zero `#include "app/"` (no upward edge); CUDA toolkit headers are PRIVATE to steppe_device (device CMake:65) so core physically cannot include a CUDA header; public headers (include/steppe, 14 files / ~2000 LOC) include ZERO src internals.
- The DRY single-source discipline is exemplary and well-documented: one f2 estimator (core/internal/f2_estimator.hpp), one block-partition rule, one launch-math home, STEPPE_HD for host/device-shared primitives, and the CMake files explain every cross-target link edge.
- src/app is the correct and only wiring layer (5 TUs include io, 15 include device).

Overall recommendation, in priority order: (1) break up cuda_backend.cu — the single 5400-line class is the one finding that materially hurts navigability for a new engineer and is the headline structural debt; (2) segregate the ComputeBackend seam into role interfaces, which also shrinks cpu_backend.cpp; (3) rename/split the misleading core/qpadm/ grab-bag; (4) align the access/extract target dirs with their source files. None of these are correctness or layering VIOLATIONS — the architecture is sound; this is decomposition debt that accrued as 9 tools were grown through one backend class and one tool-named directory.
