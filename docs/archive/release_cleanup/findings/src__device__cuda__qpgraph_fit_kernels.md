### G4
- [G4.src__device__cuda__qpgraph_fit_kernels][MED] qpgraph_fit_kernels.cu:315,325,341,370,444 — `D = t.nadmix` drives writes into fixed-size stack arrays `th[kMaxThetaDev]`, `thp/thm[kMaxThetaDev]` (325), `thn[kMaxThetaDev]` (341), all sized `kMaxThetaDev=16`. The loops `for (int k = 0; k < D; ++k)` write `thp[k]/thm[k]/thn[k]` and `th[d]` for `d < D` with NO guard that `D <= kMaxThetaDev`. Neither host launcher (`launch_qpgraph_fleet` 452, `launch_qpgraph_fleet_batch` 482) rejects `nadmix > 16`, so a topology with more than 16 admixture nodes silently overruns the per-thread stack (corruption, not a crash). The header comment "production nadmix; comfortably large" (cuh:301 / .cu:301) is an assumption, not an enforced bound. Suggested: assert/clamp `nadmix <= kMaxThetaDev` in the host launchers (or in-kernel return a sentinel score) so an oversized topology fails fast instead of corrupting the stack.
- [G4.src__device__cuda__qpgraph_fit_kernels][LOW] qpgraph_fit_kernels.cuh:46-65 — `make_layout` accumulates the slab offset in a plain `int o`, summing several `nedge*nedge` (cc, ccs, nn_Ap, nn_lu) and `npair*nedge` (ppwts, Wm) terms, all int multiplies; `dbl_total` is stored as `int`. For the qpGraph regime (small leaf counts) this never approaches 2^31, but the per-thread total is an `int` and the products are int arithmetic, so a pathologically large topology would overflow before the `long` cast at .cu:368/440. Lower risk than the resident-f2 P~2500 path, but the index type is narrower than the multiply could need. Suggested: compute the offsets / `dbl_total`/`int_total` in `long` (or `std::size_t`) and keep the `long` stride cast on the way out.

### G11
- [G11.src__device__cuda__qpgraph_fit_kernels][LOW] qpgraph_fit_kernels.cu:361-364,390-393,410-418 — kernel pointer parameters that are read-only (`f_obs`, `qinv`, the packed arenas `g_pwts0`/`g_pe_*`/`g_pae_*`/`g_cmb*`, `theta`) and the write-only outputs (`out_theta`, `out_score`, `out_bl`, `out_f3`, `g_dbl`, `g_int`) carry no `__restrict__`. The inner objective re-reads `qinv`/`f_obs`/`ppwts` heavily in tight loops (228, 233, 240, 244, 284); `__restrict__` on the non-aliasing read-only inputs would let the compiler keep them cached. Suggested: add `const ... __restrict__` to the read-only kernel pointers and `__restrict__` to the disjoint outputs.
- [G11.src__device__cuda__qpgraph_fit_kernels][LOW] qpgraph_fit_kernels.cu:361,390,410 — `QpGraphDeviceTopo t` (~ a dozen ints + a double + 8 pointers) and `ScratchLayout L`/`Lmax` (≈21 ints) are passed to the kernels by value. Within the 4KB param limit so not a fault, but the idiom for large read-only by-value kernel structs is `__grid_constant__ const` (constant-bank, avoids the per-thread param copy). Suggested: mark the by-value struct params `__grid_constant__ const`.
