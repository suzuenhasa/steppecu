## Role & layering

`FilterPlan` is the plain aggregate the M2 filter front-end is *intended* to hand across the `io`‚Üí`app` boundary so the S0' "harmonized + filtered tile produce" stage (architecture ¬ß5) can apply the cheap in-tile filters without a second data pass. It carries four things:

- `snp_keep` ‚Äî one `bool` per SNP, the resolved SNP-global keep mask (produced upstream by `snp_filter::build_snp_keep_mask`);
- `n_snp_kept` ‚Äî a cached popcount of `snp_keep`;
- `kept_samples` ‚Äî ascending kept-sample indices over the individual axis (the `--mind` S-1 result, produced by `mind_prepass::run_mind_prepass`);
- `in_tile` ‚Äî a by-value copy of the live `FilterConfig` thresholds the S0' apply still needs.

**One disambiguation the prior draft blurred.** Architecture ¬ß5 distinguishes two `io`-leaf plan artifacts: the **S-2 merge plan** (`io/merge`, line 239 ‚Äî "harmonized SNP set / allele-polarity map / sample-pop column maps / a `.missnp`-equivalent dropped-SNP list ‚Ä¶ Merge is a *plan*, not an on-disk rewrite") and the **S-1/S0' filter producers** (`include_exclude` + `filter_plan` + `snp_filter` + `mind_prepass`, grouped together at line 166). `FilterPlan` is the **filter** plan (single dataset, S0'/S-1), *not* the multi-dataset merge plan; its banner borrows the ¬ß1 "merge is a plan, not an on-disk rewrite" slogan to assert the no-rewrite property, but the type is a sibling of, not the same as, the M6 `io/merge` artifact. This matters for the milestone column (below) and for 9.2.

Layering is **correct and minimal**. Three includes (`<cstddef>`, `<vector>`, `steppe/config.hpp`), all downward. `src/io/CMakeLists.txt` links `steppe_io` only against `steppe::api` (which carries `config.hpp`) PUBLIC and `steppe::warnings` PRIVATE, with `core`/`core_internal`/`device` deliberately absent (lines 34‚Äì37) ‚Äî so this struct physically cannot reach a `core` domain rule, a `core/internal` facade, or a CUDA header, exactly as the ¬ß4 dependency-direction rule (architecture.md line 192) requires, and it crosses the leaf boundary unchanged. The CMake comment (lines 21‚Äì23) correctly classifies it as a header-only struct with no `.cpp`. The deliberate omission of block info (deferring to the shared `core::block_partition_rule`) is the right ¬ß8 single-source decision. All verified correct ‚Äî layering is not where the problems are, and it constrains the *fixes* (see 2.1/11.1: the io leaf cannot reach `std::expected`/`Error`/`STEPPE_LOG`).

The substantive defects are structural: (1) the struct is **never instantiated or read** (dead code; invariants asserted-but-untested, contra ¬ß2 testability and ROADMAP ¬ß6); (2) a real **axis/representation gap** ‚Äî `snp_keep` is documented as dataset-wide but its only producer emits a *per-tile* mask, and the plan records *no denominator* (total-M, total-sample) to validate either index array; (3) several invariants the comments lean on (`n_snp_kept == popcount(snp_keep)`; `kept_samples` ascending + unique + in-range; `in_tile` is the *same* config that built the mask) have **no constructor or validator** to enforce them, contra ¬ß2 fail-fast / single-source; (4) the by-value full-`FilterConfig` embedding drags resolved-away string id lists across the seam and is a **double-filter footgun**; (5) the aggregate offers **no accessors** (`empty()`, derived `n_snp_kept()`, `validate()`) where its sibling `F2BlockTensor` does, breaking house style.

The header *reads* like 9.5/10 work ‚Äî the comment density, the architecture cross-references, and the layering are exemplary. ## Findings

### (1) Correctness & bugs

**1.1 ‚Äî `FilterPlan` is never constructed or consumed anywhere in the repository (dead code). [HIGH, effort M, before-M4.5: no (M5/M6)]**
Location: whole file.
A full grep over `*.hpp/*.cpp/*.cu/*.cuh/CMakeLists.txt` (excluding `docs/`) returns exactly three textual hits, all inert: the `FilterPlan` definition itself (`filter_plan.hpp:3,14,37`) and the CMake comment at `src/io/CMakeLists.txt:22`. No `snp_filter`/`mind_prepass`/`include_exclude` TU assembles a `FilterPlan`; no `app`/`core`/test reads one (verified: `grep -rn FilterPlan` returns no constructor, no field access, no parameter). The header's stated reason for existing ‚Äî "everything the S0' harmonized+filtered tile produce needs to apply the cheap in-tile filters ‚Ä¶ with no second pass over the data" (lines 6‚Äì9) ‚Äî is unrealized. Why it matters: ROADMAP ¬ß3 marks **M2 DONE** and ¬ß6 requires "the new kernel has a CPU-reference-equivalence test"; ¬ß2 makes testability a first principle. Every `///` invariant in this file ("Cached so callers do not re-scan," "ascending," "the SAME ones snp_filter used to build `snp_keep`") is therefore an *unverified* promise. Fix: either (a) add the `io`-leaf factory that assembles a `FilterPlan` from `build_snp_keep_mask` + `run_mind_prepass` + the threshold snapshot, with a `tests/unit/test_filter_plan.cpp` asserting the 1.3/1.4/1.5 invariants and the no-op-all-keep parity; or (b) if it is deliberately ahead of its consumers, say so in the banner and name the wiring milestone (M5/M6), so a reader does not mistake a stub for live code. As written it is in neither state ‚Äî it looks live but is inert.

**1.2 ‚Äî `snp_keep` is documented as dataset-wide, but its only producer computes a per-tile mask, and the plan records no denominator to validate against. [MED (latent HIGH at wiring time), effort M, before-M4.5: no (M5/M6)]**
Location: `snp_keep` (line 43, "Length == number of SNPs considered (M)"); producers `build_snp_keep_mask` (`snp_filter.cpp:61‚Äì108`), `run_mind_prepass` (`mind_prepass.cpp:17`).
`build_snp_keep_mask` returns a mask of length `in.M` (`snp_filter.cpp:66`), where `in.M` is `DecodedTileSummaryInput::M` = "number of SNPs **in the tile**" (`snp_filter.hpp:45`), and its pooled-MAF / missing-fraction derivation runs over the per-pop `Q/V/N` of *that tile* (`derive_per_snp_summary`, `snp_filter.cpp:32‚Äì48`), which already presupposes `--mind`'s kept samples are applied (`pop_individuals` is "intersected with the kept-sample set," `snp_filter.hpp:50‚Äì52`). So `snp_keep` is naturally a *per-tile* product produced *after* `kept_samples`. Yet `FilterPlan::snp_keep` is documented as a *whole-dataset* product (length = "number of SNPs considered (M)"), with no recorded stitching of per-tile masks into a dataset mask and **no field recording the dataset's true total-M or total-sample-count**. `run_mind_prepass` correctly produces `kept` over the *individual* axis (`mind_prepass.cpp:59‚Äì64`, ascending by construction), which the plan rightly calls "the whole sample axis." Why it matters: ¬ß8 single-source + ¬ß1 "merge is a plan." A plan whose two index arrays live on axes the producers compute at different stages, with no denominators to check against, is an integration hazard: the first wiring code will guess the per-tile‚Üídataset stitching and may get file-order vs tile-order wrong ‚Äî a silent SNP misalignment that the ¬ß5-S2 `V¬∑V·µÄ` masking would *not* catch (it would faithfully mask the wrong columns). Fix: state the axis contract precisely (does `snp_keep` index *raw* `.snp` file order or *post-membership* order? ‚Äî since `derive_per_snp_summary` runs over the decoded-tile pop axis, it is per-tile, file-order-within-tile) and add `std::size_t n_snp_total` + `std::size_t n_sample_total` so a validator can assert `snp_keep.size() == n_snp_total` and (when non-empty) `kept_samples.back() < n_sample_total`. Marked MED now because nothing yet stitches masks (no live integration to break); becomes HIGH the moment something composes per-tile masks ‚Äî i.e. exactly the contract to pin before wiring.

**1.3 ‚Äî `n_snp_kept` is a denormalized cache with no enforced relationship to `snp_keep`. [MED, effort S, before-M4.5: no (M5/M6)]**
Location: `n_snp_kept` (line 47, "== count of true in `snp_keep`. Cached so callers do not re-scan").
A classic redundant-field hazard: two public members must agree (`n_snp_kept == std::count(snp_keep.begin(), snp_keep.end(), true)`), and nothing enforces it. `snp_keep` is a public mutable `std::vector<bool>`; a caller that flips one bit leaves `n_snp_kept` stale, and the ¬ß11.2 VRAM-budget math (resident-set sizing uses the kept-SNP count) then silently uses a wrong count. Why it matters: ¬ß2 "no global mutable state ‚Ä¶ the precondition for thread-safety and unit-testability" extends to *no derivable mutable state inside a value object*. The "callers do not re-scan" justification is weak: `std::count` over a bit-packed `vector<bool>` is sub-millisecond at M‚âà600k, run once per run (not per tile) ‚Äî the same M‚âà584k the M2 oracle already iterates per `drop_columns`/`mask_columns` call without concern. Fix: replace the field with an on-demand `[[nodiscard]] std::size_t n_snp_kept() const`, or construct-once via the 1.1 factory and never re-expose the mask for mutation.

**1.4 ‚Äî `kept_samples` ascending/unique/in-range is documented but unenforced; consumers that binary-search or set-difference it will silently misbehave on a malformed plan. [MED, effort S, before-M4.5: no (M5/M6)]**
Location: `kept_samples` (lines 49‚Äì53, "ascending," "the result of the conditional S-1 --mind pre-pass").
The producer `run_mind_prepass` does emit ascending, unique, in-range indices by construction (it `push_back`s `g` in strictly increasing order in a single `for (g=0; g<n_ind; ++g)` loop, `mind_prepass.cpp:59‚Äì64`), so the *current* producer is correct. But `FilterPlan` is a public aggregate any caller can fill by hand or mutate after construction; the "ascending" promise is a comment, not an invariant. A downstream consumer that assumes monotonicity (e.g. to do an O(P) merge against `pop_offsets`, or `std::lower_bound` to test membership) will return wrong results on an out-of-order or duplicate-bearing plan with no diagnostic. Why it matters: ¬ß2 fail-fast. Fix: a debug-only `validate()` asserting `std::is_sorted` + adjacent-unique (`std::adjacent_find`) + `kept_samples.back() < n_sample_total` (the 1.2 denominator), called by the 1.1 factory.

**1.5 ‚Äî A default-constructed `FilterPlan` is indistinguishable from "filtered everything out": `snp_keep` empty, `n_snp_kept == 0`, `kept_samples` empty all read as a valid plan that drops every SNP and every sample. [MED, effort S, before-M4.5: no (M5/M6)]**
Location: in-class initializer (`n_snp_kept = 0`, line 47) + the implicitly-defaulted empty vectors.
The no-op / parity-path default for these filters is **keep everything** (`config.hpp:197‚Äì199` "Defaults are no-ops (keep everything)"; `mind_prepass.cpp:30,60‚Äì62` keeps all samples when `mind_max_missing >= 1.0`). But a *default-constructed* `FilterPlan` has empty `snp_keep`, zero `n_snp_kept`, and empty `kept_samples` ‚Äî which a naive consumer reads as "drop all SNPs and all samples," the exact opposite of the no-op intent. There is nothing (no `populated` flag, no totals) to distinguish "validly kept nothing" from "never built." Why it matters: ¬ß2 fail-fast ‚Äî a value object whose zero-state is a silently-wrong-and-dangerous state is a footgun. Fix: make the plan factory-only (no public default ctor, or carry the 1.2 totals so a validator can tell "kept none" from "never built"); have the factory surface a genuinely all-dropped result via the io-native diagnostic (see 2.1 ‚Äî `throw std::runtime_error`, *not* a non-existent logger).

### (2) Edge cases & failure modes

**2.1 ‚Äî All-SNP-dropped and all-sample-dropped are degenerate cases with no surfaced diagnostic; AND the prior draft's recommended diagnostic path is layering-illegal for an io-leaf. [MED, effort S, before-M4.5: no (M5/M6)]**
Location: whole struct (no construction logic exists to detect it).
If every SNP fails the active filters (`n_snp_kept == 0`) or `--mind` drops every sample (`kept_samples.empty()`), the downstream S2 GEMM path gets an empty SNP axis or an empty sample axis ‚Äî `f2_blocks` is all-zero / `Vpair` all-zero, and the ¬ß5-S2 `(Vpair > 0) ? ‚Ä¶ : 0` guard returns 0 silently. The user sees zeros, not an error. Why it matters: ¬ß2 fail-fast; ROADMAP ¬ß6.
**Correction to the prior draft's fix.** The prior draft recommended returning `std::expected<FilterPlan, Error>` or logging `STEPPE_LOG_WARN`, asserting "`Error`/`Status` already exists at `include/steppe/error.hpp`." I verified the actual state and **both halves are wrong for the io leaf**: (i) `include/steppe/error.hpp` defines `enum class Status` only ‚Äî there is **no `Error` type** (the richer internal `Error` is described in architecture.md ¬ß10/¬ß16 as future and does **not** exist in the tree; `grep -rn 'struct Error|class Error|using Error'` over `src/`+`include/` returns nothing); (ii) the internal `std::expected<T,Error>`/`STEPPE_TRY` mechanism lives in `internal/expected.hpp` (architecture.md line 522) which is `core`-internal and **never crosses the ABI** ‚Äî the io leaf links only `steppe::api`+`steppe::warnings` (CMake lines 34‚Äì37) and cannot include it; `grep` confirms `internal/expected.hpp` does not even exist yet; (iii) `STEPPE_LOG_WARN` / `internal/log.hpp` likewise **does not exist** ‚Äî its only mention is a comment at `src/device/cuda/handles.hpp:26` placing it in `internal/`, again unreachable from the io leaf. The io leaf's *actual* error convention is `throw std::runtime_error` (`include_exclude.cpp:17,39`). Fix (corrected): the 1.1 factory should `throw std::runtime_error("filter plan dropped every SNP/sample ‚Äî check config")` on a fully-empty keep set (matching the leaf's existing convention), or ‚Äî if a non-throwing path is wanted ‚Äî return the plan plus an explicit `all_dropped` flag the `app` caller maps to `Status::InvalidConfig` at the api boundary (the boundary is where `Status` is reachable). Do **not** reach for `std::expected`/`Error`/`STEPPE_LOG` from this leaf.

**2.2 ‚Äî Integer-width story is split across the struct and its producers: `long M` (producer) vs `std::size_t` (plan length) vs `int P`, requiring `M < 0 ? 0 : M` guards that this struct cannot honor. [LOW, effort S, before-M4.5: no]**
Location: `snp_keep`/`n_snp_kept` use `std::size_t`; the producer `DecodedTileSummaryInput::M` is `long` (`snp_filter.hpp:45`) and `snp_filter.cpp:20,66` defensively guards `M < 0` (`static_cast<std::size_t>(M < 0 ? 0 : M)`). `kept_samples` is `std::vector<std::size_t>`.
`FilterPlan` lands on `std::size_t` (correct: ROADMAP ¬ß4 keeps `size_t` for the index cast, "free, mandatory above P‚âà32k"), but the producer speaks signed `long` and must guard against negatives ‚Äî a mismatch that means the conversion happens *outside* this struct, unvalidated. There is no overflow risk in `FilterPlan` itself (`std::vector::size()` is `size_t`; popcount fits), but the split is a single-source smell. Why it matters: ¬ß2 single-source, ¬ß8. Fix: settle on one unsigned index type across the M2 chain (or typed `SnpIndex`/`SampleIndex` newtypes), removing the negative-guard dance.

**2.3 ‚Äî `in_tile` is a deep `FilterConfig` copy carrying three resolved-away string id lists + `prune_in_path`; copying a `FilterPlan` deep-copies all of them, and a consumer re-reading them double-filters. [MED, effort M, before-M4.5: no (M5/M6)]**
Location: `in_tile` (lines 55‚Äì59, "Carried by value so the plan is self-contained").
`FilterConfig` holds `include_snp_ids`, `exclude_snp_ids` (both `std::vector<std::string>`) and `prune_in_path` (`std::string`) ‚Äî verified `config.hpp:257,261,267`. Those are **already resolved** into `SnpMembership` (and thus into `snp_keep`) by the time a plan exists (`build_snp_keep_mask` takes a pre-built `SnpMembership&`, `snp_filter.hpp:101‚Äì105`). Carrying them by value means: (a) every `FilterPlan` copy deep-copies potentially-large id vectors (the `.snp` can be ~584k ids, `include_exclude.hpp:32`) for fields that are dead at apply time; (b) a footgun ‚Äî a consumer that constructs a `SnpMembership` from `plan.in_tile` and re-applies it at S0' double-filters SNPs that were already dropped, or worse, with a different result if `prune_in_path` changed on disk (the path, not its resolved contents, is what is carried). The banner's claim that the plan is "plain std::vector POD" (line 11) and "self-contained" is undermined by embedding a config that points at an external `prune.in` *path*. Why it matters: ¬ß2 single-source ("implemented exactly once"); the membership rule is supposed to live only in `include_exclude.hpp` (its header line 12 says "composed in one place so snp_filter does not re-derive it"). Fix: replace `FilterConfig in_tile` with a small trivially-copyable threshold sub-struct (`maf_min`, `geno_max_missing`, `drop_monomorphic`, `transversions_only`, `autosomes_only` ‚Äî the only knobs still live in-tile per `build_snp_keep_mask`'s flag checks, `snp_filter.cpp:87‚Äì96`) and *do not* carry the membership/mind id lists; rename it `apply_thresholds` to signal it is the *remaining* in-tile work, not the whole config.

**2.4 ‚Äî Empty-`snp_keep` with non-zero `n_snp_kept` (or vice versa) is representable and undetectable. [LOW, effort S, before-M4.5: no]**
Covered structurally by 1.3/1.4; noting the specific degenerate pairing (`snp_keep.empty() && n_snp_kept > 0`, and `snp_keep` non-empty with `n_snp_kept` wrong) as a concrete invariant a `validate()` must reject.

### (4) CUDA idioms / RAII / streams / launch config vs ¬ß7

N/A by design ‚Äî and the prior review's correction here is the most important verification in this unit, so I re-derive it with a primary-source citation.
There is **no CUDA in this file and none should be added.** A candidate finding ‚Äî "`std::vector<bool>` is the wrong host type because the keep mask must be `cudaMemcpyAsync`'d to the device as a `V`-column drop, and `vector<bool>` has no contiguous `bool*` to memcpy (it is a bit-packed proxy-iterator specialization)" ‚Äî is *type-accurate but not applicable here*, and I reject it. The C++ standard (eel.is/c++draft/vector.bool, normative) does specify the specialization: "A space-optimized representation of bits is recommended instead," "reference is a class that simulates a reference to a single bit," and "There is no requirement that the data be stored as a contiguous allocation of bool values" (no `data()`‚Üí`bool*`). So the *premise* (no contiguous memcpy source) is true of the type. But the *consequent* is false in this codebase: I confirmed the actual consumption path in `tests/reference/test_filter_oracle.cu` ‚Äî `drop_columns` (lines 94‚Äì115) and `mask_columns` (lines 120‚Äì130) take the `std::vector<bool>& keep` and use it *host-side* to gather/zero the FP64 `Q/V/N` columns into contiguous `std::vector<double>` (`Qk/Vk/Nk` and `Qm/Vm/Nm`, lines 99‚Äì101/122), and *those* `double` arrays are what cross to the GPU via `MatView` (lines 272/298), never the bool buffer. The producer `build_snp_keep_mask` returns `std::vector<bool>` (`snp_filter.hpp:101`), and the future ‚ö°-3 fusion (`TODO.md:107`) makes the *predicates* `__host__ __device__` and re-evaluates them as a validity multiply inside `decode_af` ‚Äî it re-derives the decision device-side, it does not ship this bool buffer. So `vector<bool>` is neither span-fed to a kernel nor `cudaMemcpyAsync`'d, it is the *consistent* type across the whole M2 mask chain, and it is memory-optimal (the bit-packing the standard recommends: ~73 KB vs ~584 KB for `uint8_t` at M‚âà584k). No CUDA/RAII finding stands. (See 8.1 for the one legitimate, host-only reason one might still prefer `uint8_t` ‚Äî host spanability ‚Äî kept strictly separate from the rejected CUDA rationale.)

### (5) Magic numbers & hardcoded values vs ROADMAP ¬ß4

**Clean ‚Äî no finding.** The only literals in the file are the zero-initializer `n_snp_kept = 0` (a true initializer, not a tunable) and the include-guard token. Every threshold the plan represents is sourced from the embedded `FilterConfig`, which itself names every constant in `config.hpp` (`kAutosomeChromMin/Max`, etc.) and documents every default. This is exactly the ROADMAP ¬ß4 / ¬ß5 "no literal may survive except true mathematical constants" posture. Explicitly N/A ‚Äî and correctly so.

### (6) Decomposition / single-responsibility vs ¬ß2

**6.1 ‚Äî The struct conflates three axes (SNP keep, sample keep, in-tile thresholds) into one flat aggregate with no sub-types, so the per-axis invariants cannot be expressed by type. [LOW, effort M, before-M4.5: no]**
Location: whole struct.
`snp_keep`+`n_snp_kept` form one cohesive thing (a SNP-axis keep mask with its popcount); `kept_samples` is a separate sample-axis set; `in_tile` is config. Flattening them means the invariants ("popcount matches," "samples sorted/unique/in-range") have nowhere to attach. Why it matters: ¬ß2 separation of concerns + testability. Fix (also serves 1.3/1.4): factor into two self-validating value types ‚Äî `SnpKeepMask { std::vector<bool> mask; std::size_t n_total; std::size_t kept() const; }` and `SampleKeepSet { std::vector<std::size_t> indices; std::size_t n_total; bool validate() const; }` ‚Äî and let `FilterPlan` aggregate those two plus `apply_thresholds`. The single-responsibility win: each sub-type owns and can test its own invariant.

**6.2 ‚Äî `ploidy` lives on the *producer's* input struct (`DecodedTileSummaryInput::ploidy`, `snp_filter.hpp:54‚Äì57`) but is a dataset-wide policy the *plan* arguably should record for the S0' apply. [LOW, effort S, before-M4.5: no]**
Location: absence of any ploidy field on `FilterPlan`.
The missing-fraction and `N`-to-individual conversion that decided `snp_keep` depend on `ploidy` (`snp_filter.cpp:30,47`: `nonmissing_indiv += npn / ploidy_d`; ROADMAP ¬ß2: "must be honored," config-driven, default 2, never inferred). The plan carries the *thresholds* (`in_tile`) but not the *ploidy* under which they were evaluated, so a consumer cannot reconstruct the denominator the decision used. This is minor today (ploidy is constant 2 for v66 HO ‚Äî ROADMAP ¬ß3 line 67 "No pseudo-haploids in v66 HO"), but it is a latent single-source gap for the pseudo-haploid case. Note only ‚Äî fold into 1.2's "record the denominators" if a factory materializes.

## Considered & rejected

- **"`std::vector<bool>` is the wrong type because the mask must be `cudaMemcpyAsync`'d to the device."** Rejected ‚Äî premise true of the *type* (C++ standard, eel.is/c++draft/vector.bool: bit-packed, proxy `reference`, no contiguous `bool*`/`data()`), but the consequent is false *here*: the mask is consumed entirely host-side; `test_filter_oracle.cu`'s `drop_columns`/`mask_columns` (lines 94‚Äì130) apply it to gather/zero host FP64 `Q/V/N`, and only the resulting `std::vector<double>` cross to the GPU. `vector<bool>` is also the *consistent* type with the producer (`build_snp_keep_mask`, `snp_filter.hpp:101`) and the oracle test, and is memory-optimal for a 584k-entry mask. See ¬ß4 / ¬ß8.1.
- **"Use `std::bitset` or a fixed-width mask."** Rejected ‚Äî the SNP count is a runtime quantity (~584k, varies per dataset); `std::bitset<N>` needs a compile-time `N`. `std::vector<bool>` is correct.
- **"Carry `block_id[]` in the plan so S2 can bin directly."** Rejected ‚Äî block assignment is the shared `core::block_partition_rule` (¬ß5, ¬ß8; architecture.md line 192 names it the single cross-leaf exception); duplicating it here is the exact smell ¬ß5 warns against. The plan's deliberate omission (lines 35‚Äì36) is correct.
- **"Move it to `include/steppe/` so the public API can return it."** Rejected ‚Äî it is an internal `io`‚Üí`app` hand-off (9.1/9.2); promoting it imposes ¬ß16/ADR-0008 C-ABI burden for no external consumer. Expose a counts summary if bindings ever need one.
- **"Add a `valid`/`built` bool defaulting false."** Partially folded into 1.5 (recommend factory-only construction or explicit totals instead) ‚Äî a bare bool flag that a caller can also set is a weaker fix than removing the dangerous default state, so the standalone "add a bool" version is rejected in favor of the factory-only / totals approach.
- **"`<algorithm>` is missing (the comments imply `std::count`)."** Rejected for *this header* ‚Äî the header itself performs no `std::count`; the popcount is described in a comment as the cache's definition, not implemented here. IWYU is satisfied: the file uses `bool`/`std::vector` (`<vector>`), `std::size_t` (`<cstddef>`), and `FilterConfig` (`steppe/config.hpp`) ‚Äî all three includes present and used, none missing, none superfluous. (When the 1.1 factory is added, *it* will need `<algorithm>` ‚Äî but that is the factory's TU, not this header.)
- **"`n_snp_kept` should be `int` to match `F2BlockTensor::n_block`."** Rejected ‚Äî SNP counts reach ~584k and the design pins `size_t` for the index domain (ROADMAP ¬ß4); `int` would be a regression. `F2BlockTensor::n_block` is `int` because `n_block ‚âà 757` (verified ROADMAP ¬ß3 line 70: real-AADR ‚Üí 757 blocks; autosome ‚Üí 719), a different, small domain.
- **"The comment listing of filters is incomplete (omits the unconditional class drop)."** Kept but downgraded to a minor doc nit (7.4) rather than rejected ‚Äî it is technically an omission, but defensible since the class drop is a precondition documented in the single-home `filter_decision.hpp`.
- **"Return `std::expected<FilterPlan, Error>` from the factory (per the prior draft)."** Rejected as written ‚Äî there is **no `Error` type** in the tree (only `enum class Status` in `error.hpp`), and `std::expected`/`internal/expected.hpp` is a `core`-internal mechanism the io leaf cannot link (architecture.md lines 522/641; CMake lines 34‚Äì37). The corrected recommendation (2.1) is the io-native `throw std::runtime_error` (matching `include_exclude.cpp:17,39`), with `Status` mapping done at the `app`/api boundary. This is a *correction of the prior draft's fix*, not of a finding.
- **"Route the all-dropped diagnostic through `STEPPE_LOG_WARN` (per the prior draft)."** Rejected as written ‚Äî `STEPPE_LOG_WARN`/`internal/log.hpp` does not exist (only a forward-reference comment at `src/device/cuda/handles.hpp:26`), and would be `core`/internal, unreachable from the io leaf. Corrected in 2.1/11.1.
- **"`FilterPlan` IS the ¬ß1 merge plan."** Rejected ‚Äî architecture ¬ß5 distinguishes the S-2 `io/merge` plan (line 239) from the S-1/S0' filter producers (line 166). `FilterPlan` is the *filter* plan; the merge plan is a separate (M6) `io/merge` artifact. The prior draft occasionally conflated them; corrected in Role & layering.
- **"The implicit special members are a bug."** Rejected ‚Äî implicit copy/move is *correct* for an aggregate and not itself a defect; the only real issue is the cost of copying `in_tile` (8.1) and the future need to make construction explicit once a factory exists (7.5). Filed as a LOW note, not a defect.

## Good patterns to keep

- **Exemplary documentation and invariant articulation.** The banner ties every field to its precise architecture clause (¬ß1 "merge is a plan," ¬ß5 S0'/S-1, ¬ß5-S2 `V¬∑V·µÄ` masking) and states the SNP-global / sample-global invariant crisply ("a per-(pop, SNP) drop ‚Ä¶ the plan cannot even express one"). - **Correct, minimal, downward-only layering.** Three includes, all downward; `src/io/CMakeLists.txt` links only `steppe::api` (PUBLIC) + `steppe::warnings` (PRIVATE) ‚Äî no `core`, no `core_internal`, no `device`, no CUDA, no upward edge. This is also what *constrains* the fixes (no `std::expected`/`Error`/`STEPPE_LOG` from here) ‚Äî the discipline cuts both ways and is worth preserving.
- **Zero magic numbers.** Every threshold is sourced from the embedded `FilterConfig`, which itself names every constant in `config.hpp`. The only literal is a zero-initializer. Exactly the ROADMAP ¬ß4/¬ß5 posture.
- **Type consistency with the M2 producer.** `snp_keep` as `std::vector<bool>` matches `build_snp_keep_mask`'s return type and the `test_filter_oracle.cu` apply path ‚Äî the consistent (and, per the C++ standard's recommended bit-packing, memory-optimal) choice across M2. - **Deliberate omission of block info.** Refusing to carry block ids (deferring to the shared `core::block_partition_rule`) is the right ¬ß8 single-source decision and avoids the smell ¬ß5 explicitly warns against.
- **The no-op-is-keep-everything sentinel discipline.** Pinning "`mind_max_missing >= 1.0` ‚áí all samples kept" in the field doc (consistent with `mind_prepass.cpp:30,60‚Äì62`) preserves the parity-path default ‚Äî carry this convention front-and-center into the factory when it is added.
- **Capability-agnostic by construction.** A host-side, parity-neutral QC artifact upstream of all GPU work ‚Äî correctly *not* entangled with the capability tiers. Keep the capability log in `DeviceConfig`/`Resources`, not here.
