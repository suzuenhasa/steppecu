# Review — `io/eigenstrat_format` (packed `.geno` format constants + header parse)

Unit: `src/io/eigenstrat_format.hpp` (116 lines) + `src/io/eigenstrat_format.cpp` (85 lines).
Reviewer scope: this one unit, read line by line, against `docs/architecture.md` §2/§4/§5/§7/§8/§9/§11/§12/§13, `docs/ROADMAP.md` §4/§5/§6, and `docs/TODO.md` capability-tier section. Library-behavior claims are cited to primary docs.

## Role & layering

This is the single-home for the packed-genotype on-disk **format constants** the ROADMAP §4 magic-number inventory explicitly promotes ("TGENO header `48`, `ceil(nsnp/4)` → `io` format constants, derived from the header parse") plus the **header parse** that derives the record stride. It is correctly an `io`-leaf translation unit (architecture §4): pure host C++20, includes only `<array>/<cstddef>/<cstdint>` (header) and `<algorithm>/<cctype>/<cstddef>/<cstdint>/<string>` (TU), no CUDA, no `core`/`device` dependency, no upward edge. The layering is exemplary and the file is the textbook realization of the §4 dependency-direction rule and the §8 "single home" table row.

Consumers verified:
- `io/geno_reader.{hpp,cpp}` — primary: reads `kGenoHeaderBytes`, calls `parse_geno_header`, validates file size against the DERIVED `bytes_per_record`/`n_records`, uses `packed_bytes` for the tile stride.
- `io/filter/mind_prepass.{hpp,cpp}` — uses `code_in_byte` + `kMissingCode` for the `--mind` byte path (deliberately shares the one bit-extraction site).
- `tests/unit/test_filters.cpp`, `tests/reference/test_filter_oracle.cu`, `tests/reference/test_decode_equivalence.cu` — use `code_in_byte`/`kMissingCode`/`GenoFormat` as fixture helpers, not as the subject under test.
- Deliberate DRY split: `core/internal/decode_af.hpp` re-declares `kMissingGenotypeCode = 3` and `genotype_code()` (the `(byte>>shift)&3` rule) *separately* so `core` does not depend on the `io` leaf (architecture §4); the two are "identical by construction and pinned by the test." Consistent with §4, but it creates a second home for the same two domain facts — see Finding L1.

This unit touches **neither** the PRO-6000-capable nor the budget-5090 GPU path directly (no CUDA, no device probe, no streaming). The capability-tier obligation is therefore mostly N/A — its only legitimate contribution is in what it should expose so a future GDS/partial-file path can be tagged (see Capability-tiers).

The substance below is mostly low/medium severity: the header is genuinely close to its purpose, but the `.cpp` parser has two real defects (a comment that flatly contradicts the code it annotates, and an unbounded `n*10+digit` accumulation with no overflow guard), several robustness gaps the doc-comments overpromise, and a testability gap (zero direct unit coverage of a function whose whole job is parsing untrusted bytes). A header this load-bearing and this self-described as "verified bit-for-bit" should not ship its parser untested.

## Score: 8/10 — solid, well-documented, correctly layered; held back by an unbounded `n*10+digit` accumulation with no overflow guard, a comment that contradicts the code, doc-comments that overpromise robustness the parser does not deliver, and zero direct test coverage of the parser.

## Findings

### (1) Correctness & bugs

**C1 — comment claims a "trailing-substring check" / robustness to surrounding whitespace that the code does not implement (med, S, before-M4.5: yes).**
`eigenstrat_format.cpp:32-34`:
```cpp
// A trailing-substring check keeps it robust to
// surrounding whitespace; we match the leading token exactly.
```
There is no trailing-substring check anywhere in the function. The code does `find_first_not_of(" \t")` then `substr` up to the next space/tab and compares the token *exactly* (`magic == "TGENO"`). The comment is not merely stale — it describes an algorithm (substring containment) with materially different behavior from what runs (exact leading-token match). Why it matters: §2/§7 want comment density matching the code, but a comment that contradicts the code is worse than none — it misleads the next maintainer into believing `"...TGENO..."` or trailing-garbage magic is accepted (it is not) and into "preserving" a behavior that never existed. Fix: delete the "trailing-substring check keeps it robust" sentence; state plainly "we match the first whitespace-delimited token exactly against `TGENO`/`GENO`." Adversarial check: is the comment describing intent and the exact match a deliberate tightening? Even so it is false about the present code and must be corrected, not retained as aspiration.

**C2 — unbounded decimal accumulation: `v = v*10u + digit` can silently wrap `std::size_t` (med, S, before-M4.5: yes).**
`eigenstrat_format.cpp:54-60`:
```cpp
std::size_t v = 0;
...
v = v * 10u + static_cast<std::size_t>(text[i] - '0');
```
No digit-count cap, no overflow check. A header is only 48 bytes so a *legitimate* count cannot overflow `size_t`. But the function takes an arbitrary `std::array<char,48>` and is documented to "return Unknown on … unparsable counts" — a malformed/adversarial header such as `TGENO 999999999999999999999999999 …` (post-magic bytes all digits) wraps `size_t` to an arbitrary value with no diagnostic. `std::size_t` unsigned overflow is well-defined modular arithmetic (not UB), so this is not UB — but it is a silent *wrong* `n_ind`/`n_snp` that flows into `packed_bytes()` and `geno_reader`'s file-size validation, producing a wrong-but-plausible stride rather than the documented `Unknown`. Why it matters: architecture §10 classes malformed `.geno` as `STEPPE_ERR_IO_FORMAT` ("caller fixes input") and fail-fast (§2) wants a malformed count *rejected*, not wrapped. The §4 inventory's whole point is that the stride is "DERIVED … never hardcoded" — a silently-wrapped count makes the derivation garbage. Fix: cap the digit run (reject if `> 19` digits) or detect overflow (`v > (SIZE_MAX - d)/10`) and set `format = Unknown` on overflow. Severity med because the trigger requires a malformed header, but that is exactly the untrusted-input path this parser exists to harden. Effort S.

**C3 — a parseable-but-zero count leaves a non-Unknown header with zero stride; relies entirely on the caller (low/med, S, before-M4.5: no).**
The `got < 2` guard (`:63-66`) correctly forces `Unknown` for "fewer than two integers." But `TGENO 27594 0` (n_snp = 0, a real degenerate case) keeps `format = Tgeno`, sets `n_records = 27594`, `bytes_per_record = packed_bytes(0) = 0` — a *non-Unknown* header with a zero stride. The doc-comment (`hpp:108-110`) only promises Unknown "on a bad magic or unparsable counts"; a parseable-but-zero count is neither, so the leaf hands back an undecodable header. The save is entirely downstream in `geno_reader.cpp:43` (`if (bytes_per_record == 0 || n_records == 0) throw`). Defensible layering ("the caller decides how to fail"), but the parser's contract permits returning a header that can never be decoded. Why it matters: §2 fail-fast + single-responsibility. Fix (document-or-enforce): either (a) explicitly document that a zero count yields a non-Unknown header with zero stride and size validation is the caller's job (matches current geno_reader), or (b) treat zero n_ind/n_snp as `Unknown` here. I lean (a)+doc because the GENO-floor case (C4) means a zero stride is not always the only signal.

**C4 — GENO `bytes_per_record` floor masks a degenerate `n_ind == 0` (low, S, before-M4.5: no).**
`eigenstrat_format.cpp:80`: `h.bytes_per_record = std::max(kGenoHeaderBytes, packed_bytes(h.n_ind));`. For GENO with `n_ind == 0`, `packed_bytes(0) == 0` but the `max(48, 0) == 48` floor yields a *non-zero* stride. So `geno_reader.cpp:43`'s `bytes_per_record == 0` check cannot fire for a zero-individual GENO file — only the `n_records == 0` arm (which checks `n_snp`) would. A `GENO 0 100` header yields `n_records = 100`, `bytes_per_record = 48`: a "valid" header describing 100 records that decode to zero individuals. Narrow case (GENO is recognized-but-not-decoded in M1; `geno_reader::read_tile` rejects all GENO), so impact today is nil. Latent for the future GENO decode path. Fix: when the GENO decode lands, validate `n_ind > 0` explicitly; for now a one-line note in the GENO branch that the floor can hide `n_ind==0` prevents a future surprise.

**C5 — `noexcept` but `text.push_back`/`reserve`/`substr` can throw `std::bad_alloc` → `std::terminate` (low, S, before-M4.5: no).**
`hpp:111` and `cpp:20` declare the function `noexcept`. The body allocates a `std::string` (`reserve(48)`, up to 47 `push_back`s, one `substr`). Each can throw `std::bad_alloc`; under `noexcept` a throw calls `std::terminate` rather than propagating. For a 48-byte string this is astronomically unlikely, and `noexcept` on a probe function is a reasonable choice (header comment: "the `io` leaf never throws across the layer boundary on a format probe"). But the `noexcept` is honored by "allocation won't realistically fail for 48 bytes," not by construction. Adversarial check: is this actually wrong? No real failure mode exists at 48 bytes. The right resolution is to make the parse allocation-free (P1), which makes the `noexcept` *true by construction*. Keep `noexcept`; fix the allocation.

**C6 — `substr(pos, magic_end - pos)` with `magic_end == npos` is well-defined; verified, not a bug (no action).**
`cpp:37-38`: when the magic runs to the end with no following space (`text == "TGENO"`), `find_first_of(" \t", pos)` returns `npos`, so `magic_end - pos` is `npos - pos`, an enormous value. `std::string::substr(pos, count)` clamps `count` to `size() - pos` when it exceeds the remainder, and `pos` came from `find_first_not_of` so `pos <= size()` (cannot throw `out_of_range`). So `magic == "TGENO"` correctly. Correct *by the standard*, but correct-by-clamp, not correct-by-intent, and reads as a latent overflow — see R3. (cppreference `basic_string::substr`: "If the requested substring extends past the end of the string, i.e. `count > size() - pos`, the returned substring is `[pos, size())`.")

### (2) Edge cases & failure modes

**E1 — `std::isdigit(static_cast<unsigned char>(...))` is correctly guarded — verified good (no action).**
`cpp:52,56`: both `isdigit` calls cast to `unsigned char` first. This is exactly the documented requirement: passing a (possibly negative) `char` directly to `std::isdigit` is **undefined behavior** unless the value is representable as `unsigned char` or equals `EOF` (cppreference `std::isdigit`: "the behavior … is undefined if the argument's value is neither representable as `unsigned char` nor equal to `EOF`"). The header is binary and may contain bytes ≥ 0x80 in the hash fields; on a platform where `char` is signed those would be negative and feed UB without the cast. The cast is present and correct — a genuinely good defensive detail; keep it.

**E2 — locale dependence of `std::isdigit` (low, S, before-M4.5: no).**
`std::isdigit` consults the current C locale's `ctype` category. For the default `"C"` locale `[0-9]` is the digit set, which is what is wanted; but the function does not pin the locale, so a process that called `setlocale(LC_CTYPE, …)` could in principle change classification. No real locale reclassifies ASCII `0-9`, so the risk is theoretical. Why it matters: §12 determinism/parity — the parse must be locale-independent. Fix: since only ASCII `'0'..'9'` are ever valid, replace `std::isdigit(static_cast<unsigned char>(c))` with `c >= '0' && c <= '9'` — locale-immune, branch-predictable, removes `<cctype>` and the cast foot-gun. Low severity; cheap, strictly safer.

**E3 — embedded-NUL truncation interacts with multi-field headers — confirm intended (low, doc, before-M4.5: no).**
`cpp:27-30` stops copying at the first `'\0'`. The real header is `"TGENO 27594 584131 <hash> <hash>"` NUL-padded; the first NUL is the padding, so truncation is correct for a well-formed file. A corrupt file with an embedded NUL *between* magic and counts (`"TGENO\0 27594 …"`) truncates to `"TGENO"`, `got < 2`, returns Unknown — safe, but for a slightly surprising reason (NUL, not missing digits). Acceptable; worth one comment word ("stop at the first NUL — the AT2 padding; an embedded NUL truncates and degrades to Unknown").

**E4 — the integer scanner extracts the first two digit-runs regardless of separators → corrupt input yields plausible-wrong counts, not Unknown (low, S, before-M4.5: no).**
The scan (`cpp:51-62`) starts at `magic_end` and skips any non-digit, then reads a digit run, twice. So `"GENO 1a2 3"` parses `ints = {1, 2}` (the single token `1a2` yields runs `1` then `2`); `"TGENO 27594x 584131"` parses `{27594, 584131}`. For a real AADR header this never triggers; for a corrupt header it converts corruption into plausible-wrong derived strides rather than `Unknown`. Same family as C2. Fix: require the count tokens to be whole (a run bounded by whitespace/NUL), not "the next digits anywhere." Low severity (corrupt-input only).

**E5 — `packed_bytes(near-SIZE_MAX)` overflows the `+3` (low, S, before-M4.5: no).**
`hpp:67-70`: `(n_codes + 4 - 1) / 4`. For `n_codes > SIZE_MAX - 3` the `+3` wraps. Unreachable from a real 48-byte header (and C2's hardening removes the only path to a near-`SIZE_MAX` count), but `packed_bytes` is a public `constexpr` consumed elsewhere (`geno_reader`, and `cuda_backend.cu:304` open-codes an equivalent — L2). Document the precondition `n_codes <= SIZE_MAX - (kCodesPerByte-1)` or assert it. Low; a contract-completeness note for a public helper.

**E6 — degenerate inputs that ARE handled well (verified):** empty buffer / all-NUL / whitespace-only → `find_first_not_of` returns `npos` → `Unknown` (`:36`); magic-only / one-count → `got < 2` → `Unknown`. These paths are correct. Good.

### (3) Numerical / precision vs §12

N/A for §12 arithmetic-precision concerns (no FP, no Ozaki, no GEMM, no accumulation order) — this is pure integer/byte parsing. Two adjacent §12-relevant notes:

**N1 — `code_in_byte` is the single host-side bit-order site and must stay bit-identical to `decode_af.hpp::genotype_code`; this is a §12/§13 parity surface (med, doc, before-M4.5: yes — already partially met).** The decode/parity chain (§12: "all of this lives in the shared `__host__ __device__` feeder primitive so oracle and GEMM path can't diverge") depends on the host reader (`code_in_byte`) and the device/CPU decoder (`genotype_code`) agreeing on MSB-first bit order *and* on `kMissingCode == kMissingGenotypeCode == 3`. They currently agree, and `decode_af.hpp` documents the agreement is "pinned by the equivalence test." But the pin is *indirect* — the decode-equivalence test exercises the GPU/CPU decode path, not a direct `code_in_byte == genotype_code` assertion over all 256 bytes. See L1+T1: the two are textually duplicated and only transitively tested for equality. This is the one place where a divergence would silently corrupt every f2; it deserves a direct `static_assert`/unit test, not just transitive coverage.

No RAII concern (the function owns only a transient `std::string`; `GenoHeader` is a trivial aggregate). This is exactly the boundary the architecture wants; nothing to add.

### (5) Magic numbers & hardcoded values vs §4 / ROADMAP §4

**M1 — excellent: the §4-inventoried literals are all promoted to named constants (verified, keep).** `kGenoHeaderBytes = 48`, `kCodesPerByte = 4`, `kBitsPerCode = 2`, `kMissingCode = 3`, and the `packed_bytes`/`code_in_byte` derivations are exactly the ROADMAP §4 row "TGENO header `48`, `ceil(nsnp/4)` → `io` format constants, derived from the header parse." `code_in_byte` even derives the shift table `6,4,2,0` from `(kCodesPerByte-1-(k%kCodesPerByte))*kBitsPerCode` rather than hardcoding it. This is the model the rest of the codebase should follow.

**M2 — residual literals `0x3u`, `'0'`, `10u`, the `" \t"` set, `2` (low, mostly true-constants, before-M4.5: no).**
- `0x3u` (`hpp:78`, the 2-bit mask) and `'0'`/`10u` (`cpp:57`, ASCII→int) are *true* structural/mathematical constants (the mask of a 2-bit code is `0b11`; decimal radix is 10). The §4 carve-out ("except true mathematical constants") covers these. Borderline: `0x3u` is `(1u << kBitsPerCode) - 1` and could be named `kCodeMask` to make the "2 bits ⇒ mask 3" link explicit and remove the only remaining hex literal. Low value; arguably over-engineering for a self-evident mask — I would name it only because the file is otherwise pristine and a lone `0x3u` is the single un-derived literal.
- The whitespace set `" \t"` appears twice (`cpp:35,37`); a `constexpr std::string_view kHeaderWhitespace = " \t";` would DRY it and document "fields are space/tab delimited (not `\n`/`\r`)." The set deliberately excludes line breaks — correct for a single-line fixed header, but undocumented; worth a word.

**M3 — `static_cast<std::size_t>(kCodesPerByte)` repeated twice in `packed_bytes` (cosmetic, S, before-M4.5: no).** `hpp:68-69`. `kCodesPerByte` is `int = 4`; the casts are needed because `code_in_byte` uses it in `int` shift math (so it cannot just be `size_t`). The current explicit casts are clear and correct (mixing signed `int`/`size_t` without a cast would warn under `-Wall -Wextra -Werror`, §6). Lowest priority; mention only for completeness.

### (6) Decomposition / single-responsibility / function size vs §2

**D1 — `parse_geno_header` does three jobs in one 63-line function: tokenize, classify magic, scan two ints (low, M, before-M4.5: no).** Not large by absolute measure, but it mixes (a) NUL-trim + whitespace tokenization, (b) format classification, (c) a hand-rolled two-integer scanner with its own nested-loop digit logic. The digit scanner (`cpp:51-62`) is the most error-prone part (it is where C2/E4 live) and is a natural standalone, testable helper, e.g. `static bool parse_first_two_uints(std::string_view, std::size_t& a, std::size_t& b) noexcept`. Extracting it (1) shrinks `parse_geno_header` to "tokenize → classify → derive stride," (2) makes the integer scanner unit-testable in isolation (addresses T1), and (3) gives the overflow guard (C2) a single clean home. §2 single-responsibility + §13 testability favor the split. Low severity because the current function is comprehensible; the win is testability.

**D2 — TGENO/GENO stride derivation is a clean two-branch `if/else` (verified good).** `cpp:72-81` is tidy: each branch sets `n_records` + `bytes_per_record` from the format; the GENO `max(48, …)` floor is localized and commented with the EIGENSOFT `rlen` rule. The right size; no concern.

### (7) Readability, naming, const-correctness, [[nodiscard]]/noexcept, comments

**R1 — `[[nodiscard]]` on the declaration, not repeated on the definition (verified fine; informational).** `[[nodiscard]]` on `hpp:111`, `packed_bytes` (`:67`), `code_in_byte` (`:76`). The `.cpp` definition (`:20`) omits `[[nodiscard]]`/`noexcept` re-spelling — both are inherited from the first declaration per the standard, so this is correct (re-stating `noexcept` would be redundant; re-stating `[[nodiscard]]` is ignored). **R3 — `magic_end - pos` reads like an unsigned-underflow bug at the call site (low, S, before-M4.5: no).** As established in C6 it is well-defined, but a reader must know the `substr`-clamp rule to see that `npos - pos` is safe. **R4 — comment/code density is otherwise excellent (verified, keep).** Both files carry a strong file-level rationale tying every constant to its architecture/ROADMAP citation, the decode-convention block matches `decode_af.hpp`, and the GENO `rlen` floor is explained. **R5 — `pos`/`magic_end`/`i`/`v`/`got`/`any` are terse but scoped and locally clear (no action).** In a 63-line function the short names are fine; `v`/`any`/`got` are conventional scanner locals.

### (8) Performance

**P1 — heap allocation of a `std::string` per parse (low, S, before-M4.5: no).** `parse_geno_header` allocates a `std::string` (`reserve(48)` → heap) plus a second allocation for the `substr` magic, on every call. Called **once per file open** (`geno_reader` ctor), utterly off any hot path (§11.1: the hot path is the SNP-tile stream). Performance impact: zero. *However*, replacing the `std::string` with `std::string_view` parsing over the input `std::array<char,48>` would (a) make the `noexcept` honest-by-construction (C5), (b) remove `<string>` from the includes, (c) eliminate both allocations. The input is already a fixed `std::array<char,48>`, so a `std::string_view` (find the NUL, then `string_view{head.data(), nul}`) with `find_first_not_of`/`find_first_of` (both available on `string_view`) is a near-drop-in that is allocation- and exception-free. Do it not for speed but to strengthen the `noexcept` contract and trim an include. Low severity, S effort, a net simplification.

**P2 — no other performance concern.** `packed_bytes`/`code_in_byte` are `constexpr`, trivially inlinable, branchless. Nothing to optimize.

### (9) Layering / API / ABI vs §4

**A1 — layering correct and compiler-enforceable (verified, keep).** Header includes only `<array>/<cstddef>/<cstdint>`; TU adds `<algorithm>/<cctype>/<string>`. No CUDA, no `core`, no `device`, no `api`. The `io` leaf exactly per §4; IWYU-clean by inspection (every include used: `<array>` for the param, `<cstddef>` `size_t`, `<cstdint>` `uint8_t`; TU `<algorithm>` for `std::max`, `<cctype>` `isdigit`, `<string>` the buffer).

**A2 — `GenoHeader` is a clean aggregate POD with safe defaults (verified, keep).** All members default-initialized; `format = Unknown` is the safe default so a partially-filled header (early return) is never mistaken for valid. Good fail-safe default per §2.

**A3 — `parse_geno_header` takes `const std::array<char, kGenoHeaderBytes>&` — tight, no over-read possible (verified, keep).** Binding to the exact fixed-width type makes "must point to ≥ 48 bytes" a compile-time guarantee, not a runtime precondition — markedly better than a `const char*`/length pair the caller must police. Strong API choice. (The comment `hpp:104-105` "`head` must point to at least `kGenoHeaderBytes` bytes" is now redundant since the type *is* exactly that width — harmless, but the precondition is type-enforced; worth noting in the comment.)

### (10) Testability vs §13

**T1 — ZERO direct tests of `parse_geno_header`; `packed_bytes`/`code_in_byte` are only *transitively* covered (med, M, before-M4.5: yes — highest-value action in the review).** Grep over `tests/` shows `code_in_byte`/`kMissingCode`/`GenoFormat` are used by `test_filters.cpp`, `test_filter_oracle.cu`, `test_decode_equivalence.cu` — but as *helpers to build/probe* fixtures, not as the subjects under test. There is **no** test that feeds `parse_geno_header` a TGENO header, a GENO header, a bad magic, a truncated buffer, a zero count, a multi-int header, or a malformed/overflowing count and asserts the resulting `GenoHeader`. §13 mandates pure host numerics tested in plain `.cpp`, and §6's definition-of-done requires the CPU-reference-equivalence gate. A pure host parser of *untrusted bytes* is precisely what §13 wants unit-tested, and it has the most edge-case surface in the unit. C1/C2/E4 would all be caught by a 10-case table-driven host test. Fix: add `tests/unit/test_eigenstrat_format.cpp` covering: (a) real AADR `TGENO 27594 584131` → all fields; (b) `GENO 100 50` → the `max(48,ceil(100/4))` floor; (c) bad magic → Unknown; (d) one-count/empty/all-NUL → Unknown; (e) `packed_bytes` boundary table (0→0,1→1,4→1,5→2); (f) `code_in_byte` round-trip vs `decode_af.hpp::genotype_code` for all 256 bytes × 4 positions (directly pins N1/L1); (g) the overflow case once C2 is fixed. Med severity, M effort.

**T2 — `packed_bytes` and `code_in_byte` are `constexpr` → directly `static_assert`-testable at compile time (low, S, before-M4.5: no).** A handful of `static_assert(packed_bytes(5) == 2);` / `static_assert(code_in_byte(0b11'00'00'00, 0) == 3);` lines (in the test TU, not the header) give free compile-time coverage of the bit math and document the expected mapping inline. Complements T1.

### (11) Capability tiers (PRO-6000-capable vs budget-5090) — §TODO

**CT1 — this unit is correctly outside the capability-tier surface, but should carry the seam for the future partial-file / streaming path (low, S, before-M4.5: no).** The TODO capability table is entirely about *data movement and observability* (P2P combine, GDS ingest, ncu counters) — none of which live in a host-pure format parser. So the bulk of the obligation is **N/A because this unit performs no device work, no I/O transport, and no profiling.** The one genuine touch-point: `parse_geno_header` produces the `n_records`/`bytes_per_record` that the *future* GDS/cuFile ingest lane (M5/M7, TODO line 129) will use to compute DMA offsets, and `geno_reader` already supports a *partial-file* (`records_present_ < n_records`) case. The format layer should make the "complete records on disk vs header-claimed records" notion first-class so a GDS lane can size its `cuFileRead` against the *derived* stride without re-deriving it. Today `geno_reader.cpp:58` computes `records_present_ = data_bytes / bytes_per_record` inline; that derivation arguably belongs next to the stride it depends on (a `std::size_t records_in_file(std::size_t file_bytes, const GenoHeader&)` free function in this unit) so both the POSIX-pread path and the future GDS path share one offset/stride/record-count derivation — and so the eventual capability-tagged log line ("GDS unavailable → POSIX pread") references a single source of the geometry. Low severity, forward-looking; do it when M5 lands, not before M4.5.

**CT2 — no capability *probe* or tagged log belongs here (verified).** A format parser must not log or probe hardware (it is a leaf with no logger dependency — §10 keeps `STEPPE_LOG_*` in `core/internal`, and `io` failures surface as `std::runtime_error` per `geno_reader.hpp:18`). Correct as-is; do not add capability logging to this unit.

### (12) Cross-cutting DRY (§2/§8)

**L1 — the missing-code constant and the bit-extraction rule are duplicated between `io/eigenstrat_format.hpp` and `core/internal/decode_af.hpp` (med, M, before-M4.5: yes — the §2/§4 tension, the most architecturally interesting finding).**
- `io::kMissingCode = 3` (`hpp:62`) vs `core::kMissingGenotypeCode = 3` (`decode_af.hpp:59`).
- `io::code_in_byte` (`(byte >> ((3-(k%4))*2)) & 3`) vs `core::genotype_code` (`(byte >> ((3-(k&3))*2)) & 3`).
The same two domain facts written twice. The duplication is **deliberate and documented** (`decode_af.hpp:55-58`: "kept here so the core decoder does not depend on the `io` leaf — architecture §4 layering: core does not depend on io"). Legitimate layering reason — `core` must not `#include` an `io` header (§4 direction). So this is *not* a simple "merge them." But §2's single-source principle is in real tension with §4's no-upward-dependency, resolved here by duplication-plus-a-test-pin. Honest assessment: the resolution is acceptable, but the "pin" is currently only the *transitive* decode-equivalence test, not a direct equality assertion. Why it matters: §12 parity hinges on these never diverging; a future edit to one (e.g. "optimizing" `code_in_byte` into a lookup table with a transposed bit order) would pass every test that does not exercise both against each other on the same input. Fix options, preferred first: (a) put the *one true* bit-extraction + missing-code in a host-pure home that *both* `io` and `core` may include without violating layering (a tiny shared `packing_bits.hpp` that `io` consumes as a leaf — the same exception §8 already grants `block_partition_rule.hpp`: "host-pure, consumed by `io` *and* device kernels"); or (b) keep the duplication but add a direct cross-checking unit test/`static_assert` (T1.f) so divergence fails CI loudly. Option (a) is the cleaner §2 answer with direct precedent; (b) is the minimum.

**L2 — `cuda_backend.cu:304` open-codes a `packed_bytes`-equivalent instead of calling `io::packed_bytes` (low/med, S, before-M4.5: no — verify).** Grep shows `src/device/cuda/cuda_backend.cu:304` computes a local `packed_bytes` quantity. If it re-derives `ceil(n/4)` by hand it re-spells the exact §4 literal this unit single-homes ("computed here, never open-coded elsewhere", `hpp:64-66`). The catch: `cuda_backend.cu` is in `steppe_device` and *cannot* include `io/eigenstrat_format.hpp` (the `io` leaf is not a dependency of `device`, and shouldn't be). So the device side likely needs the same shared-home treatment as L1 — the `ceil(n/4)` packing-stride formula is a *domain rule* shared by `io` and `device`, exactly like `block_partition_rule`. Recommend confirming whether `cuda_backend.cu:304` is a true `packed_bytes` duplicate; if so, fold both into the shared host-pure packing header proposed in L1(a). Flagged as "verify" because I reviewed only the grep line, not the full `cuda_backend.cu` context — but the §4/§8 single-home mandate makes any second `ceil(n/4)` site a defect by definition.

## Considered & rejected

- **`std::isdigit` UB** — both call sites cast to `unsigned char` first (`cpp:52,56`); per cppreference this is the documented-correct guard. **Not a bug.** (Recommended `c>='0'&&c<='9'` in E2 for locale-immunity, but the current code is not UB.)
- **`substr(pos, npos - pos)` unsigned underflow** — `substr` clamps `count` to `size()-pos`, and `pos` is a valid `find_first_not_of` result so no `out_of_range`. **Well-defined; not a bug** (C6). Flagged only for readability (R3).
- **`size_t` overflow in `v*10u+digit` is UB** — rejected: unsigned overflow is well-defined modular wraparound, not UB. The defect is *silent wrong value*, not UB (re-scoped into C2).
- **`noexcept` is wrong because the body throws** — rejected as high-severity: the only throwers are `std::string` allocations of ≤48 bytes; `std::terminate` on `bad_alloc` here is effectively unreachable. Kept as low-severity C5 with the right fix (allocation-free parse, P1).
- **`GenoFormat::Geno` is dead code since M1 only decodes TGENO** — rejected: recognizing GENO so the reader "does not mis-decode" and reports rather than silently reading the wrong axis is an explicit documented goal (`hpp:17-21`), and `geno_reader::read_tile` relies on it to reject GENO. Intentional, not dead.
- **`header_bytes` member is redundant (always `kGenoHeaderBytes`)** — set to the constant unconditionally (`cpp:70`); currently always 48, but exposing it per-header is forward-compatible (a future format with a different header width) and `geno_reader` reads it via the struct rather than the constant — the right decoupling. Borderline-redundant today but a defensible seam; not a defect.
- **`packed_bytes` should reject `n_codes==0`** — rejected: `packed_bytes(0)==0` is the mathematically correct "0 codes ⇒ 0 bytes"; degeneracy handling correctly belongs to the caller (geno_reader) which has the context to decide 0 records is fatal. Re-scoped into C3/C4 as a *documentation* gap, not a `packed_bytes` bug.
- **The whitespace set should include `\n`/`\r`** — rejected: the header is a single fixed-width line; excluding line breaks is correct. Flagged only as an undocumented choice (M2).
- **`code_in_byte` MSB-first vs LSB-first might be wrong** — checked against the header's own decode-convention block and `decode_af.hpp`: both are MSB-first (`6,4,2,0`), documented as verified bit-for-bit against the on-box oracle `build_tgeno_matrix.py`. Consistent; not a bug. The only residual risk is the two homes diverging — L1.

## Good patterns to keep

- **The §4 magic-number promotion is exemplary** — every inventoried literal (`48`, `ceil(n/4)`, the 2-bit packing) is a named constant with its derivation shown, not a hardcoded number at a call site. `code_in_byte` deriving `6,4,2,0` from `(kCodesPerByte-1-(k%kCodesPerByte))*kBitsPerCode` is the right way to encode a bit layout.
- **Fixed-width array parameter** (`const std::array<char,48>&`) makes the "≥48 bytes" precondition a compile-time guarantee — strictly better than a pointer+length the caller must police (A3).
- **`std::isdigit(static_cast<unsigned char>(...))`** — the correct, UB-free guard, present at both call sites (E1).
- **Recognize-don't-mis-decode for GENO** — refusing to read a SNP-major file on the individual-major axis, reporting instead of silently producing garbage, is the fail-fast spirit of §2.
- **Safe-by-default `GenoHeader` aggregate** — `format = Unknown` and all-zero counts mean a partial/early-return header can never be mistaken for valid (A2).
- **Layering discipline** — host-pure, minimal includes, no CUDA, no upward dependency; the model `io` leaf (A1). The *documented* reason for duplicating the missing-code into `core` (rather than an accidental copy) shows the layering tension was reasoned about, even if the pin should be tightened (L1).
- **Comment-to-architecture traceability** — every constant cites its architecture/ROADMAP section and the on-box oracle it was verified against; the right density for a load-bearing format header (R4), the lone exception being the C1 contradiction.
