## Role & layering

`fstats.hpp` is the installed, CUDA-free public header carrying the M4 deliverable: a single plain-old-data aggregate `F2BlockTensor` holding the per-block f2 tensor `f2 [P×P×n_block]`, the retained per-block pairwise-valid count `vpair`, the per-block SNP counts `block_sizes`, the scalars `P`/`n_block`, and one convenience accessor `size()`.

Layering is *exactly* §4-correct. It is the seam between the GPU precompute engine (S0–S2) and the small-LA fit engine (S3–S8); it is plain host-accessible FP64 storage that crosses the CUDA-PRIVATE seam; it includes only `<cstddef>` and `<vector>` and names no device type. `backend.hpp` returns it by value from the pure-virtual `compute_f2_blocks` (`backend.hpp:164-169`), `core/fstats/f2_from_blocks.{hpp,cpp}` forwards it through the DI seam (`f2_from_blocks.cpp:41-52`), and the planned CLI/Python bindings consume it without dragging in the toolkit. Dependency direction is down-only, no cycles, `#include`-clean behind a correct guard (`STEPPE_FSTATS_HPP`). This is a faithful realization of the §4 "include/steppe/fstats.hpp (planned, M4) public F2BlockTensor handle" line and ADR-0005's `f2_blocks`-seam decision.

Because this is a near-trivial POD header, the substantive findings cluster into: (a) a real, multi-file documentation contradiction on the diagonal convention masking an uncaught GPU-vs-CPU M0 correctness divergence (F1); (b) a 2×-undercount of the resident-tensor budget the header itself points the reader at (F21, new); (c) defensive/correctness hardening of the only logic the header defines (`size()`, F4) on a degenerate input that is constructible and, at M7, deserialized from untrusted bytes; (d) the absence of any executable invariant on a struct three layers and an on-disk cache will marshal (F8); and (e) interchange-longevity gaps — versioning, labels, provenance (F16/F17/F20). Where the header is already excellent I say so explicitly and resist padding.

## Score: 8/10 — excellent, near-publishable contract header; held back by a documentation claim that is false in three files and papers over an uncaught M0 diagonal divergence, a §11.2 budget citation that undercounts the resident tensor by 2×, and a `size()` that lies on the exact degenerate input the backends write unclamped

Rationale. The doc-comments are unusually rigorous and spec-cited; the layout invariant `i + P·j + P·P·b` is stated precisely and matches both backends and all three test loops; `size()` is correctly `[[nodiscard]] noexcept const` and casts each `int` to `std::size_t` *before* multiplying (the right overflow-safe idiom, the exact ROADMAP §4 "index cast: keep size_t" decision). It is held back from the 9.5–10 bar by: F1 (high) — re-verified to be **wrong**, not merely loose, with the CPU and GPU M0 paths actually disagreeing and every test excluding the diagonal; F21 (med) — the resident-footprint doc the consumer reads here counts `f2` but not the equally-large `vpair`; F4 (low-med) — `size()` returns a near-`SIZE_MAX` value on negative `P`/`n_block`, which both backends write unclamped and M7 will read from untrusted bytes; F8/F16/F17/F20 (low-med→med) — no executable invariant, no `schema_version`, no labels/provenance decision on an artifact whose comments repeatedly call it "the cacheable, ADMIXTOOLS-compatible interchange artifact." I hold the prior author's 8.0 (down from their starting 8.5): F1 alone is a correctness-adjacent defect with three wrong copies, and F21 adds a second concrete doc defect on the budget the header surfaces.

## Findings

### (1) Correctness & bugs

**F1 [high — confirmed, upgraded from the original pass's med] The header attributes the nonzero diagonal to "the M0 compute_f2 convention," but the CPU M0 path zero-fills the diagonal and the GPU M0 path fills it with `−2·mean_het`, so the two M0 backends genuinely DISAGREE, every reference test excludes the diagonal so the divergence is uncaught, and the false attribution is replicated in three files.**

Location: doc-comment lines 35–39 ("The DIAGONAL carries the full (i,i) computation (= -2·mean within-pop het correction … the M0 compute_f2 convention), NOT a forced 0").

Verified arithmetically and line-by-line this pass:
- `F2BlockTensor` diagonal is correctly `−2·mean_het`. CPU `compute_f2_blocks` (`cpu_backend.cpp:245`) iterates `for (int j = i; …)` (includes `i==j`); `f2_estimator.hpp:76-80` gives `f2_term(p,p,hc,hc) = (p−p)² − hc − hc = −2hc`; `finalize_f2` over `Vpair(i,i)` ⇒ `−2·mean_het`. The CUDA `compute_f2_blocks`→`assemble_blocks_group_kernel` (`f2_blocks_kernel.cu:127-161`) scatters every `(i,j)` identically. **So the header's claim about `F2BlockTensor` itself is true.**
- **The attribution to "the M0 compute_f2 convention" is false for the CPU M0 path.** `CpuBackend::compute_f2` (`cpu_backend.cpp:123`) walks the **strict** upper triangle `for (int j = i + 1; …)` and explicitly "leave[s] the diagonal at its zero default … (f2(i,i) = 0)" (lines 106-108, 118-119). The CPU M0 `F2Result` diagonal is **zero**.
- **The GPU M0 path does the opposite.** `CudaBackend::compute_f2` (`cuda_backend.cu:89`) calls `launch_assemble_f2`; `assemble_f2_kernel` (`f2_block_kernel.cu:139-157`) writes **every** `(i,j)` with no `i==j` guard. I checked the `i==j` algebra against the masked GEMMs: `G(i,i) = Σ_s Q(i,s)² = sumsq_i` (Q is zero-filled where invalid), so `sumsq_i + sumsq_i − 2·G(i,i) = 0`, leaving `num = −hsum_i − hsum_i`, divided by `Vpair(i,i) = Σ_s V(i,s)` ⇒ `f2(i,i) = −2·mean_het`. The GPU M0 `F2Result` diagonal is **nonzero**.
- Therefore the two M0 `F2Result` artifacts **disagree on the diagonal** (CPU 0, GPU `−2·mean_het`), and `F2Result`'s own doc (`backend.hpp:39-52`) is silent on the diagonal.
- The divergence is **silently uncaught**: `test_f2_equivalence.cu:525` does `if (i == j) continue;` and the comment at `:512` asserts (wrongly) "The diagonal is ~0 by construction and excluded." On the GPU it is `−2·mean_het`, not ~0; it is simply never compared.
- The same false claim is copied into two more places: `cpu_backend.cpp:233-238` ("the M0 compute_f2 GPU/ref pair likewise fill the diagonal") and `test_f2_blocks_equivalence.cu:158-162` ("the M0 compute_f2 convention this M4 path must match"). The M4 test's "single-block == M0" check (`test_f2_blocks_equivalence.cu:252-253`) compares `compute_f2_blocks` against `compute_f2` **both on the GPU backend** (`*gpu`), so both carry the nonzero diagonal and the bit-for-bit check passes — it does **not** catch that the *CPU* M0 diagonal differs. The test header summary line 34 ("f2 diagonal zero") even contradicts its own body lines 158-162 ("diagonal not 0"); neither is asserted (`vpair_matches_recount` checks Vpair only).

Why it matters (architecture.md §2 single-source-of-truth; §13 oracle≡GPU at the `f2_blocks`/`F2Result` seam; Definition-of-Done §18 "agreement at the `f2_blocks` seam"). This is the public documentation of the interchange artifact making a factual claim about a sibling path the code contradicts, sitting on top of a real cross-backend numeric divergence. Two backends meant to be diffed at the `F2Result` seam (§13) produce different bytes on the diagonal, and CI is green only because all three equivalence tests exclude `i==j`. Downstream f3/f4 read off-diagonal only, so today it is latent — but the moment anyone (M7 cache round-trip; a property test on `f2(i,i)`; a future f3 that reads the diagonal; the M4.5 1-GPU-vs-G equivalence test if it ever compares against a CPU partial) compares an M0 `F2Result` across backends, they get a spurious mismatch and conclude a backend is buggy. The header is the worst place to be wrong about this because it is the artifact's canonical contract.

Concrete fix (do all three):
1. In this header, stop attributing the convention to "the M0 compute_f2 convention." State precisely: *"`F2BlockTensor` fills the diagonal with the full `(i,i)` computation `= −2·mean within-pop het correction`, for path-consistency between the CUDA grouped GEMM and the CPU per-block oracle. NOTE the sibling single-block `F2Result` is **inconsistent across backends** on the diagonal (CPU `compute_f2` zero-fills; GPU `compute_f2` computes `−2·mean_het`); the diagonal is never consumed downstream — f3/f4 read off-diagonal f2 only."* Better: pin ONE convention everywhere.
2. Pin the `F2Result` diagonal convention in `backend.hpp` and make the CPU and GPU M0 paths agree (cheapest: have `assemble_f2_kernel` write 0 when `i==j`, OR have CPU `compute_f2` use `j = i` like the per-block oracle).
3. Fix the three parroting comments (`cpu_backend.cpp:233-238`; `test_f2_blocks_equivalence.cu:34` & `:158-162`; `test_f2_equivalence.cu:512`) and add a test that compares the diagonal across backends so the divergence cannot reappear silently.

Severity: **high** (documented-but-false invariant on the public artifact, layered on an uncaught cross-backend numeric divergence). Effort: S (header doc) / M (unify convention + fix the M0 split + add a diagonal-comparing test). Before-M4.5? **Yes** — M4.5 produces `F2BlockTensor` partials per GPU and combines them; pin/correct/**test** the diagonal before it is replicated across G devices, and close the M0 split before M7 round-trips any `F2Result`/`F2BlockTensor` to disk.

**F2 [low — confirmed] The comment claims `Σ over blocks == the total SNP count` for `block_sizes`, but that holds only for the single-dataset, unfiltered, single-device path; the populating code neither enforces nor can enforce it.**
Location: lines 56–58.
Verified: `block_sizes[b]` is `end[b] − begin[b]` over the contiguous `block_id==b` run (`cpu_backend.cpp:222-223`, `cuda_backend.cu:144`). The sum equals the number of SNP columns *supplied to the call* (`Q.M`). Per §5/§5 S0′, cheap in-tile filters drop SNPs before S2; per §5/§11.4 a multi-GPU shard owns only its SNP tiles, so a device's partial sums to *its shard's* post-filter count, not the dataset total.
Why it matters: low, but this is the doc a future engineer consults for the M4.5 host-side combine; assuming `Σ block_sizes == dataset SNP count` for a per-GPU partial mis-accounts.
Concrete fix: replace "the total SNP count" with "the number of SNP columns supplied to this call (`Q.M`; after any in-tile filtering, per-device for a multi-GPU shard — §5 S0′/§11.4)."
Severity: low. Effort: S. Before-M4.5? Yes (it is the M4.5 combine's mental model).

**F3 [low — confirmed] Nothing documents that `block_sizes[b]` can be 0 / a slab can be all-zero on a shard, and that this is the `Vpair==0 ⇒ 0` zero-weight case S4 must compose correctly.**
Location: lines 47–58.
Verified: `backend.hpp:160` documents `n_block == max(block_id)+1`; the M3 rule dense-renumbers occupied bins (ROADMAP M3) so in the single-dataset path every `block_sizes[b] ≥ 1`. After sharding (M4.5/§11.4) a device may own zero SNPs for some block id ⇒ `block_sizes[b]==0` and an all-zero f2/vpair slab. `finalize_f2` (`f2_estimator.hpp:110-111`) returns 0 when `vpair==0`; §5 S2 caveat (a) requires the S4 weighting to treat that as weight 0 (compose, not double-normalize).
Concrete fix: one sentence in the `block_sizes`/`vpair` comments: "On a multi-GPU shard a block may be empty (`block_sizes[b]==0` ⇒ that slab's f2/vpair are all zero and it contributes weight 0 at S4; §5 S2 (a))."
Severity: low. Effort: S. Before-M4.5? Yes (sharding makes empty per-device blocks real).

### (2) Edge cases & failure modes

**F4 [low-med — confirmed, reachability sharpened] `size()` returns a near-`SIZE_MAX` value when `P < 0` or `n_block < 0`, and both backends write `out.P`/`out.n_block` from the caller UNCLAMPED, so such an object is constructible and returned as written; it then disagrees with `f2.size()`.**
Location: lines 67–70.
```cpp
[[nodiscard]] std::size_t size() const noexcept {
    return static_cast<std::size_t>(P) * static_cast<std::size_t>(P) *
           static_cast<std::size_t>(n_block);
}
```
Verified against the C++ working draft: converting a negative signed value to an unsigned type of width N is **well-defined** and yields the unique value congruent to the source modulo 2^N — i.e. a near-`SIZE_MAX` value for small-magnitude negatives ([conv.integral], eel.is/c++draft/conv.integral: "the result is the unique value of the destination type that is congruent to the source integer modulo 2^N"). Not UB. So for `n_block == -1`, `size()` returns `(size_t)P · (size_t)P · 0xFFFF…FF`.
Verified the write path: both backends set `out.P = P;` / `out.n_block = n_block;` **directly from the caller, unclamped** (`cpu_backend.cpp:200-201`, `cuda_backend.cu:123-124`). They clamp only the *allocation* via `n_block < 0 ? 0 : n_block` (`cpu_backend.cpp:202,204`; `cuda_backend.cu:126,129`) then early-return `if (P <= 0 || M <= 0 || n_block <= 0)` (`cpu_backend.cpp:207`, `cuda_backend.cu:130`). So a `F2BlockTensor` with `n_block == -1` (or `P == -1`) and **empty** vectors is constructible and returned, and `size()` on it is a 64-bit garbage value that does **not** equal `f2.size()` (= 0). A consumer using `size()` to bound a loop over `f2.data()` walks far off the end.
Adversarial reachability check (sharper than the original pass): the *normal* pipeline never produces this — `partition.n_block = max(block_id)+1 ≥ 1`, so `assign_blocks` cannot emit a negative. The reachability that matters is (i) the struct is a public POD any caller can build/mutate, and (ii) M7 will **deserialize these scalars from disk**, where the field values are untrusted bytes (§16 ABI surface; ROADMAP M4.5/M7). On both, fail-fast (§2) wants `size()` impossible-to-misread. This is the *only* logic in the file, so getting it right matters disproportionately.
Concrete fix (smallest, removes the lie): `if (P <= 0 || n_block <= 0) return 0;` then the cast-multiply. Matches the backends' clamp and `f2.size()`. A debug `assert(P >= 0 && n_block >= 0)` is a fine complement but not a substitute (release strips it). Pairs with `well_formed()` (F8).
Severity: low-med. Effort: S. Before-M4.5? Recommended — one-line hardening of the only logic in the file; M7's on-disk cache reads these fields from untrusted bytes.

**F6 [info, clean — confirmed] Degenerate `P==0` / `n_block==0` is correct.** `size()==0`, vectors empty, backends early-return (`cpu_backend.cpp:207`, `cuda_backend.cu:130`). No action.

**F7 [info, clean — confirmed] `std::size_t` overflow of `size()` is physically impossible.** `P²·n_block > 2^64` cannot occur: §11.2's VRAM budget bounds `P ≤ ~4266` (ROADMAP §0) and `n_block ≈ 757`, so `P²·n_block ≈ 1.38e10` — far inside 64 bits. The cast-before-multiply already prevents the `int·int` 32-bit overflow that *would* bite above `P²·n_block ≈ 2.1e9` (the real win; see F12). No fix. (Note: the *element count* fits 64 bits, but the *byte footprint* at that scale is the F21 budget concern — distinct issue.)

### (7) Readability, naming, const-correctness, [[nodiscard]]/noexcept, comments

**F8 [low-med — confirmed] An interchange/value struct three layers + a planned on-disk cache marshal should own the index math derived from its own layout rule; today it offers only `size()`, so consumers re-derive slab/offset math by hand (a §2/§8 DRY smell).**
Location: whole struct.
Verified duplication this pass: `cpu_backend.cpp:203` (`slab = P*P`) and `:243` (`base = slab*b`), `:264-267` (`base + i + P*j`); `cuda_backend.cu:125` (`slab = P*P`), `:257` (`pp_nb = slab*nb`); `f2_blocks_kernel.cu:144,158` (`Pp*Pp*k`, `Pp*Pp*id`); all three test loops recompute `i + P*j + P*P*b` inline (`test_f2_blocks_equivalence.cu:122-123, 150-151, 179-182`). The layout rule is documented once (lines 31–34, good) but *implemented* nowhere, so it is re-implemented per consumer; S3 (f3/f4), the M4.5 combine, and M7 (de)serialize will add more copies. An off-by-one in `P·P·b` is a silent cross-block corruption, not a crash.
Concrete fix: add small `[[nodiscard]] noexcept const` accessors next to `size()`, turning the documented layout into the single *executable* source of truth:
```cpp
[[nodiscard]] std::size_t slab_elems() const noexcept {
    return static_cast<std::size_t>(P) * static_cast<std::size_t>(P);
}
[[nodiscard]] std::size_t block_offset(int b) const noexcept {
    return slab_elems() * static_cast<std::size_t>(b);
}
[[nodiscard]] std::size_t flat_index(int i, int j, int b) const noexcept {
    return block_offset(b) + static_cast<std::size_t>(i)
         + static_cast<std::size_t>(P) * static_cast<std::size_t>(j);
}
[[nodiscard]] bool well_formed() const noexcept; // P>=0 && n_block>=0 &&
    // f2.size()==size() && vpair.size()==size() && block_sizes.size()==(size_t)n_block
```
Keep them minimal — the POD stays a POD (shape/index/validity only, no domain behavior). Verified these do NOT break aggregate init (see Considered & rejected: member functions don't disqualify an aggregate).
Severity: low-med. Effort: S. Before-M4.5? Recommended — the M4.5 combine and S3 contraction both index into slabs; `flat_index`/`block_offset` now prevents two more copies.

**F12 [low — confirmed] `size()`'s comment under-documents the only logic in the file and doesn't say WHY it casts before multiplying.** Line 66 ("Flat element count … (convenience)") is thinner than the surrounding (exemplary, spec-citing) density. Add: "casts each `int` to `std::size_t` *before* multiplying so the product is computed in 64-bit unsigned arithmetic — `int·int` would overflow once `P²·n_block` exceeds ~2.1e9 (ROADMAP §4 'index cast: keep size_t, free, mandatory above P≈32k')." Ties it to the existing ROADMAP §4 decision (verified at ROADMAP.md:98).
Severity: low. Effort: S. Before-M4.5? No.

**F13 [low — confirmed] State the width contract: dimensions are `int`, flat offsets are `std::size_t`, and note the cross-struct `int` vs `long` split with `DecodeResult::M`/`MatView::M`.** ROADMAP §4 calls for size_t flat-indexing; here the *fields* are `int` (matching `MatView::P`, views.hpp:57) while `size()` widens to `size_t` — correct and deliberate, but the header is the natural place to say so. Not a bug at AADR scale (a single block's SNP count never approaches `INT_MAX`; per ROADMAP §0 ~770 SNPs/block), but worth one line: "pop/block counts are `int`-bounded by the §11.2 VRAM budget; raw total SNP counts (`M`) are `long`."
Severity: low. Effort: S. Before-M4.5? No.

**F14 [info, clean — confirmed] `[[nodiscard]] noexcept const` on `size()` is correct; naming and guard match house style.** A pure side-effect-free accessor should be all three, and it is. Naming (`f2`, `vpair`, `block_sizes`, `P`, `n_block`) matches the spec vocabulary and the sibling `F2Result`. Guard `STEPPE_FSTATS_HPP` matches `config.hpp`/`backend.hpp`. ### (8) Performance

**F15 [info — confirmed] `size()` is trivially cheap; the by-value vectors are appropriate.** The struct owns three `std::vector`s by value; the CUDA backend fills them via D2H copy (appropriate for the small artifact, modulo F21's footprint note). Returning by value from `compute_f2_blocks` relies on guaranteed move/NRVO — fine. Out of scope for this header.

## Considered & rejected

- **"`block_sizes` should be `std::vector<std::size_t>` not `std::vector<int>`."** Rejected: a block's SNP count is bounded by the per-block count (ROADMAP §0: ~757 blocks over ~584k SNPs ⇒ ~770 SNPs/block) and by the §11.2 VRAM budget. `int` is correct and matches `n_block`/`P`. (I do recommend documenting the int-vs-long width contract — F13.)
- **"`size()` should return bytes / add a `bytes()` helper."** Rejected as required: `size()` is unambiguously the element count and matches `f2.size()`. A `bytes()` could aid the §11.2 budget check, but that math lives in `build()`/the allocator (§11.2), not on this POD; adding it here invites budget logic to leak onto the struct. NOTE: F21 is the *documentation* of the footprint, which IS this header's job; a `bytes()` *method* is still rejected — the fix is the doc + the §11.2 table, not a method. Optional at best.
- **"Make the struct `final` / delete copy to prevent deep copies."** Rejected: it is a value/interchange type legitimately copied; the M4.5 host-side combine *wants* to hold several `F2BlockTensor`s. Move-only would fight the §9 immutable-value posture.
- **"`F2BlockTensor` duplicates `F2Result` — collapse them."** Rejected: genuinely different artifacts — `F2Result` is one `[P×P]` block (M0); `F2BlockTensor` is the `[P×P×n_block]` stack plus `block_sizes`. A `n_block==1` `F2BlockTensor` is not interchangeable with `F2Result` (different diagonal behavior on the CPU path — F1 — and the extra `block_sizes`). The action is to fix/pin the diagonal (F1), not merge.
- **"Use `#pragma once`."** Rejected: the codebase consistently uses `#ifndef` guards (`config.hpp`, `backend.hpp`, `views.hpp`); matching house style is correct.
- **"`size()` could be `constexpr`."** Rejected as load-bearing: `P`/`n_block` are runtime fields, so no caller can evaluate it at compile time; marking it `constexpr` is legal but pointless. Marking the *proposed* `slab_elems`/`block_offset`/`flat_index` `constexpr` is harmless and slightly nicer — optional, noted in F8.
- **"Validate `P`/`n_block` in a constructor."** Rejected: the struct is an aggregate POD by design (both backends use `out.field =` init). **Verified against the C++ working draft:** an aggregate is "an array or a class with no user-declared or inherited constructors, no private/protected direct non-static data members, no private/protected direct base classes, and no virtual functions or virtual base classes" ([dcl.init.aggr], eel.is/c++draft/dcl.init.aggr). Member functions (`size()`) and default member initializers (`int P = 0;`) do **not** disqualify it; a *user-declared/provided* constructor **would**. So `size()` and the proposed F8 accessors keep aggregate init, but a validating constructor would break it. The right place for validation is a `well_formed()` predicate the caller invokes (F8), not a throwing constructor.
- **"`vpair` as `double` loses exactness for huge counts."** Rejected: `vpair` is a SNP count ≤ `M`; `double` represents integers exactly to 2^53, far above any realistic SNP count, and the doc already says "Integer-valued (carried as double)" with the §5 S2 (a) rationale (it feeds GEMM-shaped `V·Vᵀ` math, where FP64 is the natural type). Correct as-is. (Separately: an M7 *on-disk* format MAY narrow it to an integer type to halve the cache — noted under F21 — but that is a serialization choice, not an in-memory/resident one.)
- **"first-pass F5 (no documented invariant tying the sizes together) is a separate finding."** Rejected as a separate item: it is the same fix as F8's `well_formed()` + the invariant sentence; kept folded into F8/F4 (not double-listed). Invariant to document: `f2.size() == vpair.size() == size()` and `block_sizes.size() == (size_t)n_block`.
- **"The `Precision` shown in architecture.md §9 is an `enum class`, but config.hpp makes it a `struct` — flag the drift here."** Rejected for THIS header: fstats.hpp refers only to "the `Precision` knob" generically (lines 17, 41) and never names the enum-vs-struct shape, so it is consistent with the actual `struct Precision` in config.hpp:121. The arch-doc-vs-config drift is real but belongs in the config.hpp / architecture.md review, not here.
- **"`size()` should `assert` the vectors match (`f2.size()==size()`)."** Rejected as the primary fix: `size()` is `noexcept` and is the *definition* of the expected count, not a checker; coupling it to `f2.size()` would make a pure shape query depend on populated state. The check belongs in `well_formed()` (F8). A debug `assert(P>=0 && n_block>=0)` inside `size()` is a fine *complement* (F4) but not the home for the cross-vector invariant.

## Good patterns to keep

- **Citation-rich, spec-anchored doc-comments.** Every field and the file header cite the exact architecture/ROADMAP section (§4, §5 S2, §11.1/§11.2, §0) and even the byte-budget term `P²·n_block·8`. This is the documentation standard the rest of the codebase should match. The places to *correct* (not remove) are the diagonal sentence (F1) and the single-term footprint citation (F21).
- **The layout rule stated once, precisely.** "`i + P·j + P·P·b` (block-major outer, column-major within a block)" (lines 31–34) is the unambiguous single-home invariant §2 wants — it just deserves an executable companion (F8). Verified it matches `cpu_backend.cpp:264-267`, the CUDA scatter (`f2_blocks_kernel.cu:158-160`), and all three test loops.
- **`size()` casts each `int` to `std::size_t` before multiplying.** The correct overflow-safe idiom (ROADMAP §4 "keep size_t … mandatory above P≈32k", verified ROADMAP.md:98); the right default for all flat-index math, and why F8's `flat_index` casts before the `+ P·j`.
- **`[[nodiscard]] noexcept const` on the pure accessor.** Exactly right for a side-effect-free value query; the standard the proposed `well_formed()`/`flat_index()` accessors should follow.
- **Storage-is-always-FP64, precision-is-an-operation-mode** stated up front (lines 17–19, 41–44) — the load-bearing §11.2/§12 invariant, surfaced in the public header where consumers see it first. Keep verbatim (just extend the footprint half of line 19 per F21).
- **CUDA-free, minimal-include, self-contained.** Earns its place on the §4 CUDA-PRIVATE-seam side by including only `<cstddef>`/`<vector>` and naming no device type. Keep this discipline as the struct grows (resist pulling in `config.hpp` for a precision tag; carry tags as a small enum or out-of-band).
- **`vpair` retained and documented as the S4 weight, with the explicit "must compose, not double-normalize" warning** (lines 47–53) — precisely the §5 S2 caveat (a) parity trap, surfaced at the point of definition. Keep this warning verbatim.
