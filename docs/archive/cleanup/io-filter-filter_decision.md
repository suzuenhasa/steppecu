## Role & layering

`filter_decision.hpp` is the single-source-of-truth host-pure predicate library for every M2 keep/drop decision — the M2 analogue of `core/internal/f2_estimator.hpp`: nine `inline [[nodiscard]] noexcept` pure functions, one decision each, every threshold injected by the caller (sourced from `FilterConfig`). It is consumed by `snp_filter.cpp` (in-tile MAF/geno/class/monomorphic/transversion/autosome), `mind_prepass` (per-sample `--mind` via `sample_passes_mind`), and the two test TUs (`test_filters.cpp`, `test_filter_oracle.cu`). The DRY intent — "the rule lives ONCE so the in-tile path, the pre-pass, and the tests cannot diverge" (header lines 6-9) — is exactly arch §2/§8, and I confirmed it is real: `oracle_keep_mask` in `test_filter_oracle.cu` deliberately re-implements the *threshold* boundaries by hand (lines 155-166) but still calls THESE predicates for the allele-pair classes (`is_multiallelic`/`is_strand_ambiguous`/`is_transversion`), so the class logic is genuinely single-homed even in the independent oracle.

Layering is clean and correct against arch §4: an `io`-leaf header, pure host C++20, includes only `<cctype>` and `<steppe/config.hpp>` (the CUDA-free public config). It depends on nothing in `core`/`device`, includes no CUDA header, emits no upward dependency, and reads `kAutosomeChromMin/Max` from config rather than hardcoding `1`/`22` (ROADMAP §4). This is a genuinely near-model leaf; most findings below are low-severity polish, with the substantive ones concentrated in three places: the contract on non-finite/degenerate frequency (correctness vs §12 parity), the locale-dependence of `normalize_allele` (a cross-host determinism foot-gun vs the capability-tier "parity holds on both hosts" promise), and the `is_monomorphic` re-fold parameter-semantics contradiction.

## Findings

### (1) Correctness & bugs

**1.1 — `is_monomorphic(double pooled_minor_af)` takes an ALREADY-FOLDED value but re-folds it; the parameter name and the call site disagree about what is passed. [med, S, before-M4.5: yes] — CONFIRMED.**
Location: `is_monomorphic`, lines 110-112; call site `snp_filter.cpp:91`; test `test_filters.cpp:77-83`.
Re-verified against all three. The parameter is named `pooled_minor_af` and the doc (line 105) says "monomorphic iff its pooled folded MAF is exactly 0", yet the body computes `folded_maf(pooled_minor_af) == 0.0` — it folds again. `snp_filter.cpp:91` calls `is_monomorphic(sm.pooled_minor_af)` where `sm.pooled_minor_af` is *already* `folded_maf(sm.pooled_ref_af)` (set at `snp_filter.cpp:54`), so the production value is folded twice. Harmless *only because folding is idempotent on `[0, 0.5]`*: `folded_maf(x)=min(x,1-x)`, and for `x∈[0,0.5]`, `min(x,1-x)=x`; `snp_filter.cpp:54` guarantees the input is in `[0,0.5]`. But the contract is now self-contradictory: the name says "minor af" (folded), the body treats it as a "ref af" (unfolded) by folding it — the opposite of its sibling `snp_passes_maf` (line 83), which takes the *same* parameter name `pooled_minor_af` and does NOT fold. The test confirms the body's contract, not the name's: `test_filters.cpp:79` passes `is_monomorphic(1.0)` and relies on the inner re-fold to get `folded_maf(1.0)==0` (if the inner fold were removed and the body tested `pooled_minor_af==0.0` directly, `1.0==0.0` is false → the test would FAIL). So the test is written against "re-fold a ref-af" while the parameter name asserts "already-folded minor-af." Pick one.
Why it matters: arch §8 — the whole point of this file is "the rule lives ONCE"; a predicate whose parameter semantics silently differ from its sibling is a divergence trap, and the next refactor that "cleans up" the apparent double-fold by deleting the inner `folded_maf` will silently break `test_filters.cpp:79`/`:80` (which pass unfolded `1.0`).
Concrete fix (recommended): rename the parameter to `pooled_ref_af`, keep the inner fold, and update the doc to "monomorphic iff the pooled folded MAF is 0, i.e. pooled_ref_af is exactly 0 or 1" — this matches what the body and the test already do and makes the name honest. (The alternative — drop the inner fold, test `pooled_minor_af==0.0`, and change the test inputs `1.0`→`folded_maf(1.0)` — also works and makes `is_monomorphic(x) ≡ (snp_passes_maf(x,ε) false ∀ε>0)` exact, but it churns the test; the rename is lower-risk.) Severity med because it is a live contract self-contradiction in the single-source file, even though no current path is wrong.

**1.2 — NaN / non-finite `pooled_minor_af` has no defined predicate behavior; a NaN that survives at `maf_min==0` flows into the kept set and poisons the bit-identical parity gate via BOTH the drop and the mask paths. [med, S, before-M4.5: yes] — CONFIRMED and EXTENDED.**
Location: `snp_passes_maf` (83-85), `is_monomorphic` (110-112), `folded_maf` (74-77); upstream producer `snp_filter.cpp:52-54`.
`pooled_minor_af` is `folded_maf(pooled_ref_count / pooled_allele_count)`, guarded by `pooled_allele_count > 0.0` (snp_filter.cpp:53), which blocks the `0/0→NaN` all-missing case. It does NOT block a `NaN` arriving via `in.q[off]` being `NaN`, nor an out-of-range `pooled_ref_af` if a caller supplies inconsistent Q/N. IEEE-754 ordered comparisons with NaN are all false (only `!=` is true), so:
 - `folded_maf(NaN)`: `(NaN < 1.0-NaN)` is false → returns `1.0-NaN = NaN`.
 - `snp_passes_maf(NaN, maf_min)`: `NaN >= maf_min` is false → DROPPED (the safe direction — but *accidental*, falling out of IEEE semantics, not a deliberate guard).
 - `is_monomorphic(NaN)`: `folded_maf(NaN)==0.0` → `NaN==0.0` is false → NOT monomorphic.
So at `maf_min==0` a NaN-frequency SNP is KEPT by MAF (`NaN>=0` is false → wait: false → dropped). Correction on re-trace: `snp_passes_maf(NaN, 0.0)` is `NaN>=0.0` = false → the SNP IS dropped even at `maf_min==0`. So a NaN does NOT survive the MAF gate. **This narrows the auditor's claim**: the *MAF gate accidentally drops NaN at every threshold including 0*. The residual hazard is the `snp_passes_geno` path: `missing_frac` is computed at `snp_filter.cpp:55-56` and could be NaN only if `total_indiv_d>0` but `nonmissing_indiv` is NaN (i.e. some `in.n[off]` is NaN). `snp_passes_geno(NaN, geno_max_missing)` = `NaN <= geno_max_missing` = false → DROPPED. So both threshold gates accidentally drop NaN. The genuinely unpinned residual is the **masking path in the oracle test** (`mask_columns`, `test_filter_oracle.cu:120-130`): if a SNP is *kept* (because its Q was finite but some other quantity was NaN — not reachable today but the contract is silent), its Q column flows unmodified into `compute_f2` and the `memcmp`-based `f2_bit_identical` gate (lines 85-90) compares NaN bit-patterns, where `memcmp` treats NaN payloads as ordinary bytes — two computations that both produce a NaN can still differ in NaN payload and FAIL the gate, OR a quiet NaN can silently propagate.
Why it matters: arch §12 makes the f2 parity contract bit-exact via `memcmp`. The unit cannot fix the upstream cause, but a 9.5 leaf PINS the contract. The accidental "drop NaN at every threshold" is a *good* default — but it is undocumented and one IEEE-semantics edit away from inverting.
Concrete fix: add a one-line precondition to each threshold predicate ("`pooled_minor_af`/`missing_frac` is assumed finite and in `[0,1]`; on `NaN` the predicate returns `false` (drop) by IEEE comparison semantics — a deliberate fail-safe, do not change `>=`/`<=` to `!(...)`-style negations, which would invert the NaN behavior to keep"), and add a one-line note to `folded_maf` that `NaN` propagates (`1.0-NaN`). Adversarial check: reachable today? No — `decode_af` zero-fills Q where invalid and the snp_filter denominator guard blocks `0/0`. That is why this is med, not high. But "not reachable on today's path" is exactly the latent-contract class the bar asks to close, especially before M4.5 wires the filter into a merge/impute front-end where Q can come from merged sources (arch §1 imputation path).

**1.3 — `folded_maf` returns `1.0 - q` for `q > 1` (negative "MAF") and `1.0 - q < q` for `q < 0`; out-of-`[0,1]` ref-af yields a nonsense fold. [low, S, before-M4.5: no] — CONFIRMED, scope narrowed.**
Location: `folded_maf`, lines 74-77.
For `q=1.2`, `folded_maf` returns `min(1.2, -0.2) = -0.2`; for `q=-0.1`, returns `min(-0.1, 1.1) = -0.1`. A negative folded MAF makes `snp_passes_maf(-0.2, 0)` false (dropped) and `is_monomorphic(-0.2)` false. Not reachable on the guarded snp_filter path: Q ∈ [0,1] by the decode contract, and `pooled_ref_af` is a weighted average (weights `N≥0`) of values in [0,1], hence in [0,1] whenever the denominator is positive (and 0 otherwise by the guard). Listed for completeness: the function is documented as `min(ref_af, 1-ref_af)` with an *implicit* domain `[0,1]` that is never stated. Fix: one-line doc precondition "`pooled_ref_af ∈ [0,1]`" (cheap) or a debug assertion. Documentation only; no shipping bug.

**1.4 — `complement` re-normalizes its argument; the `is_strand_ambiguous` line-164 guard is correct and load-bearing despite the "Equivalently: a == complement(b)" comment. [verified-correct, informational] — CONFIRMED.**
Location: `complement` (131-139), `is_strand_ambiguous` (161-166).
Re-traced adversarially. With `a='A', b='N'`: `na='A'`, `nb='\0'`, the `if (na=='\0'||nb=='\0') return false;` at 164 fires → correct. If that guard were removed: `na ('A') == complement('N')='\0'` is false, still correct *by luck* — no real base complements to `'\0'`, so `na == complement(nb)` is true only for real self-complementary pairs. The function is correct as written. No fix; keep the explicit guard — removing it makes correctness depend on the fragile "nothing complements to `'\0'`" invariant.

**1.5 — `is_transversion` re-normalizes the alleles twice (own guard + inside `is_transition`); correct result, redundant guard + work. [low, S, before-M4.5: no] — CONFIRMED.**
Location: `is_transversion`, lines 195-200.
`is_transversion` normalizes `na/nb`, guards `na=='\0'||nb=='\0'||na==nb`, then returns `!is_transition(a,b)` — and `is_transition` re-normalizes and re-runs the identical guard. The result is correct: on the degenerate inputs `is_transversion`'s guard already excludes, so the `!is_transition` branch is unreachable for them. But the *guard must stay*: without it, `is_transversion('N','N')` would be `!is_transition('N','N') = !false = true` — a wrong `true` on garbage. So the duplication is load-bearing, not gratuitous. Four `normalize_allele` calls for two chars; negligible host cost (8.1). Fix (lowest-risk): keep the body, add a one-line comment "the guard is intentionally duplicated so a non-clean pair returns false, not `!is_transition==true`." Or compute `na,nb` once and share a `same_ring(na,nb)` helper with `is_transition`.

### (2) Edge cases & failure modes

**2.1 — `normalize_allele` handles the signedness UB correctly. [verified-correct, informational] — CONFIRMED against the standard.**
Location: `normalize_allele`, lines 123-126.
`std::toupper(static_cast<unsigned char>(a))` is the documented-safe idiom: `std::toupper` has UB "if the value of `ch` is not representable as `unsigned char` and does not equal `EOF`" (cppreference/W3cubDocs `std::toupper`), and a plain `char` may be signed, so a high-bit byte (e.g. `0xC3`) cast directly to `int` would be negative → UB. The `unsigned char` cast first is exactly the prescribed safe form. Good — keep. (But see 11.1: *correct* is not the same as *locale-independent*.)

**2.2 — Absent/short-record allele sentinel: `'N'` (substituted by `snp_filter.cpp:76-77`) normalizes to `'\0'` → `is_multiallelic` true → dropped. Correct; the contract loop between this file and the caller's sentinel is undocumented here. [low, S, before-M4.5: no] — CONFIRMED.**
Location: `is_multiallelic` (174-178); call site `snp_filter.cpp:76-77`.
`snp_filter.cpp` substitutes `'N'` when `si >= snps.ref.size()`; `normalize_allele('N')='\0'`, so `is_multiallelic('N', alt)` is true → dropped — conservative, consistent with drop-not-flip. Correct separation: this header owns *no* sentinel (it classifies anything non-ACGT as non-clean); the reader/caller owns the absent sentinel. No code fix; a one-line note in `is_multiallelic`'s doc ("absent/short-record alleles arrive as any non-ACGT char; the caller substitutes its own sentinel, e.g. 'N'") closes the contract loop with `snp_filter.cpp`.

**2.3 — `is_autosome` is total and overflow-free over `int` chrom codes; negative sentinels handled. [verified-correct, informational] — CONFIRMED.**
Location: `is_autosome`, lines 211-213.
`snp_reader.hpp` maps non-numeric labels to "a stable negative sentinel per distinct label" (doc lines 53-55). `is_autosome(negative)=false`, `is_autosome(0)=false`, `is_autosome(23/24/90)=false`. No arithmetic, no overflow. `test_filters.cpp:141-149` pins exactly these (1, 22, the named constants, 23, 24, 90, 0, -1). Correct and complete.

**2.4 — No predicate validates `maf_min`/`geno_max_missing`/`mind_max_missing` ∈ range. [N/A — by design] — CONFIRMED N/A.**
If a caller passes `maf_min=0.7` (>0.5, impossible for a folded MAF), `snp_passes_maf` silently drops every SNP. Threshold validation belongs in `ConfigBuilder::build()` (arch §9), NOT here — a pure comparator with no clamping is the right design. Flagged only to record the deliberate absence. No fix.

**2.5 — Integer width / wraparound: N/A for this file.** Index/width concerns (`long M`, `size_t` offsets, `i + P*s`) live in `snp_filter.cpp`, not here. This header takes only scalar `double`/`char`/`int`; no integer-width or wraparound surface. Explicitly N/A.

**2.6 — NEW: `complement` is a *public* symbol used only internally by `is_strand_ambiguous`; it widens the API surface for no consumer. [low, XS, before-M4.5: no] — ADDED.**
Location: `complement`, lines 131-139.
`complement` and `normalize_allele` are both public `inline` functions in `steppe::io::filter`, but the only call sites of `complement` are inside this header (`is_strand_ambiguous`). `normalize_allele` is at least re-used by every class predicate; `complement` is single-use. If the intent is a public utility, fine; if not, an unnamed-detail or a comment "exposed for testability" would scope it. The auditor missed this; severity low because exposing a pure, total, documented helper costs nothing. No action required, recorded for completeness.

### (3) Numerical / precision vs §12

**3.1 — `snp_passes_maf` `>=` and `is_monomorphic` `== 0.0` on `double`; deliberate and documented, but the exactness rationale (integer-ratio) is not stated where the `==` lives. [low→med, S, before-M4.5: yes] — CONFIRMED.**
Location: `snp_passes_maf` (84), `is_monomorphic` (111); upstream `snp_filter.cpp:52-54`.
The boundary choices are correct and well-documented (header 60-68). The precision nuance: `pooled_minor_af = folded_maf(pooled_ref_count / pooled_allele_count)` is a single FP64 division of two pooled sums, NOT a cancellation-prone subtraction — well-conditioned (arch §12 reserves native-FP64-cancellation care for the f2 numerator, not a ratio of pooled counts). `is_monomorphic`'s `== 0.0` is *exact* iff `pooled_ref_count` is exactly `0.0` or exactly `pooled_allele_count`. For a truly monomorphic site each per-pop `Q` is exactly `0.0` or exactly `1.0`, so `Q·N` is exact and the pooled sum is exactly `0` or exactly `pooled_allele_count` → `pooled_ref_af` is exactly `0.0` or `1.0` → `folded_maf` is exactly `0.0`. So `== 0.0` is exact *on the real path* — correct, but a non-obvious FP argument the file does not state where it matters. Why §12-relevant: the monomorphic decision feeds the kept-mask feeding the bit-identical f2 parity gate; an `==` on a derived double is a classic flake source, and the reason it is safe (exact integer ratios) deserves a one-line comment so a future refactor (e.g. averaging per-pop *frequencies* instead of pooling *counts*) does not silently break exactness.
Concrete fix: comment on `is_monomorphic` — "exact `== 0.0` is safe because pooled_ref_af is an exact integer ratio AC/AN for a monomorphic site; do not change the upstream pooling to a mean-of-frequencies, which rounds." Cross-ref the MAF-convention block (lines 33-49) which already pins the pooling formula. Note the oracle test (`test_filter_oracle.cu:166`) re-implements this `maf == 0.0` by hand — the comment guards both copies.

**3.2 — `folded_maf` at the symmetric peak `q == 0.5`. [verified-correct, informational] — CONFIRMED.**
`(0.5 < 0.5)` is false → returns `1.0-0.5 = 0.5` (test `test_filters.cpp:46` pins it). At `0.5 ± 1ulp` it returns the smaller, which is what "folded" means. No accumulation, single comparison — no determinism concern.

**3.3 — Determinism / accumulation order: N/A for this header.** No reduction, no accumulation, no GPU, no Ozaki/native-FP64 choice here. Every function is a single comparison or a 2-3-term branch. The accumulation §12 cares about (`Σ_pop Q·N`) lives in `snp_filter.cpp:45-47`. Keeping the predicates accumulation-free is exactly what makes them trivially deterministic and reusable by the oracle. Explicitly N/A — and correctly so.

### (11) Capability tiers (PRO-6000 vs budget-5090)

**11.1 — Capability-tier-NEUTRAL by construction (correct) — but the unit should SAY so, and it must be locale-pinned so the host-side allele class is bit-identical across the capable and budget hosts. [low→med, S, before-M4.5: yes] — CONFIRMED; this is the single highest-value change.**
Location: `normalize_allele` (123-126); cf. `docs/TODO.md` capability-tier section.
This header runs entirely host-side and touches NO GPU capability — no GDS, no P2P, no ncu, no Ozaki, no multi-GPU combine. So there is genuinely nothing to runtime-detect or degrade *here*; the filter decision is identical on both hosts. That is the right place for it to be (the parity-critical cross-GPU work is the f2 combine, arch §11.4). HOWEVER, there is one host-side determinism hazard interacting with cross-host parity: **`normalize_allele` depends on the global C locale**. cppreference/W3cubDocs `std::toupper`: "Converts the given character to uppercase according to the character conversion rules defined by the currently installed C locale," with a documented `setlocale` example where byte `0xb8` folds differently under ISO-8859-1 vs ISO-8859-15. In every *standard* locale ASCII `[A-Za-z]` folds identically, and I confirmed *no* `setlocale`/`std::locale` call anywhere in `src/`/`include/`, so the process runs in the default `"C"` locale and the allele fold is locale-stable *today*. But the contract is "the current C locale," not "ASCII." If a future host process sets a non-default `LC_CTYPE` (a real risk in a multi-dataset merge front-end reading internationalized metadata), two hosts with different locales could in principle diverge — breaking the cross-host bit-identical parity the capability-tier doc demands ("parity holds on both"). The alleles are guaranteed ASCII (`A/C/G/T`), so a locale-sensitive fold buys nothing and risks divergence.
Concrete fix: replace `std::toupper` with a branch-free locale-independent ASCII fold inside `normalize_allele`, e.g. `const char u = (a >= 'a' && a <= 'z') ? static_cast<char>(a - ('a' - 'A')) : a;`. This (a) removes the cross-host locale dependence, (b) is faster (8.2), (c) makes the allele predicates `constexpr`-able (10.3), and (d) makes them device-portable (4.1). Add a comment: "locale-independent ASCII fold — the allele alphabet is ASCII; do NOT use std::toupper, which is locale-sensitive (cppreference: 'rules defined by the currently installed C locale') and could differ across hosts (capability-tier parity, TODO.md)." Highest-value capability-tier change for this unit.

**11.2 — No logged/tagged capability hook is needed in this unit — the correct action is the *negative* one. [verified] — CONFIRMED.**
TODO.md asks each unit to add a capability probe + tagged log line where it touches a tiered capability. This unit touches none, so a log line would be noise. The right action: state in the header that the filter decision is capability-tier-neutral and host-deterministic (after the 11.1 locale fix). No probe, no tag — just the locale fix and a one-line "tier-neutral" note.

## Considered & rejected

- **"`folded_maf` should use `std::min`."** Rejected: `std::min(q, 1.0-q)` requires `<algorithm>` (adding an include) and has the *same* NaN behavior as the ternary. The hand-written ternary is clearer and avoids a header. The NaN concern (1.2) is orthogonal to ternary-vs-min. Not a defect.
- **"NaN survives the MAF gate at `maf_min==0` and reaches the kept set."** Partially rejected / corrected (see 1.2): on re-trace, `snp_passes_maf(NaN, 0.0)` is `NaN>=0.0` = false → the SNP is DROPPED even at threshold 0, and `snp_passes_geno(NaN, ...)` likewise drops. So the auditor's "NaN flows into the kept set at maf_min=0" overstated it — both threshold gates accidentally drop NaN. The residual real hazard is the *masking path* + `memcmp` NaN-payload comparison, and the fact that the safe drop is accidental (IEEE), not deliberate. Finding retained at med with the scope corrected.
- **"Make everything `constexpr`."** Partially upheld (10.3). The threshold predicates CAN be `constexpr` (only comparisons). The allele predicates CANNOT as written, because `normalize_allele` calls `std::toupper`, which is NOT `constexpr` in any standard version (verified C++23/26: no constexpr `std::toupper`). So "make everything constexpr" is rejected *as written*, but "make the threshold predicates constexpr for a no-op-default `static_assert`" is upheld — and if 11.1 replaces `std::toupper` with an ASCII fold, the allele predicates become `constexpr`-able too (a secondary win).
- **"`is_strand_ambiguous` should also drop multiallelic via one call."** Rejected: the two predicates are deliberately orthogonal (palindromes vs non-clean pairs); `snp_filter.cpp:82` OR-combines them. Merging would conflate two distinct drop reasons and lose the ability to report *why* a SNP dropped. Keep separate — good separation of concerns.
- **"The line-164 guard in `is_strand_ambiguous` is redundant."** Rejected — see 1.4: redundant *by luck* (no base complements to `'\0'`); removing it makes correctness depend on that fragile invariant. Keep the explicit guard.
- **"Missing `#include <algorithm>` / `<cmath>` / `<cstddef>`."** Rejected: the file uses no `std::min`/`std::max`, no `<cmath>` function, and no `std::size_t`/`std::ptrdiff_t` in the header. `<cctype>` + `<steppe/config.hpp>` are exactly the needed includes. IWYU-clean (re-verified).
- **"The MAF-convention block duplicates config.hpp's docs — DRY violation."** Rejected as a defect: arch §8 single-home is about *code*, not prose; this file is the single documented home of the convention and config.hpp/snp_filter.hpp point *to* it. Prose redundancy with a clear canonical source is acceptable; noted as a sync-on-edit caution (7.3), not a violation.
- **"`int chrom` should be a stronger type / enum."** Rejected: chromosome codes are an open integer space (1-22 autosomes, 23/24 sex, 90 MT, negative sentinels per `snp_reader`). An enum cannot represent the open negative-sentinel space; `int` matches `SnpTable::chrom`. Keep.
- **"`is_transition`'s `a_purine == b_purine` is unclear / could be wrong."** Rejected: re-verified the truth table — purine={A,G}, pyrimidine={C,T}; a transition is a within-ring substitution, so `a_purine == b_purine` (same ring class) ⇒ transition is the minimal correct formulation, and the guard already excludes `na==nb`, so "same ring AND distinct" ⇒ a real A↔G or C↔T transition. The "same ring class ⇒ transition" comment (line 189) makes it clear. `test_filters.cpp:123-138` pins all eight ts/tv ordered pairs. Correct.
- **"`complement` being public is an API-surface leak — make it `detail`."** Considered, retained as a low/informational note (2.6) rather than a defect: exposing a pure, total, documented helper has no ABI/cost, and the doc justifies it. Not actionable; recorded for the next editor.

## Good patterns to keep

- **Single-source predicate library, one decision per function, all thresholds injected** — the DRY backbone (arch §2/§8). The in-tile path, the `--mind` pre-pass, and both test layers call THESE; even the *independent* oracle re-uses the allele-class predicates, so the class logic cannot diverge.
- **Boundary sides pinned and documented** (`>=` for MAF, `<=` for the missing caps, exact `==0` for monomorphic), with the no-op-default property called out explicitly — the contract that prevents off-by-a-boundary parity flakes.
- **Drop-not-flip enforced by shape** — predicates return one bool per SNP / per sample, so the V·Vᵀ-masked-GEMM column invariant (arch §5 S2) is impossible to violate at this layer.
- **The honest CONVENTION FLAG** (lines 147-160) distinguishing the genetics-correct palindrome class (A/T, C/G) from the brief's transition/transversion example, with the correct implementation of *both* and a measured note ("zero A/T·C/G pairs on the AADR HO panel; the tens-of-thousands the brief means are transversions"). Exactly the "flag the genuinely-ambiguous-against-the-brief point" discipline the project wants — and the oracle test (f) measures it on real data.
- **Correct signedness handling** in `normalize_allele` (`static_cast<unsigned char>` before `std::toupper`) — the documented-safe idiom, verified against the standard's UB clause.
- **Autosome range sourced from `config.hpp`** with an explicit "no bare 22 here" comment — model magic-number discipline (ROADMAP §4).
- **Uniform `[[nodiscard]] inline … noexcept`** on every predicate, justified by their pure-comparator nature.
- **Belt-and-suspenders degenerate-input guards** (the `'\0'` early-outs) making every predicate total over arbitrary `char` input — no UB, no surprising `true` on garbage (the load-bearing guards in `is_strand_ambiguous`/`is_transversion`).
- **CUDA-free, IWYU-clean `io` leaf** — correct layering, minimal includes, zero upward dependency.
