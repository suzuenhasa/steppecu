## Role & layering

`backend.hpp` is the one door `core` uses to reach the GPU (architecture.md ¬ß2, ¬ß4): `core` is pure host C++20 and never includes a CUDA header. The header delivers on that contract ‚Äî it includes only `<cstddef>`, `<cstdint>`, `<vector>`, `steppe/config.hpp`, `steppe/fstats.hpp`, `core/internal/views.hpp`, all CUDA-free (verified: `f2_block_kernel.cuh` includes `backend.hpp`, never the reverse; `decode_af_kernel.cuh` only *mentions* it in a comment and takes raw pointers; no CUDA header leaks into the six includes). The plain-data result types (`F2Result`, `DecodeResult`, `DecodeTileView`) are deliberately `std::vector`/raw-pointer structs so they cross the seam, and the abstract `ComputeBackend` has exactly the two implementations ¬ß8 prescribes. The layering intent is met.

The substantive concerns, in priority: (1) **CPU and GPU `compute_f2` disagree on the diagonal** and the doc is silent while the durable artifact (`F2BlockTensor`) claims the diagonal is "kept consistent across paths" ‚Äî false for the M0 pair (1.2); (2) **GPU `compute_f2` is the only method missing the empty-input guard** the contract implies (1.6); (3) the interface declares contracts the implementations do not enforce ‚Äî P/M agreement, `0 <= block_id < n_block`, monotonicity, `n_block == max+1` ‚Äî UB or silent-wrong on violation (2.1/2.5); (4) the **factories are declared in no header** and six consumers forward-declare them ad hoc across two namespaces (9.1); (5) `block_id` is a bare `const int*` with its length implied, a ¬ß7 span-violation extended to the seam, rationalized by a self-contradicting comment (9.2/9.5); (6) no capability-tier hook, which the TODO names as an explicit M4.5 cross-cutting addition (11.1); (7) the M0/M4 method split is redundant once you see `compute_f2 == compute_f2_blocks(n_block=1)` (6.1).

## Score: 7.5/10 ‚Äî strong, exemplary documentation and clean, verified CUDA-free layering; held back by a real (test-masked) CPU/GPU diagonal divergence, a GPU-only missing empty-input guard, UB-on-malformed-input from documented-but-unenforced preconditions, factories declared nowhere, a bare-pointer length-eliding seam parameter rationalized by a false comment, redundant M0/M4 methods, and no capability-tier seam.

(Down 0.5 from the prior second-pass 8.0: the GPU-only missing empty-input guard (1.6) is a *second* real CPU/GPU divergence on the exact seam this unit owns, alongside the diagonal (1.2) ‚Äî two substantiated correctness divergences between the oracle and the GPU, both on the M0 `compute_f2` the header is built around, is below the 8 band. The documentation and layering remain genuinely excellent, which keeps it well above mid.)

## Findings

### (1) Correctness & bugs

**1.1 ‚Äî [med, CONFIRMED] `DecodeTileView` carries no SNP‚Üírecord offset; the byte-address formula is tile-relative and only coincidentally consistent with the whole-tile decode path ‚Äî M5-hostile, would mis-decode silently. (lines 65‚Äì81)**
The doc fixes SNP `s` of gathered individual `g` at byte `g¬∑bytes_per_record + s/4`, bit position `s%4` (MSB-first). Re-verified against the kernel (`decode_af_kernel.cu:71-72`: `byte_in_rec = s/4u`, `pos_in_byte = s & 3`), the CPU path (`cpu_backend.cpp:309-310`, identical), and the bit primitive (`decode_af.hpp:66-69`: `shift = (3-(k&3))*2`): all three agree with the doc, and `s` is a *tile-local* index `0..n_snp-1`. But `bytes_per_record` is "stride between individual records" ‚Äî in a real TGENO file that is `ceil(total_snp_in_file/4)`, not `ceil(n_snp/4)`. There is no `snp_offset`/`first_snp` field, so a tile that is not the file's first tile (M5 streaming, the milestone right after M4.5) cannot be expressed: the within-record bit position is *absolute* in the file, but the tile-local `s/4`,`s&3` math only addresses bytes from record start. Not a live bug at M1 (the whole ~100k-SNP block is one tile, `snp_offset==0`). Why it matters: M5's first need is "decode SNPs `[t0,t1)` of these records," and the mis-decode would be silent (wrong bytes, plausible-looking Q). Fix: add `std::size_t snp_offset = 0;` and document the byte address as `g¬∑bytes_per_record + (snp_offset+s)/4`, bit position `(snp_offset+s)%4`. Severity med. Effort S. Before-M4.5? No (M5), but cheap to fix while the view is young.

**1.2 ‚Äî [HIGH, CONFIRMED + re-derived] `CpuBackend::compute_f2` and `CudaBackend::compute_f2` produce DIFFERENT diagonals for the same input, and the two `compute_f2` cross-checks that compare them MASK it by skipping `i==j`. The header's `F2Result::f2` doc is silent on the diagonal; `F2BlockTensor` (fstats.hpp:36‚Äì39) promises it is nonzero and "kept consistent across paths." (lines 39‚Äì42, 135‚Äì138)**
Verified at all four computation sites and both gates:
- `CpuBackend::compute_f2` (cpu_backend.cpp:122‚Äì123) walks `for (j = i+1; ...)` and leaves the diagonal at its zero default ‚áí **`f2(i,i) = 0`**.
- `CudaBackend::compute_f2` runs `assemble_f2_kernel` (f2_block_kernel.cu:139‚Äì157), which computes **every** `(i,j)` including `i==j`. For `i==j`: `Gij = G(i,i) = Œ£p_i¬≤`, `sumsq_i = sumsq_j = R(i,i) = Œ£p_i¬≤`, `hsum_i = hsum_j = H(i,i)`, so `num = Œ£p_i¬≤ + Œ£p_i¬≤ ‚àí 2¬∑Œ£p_i¬≤ ‚àí H(i,i) ‚àí H(i,i) = ‚àí2¬∑H(i,i)` ‚áí **`f2(i,i) = ‚àí2¬∑H(i,i)/Vpair(i,i) ‚âÝ 0`** (modulo `Vpair(i,i)==0 ‚áí 0`). Hand-re-derived against `assemble_f2_numerator` (f2_estimator.hpp:97-101); exact.
- The CPU `compute_f2_blocks` diagonal (cpu_backend.cpp:245 `for (j = i; ...)`) and the GPU grouped `assemble_blocks_group_kernel` (f2_blocks_kernel.cu:134-160, every `(i,j)`) BOTH fill the diagonal. So **three of the four `f2` computation paths fill the diagonal; only CPU `compute_f2` zeroes it.** The CPU `compute_f2` even disagrees with the CPU `compute_f2_blocks(n_block=1)` of the same backend.
- The two gates that compare CPU `compute_f2` against the GPU skip the diagonal: `test_decode_equivalence.cu:139` (`if (i == j) continue;` inside `f2_max_rel`) and `test_f2_equivalence.cu:525` (same). And `test_f2_blocks_equivalence.cu:121` skips it in `accuracy`. The blocks test's "single-block == M0" check (`:252-253`) compares **GPU** `compute_f2_blocks(n_block=1)` against **GPU** `compute_f2_block` (which calls **GPU** `compute_f2`) ‚Äî both GPU, both fill the diagonal, so it agrees and *cannot* surface the CPU divergence.

So the two reference implementations of the *same documented statistic* disagree on the diagonal, the header (`F2Result::f2`, line 41) describes the formula with no diagonal carve-out, and the durable artifact (`F2BlockTensor`, fstats.hpp:38) explicitly promises the diagonal is "kept consistent across paths" ‚Äî **false for the M0 `compute_f2` pair.** The diagonal is never consumed downstream (f3/f4 read off-diagonal), so this is not a result-corrupting bug today ‚Äî but it is an oracle/GPU divergence on the seam this unit owns, exactly the class of thing ¬ß13's reference-equivalence exists to catch and here cannot. Why it matters: architecture.md ¬ß12/¬ß13 make CPU‚â°GPU-at-the-seam the trust anchor; a silent, gate-masked disagreement undermines it, and M4.5 replicates the M4 path. Fix: pin the convention on `F2Result::f2` (state exactly what the diagonal is) and make the two `compute_f2` paths agree ‚Äî set the CPU loop bound to `j=i` (matching `compute_f2_blocks` at cpu_backend.cpp:245 and both GPU kernels), then assert the diagonal in the equivalence tests so the gate stops masking it; preferably unify on `compute_f2_blocks(n_block=1)` (6.1) and delete the divergent M0 path. Severity HIGH (oracle/GPU divergence on the seam, masked by the gate). Effort S (CPU loop bound) + S (doc + test assert). Before-M4.5? **Yes** ‚Äî pin the convention and close the test gap before M4.5 calcifies the M4 path.

**1.3 ‚Äî [low, CONFIRMED] `Vpair` "RETAINED ... compose-not-double-normalize" is a load-bearing cross-stage convention the *type* does not encode. (lines 19‚Äì21, 45‚Äì48; fstats.hpp:49‚Äì51)** The header cautions three times that the per-pair divide here and the S4 weighting "must compose to AT2's f2_blocks definition, not double-normalize" (a real ¬ß5 S2 caveat (a) / ¬ß12 constraint), but `F2BlockTensor` records no flag for which normalization was already applied; the only protection when S4 (Phase 2, unwritten) consumes it is the prose. Verified: `finalize_f2` (f2_estimator.hpp:110‚Äì112) returns `num/vpair`, so `f2` is the per-block mean already divided by `vpair`. Fix: nothing actionable until S4, but state on `F2BlockTensor` the exact invariant `f2_block = num/vpair` (already implicit at fstats.hpp:41‚Äì44 ‚Äî make it the one-line invariant S4 reads). Severity low. Effort S. Before-M4.5? No.

**1.4 ‚Äî [low, CONFIRMED] `F2Result::P` / `DecodeResult::P` are settable `int` with no enforced relation to `f2.size()`/`q.size()`.** Both backends copy `P`/`M` faithfully from the input, but a hand-constructed `F2Result` could have `P` disagree with `f2.size()`. Not a bug (the producers are correct) ‚Äî an interface-robustness gap, mirroring `F2BlockTensor::size()` which is checkable but unchecked. Fix: document `f2.size() == size_t(P)*P` (and `q.size() == size_t(P)*M`); optionally add a `[[nodiscard]] bool consistent() const`. Severity low. Effort S. Before-M4.5? No.

**1.5 ‚Äî [med, CONFIRMED] `int P` √ó `long M` is correct, but `DecodeResult::M` is `long` while `DecodeTileView::n_snp` is `std::size_t` and `decode_af` does `static_cast<long>(tile.n_snp)` (cpu_backend.cpp:289, cuda_backend.cu:288) ‚Äî a silent narrowing if `sizeof(long) < sizeof(size_t)`; the header does not state the width contract.** `MatView::M` is `long` (views.hpp:60, justified for >2¬≥¬π SNPs); `DecodeResult::M` mirrors it; but `n_snp`/`n_individuals`/`bytes_per_record` are `std::size_t` and the conversion to `long` is unchecked. On the project's LP64 Linux target `long`==`size_t`==64-bit so this is benign *today*, but the ¬ß4 ABI surface should not silently depend on `sizeof(long)==sizeof(size_t)` (it does not hold on LLP64). Fix: make `DecodeResult::M`/`MatView::M` `std::int64_t`/`std::ptrdiff_t` (explicit width) or document "LP64 assumed: `long` is 64-bit; `n_snp` must fit `long`." Severity med (latent ABI-portability + silent narrowing). Effort S. Before-M4.5? No.

**1.6 ‚Äî [med, ADDED ‚Äî missed by both prior drafts] `CudaBackend::compute_f2` is the ONLY method without the empty-input guard `if (P <= 0 || M <= 0) return out;` that the CPU `compute_f2` and every other method have ‚Äî a real CPU/GPU divergence and contract gap on the seam. (the contract is the doc at lines 130‚Äì134; impl at cuda_backend.cu:55‚Äì103)**
Verified the guard inventory directly: CPU `compute_f2` (cpu_backend.cpp:111), CPU `compute_f2_blocks` (:207), CPU `decode_af` (:300), GPU `compute_f2_blocks` (cuda_backend.cu:130), GPU `decode_af` (:298) ALL early-return on empty/degenerate dimensions. **GPU `compute_f2` (cuda_backend.cu:55‚Äì103) has no such guard.** Consequences on the empty edge:
- `P==0`: `pm=pp=0`, `DeviceBuffer<double>(0)` allocations, zero-length `cudaMemcpyAsync`, then `launch_f2_feeder(..., P=0, ...)` (grid `grid_for(0)=0`, no-op) and `run_f2_gemms(..., P=0, M, ...)` issuing `cublasGemmEx` with `m=n=0` ‚Äî a zero-dimension GEMM. The C++ standard library and CUDA tolerate zero-size allocations, but a `cublasGemmEx` with `m==0||n==0` is undefined/implementation-specific and may return `CUBLAS_STATUS_INVALID_VALUE`, which `CUBLAS_CHECK` would convert to a thrown error ‚Äî a *throw on an input the documented "empty block ‚áí empty result" contract says should return cleanly*, and that the CPU oracle DOES return cleanly. So the two backends diverge on the empty case: CPU returns an empty `F2Result`, GPU may throw.
- `M==0` (P>0): the feeder grid-y is 0 (no-op), the GEMMs have `k=0` so `C = Œ≤¬∑C` with `Œ≤=0` ‚áí zeroed outputs, assemble fills f2/Vpair with 0 ‚áí result happens to match the CPU empty result ‚Äî but only by accident of `Œ≤=0`, not by a guard.

Why it matters: ¬ß2 fail-fast and ¬ß13 reference-equivalence both require the two backends to agree at the seam, including on degenerate input; the header's "empty block: nothing to accumulate" promise (implied by the CPU path and the doc) is not honored by the GPU `compute_f2`. Fix: add the same `if (P <= 0 || M <= 0) return out;` guard at the top of `CudaBackend::compute_f2`, and state the empty-input contract on the interface (`P<=0 || M<=0 ‚áí empty/zero F2Result, no throw`). Severity med (real CPU/GPU divergence + possible throw-vs-clean-return on the seam). Effort S. Before-M4.5? Borderline-yes ‚Äî it is a one-line guard on the M0 path the M4.5 work builds on; fix with 1.2.

### (2) Edge cases & failure modes

**2.1 ‚Äî [HIGH, CONFIRMED] No precondition is enforced; the documented preconditions (P/M agreement; `block_id` length M, dense, non-decreasing; `n_block == max+1`) are honor-system only ‚Äî a violation is UB or silently-wrong output.** Re-verified the consequences against the code:
- If `V.M != Q.M` (or `N.M != Q.M`): both backends loop `s` to `Q.M-1` and read `V.element(i,s)`/`N.element(i,s)`; `MatView::element` (views.hpp:66‚Äì68) has "No bounds check (hot path)" ‚áí **OOB read / UB**. The orchestrator (`f2_from_blocks.cpp`) passes the three views through without cross-checking their `M`.
- If `block_id` is NOT non-decreasing: the block-range scan in *both* backends (cuda_backend.cu:138‚Äì146, cpu_backend.cpp:215‚Äì225) finds, for each id, only the **last** contiguous run (`begin[b]=s; end[b]=e;` overwrites), silently dropping earlier runs of the same id ‚áí **wrong f2, no error**. The header *relies* on monotonicity (line 159) but neither it nor the implementations even debug-assert it.
- If `block_id[s] >= n_block`: the scan writes `begin[block_id[s]]` / `out.block_sizes[block_id[s]]` **out of bounds** (sized to `n_block`) ‚áí **UB**. (See also 2.5 for negative.)
- `n_block > max(block_id)+1`: trailing slabs stay zero (harmless, undocumented). `n_block < max+1`: ids index out of bounds as above.

Why it matters: ¬ß2 fail-fast ("invalid configs surface immediately ... not as silent corruption") applied to the hottest interface in the codebase. The interface cannot run code, but it can (a) state these as hard preconditions whose violation is UB, and the implementations should `STEPPE_DEBUG_ONLY`-assert P/M agreement + monotonicity + range at entry. Best: the TODO already lists `[high] block_ranges(block_id, M, n_block) ‚Üí vector<BlockRange>` in `core/domain/` (verified at TODO ¬ßA line 80) ‚Äî extract it, validate monotonicity/range once, unit-test it, and feed both backends the validated ranges (this also kills the duplicated scan, the same TODO item). Severity HIGH (UB on malformed input from a documented-but-unenforced contract). Effort M. Before-M4.5? **Yes** ‚Äî M4.5 shards `block_id` per device and will replicate the unguarded scan.

**2.2 ‚Äî [med, CONFIRMED] `n_block` is signed `int`; both backends defensively treat negative as 0 (`n_block < 0 ? 0 : n_block`, cpu_backend.cpp:202/204, cuda_backend.cu:126/129) then early-return on `<= 0` ‚Äî but the interface neither documents `n_block >= 0` nor the empty/zero case.** A reader of `backend.hpp` alone would not know `n_block==0` yields an empty `F2BlockTensor`, nor that negative is silently clamped. Line 160 says only "number of distinct blocks (== max(block_id)+1)". The silent clamp *hides* a caller bug (a negative `n_block` is always a programming error, not a domain-empty input). Fix: document `n_block >= 0`; `0` ‚áí empty result; make negative a debug-assert (per 2.1) rather than a silent clamp. Severity med. Effort S. Before-M4.5? No.

**2.3 ‚Äî [med, CONFIRMED ‚Äî divide-by-zero re-verified] `DecodeTileView` degenerate/overflow cases are unconstrained at the interface: `n_pop <= 0`, null/short `pop_offsets`, `ploidy <= 0`, `bytes_per_record == 0`.** Re-verified:
- `n_pop`/`P <= 0`: both backends guard (`if (P <= 0 || M <= 0) return out;`, cpu_backend.cpp:300, cuda_backend.cu:298) ‚áí safe, but the doc never states `n_pop >= 0` nor that `pop_offsets` has length exactly `n_pop+1` and is non-null. The CUDA path computes `n_off = P+1` and uploads `pop_offsets` (cuda_backend.cu:306, 315) ‚Äî safe only because `P>0` is already guaranteed by the guard; a null `pop_offsets` with `P>0` is a null-deref upload.
- `ploidy`: `finalize_af` (decode_af.hpp:96‚Äì108) computes `n = ploidy¬∑an`, `q = ac/n` inside the `an > 0` branch. With `ploidy == 0` and `an > 0`: `n = 0`, `q = ac/0 = ¬±inf/nan` ‚Äî **a live divide-by-zero producing silent NaN**, NOT guarded by `finalize_af` (the only guard is `an > 0`). The interface doc (lines 78‚Äì80) says "2 diploid ... 1 pseudo-haploid" but does not forbid 0/negative. Confirmed by reading `finalize_af` line by line. Fix: document and (debug-)assert `ploidy >= 1` (ideally `‚àà {1,2}`); add a guard in `finalize_af`. This is a real bug class, not cosmetic.
- `bytes_per_record == 0` with `n_snp > 0`: `packed_bytes = n_individuals ¬∑ 0 = 0` (cuda_backend.cu:304‚Äì305) ‚áí the device upload is zero-length, but the kernel still reads `packed[g¬∑0 + byte_in_rec]` = `packed[byte_in_rec]` from a 0-size buffer ‚áí **OOB read**. Fix: document `bytes_per_record >= ceil((snp_offset+n_snp)/4)`.

Severity med (a real NaN-producing divide-by-zero on `ploidy==0`, plus undocumented degenerate shapes). Effort S. Before-M4.5? No, but the `ploidy` guard is worth doing whenever `decode_af` is next touched.

**2.4 ‚Äî [N/A, verified positive] Empty-block / `Vpair==0` is correctly handled in the shared primitive (`finalize_f2`: `vpair > 0.0 ? num/vpair : 0`, f2_estimator.hpp:110‚Äì112) and `finalize_af` (`an>0` branch).** The header references both correctly (lines 46‚Äì47, 184‚Äì185). Retained for completeness ‚Äî not a defect.

**2.5 ‚Äî [med, ADDED ‚Äî broadens 2.1] The block-id range precondition must be `0 <= block_id[s] < n_block`, not just bounded above: a NEGATIVE `block_id[s]` is an OOB *write* `begin[negative]`/`out.block_sizes[negative]` in both backends' scans, and `block_of` itself can return a negative bin.** The auditor's 2.1 covered `block_id[s] >= n_block`; it did not cover negatives. Verified: both scans index `begin[static_cast<std::size_t>(b)]` (cpu_backend.cpp:220, cuda_backend.cu:143) ‚Äî a negative `b` casts to a huge `size_t` ‚áí heap corruption / UB. And `block_of` (block_partition_rule.hpp:56-58) returns `floor(genpos/bsize)`, which the file itself documents (lines 107-108) can be `-1` for negative positions; `assign_blocks` densifies these into non-negative ids, so the production path is safe ‚Äî but the seam takes a raw `block_id` and a hand-built or future-sharded `block_id` could carry a negative. Fix: the interface precondition should read `0 <= block_id[s] < n_block` for all s; the `block_ranges()` validator (2.1) must reject negatives. Severity med (OOB write, broader than 2.1's top-bound). Effort S (folds into 2.1's validator). Before-M4.5? Yes (with 2.1).

### (3) Numerical / precision vs ¬ß12

**3.1 ‚Äî [low, CONFIRMED] The precision contract is accurate but does not state that `EmulatedFp64` is NOT bit-identical to the `Fp64` oracle at the seam ‚Äî only "agree within the tight tolerance tier" (lines 132‚Äì134).** Verified: `CpuBackend` ignores `precision` (cpu_backend.cpp:99, 194 ‚Äî always long-double) and `config.hpp:135‚Äì139` is explicit that `EmulatedFp64` is "accuracy-approximate, NOT bit-identical." So "both backends agree" means *CPU-long-double* vs *GPU-EmulatedFp64{40}* at the tight tier (few ULP), not bit equality ‚Äî and the blocks test itself relies on this (`emu_engaged` check, test_f2_blocks_equivalence.cu:235-236, fails LOUDLY if emu == native bit-for-bit, i.e. emulation silently fell back). ## Considered & rejected

- **Original auditor 7.1's "tests hold a concrete `CudaBackend`/`CpuBackend` ‚áí the `[[nodiscard]]` call is unguarded."** **Rejected as factually wrong** (corrected into 7.1). Verified all call sites hold `std::unique_ptr<ComputeBackend>` (base static type) and call via `cpu->`/`gpu->`/`ComputeBackend&` ‚áí overload resolution selects the *base* declaration, which has `[[nodiscard]]` ‚áí the guard fires. Confirmed against [dcl.attr.nodiscard]: the attribute is consulted on the statically-selected reachable declaration, not inherited by overrides.

- **"Delete the move ctor/assign on `ComputeBackend` ‚Äî shouldn't a polymorphic base be movable?"** Rejected. A polymorphic base held by `unique_ptr` should NOT be value-movable (slicing). All-deleted copy/move + virtual `= default` dtor is the correct rule-of-five (¬ß9, 4.1).

- **"`MatView` passed by `const&` though it is a 16/24-byte POD ‚Äî pass by value?"** Rejected. `const&` vs by-value is a wash for a tiny POD; `const&` is idiomatic and matches the codebase. Not worth a note.

- **"`element()` has no bounds check (views.hpp:64‚Äì65) ‚Äî the backends read it in hot loops; UB."** The OOB risk is real but gated by the P/M precondition (2.1); the missing bounds check is a deliberate, documented hot-path choice in `views.hpp`, not a `backend.hpp` defect. Folded into 2.1.

- **"`std::vector<double>` for `vpair` wastes memory ‚Äî counts are integers."** Rejected. `fstats.hpp:51` documents "Integer-valued (carried as double)" deliberately: `vpair` is the S4 *weight* feeding FP64 arithmetic; FP64 avoids an int‚Üídouble conversion at every weighting step and keeps the seam type-uniform. By-design.

- **"Use `std::span`/`mdspan` for ALL array params including the `F2Result::f2` output vectors (¬ß7)."** Partially rejected. ¬ß7's view-not-owner rule is about *kernel* parameters and the device currency; *result* types must own their storage to outlive the call (the caller keeps them). So `std::vector` outputs are correct; only the *input* `block_id` raw pointer warrants the span treatment (9.2).

- **"Unify `F2Result` and `F2BlockTensor` return types."** Rejected as stated. `F2Result` is intentionally the lighter flat `[P√óP]` (no `block_sizes`); `F2BlockTensor` is the durable public artifact. The right fix is method consolidation (6.1), not return-type unification ‚Äî a single-block call legitimately needs no `block_sizes`.

- **"`int n_pop`/`int P`/`int n_block` should be unsigned `std::size_t`."** Rejected. Signed `int` for P/n_block matches `MatView::P`, `BlockPartition`, `F2BlockTensor` codebase-wide; the ranges (P ‚â§ ~4,266; n_block ~757 on whole-genome AADR) are far within `int`; and signed lets the `<= 0` guard be a clean early-return (unsigned would make it underflow-prone). Width is correct; only the *documented domain* is missing (2.2/2.3/2.5).

- **"`P¬≤¬∑n_block` flat element count overflows `int`."** Rejected. ## Good patterns to keep

- **The CUDA-free seam is real and verified** ‚Äî no CUDA leaks transitively (`f2_block_kernel.cuh`‚Üí`backend.hpp`, never the reverse; `decode_af_kernel.cuh` only references it in a comment); `MatView`/`F2BlockTensor`/plain structs cross the boundary; the ¬ß4 layering holds by construction (9.4). The load-bearing architectural win, correctly implemented.
- **Textbook polymorphic-base ownership** ‚Äî all copy/move deleted + virtual `= default` dtor, held by `unique_ptr` per ¬ß9 (4.1). Replicate for any future backend facet.
- **Exemplary documentation density** ‚Äî every field/method cites the arch ¬ß/ROADMAP ¬ß and the measured rationale (mantissa bits, Vpair-as-S4-weight, the compose-not-double-normalize caveat, the pseudo-haploid `N` convention, the TGENO packing). Matches `config.hpp`/`views.hpp`; exactly what TODO ¬ßC says to preserve.
- **Shared-formula discipline advertised at the seam** ‚Äî the header points callers at `core/internal/f2_estimator.hpp` / `decode_af.hpp` as the single source so oracle and GPU "cannot diverge" on the formula (lines 132‚Äì134, 176‚Äì178), reinforcing the ¬ß13 trust seam from the interface side. (The diagonal divergence in 1.2 is a *loop-bound* divergence, not a formula divergence ‚Äî the shared primitive is correctly shared; the bug is which `(i,j)` each backend's `compute_f2` iterates and whether the empty guard fires.)
- **`long M` / `int P` width discipline** carried faithfully from `MatView` (7.4) ‚Äî SNP axis overflow-safe, population axis not over-engineered; flat counts promoted to `size_t` before multiply (rejected-overflow item).
- **Vpair retained, not discarded** ‚Äî the interface makes the S4 jackknife weight a first-class output and documents the AT2 caveat, the correct anticipation of Phase 2 (where most reimplementations get it wrong).
