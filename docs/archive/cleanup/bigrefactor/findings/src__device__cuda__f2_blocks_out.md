Suggested: route the restore through the non-throwing `STEPPE_CUDA_WARN(cudaSetDevice(d))` (check.cuh:227) so a failed restore emits one diagnostic line instead of vanishing, matching the project's recoverable-status discipline.

## Group 15 — Memory: transfers

- [15.1][MED] src/device/cuda/f2_blocks_out.cu:78,93-96 — `read_block_to_host` is the FIT's per-block tile reader: each call does TWO synchronous `cudaMemcpy(...DeviceToHost)` (f2 slab + vpair slab) for ONE block `b`. The fit consumes the result block-by-block (n_block up to ~757), so the Resident arm's D2H is incurred once per block per call — exactly the "cudaMemcpy in a loop, should be kept-resident/batched" anti-pattern. MEMORY context flags this same seam as the multi-GPU speed wall (the f2_blocks D2H bounce) whose real fix is the fit reading f2_blocks resident from VRAM rather than D2H-bouncing per block. The per-block round-trip also serializes the launch/copy/sync for each tiny P²-slab. Suggested: provide a resident-consuming fit path (let the GPU fit read `resident.f2_device()`/`vpair_device()` in place, no D2H), or — if a host copy is unavoidable — expose a batched multi-block read that hoists the device-set + issues one larger D2H (or a strided/2D copy) instead of two-per-block.
- [15.3][MED] src/device/cuda/f2_blocks_out.cu:93-96 (and .cu:19) — the Resident-arm D2H copies into the caller-provided `double* f2_slab_out`/`vpair_slab_out` (.cu:78), which are plain PAGEABLE host buffers; the synchronous `cudaMemcpy(...DeviceToHost)` over pageable memory runs at roughly half the bandwidth of a pinned-buffer transfer. This matters precisely because of 15.1 (per-block, P²·8 bytes, potentially hundreds of blocks). The TU even includes `device/cuda/pinned_buffer.cuh` with the comment "RegisteredHostRegion (pin the D2H; graceful degrade)" (.cu:19) but never uses it — the perf intent was recognized and left unrealized (also filed as [3.3]/[8.2] for the dead include/comment). Suggested: stage the D2H through a pinned/registered host region (the included `RegisteredHostRegion`, graceful-degrade if pinning fails) before the memcpy into the caller buffer, or have the caller supply a pinned target; otherwise drop the include/comment if the perf path is abandoned.

## Group 22 — Performance: compute & launch

- [22.4][MED] src/device/cuda/f2_blocks_out.cu:78,88-96 — `read_block_to_host` is the FIT's per-block tile reader (called once per block, n_block up to ~757); EACH call re-issues the full per-call CUDA runtime sequence `cudaGetDevice(&prev)` (.cu:89) + `cudaSetDevice(resident.device_id)` (.cu:91) + the RAII-guard restore `cudaSetDevice(prev)` (.cu:90) AROUND the two synchronous `cudaMemcpy(...DeviceToHost)` (.cu:93-96). Across a multi-block fit loop the device save/set/restore is loop-invariant per the resident buffer's home device, yet it is paid block-by-block (3 runtime calls + 2 blocking copies × n_block), so the per-call/per-transfer overhead is amplified by the call count exactly the way 22.4 targets. This is the same seam already filed as [15.1][MED] (per-block D2H anti-pattern; the multi-GPU speed wall) — the 22.4 angle is that the FIX is a batched/resident read that issues the device-bind ONCE and one larger transfer (or a resident in-VRAM fit consume), collapsing the repeated per-block launch/copy/sync. Cross-references [15.1]/[15.3]; not re-counting the transfer-bandwidth/pageable aspects. Suggested: expose a multi-block (or whole-tensor) read that hoists the `cudaGetDevice`/`cudaSetDevice` pair out of the per-block path and issues one batched D2H, or let the GPU fit read `resident.f2_device()`/`vpair_device()` in place (no per-block D2H at all).
