## Group 13 — Error handling

- [13.3][HIGH] block_sink.cu:91-98 — HostRamSink::writer_loop checks `cudaEventSynchronize(s.done)` but on failure only emits a `STEPPE_LOG_WARN` (line 93) and then UNCONDITIONALLY falls through to the two `std::memcpy`s (lines 95-98) that copy the pinned slot into `host_.f2`/`host_.vpair`. If the per-slot D2H event sync failed, the slot may hold stale/partial bytes, so a swallowed error silently corrupts the f2/vpair tensor (a §12 parity violation that surfaces as a wrong statistic, not a thrown/observable failure). DiskSink::writer_loop (lines 246-249) treats the SAME `cudaEventSynchronize` failure as fatal — it throws, records `writer_failed_`, and re-throws at finish() — so the two tiers are inconsistent on the identical operation, and the wrong one (HostRam) is the silently-corrupting one. Suggested: make HostRam fail-fast like Disk — on a non-success event sync, record a writer-failed flag and re-throw at finish() instead of memcpy-ing an undrained slot; do NOT proceed with the copy.
- [13.1][MED] block_sink.cu:91-94 — when `cudaEventSynchronize(s.done)` fails in HostRamSink::writer_loop the nonzero status is logged but the resulting STICKY device error is never cleared (no `cudaGetLastError()` drain, unlike pinned_buffer.cuh:173 which deliberately clears a tolerated sticky error). Because this WARN-and-continue path keeps running, the next `STEPPE_CUDA_CHECK(cudaMemcpyAsync(...))` issued on the compute thread (spill_block, lines 115-119) can throw on a sticky error that actually originated in the writer's event sync — a misattributed fault. (Moot if 13.3 is fixed to fail-fast.) Suggested: if HostRam is kept tolerant, clear the sticky error after the failed sync (`(void)cudaGetLastError();`) so it is not later misattributed; better, fail-fast per 13.3.
- [13.3][LOW] block_sink.cu:91 vs 246 — the WARN-vs-throw inconsistency is itself the issue beyond the data-corruption angle: the same `cudaEventSynchronize(s.done)` call is guarded two different ways in the two sinks with no shared helper, so a future writer_loop edit can only fix/regress one tier. Suggested: route both writer drains through one shared event-wait helper with a single, fail-fast policy (folds naturally into the Group 7 StagingRing extraction).

## Group 14 — Memory: allocation & lifetime

- [14.4][MED] block_sink.cu:115-119 (HostRamSink) and 275-279 (DiskSink) — in `spill_block` the async D2H is ISSUED into the pinned slot (`cudaMemcpyAsync` lines 115/117 resp. 275/277) BEFORE the slot is enqueued to `ready_` (lines 122/282). The slot is enqueued only after BOTH `cudaMemcpyAsync`s and the `cudaEventRecord` succeed. If the second `cudaMemcpyAsync` or the `cudaEventRecord` (`STEPPE_CUDA_CHECK`, which THROWS a CudaError per check.cuh:154/214) fails, `spill_block` unwinds with the FIRST D2H still in flight on `stream` and the slot never pushed to `ready_`. The exception propagates to the orchestrator, which destroys the sink; the dtor (lines 137-151 / 328-344) only stops+joins the writer — and the writer only `cudaEventSynchronize`s slots that ARE in `ready_` (writer_loop lines 88-91 / 244-246), so this issued-but-not-enqueued slot is NEVER synchronized. The dtor then runs no `cudaStreamSynchronize(stream)`/`cudaDeviceSynchronize()` and lets `slots_` destruct, which calls `PinnedBuffer::reset()` -> `cudaFreeHost` (pinned_buffer.cuh:112) on the pinned destination while its D2H DMA may still be running = use-after-free of pinned host memory by an in-flight cross-stream transfer (UB). Suggested: wrap the issue sequence in a try/catch (or guard the dtor) that `cudaStreamSynchronize(stream)`s — or device-syncs — before the pinned ring is freed when a spill faults mid-issue, so no DMA outlives its pinned target.
- [14.4][LOW] block_sink.cu:137-151, 328-344 — neither destructor performs ANY stream/device synchronize before destroying the events and freeing the pinned ring; the sole completion guarantee is the writer thread's per-enqueued-slot `cudaEventSynchronize`. On the normal path that is sufficient (every issued D2H is also enqueued and thus drained before join). But it is a fragile invariant with no defensive backstop — the moment any code path issues a D2H without enqueueing (the 14.4 error path, or a future edit) the pinned free races the DMA. Suggested: add a one-line `cudaDeviceSynchronize()` (warn-not-throw) in each dtor before the event-destroy/pinned-free, as a cheap belt-and-suspenders barrier against an undrained in-flight transfer; this is teardown, not the hot path, so the sync cost is irrelevant.
- [14.1][LOW] block_sink.cu (whole TU) — no alloc/free family mismatch: the only owned device-runtime resources are the pinned staging buffers (`PinnedBuffer<double>`, cudaHostAlloc paired with cudaFreeHost via RAII, pinned_buffer.cuh:80/112) and the CUDA events (cudaEventCreateWithFlags at 59/214 paired with cudaEventDestroy at 145/337). The POSIX `fd_` (open/close) and `read_handle_` (fopen/fclose) are correctly paired with their own families. The device f2/vpair slabs are NOT owned here (the stream loop owns them; spill_block takes non-owning `const double*`), so the sink never frees device memory. No issue.
- [14.2][LOW] block_sink.cu:54-59, 209-214 — all pinned-host allocation happens ONCE in `begin()` (the persistent ring), never in the hot `spill_block` path; there is no plain `cudaMalloc`/`cudaFree` on any path. - 14.3 (async/sync free pairing): N/A — this unit uses no `cudaMallocAsync`/`cudaFreeAsync`. No issue.
- [14.5][LOW] block_sink.cu:181-220 (DiskSink::begin) — error-path frees are otherwise sound: if `cudaEventCreateWithFlags`/`PinnedBuffer` throws inside the slot loop (line 214/211-213) the already-opened `fd_` is reclaimed by the dtor (`if (fd_ >= 0) ::close(fd_)`, line 343) and any created events are destroyed by the dtor's slot loop (334-341); the writer thread is started only at the end of begin() so a mid-begin throw leaves it non-joinable. ## Group 16 — RAII: ownership & wrapper hygiene

- [16.1][MED] block_sink.cuh:69-74, block_sink.cu:59/214 (create) & 143-150/334-341 (destroy) — `SinkSlot` mixes two RAII-wrapped members (`PinnedBuffer<double> f2`/`vpair`, move-only with null-on-move) with a RAW owned `cudaEvent_t done`. The event IS a real owned CUDA resource (created via `cudaEventCreateWithFlags` in each `begin()`, freed via a hand-rolled `cudaEventDestroy` loop in each dtor), but unlike the pinned buffers it is NOT wrapped in a move-only RAII owner — its lifetime is managed manually by the sink, separately from the slot it lives in, in code copy-pasted into both destructors (the Group 7 7.1 dup). Per task 16.1 (wrap EVERY resource, events included — they leak/double-free identically to device memory), the event should have its own RAII wrapper. Suggested: add a small move-only `CudaEvent` wrapper (cudaEventCreate/cudaEventDestroy, null-on-move, dtor warns-not-throws, mirroring `PinnedBuffer`) and make `SinkSlot::done` that type; the two dtor teardown loops then vanish and the event lifetime is owned in one place.
- [16.2][MED] block_sink.cuh:69-74 — because `SinkSlot` has move-only `PinnedBuffer` members, the implicit copy ops are deleted and `SinkSlot` is implicitly MOVE-only; but the implicit move just member-wise moves, which for the raw scalar `cudaEvent_t done` is a COPY that does NOT null the moved-from. A moved-from `SinkSlot` would retain the same `cudaEvent_t` handle as the destination → if any code path ever moves a `SinkSlot` (e.g. a future `slots_` reallocation, an `erase`/`insert`, or moving a slot between rings), two slots own the one event and the dtor loop double-`cudaEventDestroy`s it (UB / double-free). Today `slots_` is `resize()`d once to exactly `kStreamStagingSlots` in `begin()` and never regrown (block_sink.cu:54,209), so no move occurs and this is latent, not live. Suggested: same fix as 16.1 — wrap the event so its move nulls the moved-from; that makes `SinkSlot`'s implicit move correct by construction (the raw-handle-survives-move hazard is exactly what the RAII wrapper exists to prevent).
- [16.3][LOW] block_sink.cuh:80-106 (HostRamSink) & 117-156 (DiskSink) — both sinks have a user-declared freeing destructor (block_sink.cu:137/328) — the rule-of-five trigger — yet neither EXPLICITLY declares/deletes the copy or move operations. They are non-copyable and non-movable only IMPLICITLY, via their `std::mutex mtx_` / `std::condition_variable` members (non-copyable, non-movable), which suppress all four special members. So the classes are SAFE today (a copy/move won't compile), but the freeing-resource ownership (the `cudaEvent_t`s, `fd_`, `read_handle_`, `std::thread`) is protected only by an incidental member rather than by intent. Per task 16.3, a class with a freeing destructor should make its rule-of-five posture explicit. Suggested: add `HostRamSink(const HostRamSink&) = delete; HostRamSink& operator=(const HostRamSink&) = delete;` (and the move pair = delete) to both sinks so the non-copy/non-move ownership is documented and robust against a future member change that would otherwise silently re-enable a wrong-shaped special member.
- 16.4 (single clear ownership; raw kernel pointers non-owning, never freed; no owning wrapper by value): OK. `spill_block`'s `const double* f2_dev`/`vpair_dev` (block_sink.cuh:59-60) are documented non-owning device views and are never freed here (the stream loop owns them, header lines 56-58). `DiskSink::take_descriptor` (block_sink.cu:318-326) MOVES `read_handle_` ownership to the descriptor and nulls the source (line 325), and the dtor's `if (read_handle_)` guard (line 342) prevents a double-`fclose` — a correct null-on-transfer. `fd_` is single-owned (open in begin, close in finish OR the dtor error path, guarded by `fd_ >= 0`, lines 309-310/343). No owning wrapper is passed by value. No issue.
- 16.5 (don't reinvent the wrapper): OK — the pinned staging correctly reuses the project's `PinnedBuffer<double>` (block_sink.cuh:32,69-71) rather than re-rolling a pinned-memory owner. The ONLY hand-rolled lifetime management is the `cudaEvent_t` (the 16.1 finding) and the POSIX `fd_`/`FILE*` (which have no project wrapper and are legitimately managed inline). No spurious reinvention. (See 16.1 for the event wrapper that is the one genuinely-missing RAII type.)

## Group 17 — RAII: lifetime & deleter pitfalls (CUDA-specific)

- [17.5][MED] block_sink.cu:42-64 (HostRamSink::begin) & 181-220 (DiskSink::begin) create the per-slot `cudaEvent_t`s (`cudaEventCreateWithFlags`, lines 59/214) and the `PinnedBuffer<double>` pinned slabs (lines 57-58/212-213) bound to whatever device is CURRENT at begin() time, but the sink RECORDS NO device id (no `cudaGetDevice` member; grep confirms zero `cudaSetDevice`/`cudaGetDevice` in the unit). The destructors (block_sink.cu:137-151 / 328-344) then `cudaEventDestroy(s.done)` (lines 145/336) on whatever device is current at teardown. Under the documented multi-GPU fan-out (block_sink.cuh §11.4 reference; 2x-GPU box per project notes), if the sink is created with device A current and the destructing thread has device B current (a different device set since), the `cudaEventDestroy` runs against the wrong device/context. (The paired `cudaFreeHost` in `PinnedBuffer::reset`, pinned_buffer.cuh:112, is device-agnostic — `cudaHostAlloc` is process-wide pinned host memory — so only the EVENT teardown is exposed, not the pinned free.) Task 17.5: a device-owned-resource deleter must restore the alloc device. Suggested: record `cudaGetDevice(&dev_)` in begin(), and in each dtor record-current / `cudaSetDevice(dev_)` / event-destroy / restore (warn-not-throw) so the event teardown targets its creating device regardless of the caller's current device.
- [17.4][MED] block_sink.cu:137-151 (HostRamSink dtor) & 328-344 (DiskSink dtor) — RAII frees-at-scope-exit do NOT guarantee async D2H work is finished: neither dtor issues any `cudaStreamSynchronize(stream)`/`cudaDeviceSynchronize()` before `slots_` destructs and `PinnedBuffer::reset` → `cudaFreeHost` (pinned_buffer.cuh:112) reclaims the pinned D2H destination. On the normal path every issued D2H is also enqueued and the writer's per-slot `cudaEventSynchronize` (lines 91/246) drains it before `writer_.join()`, so the pinned free is safe. But the only completion guarantee is the writer's per-ENQUEUED-slot sync; if `spill_block` faults mid-issue (the second `cudaMemcpyAsync`/`cudaEventRecord` throws via STEPPE_CUDA_CHECK, lines 115-119/275-279) the FIRST D2H is left in flight on `stream` and the slot is NEVER pushed to `ready_` (lines 122/282), so the writer never synchronizes it — the dtor then frees the pinned slab under the still-running DMA = use-after-free of pinned host memory (UB). This is the Group 17 lifetime-tie lens on the same hole captured by 14.4/14.5; the RAII fix is to TIE the pinned-ring lifetime to stream completion rather than rely on the enqueue invariant. Suggested: add a `cudaDeviceSynchronize()` (or `cudaStreamSynchronize` of the spill stream) — warn-not-throw — at the top of each dtor (and on the spill_block fault path) before the event-destroy/pinned-free, so no in-flight DMA outlives its pinned target; teardown cost is irrelevant off the hot path.
- 17.1 (non-throwing dtor / teardown-order): OK. Both dtors swallow CUDA teardown failures via the warn-not-throw sink — `cudaEventDestroy` is checked and only `STEPPE_LOG_WARN`'d (lines 145-148 / 336-339), and the pinned `cudaFreeHost` is likewise swallowed in `PinnedBuffer::reset` (pinned_buffer.cuh:113-116).
