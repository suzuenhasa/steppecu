## Group 20 — Performance: memory access

- [20.1][MED] f2_block_kernel.cu:112-133 — `f2_feeder_kernel` rides the SNP axis `s` on `threadIdx.x` (cu:112) and the population axis `i` on `threadIdx.y` (cu:113), but every array is column-major `[P × M]` with `idx = i + Pl*s` (cu:117) / `sidx = i + 2*Pl*s` (cu:118), so the CONTIGUOUS dimension is `i` (pop), not `s`. Adjacent lanes in a warp vary `threadIdx.x` (= `s`) at fixed `i`, so consecutive lanes hit `Q_raw[idx]`/`V_raw[idx]`/`N_raw[idx]` and write `Q_masked`/`V_out`/`S` at addresses STRIDED by `P` doubles (cu:120,124,128,130-133) — each lane lands in a separate 32-byte sector ⇒ fully uncoalesced loads AND stores. This is the bandwidth-bound feeder (cu:99-100 says so), the one kernel where coalescing actually costs throughput, and the codebase is elsewhere explicit about exactly this (launch_config.hpp:155-161: the decode kernel deliberately rides SNP on a 32-wide x edge "keeping the packed-byte reads contiguous/coalesced"; here the same SNP-on-x choice is anti-coalesced because the f2 matrices are pop-contiguous, not SNP-contiguous). The orientation is forced by the grid.y 65 535 cap fix (X-7/B6 — M must ride the only 2^31 axis, x), so the trade is real, not free. Suggested: don't reorient (the cap fix is load-bearing); instead either swap the [P×M] feeder buffers to row-major / [M×P]-equivalent so `i` runs along x, or stage a 16×16 tile through shared memory and transpose on store, so the global loads/stores coalesce while M keeps the x grid axis. MED — scale-dependent throughput on the hot bandwidth-bound pass, not a wrong result.
- [20.1][LOW] f2_block_kernel.cu:168,170 — In `assemble_f2_kernel` the four "own-orientation" reads `G[si+sj*Pp]` (cu:165), `Vpair[si+sj*Pp]` (cu:166), `R[si+sj*twoP]` (cu:167, sumsq_i) and `R[(Pp+si)+sj*twoP]` (cu:169, hsum_i) are coalesced (lanes vary `i` = `threadIdx.x`, the column-major stride-1 dimension), but the two TRANSPOSED reads `R[sj+si*twoP]` (cu:168, sumsq_j) and `R[(Pp+sj)+si*twoP]` (cu:170, hsum_j) index with `si` as the column factor, so consecutive lanes (varying `i`) stride by `twoP` doubles ⇒ uncoalesced. Suggested: leave as-is, or (only if assemble ever profiles hot) stage R through shared memory so both R(i,j) and R(j,i) come from one coalesced tile load; LOW — small fixed-size buffer, off the dominant cost.
