## Role & layering

`stream.hpp` defines two owning, move-only RAII wrappers in `namespace steppe::device`: `Stream` (wraps `cudaStream_t`) and `Event` (wraps `cudaEvent_t`). It is a CUDA header (includes `<cuda_runtime.h>` directly, and transitively via `check.cuh`) and is correctly self-documented as PRIVATE to `steppe_device`, never seen by `core`/`api`/CLI — satisfying the architecture.md §4 dependency-direction rule and §7 "anything touching the CUDA runtime stays `__host__`-only in the platform layer." The shape is exactly architecture.md §7's reference template (line 449–450: "Stream, Event, CusolverDnHandle, CurandGenerator, PinnedBuffer<T> all follow this exact shape: move-construct + move-assign + dtor→destroy()-with-debug-log. No exceptions.") and matches the `[BUILT] Stream / Event RAII` entry at §4 line 154.

**Crucial layering context — this file is dormant infrastructure.** A full-tree grep this pass (`grep -rn` over `*.cu|*.cuh|*.hpp|*.cpp`) confirms: `stream.hpp` is `#include`d by **nothing** (the only hits for `stream.hpp` are inside `stream.hpp` itself); `device::Stream` / `device::Event` are referenced **nowhere** (the `Stream`/`Event` hits in `handles.hpp`/`f2_block_kernel.cu`/`f2_blocks_kernel.cu` are `cublasSetStream(handle, stream)` on a raw `cudaStream_t`, not this type); the file is **not** a source in `add_library(steppe_device ...)` in `src/device/CMakeLists.txt` (it's a header, so it compiles only if a TU includes it — none does); and `tests/CMakeLists.txt` plus all four test files reference it **zero** times. `cuda_backend.cu` runs M0–M4 on a raw `cudaStream_t stream_ = nullptr` member (the legacy default stream) and synchronizes with bare `cudaStreamSynchronize(stream_)` (lines 101, 208, 271, 282, 330), with a comment (lines 334–338) explicitly deferring "a dedicated RAII `Stream`" to "the streaming pipeline (architecture.md §11.1)." TODO.md §74 confirms this is tracked: **"[high] Default-stream debt — `cuda_backend.cu` uses `cudaStream_t stream_ = nullptr` (RAII `Stream` exists, unused) ⇒ every `cudaMemcpyAsync` is host-synchronous + no lane to per-device-ify."** The §7/§11.1 consumers `pinned_buffer.hpp` and `allocator.cu` do **not yet exist** in the tree (verified: `find` returns nothing) — so the lanes this type is meant to serve are genuinely unbuilt.

So this file is *staged infrastructure*: correct to land early, but its real consumers — the M5 copy/compute double-buffer of §11.1, the M4.5 P2P-or-host combine of the capability table — do not exist yet. That makes **"is this the right shape for the consumer that will arrive"** the single most important review axis, and it is where the file is weakest: as written, the `Stream` half *cannot express the very lane separation its own usage comment describes* (Findings 1.1, 4.1). A header with zero callers also has **zero compilation coverage** (10.1): these classes have never been type-checked by any build.

This is a small, careful header. Per the mandate I will not pad, but I also will not stop early. The file is near-perfect *as a default-stream timer/ordering wrapper* and materially incomplete *as the §11.1 streaming primitive its own comments promise*. The findings are mostly med/low, with two high-severity items that are high precisely because they are **latent** — they surface as wrong-but-passing behavior the moment someone builds the overlap pipeline on top of this type.

## Score: 8/10 — correct, idiomatic, leak-free RAII matching the §7 template, but ships a misleading docstring about its own stream semantics, omits the non-blocking / priority knobs the §11.1 pipeline it exists for actually requires, lets `elapsed_ms` fail only at runtime, has zero compile coverage, and carries no capability-tier surface

Rationale: As an RAII wrapper it is genuinely good — move-only done right (move-assign present, the explicit fix for the "old draft" deleted-move-assign bug; self-assignment guarded), destructors `noexcept` and never silent, `[[nodiscard]]` on the value accessors, principled `const`-correctness, and the `cudaEventDisableTiming` default is the documented performance-optimal path for ordering (verified). It loses points for, in descending impact: (1) a `Stream` that can *only* create the implicitly-synchronizing default-flavour stream, so the concurrent copy/compute lanes of §11.1/§7 **cannot be built on it** (4.1) — and a docstring (1.1) that obscures this; (2) an `elapsed_ms` that compiles against a timing-disabled (or unrecorded, or cross-device, or non-NULL-stream) event and only throws/over-reports at runtime (1.2/1.4/1.6); (3) zero capability-tier surface at the exact M4.5/M5 seam where the tiers degrade (11.1); (4) zero compilation/test coverage (10.1). None are memory-safety bugs — hence not lower — but several are "before-M4.5" because M4.5/M5 are precisely when this type stops being dormant. I hold the score at **8/10**: the first-pass score was correct; verification did not move it, but it sharpened *why*.

## Findings

### (1) Correctness & bugs

**1.1 [HIGH] The `Stream` docstring ("non-blocking-capable stream") is misleading; `cudaStreamCreate` produces the implicitly-synchronizing default-flavour stream, not a non-blocking one.** `Stream`'s class comment (lines 48–50): *"A default-constructed Stream creates a fresh non-blocking-capable stream."* The constructor (line 54) calls `cudaStreamCreate(&s_)`. Verified against the official Runtime API ([CUDART Stream Management](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__STREAM.html)): the only documented way to get non-blocking behavior is `cudaStreamCreateWithFlags(&s, cudaStreamNonBlocking)`, where the flag *"Specifies that work running in the created stream may run concurrently with work in stream 0 (the NULL stream), and that the created stream should perform no implicit synchronization with stream 0"* (quoted verbatim, NVIDIA Stream Management). The CUDA-7-streams guidance is blunter: *"Before CUDA 7, the default stream is a special stream which implicitly synchronizes with all other streams on the device"* and *"You can create non-blocking streams which do not synchronize with the legacy default stream by passing the `cudaStreamNonBlocking` flag to `cudaStreamCreate()`"* ([GPU Pro Tip: CUDA 7 Streams](https://developer.nvidia.com/blog/gpu-pro-tip-cuda-7-streams-simplify-concurrency/), both verbatim). So a stream from `cudaStreamCreate` **implicitly synchronizes with the legacy default stream** — the opposite of "non-blocking." The phrase "non-blocking-**capable**" is weasel-worded — it is neither true (the stream is *not* non-blocking) nor verifiable. Why it matters: a reader building the §11.1 copy/compute overlap trusts this comment, expects the lanes to run concurrently, and instead gets serialization through stream 0's implicit sync points — the overlap silently never happens, the pipeline "works" (just slowly), and it ships. This is the canonical CUDA streams pitfall. Fix: subsumed by 4.1 — make the default non-blocking and correct the comment to "a non-blocking stream (no implicit sync with the default stream)." If 4.1 is deferred, at minimum correct the comment to "a blocking stream that implicitly synchronizes with the default (NULL) stream." Severity **high**, effort S, before-M4.5 **yes**.

**1.2 [MED] `elapsed_ms` on a default-constructed (timing-disabled) `Event` is a latent runtime throw, not a compile error.** `Event`'s default is `enable_timing = false` → `cudaEventDisableTiming` (lines 103–106). `elapsed_ms` (lines 143–147) calls `cudaEventElapsedTime`. Verified: *"If either event was created with the `cudaEventDisableTiming` flag, then this function will return `cudaErrorInvalidResourceHandle`"* ([CUDART Event Management](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__EVENT.html), verbatim). So `Event a, b; b.elapsed_ms(a);` compiles cleanly and throws `CudaError` at runtime, surfacing only the raw `cudaErrorInvalidResourceHandle` string — opaque to a caller who simply forgot the bool. **Precision sharpened this pass:** the same `cudaErrorInvalidResourceHandle` *also* fires if `cudaEventRecord()` was never called on either event ("If `cudaEventRecord()` has not been called on either event, then `cudaErrorInvalidResourceHandle` is returned" — verified). Note this is *distinct* from a constructed-but-unrecorded event under `cudaEventQuery`/`cudaEventSynchronize`, where the docs say *"Before the first call to `cudaEventRecord()`, an event represents an empty set of work, so for example `cudaEventQuery()` would return `cudaSuccess`"* (verbatim) — i.e. ordering/query ops treat an unrecorded valid event as a no-op, but `elapsed_ms` specifically rejects it. For a type whose whole job is to make misuse hard, gating timing/ordering on a runtime bool is the weak spot (§2 fail-fast, §13 testability want misuse caught at the type level). Fix, in preference order: (a) split into `OrderingEvent` / `TimingEvent` so `elapsed_ms` only exists where valid; (b) store the timing flag and throw a *descriptive typed* error ("elapsed_ms requires enable_timing=true") instead of the raw runtime handle error; minimally (c) replace the `bool` with an enum (see 7.5). Severity **med**, effort M (a) / S (b/c), before-M4.5 no (timing is a §11.3 profiling convenience, off the parity path).

**1.3 [LOW] The `record` non-`const` vs `wait`/`synchronize`/`elapsed_ms` `const` split is *principled* (correct) but undocumented.** `record` (line 128) is non-`const`; `wait` (134), `synchronize` (139), `elapsed_ms` (143) are `const`. This is right, not arbitrary: verified that *"cudaEventRecord() can be called multiple times on the same event and will overwrite the previously captured state"* ([Event Management](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__EVENT.html), verbatim) — recording mutates the event's observable state that a later `elapsed_ms`/`synchronize` depends on, so non-`const` is correct; `wait`/`synchronize`/`elapsed_ms` read that state and enqueue/block without changing it, so `const` is correct. The only reader-confusion risk is that `Event::wait(const Stream&)` and `Stream::synchronize() const` have *observable side effects on other objects / the device* while being `const`; a one-line comment ("`const` here means 'does not change this object's logical state'; ordering/blocking side effects on streams/device are expected") removes any surprise. Not a bug. Severity low, effort S, before-M4.5 no.

**1.4 [LOW] `elapsed_ms` has a runtime-only failure beyond timing-disabled: `cudaErrorNotReady` for recorded-but-incomplete events.** Verified: *"If `cudaEventRecord()` has been called on both events but one or both of them has not yet been completed... `cudaErrorNotReady` is returned"* ([Event Management](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__EVENT.html), verbatim). `elapsed_ms` does not `synchronize()` the end event first, so a caller timing a still-running stream gets a thrown `CudaError(cudaErrorNotReady)` rather than a wait-then-measure. This is *arguably correct* (the wrapper shouldn't hide a needed sync), but the docstring ("Milliseconds elapsed from `start` to this event") implies it just returns a number; it should say the end event must have completed (call `synchronize()` first) or the call throws `cudaErrorNotReady`. Folds into the 1.2 fix (the typed/documented contract). Severity low, effort S, before-M4.5 no.

**1.5 [LOW] `elapsed_ms` returns `float` (≈0.5 µs resolution) and is sign-sensitive to argument order.** Verified: *"Computes the elapsed time between two events (in milliseconds with a resolution of around 0.5 microseconds)"* ([Event Management](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__EVENT.html), verbatim). `cudaEventElapsedTime(&ms, start, end)` (line 145) returns time *from `start` to `end`*; if a caller passes them reversed (`start.elapsed_ms(end)` reading as "from end to start") the result is **negative**, with no guard. The member form `end.elapsed_ms(start)` reads correctly, but nothing enforces it. Combined with the `float` ~0.5 µs granularity (fine for the §11.3 ~0.3 s GEMM, useless for sub-µs kernels), the docstring should note both (resolution + which operand is the start). Minor. Severity low, effort S, before-M4.5 no.

**1.6 [LOW — added this pass] `elapsed_ms` can over-report when either event was recorded on a non-NULL stream — an asynchronous-record caveat the docstring omits.** Verified, NVIDIA Event Management (verbatim): *"If either event was last recorded in a non-NULL stream, the resulting time may be greater than expected (even if both used the same stream handle). This happens because the `cudaEventRecord()` operation takes place asynchronously and there is no guarantee that the measured latency is actually just between the two events."* This is directly load-bearing for §11.3 profiling: the moment the M5 pipeline times a GEMM on the *dedicated compute stream* (not the NULL stream), `elapsed_ms` may include queueing latency, not just the GEMM. The current docstring ("Milliseconds elapsed from `start` to this event ... §11.3 profiling") gives no hint of this; a one-line note ("on a non-NULL stream the measured interval may exceed the true device interval because record is asynchronous") prevents a confusing over-reported timing in the exact profiling path this type exists to serve. Folds into the 1.2/1.4 docstring contract. Severity low, effort S, before-M4.5 no.

### (2) Edge cases & failure modes

**2.1 [MED] Operations on a moved-from (null-handle) object are silently UB for `Stream` but throwing for `Event` — and inconsistently so.** After `Stream b = std::move(a);`, `a.s_ == nullptr`. Calling `a.synchronize()` runs `cudaStreamSynchronize(nullptr)` — which is **not** an error: `0`/`NULL` is the legacy default stream, so the runtime call accepts it and returns `cudaSuccess`, **silently synchronizing the wrong stream** instead of failing. By contrast, a moved-from `Event`'s `record`/`wait`/`synchronize`/`elapsed_ms` pass a `nullptr` `cudaEvent_t` — an *invalid handle* (distinct from a constructed-but-unrecorded event, which the docs treat as an "empty set of work"; see 1.2), so it returns `cudaErrorInvalidResourceHandle` and *throws*. So moved-from misuse is inconsistent: silently-wrong for `Stream`, throwing for `Event`. The destructors correctly guard `if (s_)` / `if (e_)` (lines 81, 153), so teardown of a moved-from object is safe; only the *operations* are unguarded. Why it matters: §2 fail-fast + §13 testability — a moved-from `Stream` quietly synchronizing stream 0 is a brutal debugging session, and it directly undermines the §12 single-stream reasoning (you think you synced the statistic stream; you synced the NULL stream). Fix: document the standard "moved-from is valid-but-unspecified; only safe to destroy or reassign" contract in the class comment, and add a debug-only `assert(s_ && "operation on moved-from Stream")` / `assert(e_)` in each operation — consistent with the debug teardown warning already present, and free in release. Severity **med**, effort S, before-M4.5 no.

**2.2 [LOW] Future flag/priority extension on `Stream` must avoid an `int`/`unsigned` ambiguity with the default ctor.** `Stream()` takes no args. The obvious future signatures (`Stream(unsigned flags)` / `Stream(int priority)`, see 4.1/4.2) risk `Stream s(0)` ambiguity. Flagged now so the eventual API is designed with **named enums** (`StreamFlags::NonBlocking`) or **named factories** (`make_copy_stream()`), not a bare `unsigned`/`int` overload set. Severity low, effort S, before-M4.5 **yes** (design before the first knob lands).

**2.3 [LOW] Shutdown-order teardown (`cudaErrorCudartUnloading` / `cudaErrorContextIsDestroyed`) is *safe* but, in release, fully silent and unclassified.** If a `Stream`/`Event` outlives the CUDA context (static-duration wrapper, or destruction after `cudaDeviceReset`), `cudaStreamDestroy`/`cudaEventDestroy` may return `cudaErrorCudartUnloading`. The destructor correctly does not throw and routes to the debug warning — so this is *safe* — but in release the macro is `((void)0)` (fully silent) and the debug text won't distinguish "real leak" from "benign shutdown ordering." Acceptable for now and matches the siblings; the eventual `log.hpp` wiring (6.1/10.3) should classify these. Severity low, effort S, before-M4.5 no.

### (3) Numerical / precision vs §12

**3.1 [LOW] No FP arithmetic here, but this type is the natural enforcement point for the §12 single-stream bit-stability invariant, and it offers nothing to express it.** There is no arithmetic in this file, so cancellation/accumulation-order concerns don't apply directly. But §12/§7 state the constraint verbatim (architecture.md line 489): *"the statistic-bearing GEMM/SYRK/SVD/Cholesky must run on a single stream when bit-stable goldens are required, because cuBLAS reproducibility does not hold across concurrent streams (§12). Independent throughput-only work may use multiple streams."* `build()` (§9, line 620) enforces this at config time (rejects *"`deterministic == true` with `stream_count > 1` on the statistic path"*; `config.hpp` line 179 hardcodes `stream_count = 1`). Today nothing in `Stream` distinguishes the **statistic/parity stream** (which §12 forbids running concurrently with the matmuls) from a **throughput stream**. When the §11.1 pipeline lands and starts handing `Stream`s to cuBLAS, the single-stream invariant will be upheld by convention in `cuda_backend.cu`, not by the type. Suggestion (design-time): a lightweight `Stream::Kind { Statistic, Throughput }` tag or a `make_statistic_stream()` factory makes the §12 constraint expressible and CI-greppable rather than a comment. Not actionable until the consumer exists; recorded so the §12 boundary isn't lost. Severity low, effort M, before-M4.5 no (lands with multi-stream).

### (4) CUDA idioms / RAII / stream & async semantics / launch config / occupancy vs §7

**4.1 [HIGH] No non-blocking-stream knob — the type cannot create the concurrent lanes its own usage comment (and §7/§11.1) require.** Header comment lines 14–18: *"one Stream per independent lane … separate copy / search lanes for throughput; express cross-stream dependencies with an Event."* §7 line 489: *"One `Stream` per independent lane."* TODO §106 (M5): *"replace `geno_reader`'s pageable `ifstream` ... with large sequential reads into `PinnedBuffer` ring slots + `cudaMemcpyAsync` on a dedicated copy stream, overlapped with the ~0.3 s GEMM via `Event`; end-to-end → max(read, gpu) not their sum."* For those lanes to actually overlap they must be **non-blocking** (`cudaStreamNonBlocking`); streams created by `cudaStreamCreate` each implicitly synchronize with the NULL stream and therefore serialize against each other through stream 0 (verified, [Stream Management](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__STREAM.html)), defeating overlap. As written, `Stream` exposes *only* the blocking constructor, so **the §11.1 streaming pipeline literally cannot be built correctly on top of this type without first modifying it.** Fix: make the default non-blocking — `explicit Stream(unsigned flags = cudaStreamNonBlocking) { STEPPE_CUDA_CHECK(cudaStreamCreateWithFlags(&s_, flags)); }` — or provide named factories. (Note: if the *statistic* path is deliberately the legacy default stream `nullptr` for §12 single-stream parity — which is what `cuda_backend.cu` does today — then non-blocking only matters for the throughput/copy lanes; but those are exactly the lanes `Stream` is supposed to represent.) Severity **high**, effort S, before-M4.5 **yes** — this is the blocker for the very milestone the file was created for, and the prerequisite for the 11.1 capability surface.

**4.2 [MED] No stream-priority support — the §11.1 copy/compute split is the textbook use for `cudaStreamCreateWithPriority`.** Verified (verbatim, [Stream Management](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__STREAM.html)): *"Lower numbers represent higher priorities"*; *"Priorities provide a hint to preferentially run work with higher priority when possible, but do not preempt already-running work"*; *"If the specified priority is outside the numerical range returned by `cudaDeviceGetStreamPriorityRange`, it will automatically be clamped to the lowest or the highest number in the range."* Prioritizing compute over copy (or parity over throughput) is the standard overlap-tuning lever; `Stream` offers no hook. Because priorities are only a hint, this is a perf item, not a correctness blocker. Fix: an optional priority parameter or a `make_stream(priority)` factory that queries the range and clamps. Severity med, effort M, before-M4.5 no (a §11.3 tuning lever, lands when overlap is measured).

**4.3 [MED] No per-thread-default-stream (`cudaStreamPerThread`) awareness in the contract.** Verified (verbatim, [GPU Pro Tip: CUDA 7 Streams](https://developer.nvidia.com/blog/gpu-pro-tip-cuda-7-streams-simplify-concurrency/)): per-thread default stream is enabled by *"the `nvcc` command-line option `--default-stream per-thread`, or `#define` the `CUDA_API_PER_THREAD_DEFAULT_STREAM` preprocessor macro before including CUDA headers"*, and *"With per-thread default streams, the default stream in each thread behaves the same as a regular stream, as far as synchronization and concurrency goes. This is not true with the legacy default stream."* The choice changes whether `nullptr` (the backend's `stream_`, and a moved-from `Stream`, 2.1) means the *legacy* shared NULL stream (synchronizes with all streams) or a *per-thread* regular stream. The §12 single-stream determinism argument and the §11.4 SPMG "one host thread + per-device streams per device" model (TODO §21) both depend on which semantics are in force. `stream.hpp` need not *handle* this, but the header should note that default-stream semantics are a build-flag-dependent global the wrapper does not control — otherwise the §12 reasoning ("statistic path is the default stream") is silently build-flag-dependent. Severity med (a contract/documentation gap that could undermine §12), effort S, before-M4.5 no.

**4.4 [LOW] `Event::wait` hardcodes `0` for the `cudaStreamWaitEvent` flags argument.** Line 135: `cudaStreamWaitEvent(stream.get(), e_, 0)`. The third arg is a flags field; `0` is `cudaEventWaitDefault` and is correct for ordinary events (verified — the default flag; `cudaEventWaitExternal` is the only other value, for stream-capture/graphs, see 4.7). But per ROADMAP §4 ("no literal may survive except true mathematical constants"), a bare `0` for an enum-flags argument is exactly the kind of literal the project replaces with the named constant elsewhere. Use `cudaEventWaitDefault` for self-documentation. Severity low, effort S, before-M4.5 no.

**4.5 [N/A] Post-launch `cudaGetLastError` check is correctly *not* applied here.** `check.cuh` provides `STEPPE_CUDA_CHECK_KERNEL()` for `<<<>>>` launches; the event/stream ops here are plain runtime calls already wrapped in `STEPPE_CUDA_CHECK`, which is correct (no kernel launches in this file). N/A, not a finding — listed so the §7 post-launch-check checklist is visibly covered.

**4.6 [POSITIVE] RAII shape matches §7 verbatim.** Move-construct + move-assign both present (56–64, 109–117); copy deleted (66–67, 119–120); self-assignment guarded (`if (this != &o)`); `destroy()` idempotent and null-safe; destructors `noexcept` via `destroy() noexcept`. Verified that *"In case the device is still doing work in the stream `stream` when `cudaStreamDestroy()` is called, the function will return immediately and the resources associated with `stream` will be released automatically once the device has completed all work in `stream`"* ([Stream Management](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__STREAM.html), verbatim) and *"An event may be destroyed before it is complete... the call does not block on completion of the event, and any associated resources will automatically be released asynchronously at completion"* ([Event Management](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__EVENT.html), verbatim) — so destroying an in-flight stream/event is well-defined and the wrapper correctly does **not** pre-synchronize. Recorded as a positive (see "Good patterns").

**4.7 [LOW — added this pass] No `cudaEventWaitExternal` / stream-capture awareness — a forward gap for the §7 CUDA-graph fit pipeline.** §7 (line 489) commits the project to capturing the per-model fit into a CUDA graph: *"capture the per-model fit into a CUDA graph (`cudaStreamBeginCapture` → `cudaGraphInstantiate` → `cudaGraphLaunch`)."* During stream capture, the `cudaStreamWaitEvent` flags argument is exactly where `cudaEventWaitExternal` matters — verified that the flag *"captures the event in the graph as an external event node when performing stream capture"* (NVIDIA stream/event docs). `Event::wait` hardcodes `0` (4.4) and has no way to request external-wait semantics, so when the fit engine wraps these ops in `cudaStreamBeginCapture`, a cross-graph event dependency cannot be expressed through this type. Not actionable until the GraphPipeline lands, but the `wait(stream, flags = cudaEventWaitDefault)` signature (folding into 4.4's named-constant fix) keeps the door open instead of baking `0` in. Severity low, effort S, before-M4.5 no (lands with the graph fit engine, post-M5).

**4.8 [LOW — added this pass] The cross-device `record`/`elapsed_ms` contract should be documented: record requires same context, `elapsed_ms` requires same device, `wait` works cross-device.** Verified three distinct same-scope rules that this type silently inherits and must respect when §11.4 multi-GPU lands: (a) `cudaEventRecord` — *"`event` and `stream` must be on the same CUDA context"* (verbatim); (b) `cudaEventElapsedTime` — *"Event and stream must be on the same device"* (verbatim; cross-device `elapsed_ms` is invalid); (c) `cudaStreamWaitEvent` — *"will succeed even if the input stream and input event are associated to different devices ... can therefore be used to synchronize multiple devices with each other"* (verbatim, [Multi-GPU Systems](https://docs.nvidia.com/cuda/cuda-programming-guide/03-advanced/multi-gpu-systems.html)). So `Event::wait` is *deliberately* the cross-GPU primitive (this validates the device-binding rejection below), but `Event::record(stream)` requires the event and stream share a context, and `Event::elapsed_ms` requires same device — neither is documented in the header, and both are exactly the §11.4 SPMG hazard. A one-line note on each method ("record: event & stream must share a context; elapsed_ms: both events same device; wait: may be cross-device") makes the multi-GPU contract explicit at the type. Severity low, effort S, before-M4.5 no (documentation, but lands the §11.4 contract).

### (5) Magic numbers & hardcoded values vs ROADMAP §4

**5.1 [LOW] The only literals are `0.0f` (elapsed_ms init, line 144) and `0` (wait flags, line 135).** The `0.0f` is a defensive initializer that `cudaEventElapsedTime` overwrites on success and is never read on the throw path — harmless; could be `float ms;` (the call writes it on success, throws otherwise), but keeping the explicit init is also defensible. The `0` flags is 4.4/4.7. No sample-size / precision / launch-geometry magic numbers exist here — those §4 inventory targets (`kBlockGroupPadBase`, `kCublasWorkspaceBytes`, the `0.80` VRAM fraction → `kMaxVramUtilizationFraction` per TODO §83) live in `config.hpp` / the kernel TUs, not here. Severity low, effort S, before-M4.5 no.

**5.2 [LOW] The `"[steppe][warn] %s at teardown: %s\n"` format string is duplicated across three headers** — see DRY (6.1). The literal is fine; the duplication is the issue.

### (6) Decomposition / single-responsibility / function size vs §2

**6.1 [MED] DRY: the debug-only teardown-warning macro is copy-pasted into three sibling headers with three names and three signatures.** `stream.hpp` → `STEPPE_STREAM_WARN_ON_TEARDOWN(what, errstr)` (38–44); `handles.hpp` → `STEPPE_HANDLES_WARN_ON_TEARDOWN(what, statusname)` (29–35); `device_buffer.cuh` → `STEPPE_DEVICE_BUFFER_WARN_ON_TEARDOWN(errstr)` (33–40). All three: `#if defined(NDEBUG)` → no-op, else `#include <cstdio>` + `std::fprintf(stderr, "[steppe][warn] …")`. Three copies of one construct, each with its own `#include <cstdio>` and `#if`. ROADMAP §5 ("DRY helpers — one `CUDA_CHECK`, one loader …") and architecture.md §2 (single-source) both argue for one home; TODO.md §92 already lists "**triplicated teardown-warning macros → one shared**." Every one of these comments admits the real design is a single `STEPPE_LOG_WARN` from `internal/log.hpp` "once that facade is wired into steppe_device." Fix: hoist a single `STEPPE_DEVICE_WARN_ON_TEARDOWN(what, detail)` into `check.cuh` (the existing single CUDA-error home, already included by all three) or a tiny `device/cuda/teardown_warn.hpp`; then the eventual `log.hpp` swap is a one-line change in one place. Severity med, effort S, before-M4.5 no (cleanup, but exactly the "single-home" discipline the project grades on, and already on the TODO).

**6.2 [POSITIVE] Function sizes and single-responsibility are otherwise correct.** Both classes are tiny; each method does one thing; `destroy()` is the single shared per-class teardown helper. No decomposition concerns.

**6.3 [LOW] Macro-hygiene gap: `STEPPE_STREAM_WARN_ON_TEARDOWN` expands to a bare statement, not a `do { … } while (0)`, unlike the project's own `STEPPE_CUDA_CHECK_KERNEL`.** Compare `check.cuh` lines 135–146, where the multi-statement macro is correctly wrapped in `do { … } while (0)`. The teardown macro is single-statement so it is *currently* safe (always invoked inside `{ }` braces, lines 84/156), but a future `if (cond) STEPPE_STREAM_WARN_ON_TEARDOWN(...); else …` would silently misparse, and the no-op `((void)0)` branch already relies on caller braces. Also, none of the three macros are `#undef`'d, so they leak into every TU that includes the header (currently none, but they will once a consumer lands). When de-duplicated (6.1), give the single macro `do{}while(0)` wrapping (or — strictly better — make it an `inline` function in `check.cuh`'s `detail` namespace: no macro, real type-checking, testable). Severity low, effort S, before-M4.5 no (folds into 6.1).

### (7) Readability, naming, const-correctness, `[[nodiscard]]`/noexcept, comment density

**7.1 [POSITIVE] `[[nodiscard]]` correctly applied to the value accessors (`get`, `elapsed_ms`) and correctly absent from the `void` ordering/blocking ops.** Good.

**7.2 [POSITIVE] `noexcept` discipline is correct.** `synchronize`/`record`/`wait`/`elapsed_ms` are correctly **not** `noexcept` (they call `STEPPE_CUDA_CHECK`, which throws — matches §7); `get()` is `noexcept` (returns a handle); `destroy()` and the destructors are `noexcept` (teardown must not throw). Recorded to show the axis was checked.

**7.3 [LOW] Local-name inconsistency: `Stream::destroy` uses `const cudaError_t e` (line 82) while `Event::destroy` uses `const cudaError_t err` (line 154).** Cosmetic; pick one. The member naming `s_`/`e_` is terse but consistent with the siblings (`h_`, `ptr_`) — acceptable house style. Severity low, effort S, before-M4.5 no.

**7.4 [LOW] Comment density is high and accurate — except 1.1 (factually misleading) and a third copy of the "old draft deleted move-assign" backstory.** The header comment (lines 10–13) re-litigates the deleted-move-assign bug already told in `device_buffer.cuh`, `handles.hpp`, and architecture.md §7 (lines 387, 432). Good context, but the third copy — point to architecture.md §7 instead. The "non-blocking-capable" line (1.1) is the one that *actively misleads* and must be fixed. Severity low (for the duplication), effort S, before-M4.5 no.

**7.5 [LOW] `Event`'s `enable_timing` is a bare `bool` — `Event(true)` at a call site is self-undocumenting.** The classic boolean-parameter smell. An enum (`EventTiming::Enabled/Disabled`) or the two-type split (1.2) removes it. (`explicit` correctly prevents `Event e = true;` — good — but doesn't help readability at the call site.) Severity low, effort S, before-M4.5 no.

**7.6 [LOW] `Stream` and `Event` are not `final`.** They are leaf owning types with public **non-virtual** destructors; deriving and deleting through a base pointer would slice / skip the derived destructor. Idiomatic for these wrappers is `final` (no intended subclassing). The siblings (`CublasHandle`, `DeviceBuffer`) are also not `final`, so this is consistent house style, not a regression — but marking the family `final` is a cheap correctness-of-intent improvement and forecloses the slicing footgun. Severity low, effort S, before-M4.5 no.

**7.7 [LOW] No `[[nodiscard]]` on the types themselves.** `[[nodiscard]] class Stream { … };` would make `Stream{};` (constructing and immediately discarding a stream — almost certainly a bug, leaking the just-created stream at end of full-expression) a warning. Minor, but consistent with the project's `[[nodiscard]]`-everywhere posture and free. Severity low, effort S, before-M4.5 no.

### (8) Performance

**8.1 [POSITIVE] `cudaEventDisableTiming` default is the documented performance-optimal choice for ordering.** Verified verbatim: *"Events created with this flag specified and the `cudaEventBlockingSync` flag not specified will provide the best performance when used with `cudaStreamWaitEvent()` and `cudaEventQuery()`"* ([Event Management](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__EVENT.html)). The default is correct and fast.

**8.2 [LOW] `synchronize()` host-blocks by busy-wait; there is no `cudaEventBlockingSync` option to yield the CPU.** Verified verbatim: with the flag, *"A host thread that uses `cudaEventSynchronize()` to wait on an event created with this flag will block until the event actually completes"*; without it, *"the CPU thread will busy-wait until the event has been completed by the device"* ([Event Management](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__EVENT.html)). For the §11.1 pipeline where the TODO ⚡ section wants the host to be a "sequencer + metadata janitor — never a data mover or a per-SNP reducer," and **especially on the budget-5090 box where CPU is the scarce resource** (TODO §102/§135: the design must stay fast on the cheap vast box), a spinning sync burns a core that should be sequencing the next tile. Exposing `cudaEventBlockingSync` (or a blocking-sync `Event` variant) is a real lever there. Minor for default-stream M0–M4. Severity low, effort S, before-M4.5 no (lands with M5 ingest).

**8.3 [POSITIVE/N/A] No wrapper-overhead concern.** Thin one-line inlines; the only "cost" is the intended implicit-sync serialization of 1.1/4.1, which is a correctness/throughput problem, not wrapper overhead.

### (9) Layering / API / ABI vs §4

**9.1 [POSITIVE] Layering correct and well-justified.** CUDA header (includes `<cuda_runtime.h>`), correctly documented PRIVATE to `steppe_device`, in `namespace steppe::device`, never crossing into `core`/`api`/CLI (verified: `src/device/CMakeLists.txt` lines 36–38 keep `CUDA::cudart`/`CUDA::cublas` PRIVATE; `steppe::api`/`steppe::core_internal` PUBLIC are CUDA-free). The `.hpp` (not `.cuh`) extension is explained (matches architecture.md §4's listed filename, line 154 "[BUILT] Stream / Event RAII") and the comment flags it is nonetheless a CUDA header — the correct disclosure. Header-only inline classes ⇒ no ABI surface.

**9.2 [POSITIVE] `Stream::get()` / `Event::get()` leak raw handles — acceptable and necessary.** cuBLAS / kernel launches need the raw `cudaStream_t`/`cudaEvent_t`; they're non-owning `const` accessors. This is the §7 "owning types hand out views" idiom adapted to opaque CUDA handles (which can't be `span`/`mdspan`). Every `launch_*`/`run_*` wrapper (`f2_block_kernel.cuh`) and `CublasHandle{cudaStream_t}` (`handles.hpp` line 44) takes exactly `cudaStream_t`, so `Stream::get()` is a drop-in once 4.1/11.1 are closed.

### (10) Testability vs §13

**10.1 [MED] Zero compilation coverage and zero unit test — these classes have never been type-checked by any build.** Verified: `tests/` has `unit/` (`test_filters.cpp`, `test_block_partition.cpp`, `test_f2.cpp`) + `reference/` (four `.cu` equivalence tests), and `tests/CMakeLists.txt` references `Stream`/`Event`/`stream` **zero** times; no TU `#include`s `stream.hpp`, so the classes are not even instantiated/compiled in CI — a syntax or `noexcept` regression would pass green. ROADMAP §6 definition-of-done (line 131) requires *"compute-sanitizer (memcheck+racecheck) clean"* and §135 *"test → commit between successes"* — every shipped unit is implicitly exercised. A tiny GTest is cheap and would catch the real bugs above: (a) construct/destroy/move round-trip clean under compute-sanitizer (memcheck leak-check); (b) `record → wait → synchronize` across two streams actually orders work (a kernel writing a sentinel the second stream reads); (c) **`elapsed_ms` on a timing-disabled event throws** — locks in the 1.2 contract whichever way it resolves; (d) move-from then destroy is safe; (e) `elapsed_ms` on a timing-enabled, synchronized pair returns a finite, non-negative float. Without (c), the timing/ordering footgun stays unguarded. Severity **med**, effort M, before-M4.5 no (but should land with the first real consumer; even a "does it compile" test would close the zero-coverage gap immediately).

**10.2 [LOW] The debug teardown warning is not observable/testable.** It's a `std::fprintf(stderr, …)` compiled out in release and not routed through an injectable sink, so a test can't assert "a teardown failure was reported." The eventual `STEPPE_LOG_WARN` facade (referenced in the macro comment) should be injectable for exactly this. Same gap as the siblings; folds into 6.1. Severity low, effort M, before-M4.5 no.

**10.3 [LOW] The `std::fprintf(stderr, …)` teardown warning is a (knowingly-temporary) violation of the §10 "never `printf`/`std::cout` in library code" rule.** Architecture.md §10: *"Never `printf`/`std::cout` in library code. `internal/log.hpp` wraps spdlog behind `STEPPE_LOG_*`."* The header's own comment (lines 33–37) acknowledges this is a stopgap until `log.hpp` is wired. It's *debug-only* and *acknowledged*, so it is not a silent violation — but it is the concrete reason 6.1's "swap to `STEPPE_LOG_WARN`" must actually happen before this code goes hot, and CI's anti-`printf` grep gate (if/when added) will trip on it across all three headers. Recorded so it isn't lost when 6.1 is done. Severity low, effort S, before-M4.5 no.

### (11) Capability tiers (PRO-6000 vs budget-5090)

**11.1 [HIGH-for-M4.5] `Stream`/`Event` are the exact primitives the capability-tier table is built on, yet they carry no probe, no tag, and no degrade hook.** TODO.md's capability table (§125–137) makes both types load-bearing for two tiers:
- **M4.5 combine:** capable = *"P2P device-resident: GPU0 pulls peer partials via `cudaMemcpyPeer` (byte-exact DMA), sums fixed `g=0..G-1` on-device"*; budget fallback = host-staged fixed-order combine, with the explicit logged reason *"P2P combine unavailable (no peer access) → host-staged fixed-order combine."* The decision is gated on `cudaDeviceCanAccessPeer` (verified: *"Peer-to-peer memory access is supported between two devices if `cudaDeviceCanAccessPeer()` returns true"* — [Multi-GPU Systems](https://docs.nvidia.com/cuda/cuda-programming-guide/03-advanced/multi-gpu-systems.html)), and the cross-GPU ordering uses `Event`s. **Verified verbatim this pass:** `cudaStreamWaitEvent` *"will succeed even if the input stream and input event are associated to different devices ... can therefore be used to synchronize multiple devices with each other"* — so `Event` *is* the correct cross-GPU ordering primitive, and it correctly needs **no** device-id member to serve `wait` cross-device (this validates the "Considered & rejected" device-binding item; the same-device/same-context caveats for `elapsed_ms`/`record` are captured in 4.8).
- **M5 ingest:** the pinned double-buffer overlap that *is* the CPU-insensitive spine runs on *"a dedicated copy stream ... overlapped with the ~0.3 s GEMM via `Event`"* (TODO §106).

The cross-cutting requirement is explicit (TODO §137): *"a capability probe + capability-tagged results (every run records which path it took + why it degraded), slotting into `DeviceConfig`/`Resources`."* Today `stream.hpp` has zero surface for this: no way to ask whether a non-blocking/prioritized stream was granted, no notion of a copy vs compute (vs statistic) lane, no role a `Resources`/`DeviceConfig` could log in the degrade line. The fix is **not** to put probing logic in this header (that belongs in `DeviceConfig`/a capability module) — it is to (a) expose the constructors (non-blocking/priority, 4.1/4.2) the tiers need, and (b) design `Stream` so the orchestration layer can tag a lane's role for the logged degrade line (the §11.4 SPMG path needs `StreamPool` = "1 statistic stream + N search streams" per §9 line 606, which this single class must be able to populate). **Without 4.1 especially, the budget-5090 fallback — pinned double-buffer overlap on a dedicated copy stream — cannot be expressed at all.** Severity **high relative to M4.5/M5**, effort M, before-M4.5 **yes** (the non-blocking ctor is the prerequisite; tagging can follow).

**11.2 [POSITIVE, with the 3.1 caveat] Parity neutrality holds.** Nothing here changes a reported number — it's data-movement/observability only, exactly the "parity-neutral lever" class the TODO defines (§125: *"every lever is parity-neutral: data-movement/observability only"*). The §12 single-stream caveat (3.1) is the one place a *misuse* of this type could break parity, and that is a usage-site invariant (`build()` enforces `stream_count==1`), not a property of the wrapper.

## Considered & rejected

- **"Destructor should `cudaStreamSynchronize`/`cudaEventSynchronize` before destroy."** Rejected (re-verified verbatim): `cudaStreamDestroy` *"will return immediately and the resources ... released automatically once the device has completed all work in `stream`"*; `cudaEventDestroy` — *"the call does not block on completion of the event, and any associated resources will automatically be released asynchronously at completion."* Destroying in-flight is well-defined; a destructor sync would block teardown and could throw (destructors must not). The no-sync destroy is correct.
- **"`destroy()` should reset the handle only inside the `if`."** Rejected: setting `s_ = nullptr` / `e_ = nullptr` unconditionally after the `if` (lines 88, 160) is harmless and keeps the moved-from/destroyed state consistent. Correct.
- **"Self-assignment guard in move-assign is unnecessary for move."** Rejected: `if (this != &o)` (59, 113) prevents `destroy()`-then-`std::exchange(o.s_, nullptr)` from nulling the just-destroyed handle on pathological self-move. Matches the siblings and architecture.md §7's reference shape (lines 401, 433). Keep.
- **"`Stream`/`Event` should hold a device id for multi-GPU."** Rejected *for now*, and **verification strengthened the rejection**: `cudaStreamWaitEvent` works cross-device without the wrapper knowing either device (confirmed verbatim above), so `Event` needs no device member to serve as the §11.4 cross-GPU ordering primitive. A `Stream` is implicitly bound to whatever device was current at `cudaStreamCreate`; the right home for explicit device binding is `DeviceConfig`/`PerGpuResources` (§9). The cross-device *elapsed_ms*/same-context *record* hazards are not zero — they are captured as a documentation finding (4.8) rather than a device-id member. Re-evaluate at M4.5.
- **"`enable_timing` default should be `true` so `elapsed_ms` always works."** Rejected: `cudaEventDisableTiming` is the documented performance-optimal default for the *ordering* use case (the primary use per §7, verified 8.1); timing is the secondary §11.3 case. The 1.2 fix is to make *misuse* loud, not to flip the default.
- **"`elapsed_ms` should be a `static`/free function taking two events."** Rejected as cosmetic: the member form `end.elapsed_ms(start)` reads acceptably ("this event is the end point"); the real fix is the timing-disabled/not-ready guard (1.2/1.4), not relocating the function.
- **"The `#include <cstdio>` inside the `#else` branch is a hidden dependency."** Rejected as its own finding: correctly scoped to debug-only and in the preprocessor region before the namespace; the real issue is the macro duplication (6.1), of which this include is one symptom.
- **"`elapsed_ms`'s `0.0f` init is dead/should be removed."** Considered, rejected as actionable: `cudaEventElapsedTime` overwrites it on success and the throw path never reads it, so the init is harmless either way — recorded as a trivial nit in 5.1, not a standalone finding.
- **"`synchronize()`/`wait()` on a moved-from `Event` is UB like `Stream`."** Rejected as *equivalent-to-Stream* — it is actually *better*: a null `cudaEvent_t` is an invalid handle that throws `cudaErrorInvalidResourceHandle`, so moved-from `Event` ops fail loudly, unlike moved-from `Stream` ops which silently hit the NULL stream. This *asymmetry* is the finding (2.1); the `Event` side is not independently broken.

## Good patterns to keep

- **The RAII shape is architecture.md §7's reference template, done correctly:** move-construct + move-assign both present (the explicit fix for the "old draft" deleted-move-assign bug), copy deleted, self-assignment guarded, `destroy()` idempotent and null-safe, destructors `noexcept` and never silent (debug teardown warning), and — verified verbatim against the docs — **no destructor sync** (correct, since `cudaStreamDestroy`/`cudaEventDestroy` return immediately and release after completion).
- **`cudaEventDisableTiming` by default** — the documented performance-optimal choice for ordering (§7/8.1; verified verbatim), with timing opt-in for §11.3.
- **`[[nodiscard]]` on the value accessors; correct `noexcept` discipline** (`get()` noexcept; throwing ops not noexcept; teardown noexcept).
- **Layering disclosure** — the comment explicitly flags that the `.hpp` is nonetheless a CUDA header PRIVATE to `steppe_device`, pre-empting the §4 violation a reader might otherwise introduce; matches the `[BUILT]` entry in architecture.md §4 line 154.
- **Every CUDA call routes through the single `STEPPE_CUDA_CHECK`** (no bespoke error handling), and teardown failures route to a warning rather than being swallowed — fail-fast that does not become fail-silent (§2/§7/§10).
- **`Event` is the correct cross-GPU ordering primitive with no device-id baggage** — verified that `cudaStreamWaitEvent` synchronizes across devices, so the deliberately device-agnostic `Event` is the right §11.4 building block; correctness, not omission.
- **Clean seam with the consumers:** `get()` returns exactly the `cudaStream_t` every `launch_*`/`run_*` wrapper and `CublasHandle{stream}` expects, so adopting this type in `cuda_backend.cu` is a drop-in once the non-blocking/tagging gaps (4.1/11.1) are closed — paying TODO §74's "Default-stream debt."
