# X1 — Consistency / same-thing-N-ways

Whole-codebase sweep of `src/ include/ bindings/` for recurring operations done
more than one way without a reason. The headline: steppe is, on this axis,
**exceptionally disciplined** — most recurring ops have a single documented
canonical home that is followed everywhere. The deviations are few, small, and
cluster almost entirely in **one file** (`qpgraph_fit_kernels.cu/.cuh`), which
reads like it was authored against a slightly older set of conventions and never
re-aligned. No HIGH issues. Everything below is MED/LOW polish.

## Findings

- [X1][MED] src/device/cuda/dates_kernel.cu:336,346,355,363,372,380,390,400,412 (9 sites); src/device/cuda/dstat_kernel.cu:275,286 (2); src/device/cuda/qpgraph_fit_kernels.cu:464,478,505 (3) — **Post-launch kernel check done two ways.** The canonical idiom (check.cuh:247-259, used at **62 of 78** launch sites across 11 files) is `STEPPE_CUDA_CHECK_KERNEL()`, explicitly documented to be placed "immediately after every `kernel<<<...>>>(...)`" and to add the debug-only `cudaDeviceSynchronize` that attributes async kernel faults to the launch under compute-sanitizer. These **14** sites instead open-code the weaker `STEPPE_CUDA_CHECK(cudaGetLastError())`, which catches a bad launch config but drops the debug-sync async-fault attribution the macro exists to provide. (The 3 qpgraph sites do follow with a full `cudaStreamSynchronize`, so they are functionally covered there — but still a third idiom; the dates/dstat sites are not.) Suggested: replace the 14 bare `STEPPE_CUDA_CHECK(cudaGetLastError())` post-launch calls with `STEPPE_CUDA_CHECK_KERNEL()`.

- [X1][MED] src/device/cuda/qpgraph_fit_kernels.cuh:1 and src/device/cuda/qpgraph_fit_kernels.cu:1 — **Namespace nesting done two ways.** Every other device CUDA TU (38 of 40 `.cu`/`.cuh` files) lives in `namespace steppe::device`; only this header+impl pair uses the deeper `namespace steppe::device::cuda` (closing comment `}  // namespace steppe::device::cuda` at qpgraph_fit_kernels.cu:509). backend_factory.hpp:16 even documents the intent — "both factories live in `namespace steppe::device`" — and the launch_* symbols are reached the same way as every other kernel's, so the extra `::cuda` segment is gratuitous and inconsistent. Suggested: drop the `::cuda` segment so this pair joins the `steppe::device` convention used by the other 19 device-cuda files.

- [X1][LOW] src/device/cuda/qpgraph_fit_kernels.cuh:44 — **`STEPPE_HD` macro vs raw `__host__ __device__`.** host_device.hpp:8-16 establishes `STEPPE_HD` as THE single-source host/device qualifier (so a shared primitive compiles in both spaces from one spelling); f2_estimator.hpp, decode_af.hpp, snp_summary_reduce.hpp, filter_decision.hpp, launch_config.hpp all use it. `make_layout` here is exactly such a shared host/device helper but is spelled with the raw `__host__ __device__ inline`. It compiles (the `.cuh` is always seen by nvcc), but it bypasses the documented single-source macro. (Same lone-deviator file as the two findings above.) Suggested: `STEPPE_HD inline ScratchLayout make_layout(...)`.

- [X1][LOW] src/device/cuda/qpadm_fit_kernels.cuh:37,53,69,81,357 — **Device-pointer parameter naming (`d_` prefix) done two ways within one header.** The codebase-wide convention is to prefix raw device pointers with `d_` (used on essentially every launch_* device-pointer param, including this same file's `d_keep`, `d_X`, etc.). These five resident-tensor inputs are named `f2` / `vpair` with no prefix (`const double* f2`, `const double* vpair`), so the same file mixes `f2`/`vpair` and `d_keep` side by side. Cosmetic (the params ARE device pointers), but it reads as a local naming drift. Suggested: rename to `d_f2` / `d_vpair` to match the surrounding `d_`-prefix convention.

- [X1][LOW] include/steppe/qpadm.hpp:221 (`Precision::Kind precision_tag` on `QpWaveResult`) vs the other result structs — **the "precision actually used" tag is present on some result PODs and absent on others.** `QpWaveResult` carries `precision_tag = Precision::Kind::Fp64`; several sibling result types (e.g. `F4Result`, `F3Result`, `DstatResult`) report `est/se/z/p + Status status` with no precision tag. Whether each statistic needs the tag is a real design question, but as it stands the "record which precision lane ran" convention is applied unevenly across the otherwise-parallel result family. Suggested: decide once whether `precision_tag` belongs on the whole result family or only the fit/rank path, and apply uniformly.

- [X1][LOW] src/app/pop_resolver.hpp:29,37,73 (`error`, `error`, `error_`); src/core/config/config_builder.hpp:120 (`error_message_`); src/core/qpadm/qpgraph_model.hpp:93 (`error` + an `ok()` method at :94); src/app/cmd_qpgraph.cpp:149 (local `err`) — **the "outcome + reason string" POD names its reason field four ways.** The dominant form across app/io result PODs is a public `bool ok = false; std::string error;` pair (f2_dir_io.hpp:51-53, f2_dir_writer.hpp:85-87, pop_resolver.hpp:27-29). Deviations: a `private: std::string error_;` (pop_resolver.hpp:73), a `mutable std::string error_message_;` (config_builder.hpp:120), an `ok()`-as-method-over-`error.empty()` instead of an `ok` field (qpgraph_model.hpp:93-94), and a one-off local named `err` (cmd_qpgraph.cpp:149). Each is locally defensible, but a reader cannot predict the field name. Suggested: standardize the field name to `error` and the success predicate to either a `bool ok` field or a consistently-named `ok()` accessor, repo-wide.

- [X1][LOW] src/core/CMakeLists.txt (steppe_core) and src/device/CMakeLists.txt (steppe_device) — **`target_compile_features(... cxx_std_20)` set on 5 targets, omitted on 2.** steppe_io, steppe_app, steppe_access, steppe_extract, and steppe_core_internal each call `target_compile_features(<t> {PUBLIC|PRIVATE|INTERFACE} cxx_std_20)`; the two STATIC libs steppe_core and steppe_device do not (they inherit the standard transitively via linked INTERFACE targets). It works, but the same compile-feature declaration is present-or-absent without a stated reason. Suggested: add the explicit `target_compile_features` line to steppe_core and steppe_device so all targets declare their language standard the same way.

## The canonical form / recommendation

The recurring ops and their (strong, well-documented) canonical homes:

- **CUDA/cuBLAS/cuSOLVER fault checking** — `STEPPE_CUDA_CHECK` / `CUBLAS_CHECK`
  / `CUSOLVER_CHECK` (throwing) + `STEPPE_CUDA_WARN` (non-throwing, recoverable),
  single home check.cuh. Universally followed; zero `exit()`/raw-status drift.
- **Post-launch kernel check** — `STEPPE_CUDA_CHECK_KERNEL()` (check.cuh:254).
  62/78 sites; the 14 exceptions are the MED finding above.
- **Device / pinned-host allocation** — only the THREE allowlisted TUs
  (device_buffer.cuh, pinned_buffer.cuh, the documented allocator home) call
  `cudaMalloc`/`cudaFree`/`cudaHostAlloc`/`cudaHostRegister`; the grep gate is
  clean. Everything else takes non-owning views. **No deviation.**
- **RAII resource wrapper shape** — move-only, copy-deleted, `std::exchange`
  move + `noexcept destroy()`-routes-nonzero-status-to-`STEPPE_LOG_WARN`,
  dtor-never-throws. DeviceBuffer, PinnedBuffer, RegisteredHostRegion, Stream,
  Event, CublasHandle, CusolverDnHandle, GesvdjInfo, MathModeScope,
  CusolverMathModeScope — **all 10 identical.** Exemplary.
- **pImpl opaque device handle** — `struct Impl; std::unique_ptr<Impl> impl;`
  with `_impl.cuh` definition. DevicePartial, DeviceF2Blocks, DeviceDecodeResult
  — **all 3 identical.** No deviation.
- **Logging / diagnostics** — `STEPPE_LOG_WARN` (log.hpp) is the only library
  sink; all `printf`/`fprintf`/`cout` live in `src/app/` (the CLI, which by §10
  legitimately owns stdout/stderr) + the bindings. **No library-code deviation.**
- **Error/status propagation** — intentionally stratified and consistent within
  each layer: `Status` enum value-fields at the compute seam (never throw for
  domain outcomes), `BuildResult<T>` at config, `bool ok + std::string error`
  PODs in app/io, typed exceptions for device faults, `nb::value_error/key_error`
  in bindings. The only drift is the LOW reason-field-naming finding.
- **Backend dispatch** — purely via the `ComputeBackend` virtual; **zero**
  `dynamic_cast`/`typeid`/`is_cuda` branching in core. Exemplary.
- **Kernel-launch wrapper idiom** — `launch_<name>(...args..., cudaStream_t
  stream)` with the stream **always last** (0 deviations) and device pointers
  `d_`-prefixed (1 local drift, the LOW finding). ~65 launch_* functions, uniform.
- **Public entry points** — every `run_*` is `[[nodiscard]]`, takes `f2` first
  and `Resources&`/`Options` last, and ships a `DeviceF2Blocks` + a host
  `F2BlockTensor` overload pair. **Strikingly uniform** across 11 headers.
- **Include style** — 704 project-header includes, all quote-style root-relative
  (`"core/..."`, `"device/..."`); **zero** `<>` and **zero** `../`. Perfect.
- **CMake target pattern** — `add_library(... ALIAS steppe::...)`,
  `target_include_directories(... "${CMAKE_CURRENT_SOURCE_DIR}/..")`, strict
  PUBLIC/PRIVATE link discipline, find_package-first-then-CPM for deps. Uniform
  but for the LOW `target_compile_features` omission on 2 targets.
- **`__host__ __device__` qualifier** — `STEPPE_HD` macro (host_device.hpp);
  1 raw-spelling deviation (the LOW qpgraph finding).
- **`assert`** — `STEPPE_ASSERT` only; **zero** raw `assert(`. No deviation.

Overall recommendation: this dimension is in release shape. The single highest-
value cleanup is the MED kernel-check normalization (14 sites → the documented
`STEPPE_CUDA_CHECK_KERNEL()`), because it restores the compute-sanitizer
async-fault attribution the canonical macro was built to guarantee. Re-aligning
`qpgraph_fit_kernels.cu/.cuh` (namespace + `STEPPE_HD`) folds the lone structural
outlier back into the device-layer conventions. The rest is naming polish.
