# X2 — Comment signal-to-noise

Scope: src/ include/ bindings/ (docs/ excluded). READ-ONLY identify-only.

This is the dominant cross-cutting issue in the codebase. The code is correct and
the comments are accurate, but they are systematically over-written: design-doc
essays, AT2/architecture.md cross-reference chains, and PR-review archaeology are
inlined into source. Whole-codebase metrics:

- 830 comment lines cross-reference `architecture.md` / `design §` / `ROADMAP §` /
  `perf-discovery` (grep count over src include bindings).
- 42 files carry internal cleanup-ticket archaeology in comments — `cleanup X-7/B6`,
  `cleanup X-6/B2`, `group-7` (31 hits), `group-5` (18), `cleanup §(2)`, etc. These
  are PR/review-process labels with zero meaning to a future reader (grep counts).
- ~49 files open with a 25+ line file-header essay (header-block-from-line-1 scan).
- 9,869 ALL-CAPS emphasis tokens in comments — a pervasive "shouting" style that
  flattens emphasis to noise (when everything is CAPS, nothing is).
- `src/device/cuda/cuda_backend.cu` is 31% comments (1749 / 5648 lines).

Note: the high comment:code RATIO files at the top of the sweep are almost all
public headers (host_device.hpp 0.94, plink_reader.hpp 0.87, config.hpp), where
API doc-comments are appropriate — ratio alone is NOT the smell. The smell is
LENGTH/ALTITUDE: multi-paragraph design narrative + cross-ref chains + ticket IDs
buried over a few lines of code. Findings target that.

## The canonical form / recommendation

The codebase has the OPPOSITE of the usual comment problem: comments are accurate,
"why"-focused, and never restate the mechanics — but they are written at DESIGN-DOC
altitude and inlined into source, then layered with cross-ref chains and PR ticket IDs.

Canonical comment for this codebase:
1. WHY + the non-obvious invariant, in as few lines as the point needs.
2. ONE cross-reference, not a chain: "see docs/architecture.md §11.4" — not a paragraph
   re-narrating §11.4's content. The doc is the single source; the comment points, it
   does not mirror.
3. NO PR/review archaeology: strip every `cleanup X-#/B#` and `group-#` token. The
   rationale they annotate can stay; the ticket label is noise.
4. NO multi-paragraph algorithm/statistic DERIVATIONS or bit-identity PROOFS in source —
   those go to docs/ (research/architecture), with a one-line pointer at the code.
5. NO self-changelog ("the doc once said X — WRONG"); state the current fact only.
6. Reserve ALL-CAPS for the rare genuinely load-bearing word; the current ~10k caps
   tokens have flattened emphasis into background noise.

Recommended order of attack (mechanical → judgment):
- (a) Codebase-wide strip of `cleanup <id>` / `group-<n>` ticket tokens (42 files, low
  risk, high signal gain).
- (b) The HIGH design-doc essays (gate banner, p2p_combine, dstat_kernel, decode_af,
  cuda_backend Stream essay, the model_search TODO) → relocate to docs/ with a 1-line
  pointer. These six alone account for the bulk of the over-write.
- (c) The MED header essays (block_partition_rule, f2_block_kernel, dates_curve doc) →
  trim to contract + pointer.

Honest assessment: NOT a clean result, but the defect is over-documentation, not
under- or mis-documentation. Every essay is correct and load-bearing in intent — the
fix is altitude/placement (move to docs/, point from code), not deletion of meaning.
