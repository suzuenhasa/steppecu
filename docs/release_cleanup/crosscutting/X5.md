# X5 — Idiomatic modern C++20 / CUDA

Scope: whole-codebase sweep of `src/`, `include/`, `bindings/` (194 files; `docs/`
excluded). FP64/parity choices (native + Ozaki emulated FP64, fixed-order
deterministic reductions, single statistic stream) treated as intentional §12
design and NOT flagged.

Headline: this is a strong "knows modern C++/CUDA" signal. The dated-C smells the
focus asks for are essentially ABSENT. Findings below are a thin tail of LOW
polish items, not structural problems.

## What the sweep confirmed clean (whole-codebase, with evidence)

- **No C-style casts.** `grep` for `(int)/(double)/(size_t)…expr` over all .cpp/.hpp/.cu/.cuh
  returned only 7 hits, ALL of which are the substring `(double)` inside doc
  comments (e.g. `resources.hpp:106`, `block_sink.cuh:276`). Zero in code. Casts
  are `static_cast`/`reinterpret_cast` throughout.
- **No plain `enum`, no `typedef`, no `NULL`, no `goto`, no `using namespace` in
  headers, no `0`-for-pointer.** All grep counts zero. `enum class` is the rule
  (16 uses, e.g. `include/steppe/error.hpp:22`, `cli_args.hpp:34`).
- **No raw owning `new`/`delete` in host logic.** The only `new` sites are in
  `bindings/module.cpp` (lines 352/878/952 `new F2Handle()` paired with nanobind
  `rv_policy::take_ownership`, line 1102 `new double[n]` paired with an
  `nb::capsule` deleter) — these are the documented nanobind ownership-transfer
  and zero-copy idioms, not leaks. No host `malloc/free`.
- **RAII done correctly.** `DeviceBuffer<T>` (`device_buffer.cuh`), `PinnedBuffer<T>`
  (`pinned_buffer.cuh`), and the cuBLAS/cuSOLVER/stream handles (`handles.hpp`,
  `stream.hpp`) are move-only with full rule-of-5 (`= default`/`= delete`/`noexcept`
  move, `~` calling `reset()`), `[[nodiscard]]` accessors. Raw `T* ptr_ = nullptr`
  members live ONLY inside these wrappers — exactly where a raw pointer belongs.
- **Modern CUDA, not reinvented.** Uses `cub::DeviceScan::ExclusiveSum`,
  `cub::DeviceSelect::Flagged`, `cub::DeviceRadixSort`, `cub::BlockReduce`
  (`cuda_backend.cu:1404+`, `qpfstats_jackknife_kernel.cu:19`) rather than
  hand-rolling primitives. No `volatile` warp-sync anti-pattern (zero hits).
  `atomicAdd` is the correct block-shared-then-one-atomic reduction
  (`dates_kernel.cu:51` comment + 78/79). Flat slab indices use `size_t`/`long`
  with explicit widening casts (`f2_blocks_kernel.cu:168-188`, 276/283/291) — the
  P*P*n_block > 2^31 overflow class is handled (size computations consistently
  `static_cast<std::size_t>` before multiply: `device_f2_blocks.hpp:50`,
  `f2_blocks_out.cu:162`, `p2p_combine.cu:102/190`, etc.).
- **Modern vocabulary types in use:** `std::span` (308 hits; 16 in public headers,
  used as read-only param type), `[[nodiscard]]` (484), `constexpr` (198),
  `std::optional`/`variant` (66), `std::array` (94), `enum class` (16). Zero raw
  fixed C arrays for data. `static_assert` pins ABI/layout invariants
  (`f2_disk_format.hpp:42`, `block_sink.cu:179`). Build enforces
  `CMAKE_CXX_STANDARD 20` + `CMAKE_CUDA_STANDARD 20` (`CMakeLists.txt:26-29`).
- **`#define` is justified everywhere it appears:** CUDA check macros needing
  `__FILE__`/stringize (`check.cuh:230`), compile-time feature gates for `#if`
  (`handles.hpp:85/87`, `f2_block_kernel.cu:79`), a SHA round macro
  (`f2_dir_writer.cpp:167`). None should be `constexpr`/inline.

## Findings

- [X5][LOW] bindings/module.cpp:1103 — manual element copy
  `for (std::size_t i = 0; i < n; ++i) buf[i] = src[i];` into the freshly-`new`'d
  capsule buffer. This is the single clear hand-rolled-copy site in the codebase.
  Suggested: `std::copy_n(src, n, buf);` (or `std::memcpy`, contiguous `double`).

- [X5][LOW] No C++20 `concept` definitions / `requires`-clauses anywhere (all 37
  "requires/concept" grep hits are the English words in comments and error
  strings, e.g. backend.hpp:743). The few templates (`DeviceBuffer<T>`,
  `PinnedBuffer<T>` in device_buffer.cuh:42 / pinned_buffer.cuh:73, and
  build_result.hpp:36) are unconstrained `template <class T>`. Low value: only 17
  templates total, mostly RAII buffers over arithmetic/POD; a
  `requires std::is_trivially_copyable_v<T>` on the buffer templates would
  document intent and improve error messages, but this does not "earn it" broadly.
  Suggested: optionally constrain the two device buffer templates; otherwise leave.

- [X5][LOW] Index-style `for (std::size_t i = 0; i < v.size(); ++i)` dominates over
  range-for (372 index loops vs 11 range-for). Most are load-bearing — math /
  fixed-order reductions (cpu_backend.cpp ~32, small_linalg.hpp ~16) or parallel
  ("zip") iteration over two vectors where the index is genuinely needed
  (result_emit.cpp:284/314 indexes `results[i]` AND `left_labels[i]` together;
  C++23 `views::zip` not available). A residual handful iterate a single container
  using only `v[i]` and could be range-for, but no single egregious site stands
  out. Suggested: prefer range-for for new single-container loops; not worth a
  sweep of existing code.

- [X5][LOW] No `std::format`; CLI/diagnostics use `std::printf`
  (`cmd_extract_f2.cpp:245+`, `cmd_qpfstats.cpp:112+`, `cli_parse.cpp:753`) and
  iostreams (`result_emit.cpp`). The `printf` calls are `std::`-qualified and
  type-correct, but `std::format`/`std::print` would be the C++20/23-idiomatic
  choice and eliminate format-string/arg mismatch risk. Minor; nvcc `std::format`
  support has historically lagged, so iostreams/printf is a defensible call.
  Suggested: consider `std::format` for new host-side formatting; not a defect.

## The overall recommendation

X5 is effectively a pass. The portfolio signal is unambiguously "this person
writes idiomatic modern C++20 and idiomatic CUDA": correct RAII (move-only,
rule-of-5, `[[nodiscard]]`), `enum class`, `std::span`/`std::optional`/`std::array`,
overflow-safe `size_t` index widening at scale, and cub-based GPU primitives
instead of reinvented reductions — with the dated-C smells (C casts, raw
new/delete in logic, `typedef`/`NULL`/plain-enum/`#define`-constants) entirely
absent. The four findings are all LOW polish: one one-line `std::copy_n`
(bindings:1103), an optional concept on two buffer templates, a stylistic
range-for / `std::format` preference. None block release. The single concrete
fix worth making is bindings/module.cpp:1103 -> `std::copy_n`.
