# X7 — Senior first-impression (per subsystem)

Read-only whole-codebase pass over `src/ include/ bindings/` (docs/ excluded). The
question: opened cold in an interview, what makes a staff engineer wince? The
*engineering* is strong (the precision policy, the backend seam, the IO leaf, the
Python facade are all genuinely good). The winces are almost entirely about VOICE —
the shipping source is narrated as a private engineering journal with footnotes to an
internal tracker, and the dominant tells are cross-cutting, not per-file. Honest
finding: one stale/contradicted comment is a real defect; the rest is pervasive
internal-process bleed into comments.

## Findings

- [X7][HIGH] cross-cutting (90 lines / 25 files) — Internal cleanup-tracker ticket IDs
  are footnoted directly in shipping comments: `cleanup X-1/B1`, `X-2/B4`, `X-5/B5`,
  `X-7/B6`, `X-13/B26`, `cleanup 00-overview §(2).1`. They reach the PUBLIC-facing
  seam header (`src/device/backend.hpp:56,74,513`) and a PUBLIC include
  (`include/steppe/config.hpp:249` "cleanup X-8/B16"). These reference the very
  release-cleanup process being run now — an external reader cannot resolve `X-8/B16`
  and reads them as dangling footnotes on nearly every non-trivial decision. This is
  the single worst first-impression signal. Suggested: strip ticket IDs from
  shipping comments; keep the *rationale*, drop the `cleanup X-N/B-N` provenance tag.

- [X7][HIGH] `include/steppe/config.hpp:7-253` (40 tag-lines) — The PUBLIC header — the
  first file a portfolio reviewer or API consumer opens — is saturated with private
  internal references: `ROADMAP §0/§4`, `M4`/`M5` milestones, `group-5 5.3/5.5`,
  `the B26 2× term`, `cleanup X-8/B16`, `the spike's magic numbers`. None resolve
  outside the repo. A public API surface should read self-contained. Suggested:
  rewrite public-header comments to be self-describing (what/why), with zero
  ROADMAP/milestone/ticket citations.

- [X7][HIGH] `src/app/cli_parse.cpp:356` — STALE + self-contradicted comment. Inside the
  *qpadm* callback: "qpadm-rotate + extract-f2 are likewise wired; only qpwave
  (M(cli-2)) remains a scaffold no-op." But qpwave is FULLY implemented
  (`cmd_qpwave.cpp:49` `run_qpwave_command` is a complete GPU path) and dispatched at
  `cli_parse.cpp:462`. The same file's own header at line 13 correctly says all
  subcommands "are ALL wired to their GPU compute." A reviewer who greps "no-op" lands
  on a false claim that the file itself refutes. Suggested: delete the "qpwave remains
  a scaffold no-op" clause.

- [X7][MED] cross-cutting (`M4` 83×, `M4.5` 63×, `M5` 56×, `M(fit-2)` 26×, plus
  `B1/B2/B6/B22/B24/B26`, `group-5/group-7`, `L0`–`L3`, `N-1/N-5`) — Opaque
  milestone/issue codes are the dominant comment idiom across every subsystem. To a
  reader without the tracker these are noise that obscures otherwise-excellent
  technical prose. `architecture.md §N` is cited on 1641 lines — defensible as a spec
  pointer, but combined with the milestone codes the voice reads as "defending
  decisions to a reviewer," not "documenting for a reader." Suggested: collapse to at
  most a single canonical spec reference per decision; drop the bare milestone codes.

- [X7][MED] cross-cutting (47 lines) — R&D-history narration leaks into shipping code:
  "the spike", "spike-chosen", "the spike concluded", "Replaces the spike's raw
  cudaMalloc/cudaFree", "experiments/f2_emu_spike". E.g. `device_buffer.cuh:9`,
  `stream.hpp:5`, `handles.hpp:5`, `f2_blocks_kernel.cu:9`, `backend.hpp:703`. The
  reader does not have the spike; "this RAII type owns the buffer" is the durable
  statement, "replaces the spike's bare pairs" is throwaway archaeology. Suggested:
  state what the code IS, not what experiment it descended from.

- [X7][MED] `src/device/cuda/f2_block_kernel.{cu,cuh}` vs `f2_blocks_kernel.{cu,cuh}` —
  Two sibling kernel TUs whose names differ by a single `s` (single-block 3-GEMM path
  vs the M4 batched-over-blocks path). The distinction is real and documented, but a
  reader skimming the `cuda_backend.cu` include list cannot tell them apart at a
  glance — a classic confusable-name trap. Suggested: rename one (e.g.
  `f2_batched_kernel`) so the two paths are visually distinct.

- [X7][MED] `src/device/cuda/cuda_backend.cu` (5648 lines, 53 `override`s in one file) —
  `CudaBackend` is a god-class implementing the entire 50-virtual `ComputeBackend`
  seam (`backend.hpp`) in a single TU — f2, decode, transpose, dstat, dates, qpfstats,
  qpadm fit, qpgraph fleet, jackknife, sweep. The single-seam design is deliberate and
  the CpuBackend mirrors it, but 5.6k lines is the kind of file a staff engineer opens
  and immediately asks "why isn't this split per concern?" Suggested: note as
  intentional, or split the override bodies into `cuda_backend_{f2,fit,dstat,...}.cu`
  TUs behind the one class.

- [X7][LOW] cross-cutting (`NEVER` 106×, `THE DELIVERABLE` 19×, `MUST` 22×, `ALWAYS`
  17×, `LAW`/`HONORABLE`/`REJECTED`/`TRAP`/`WART`) — Heavy ALL-CAPS emphasis and
  charged vocabulary ("the law", "the cautionary tale", "HONORABLE", "the REJECTED
  trap") throughout. In moderation it signals load-bearing invariants; at this density
  it reads as shouting and slightly over-written. Suggested: reserve caps for the
  handful of genuine parity/precision invariants; let the rest be plain.

- [X7][LOW] `src/device/cuda/qpgraph_fit_kernels.cu:245` — `q1[e1] = rr;  // unscaled q
  for now` reads as in-progress scaffolding, but the line below (`q1[e1] /= sc[e1]`)
  completes the scaling intentionally; "for now" is misleading. Minor, but "for now"
  is exactly the phrase an interviewer greps for. Suggested: reword to "unscaled; the
  sc[] divide below scales it" — drop "for now".

- [X7][LOW] `src/device/host_ram.cpp`, `src/device/cpu/cpu_backend.cpp:1` etc. — "WHY
  THIS IS THE ORACLE, NOT A TEMPLATE THE GPU MIMICS" and similar long argumentative
  preambles. The content is correct and valuable, but the rhetorical framing
  ("not a template", repeated defenses) is the journal voice again. Low because it's
  genuinely informative. Suggested: keep the fact, trim the rhetoric.

NOT findings (checked, clean — worth recording):
- Boundary naming is EXCELLENT: CLI subcommands (`qpadm`, `qpwave`, `qpgraph`,
  `qpadm-rotate`, `extract-f2`) map cleanly to Python (`qpadm`, `qpwave`, `qpgraph`,
  `qpadm_rotate`/`qpadm_search`, `extract_f2`) via idiomatic kebab→snake. No mismatch.
- Python facade (`bindings/steppe/__init__.py`, 1010 lines) is clean, well-typed,
  pandas-shaped, consistent `_from`/`_to_dict` idioms. No journal voice. No winces.
- IO error handling is consistent: the `io` leaf uniformly throws
  `std::runtime_error`/`std::invalid_argument` (77+5), `app` translates to exit codes,
  `core` returns Status — a coherent layered convention, no double-throwing.
- No genuine dead code, no commented-out code blocks, no informal voice
  (no "let's"/"gonna"/"obviously"); all "magic" hits are legitimate file-format magic
  bytes or "not a magic number" callouts.
- The `double`/FP64 idiom is uniform and intentional (per §12) — correctly NOT flagged.

## The overall recommendation

The code is staff-quality; the *commentary* is not yet portfolio-quality. One pass of
"comment hygiene for an external reader" closes ~90% of X7:

1. (HIGH) Purge internal-tracker provenance from comments — `cleanup X-N/B-N`,
   `host-compute audit C1/C2`, bare milestone codes (`M4`/`M5`/`M(fit-2)`/`B26`/
   `group-5`/`L2`/`N-5`), and "the spike"/`experiments/` archaeology. Keep the
   rationale, drop the ticket footnote. Prioritize the PUBLIC surface first:
   `include/steppe/*.hpp` and `src/device/backend.hpp` must read self-contained.
2. (HIGH) Fix the one real defect: the stale "qpwave remains a scaffold no-op" comment
   at `cli_parse.cpp:356`.
3. (MED) Disambiguate `f2_block_kernel` vs `f2_blocks_kernel`; consider splitting the
   5.6k-line `cuda_backend.cu` per concern (or annotate why it's one TU).
4. (LOW) Dial back the ALL-CAPS/charged-vocabulary density; reserve emphasis for the
   handful of true parity/precision invariants.

The canonical comment idiom to converge on: a one-line "what this owns", a "why this
shape (constraint, not history)" if non-obvious, and AT MOST one spec pointer
(`architecture.md §N`) — zero milestone codes, zero ticket IDs, zero spike narration.
