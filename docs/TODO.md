## ÔÜí Keeping it GPU-dominant (perf research `wqd0a9o0l`)
*Answer to "a GPU thing shouldn't be punished by a bad CPU." Host-CPU sensitivity has 3 surfaces: ingest (#1), launch overhead (#2), orchestration (#3). Verdict: **#1 and #3 are M5/M4.5 problems solvable now with definitely-works-on-vast primitives; #2 is a Phase-2 problem for CUDA Graphs.** Guiding principle: **the CPU becomes a sequencer + metadata janitor ÔÇö never a data mover or a per-SNP reducer.** Full synthesis: workflow `wqd0a9o0l`.*

**Environments:** routine build/test/dev runs on the cheap **vast box** (CUDA 13, consumer 5090) ÔÇö the design must stay fast *there* (pinned double-buffer baseline, **no GDS**). For the "for sure need to" moments ÔÇö the **measure-first Nsight/`ncu` profile, GDS experiments, and final perf validation** ÔÇö use the **RTX 6000 full-host box** (professional card Ôçĺ no GeForce GDS lock; root/host Ôçĺ `nvidia-fs` + a GDS-qualified FS are controllable; full perf counters). Run it sparingly (expensive). Ôçĺ **GDS upgrades from "not worth it" to a real research bet on that box** (target: the large `.f64`/M7 cache), while the portable pinned pipeline stays the baseline everywhere. *(Which RTX 6000 ÔÇö Blackwell sm_120 vs Ada sm_89 ÔÇö TBD; matters for perf-representativeness vs the 5090.)*

**Ranked plan:**
1. **[adopt now] Measure-first Nsight gate** ÔÇö `nsys profile --trace cuda,nvtx,osrt` on the real end-to-end path; classify ingest-bound (un-overlapped copy) vs launch-bound (GPU-idle gaps) vs orchestration-bound (host gaps). Free, container-safe (the CUDA+NVTX trace needs no SYS_ADMIN), and it orders everything below. **Don't optimize against a guess.**
2. **[M5] Pinned double-buffered ingest = the CPU-insensitive spine** ÔÇö replace `geno_reader`'s pageable `ifstream` per-individual gather with large sequential reads into `PinnedBuffer` ring slots + `cudaMemcpyAsync` on a **dedicated copy stream**, overlapped with the ~0.3 s GEMM via `Event`; end-to-end Ôćĺ `max(read, gpu)` not their sum. Copy-engine DMA moves the data; the host just sequences. **Pin only small slots (tensÔÇôlow-hundreds MB), never the 4 GB .geno** (RLIMIT_MEMLOCK cap ~0.5ÔÇô1 GB in containers); fallback ladder `cudaMallocHost Ôćĺ malloc+cudaHostRegister Ôćĺ pageable+warn`. Plus `cudaMallocAsync` pool (release threshold MAX), hoist the per-tile block-scan/bucketing out of the loop, keep `f2_blocks`+handle resident, `O_DIRECT`/`fadvise(DONTNEED)` cold reads, `--ulimit memlock=-1` launch template + probe at `build()`.
3. **[M4.5] Fuse the cheap in-tile filter into the decode device pass** ÔÇö make `filter_decision.hpp` predicates `__host__ __device__`, apply the keep-mask as a validity-mask multiply inside `decode_af`; deletes a D2H + an O(P┬ĚM) host reduction + an O(M) host predicate loop per tile (the clearest host liability in `snp_filter.cpp` today). Gate on bit-for-bit parity vs `test_filter_oracle`.
4. **[Phase 2] CUDA Graphs for the fit engine** ÔÇö capture the per-model ~6ÔÇô10 tiny-kernel fit once, re-launch via `cudaGraphExecUpdate` across thousands of models (NVIDIA 1.5ÔÇô3├Ś + jitter collapse ÔÇö exactly the many-tiny-launch regime the ~1.6├Ś host inflation lives in). Prereqs: ZERO `cudaMalloc` during capture (pre-allocate all buffers + cuSOLVER/cuBLAS workspaces + devInfo), topology-class graph cache, RAII graph wrappers. Gated on the Nsight profile proving launch-bound.
5. **[M7 research bet] nvCOMP field-split cache compression** ÔÇö aggressive on the V mask (~46├Ś) and N counts (~4.9├Ś), near-raw on the FP64 f2 payload (~1.06├Ś); GPU SM-decompress (userspace `.so`, works on vast). Needs a bit-exact round-trip golden. Only once M7's cache exists.

**Decided against (do NOT re-litigate / don't engineer for the vast 5090):**
- **True GPUDirect Storage / cuFile** ÔÇö three independent disqualifiers on vast consumer 5090 (GeForce = compat-mode-only by NVIDIA policy; `nvidia-fs`/NVMe-P2PDMA need host-kernel access we don't have in a rented container; instance data is on **OverlayFS**, not a GDS-qualified FS). cuFile silently degrades to POSIX `pread`+copy = identical to today. The pinned double-buffer **is** the correct fallback (already ┬ž11.1). Reconsider only on a datacenter (Quadro/Tesla) box for the large `.f64` cache.
- Graphing the **precompute** (bandwidth-bound, ÔëĄ1.4├Ś, and the current code is capture-hostile) ┬Ě moving **block-binning to GPU** (one-time host metadata scan, saves ┬Ás) ┬Ě "optimizing" the `f2_from_blocks` 2-line forwarder ┬Ě the **Blackwell HW decompression engine** (datacenter-only; nvCOMP SM-decompress is the path) ┬Ě compressing the **FP64 payload** itself (~1.06├Ś) ┬Ě a transposed `.geno` layout (no ratio gain) ┬Ě cuSOLVERMp distributed solve (qpAdm matrices are tiny).

**Open questions (the Nsight gate + a tiny spike answer most):** is the ~1.6├Ś actually ingest/launch/orchestration-bound? real RLIMIT_MEMLOCK in the vast container? vast volume read bandwidth + does O_DIRECT succeed on overlay? does `cudaMemcpyAsync` overlap materialize on the 5090? can cuSOLVER `gesvdj`/`potrf` + Ozaki-emulated GEMM be stream-captured on this 13.0 box?

### Re-verified adversarially (`wxz1fiiln`) ÔÇö capability tiers + box roles
*"Officially unsupported Ôëá impossible" ÔÇö **confirmed**. The aikitoria fork (`open-gpu-kernel-modules @610.43.02-p2p`; geohotÔćĺnimlgenÔćĺvaldemardi lineage) enables REAL BAR1 PCIe P2P on full-host 5090s (measured **~55 GB/s**). **But NCCL #1637 Ôëá aikitoria:** the NCCL 2.26 fix merely makes NCCL FUNCTION over slow SHM on 5090s ÔÇö it does NOT enable P2P. And the decisive nuance: **steppe is P2P/NVLink-INSENSITIVE by design** (┬ž11.4 keeps cross-GPU traffic to a kBÔÇôMB `f2_blocks` broadcast + the parity reduction host-side, fixed-order) Ôćĺ P2P is architectural cleanliness, not throughput.*

**Verdicts (full-host RTX PRO 6000 = primary; vast 5090 = budget fallback):**
- **FLIP on the PRO 6000** (driver-policy/container blocks, not silicon): true **GDS/cuFile** (Quadro-class CC 12.0) ┬Ě full **`ncu --set full`** counters (`NVreg_RestrictProfilingToAdminUsers=0`) ┬Ě **O_DIRECT** ┬Ě **RLIMIT_MEMLOCK** (memlock=unlimited).

- **CONDITIONAL ÔÇö P2P:** clean **stock-driver** on the PRO 6000 (no NVLink on this SKU ÔÇö just IOMMU/ACS off); aikitoria-patched on full-host 5090s (out-of-tree, rebuild per driver ÔÇö **dev convenience only, NEVER the production parity path**).

**Capability tiers** (build for capable, degrade with an explicit logged tag ÔÇö every lever is **parity-neutral**: data-movement/observability only):
| Lever | Capable path (PRO 6000 / full-host) | Budget-5090 fallback + logged reason |
|---|---|---|
