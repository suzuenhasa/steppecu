# Review — `device-cuda-f2_blocks_kernel`

Unit: `src/device/cuda/f2_blocks_kernel.cuh` + `src/device/cuda/f2_blocks_kernel.cu`
M4 BATCHED per-block f2 path (size-grouped strided-batched). Reviewed against `docs/architecture.md` §2/§4/§5 S2/§7/§9/§11/§12/§13, `docs/ROADMAP.md` §4/§5/§6, `docs/TODO.md` capability tiers, and the official CUDA 13.x / cuBLAS 13.3 documentation (cited inline).

## Role & layering

This translation unit owns the three on-device primitives that bracket the M4 grouped GEMMs:

1. `launch_gather_group` — gather one power-of-2 size bucket's SNP columns out of the block-contiguous feeder outputs (`dQ/dV/dS` over all M SNPs) into a padded strided-batched slab layout (pad columns zero-filled).
2. `run_f2_gemms_group` — the three f2 GEMMs as `cublasGemmStridedBatchedEx` over the bucket's slabs (`G = Qg·Qgᵀ`, `Vpair = Vg·Vgᵀ`, `R = Sg·Vgᵀ`), precision-governed.
3. `launch_assemble_blocks_group` — fused numerator+divide (native FP64) scattered into the resident `[P×P×n_block]` f2 + Vpair tensors.

**Layering is correct.** The header includes `<cublas_v2.h>` and names `cublasHandle_t`, so it is correctly self-described as PRIVATE to `steppe_device` (architecture.md §4) — the device-internal seam, not the CUDA-free `ComputeBackend` seam. Kernel bodies and `<<<>>>` live only in the `.cu`; the host calls narrow `void launch_* / run_*` wrappers (architecture.md §7). It reuses the SHARED `core::assemble_f2_numerator` / `core::finalize_f2` / `core::grid_for` primitives and the SHARED `engage_f2_precision` / `f2_compute_type` policy (architecture.md §2 DRY, §13). The feeder is *not* duplicated — M4 reuses `launch_f2_feeder` verbatim. This is a genuinely well-factored unit; the structural DRY discipline is exemplary and most of the findings below are second-order.

## Score: 8.5/10 — strong, idiomatic, well-documented; held back by a shared latent determinism bug (`cublasSetStream` resets workspace) the comments actively misdescribe, two unguarded launch-dimension limits, and missing const/`[[nodiscard]]`/edge-guard polish.

## Findings

### (1) Correctness & bugs

**F1 [HIGH] — `cublasSetStream` in `run_f2_gemms_group` silently discards the determinism workspace (§12 violation; the comment claims the opposite).**
Location: `f2_blocks_kernel.cu`, `run_f2_gemms_group`, line 190 (`CUBLAS_CHECK(cublasSetStream(handle, stream));`), and the lines 191-193 comment.
The cuBLAS 13.3 docs state verbatim for `cublasSetStream`: *"Additionally this function unconditionally resets the cuBLAS library workspace back to the default workspace pool (see cublasSetWorkspace())."* ([cuBLAS 13.3 — cublasSetStream / cublasSetWorkspace](https://docs.nvidia.com/cuda/cublas/index.html)). The backend binds the determinism workspace exactly once, in the `CudaBackend` ctor (`cuda_backend.cu:51`, `cublasSetWorkspace(blas_.get(), ...)`), and never again. `run_f2_gemms_group` is then called once per bucket-chunk and *each call* issues `cublasSetStream`, which **resets the handle back to the default `cudaMalloc`/`cudaMallocAsync` workspace pool** before every group of GEMMs. The architecture makes the explicit workspace load-bearing: *"users can provide workspace via `cublasSetWorkspace()` to meet fixed-point emulation workspace requirements"* and reproducibility *"no longer holds when … fixed-point emulation is used"* unless that workspace is honored ([cuBLAS 13.3 — Fixed-point emulation / Reproducibility](https://docs.nvidia.com/cuda/cublas/index.html); architecture.md §12: *"have its workspace set for emulated-FP64 determinism"*). So in the EmulatedFp64 (default) path the GEMMs run on the auto-allocated pool, and run-to-run bit-stability is **not** guaranteed — exactly the §12 property the ctor tried to buy. The header comment (lines 61-63) and the `.cu` comment (lines 191-193) compound the bug by *asserting* the workspace is engaged once by the caller — they describe the intended invariant, but the actual code path destroys it on the very next `cublasSetStream`.
Why it matters: this is the §12 reproducibility/parity contract — the whole reason the workspace constant `kCublasWorkspaceBytes` exists (config.hpp:84-88). It is also a *shared* bug: the sibling `run_f2_gemms` (f2_block_kernel.cu:242) has the identical `cublasSetStream`-then-GEMM ordering, so M0 is affected too.
Fix: re-bind the workspace immediately after `cublasSetStream`, i.e. `CUBLAS_CHECK(cublasSetStream(handle, stream)); CUBLAS_CHECK(cublasSetWorkspace(handle, ws_ptr, ws_bytes));` — but that requires threading the workspace pointer/size into the wrapper (it currently isn't passed). Cleaner: hoist the `cublasSetStream` out of the per-call wrappers entirely and set the stream once alongside `cublasSetWorkspace` in the backend (since `stream_` is fixed for the whole `compute_f2_blocks`), so the wrapper never touches stream/workspace. Either way the invariant the comments claim must become true.
Severity: HIGH · Effort: M · before-M4.5? **yes** (parity/determinism is a §6 done-criterion and M4.5 layers multi-GPU parity on top of this).

**F2 [MED] — `gridDim.z = n_in_group` and `gridDim.y` can exceed the 65535 hardware limit with no guard or fallback.**
Location: `launch_gather_group` lines 176-178 (`grid.z = n_in_group`, `grid.y = grid_for(s_pad)`); `launch_assemble_blocks_group` lines 231-233 (`grid.z = n_in_group`).
On all current compute capabilities (incl. sm_120) the maximum y- and z-dimension of a grid is **65535**, while only the x-dimension goes to 2^31−1 ([CUDA C Programming Guide — Compute Capabilities / Technical Specifications: "Maximum y- or z-dimension of a grid of thread blocks = 65535"](https://docs.nvidia.com/cuda/cuda-c-programming-guide/index.html)). Both kernels put the batch (block) count directly on `gridDim.z`. The backend chunks each bucket to `max_blocks` by VRAM (cuda_backend.cu:241-243), and at P=768 the per-block footprint keeps `nb` far below 65535, so this does *not* currently fire — but it is an unguarded launch-dimension limit that depends on an *external* invariant in a different TU. If a future caller (smaller P, larger free VRAM, or the M5 out-of-core generalization the comments anticipate) drives `nb > 65535`, `gather_group_kernel<<<grid,…>>>` returns `cudaErrorInvalidConfiguration` — which `STEPPE_CUDA_CHECK_KERNEL()` would surface, but only as a runtime throw, not a graceful tiling. For `launch_gather_group`, `gridDim.y = grid_for(s_pad)` is also a y-axis: the largest bucket the spike saw was s_pad=2048 → ceil(2048/16)=128, well under 65535, so y is safe today but is the same latent shape.
Why it matters: §7 (launch-config helpers, fail-fast on bad config) and the capability-tier philosophy (the limit must be respected on *both* boxes; the 96 GB PRO-6000 can make `free_b` large enough to push `max_blocks` up). It is a correctness landmine masked by a current-data coincidence.
Fix: either (a) assert/document the precondition `n_in_group <= 65535` at the wrapper boundary (cheap, fail-fast, honest), or (b) better, fold the batch onto the x-axis instead of z (map `(i, c, k)` so the unbounded count rides `gridDim.x`), or loop the launch over z in ≤65535 chunks. At minimum a named constant `kMaxGridZ = 65535` and a precondition check.
Severity: MED · Effort: M · before-M4.5? no (not hit at M4 shapes), but document the precondition before M5.

**F3 [LOW] — `block_sizes[id]`/`block_offsets[id]` are read with no validity guard against a malformed `block_ids_in_group`.**
Location: `gather_group_kernel` lines 87-88, 96 (`block_sizes[id]`, `block_offsets[id]`).
`id = block_ids_in_group[k]` is trusted to be a valid `[0, n_block)` index and `block_offsets[id]+c` is trusted to land in `[0, M)`. These invariants hold by construction in the current backend (the host bucket scan), but the kernel is a public-within-`steppe_device` entry point with no documented precondition that `id` is in range or that `block_offsets[id] + sz <= M`. A bad `id` is an out-of-bounds device read (UB; compute-sanitizer would catch it but release would silently corrupt).
Why it matters: fail-fast / edge-case hardening (architecture.md §2). Low because the only caller is internal and correct.
Fix: document the precondition in the header (the `.cuh` doc-comment already describes the layout but not the in-range contract), and consider a debug-only `assert` on `id < n_block` is not possible without passing `n_block`; at minimum state the contract.
Severity: LOW · Effort: S · before-M4.5? no.

**F4 [LOW] — Verified-correct: the `R`-slab transpose indexing in `assemble_blocks_group_kernel` is right.**
Location: lines 149-152. I checked `sumsq_i = Rg(i,j)`, `sumsq_j = Rg(j,i)`, `hsum_i = Rg(P+i,j)`, `hsum_j = Rg(P+j,i)` against the column-major `[2P×P]` layout (lda=2P): element `(row, col)` at `row + col*2P`. `Rg[rSlab + si + sj*twoP]` = `(i, j)` ✓; `Rg[rSlab + sj + si*twoP]` = `(j, i)` ✓; `(Pp+si) + sj*twoP` = `(P+i, j)` ✓. This exactly mirrors the validated single-block `assemble_f2_kernel` (f2_block_kernel.cu:148-153) with the added `gSlab`/`rSlab`/`dstSlab` offsets. No bug — recorded as positive verification of the load-bearing indexing.

### (2) Edge cases & failure modes

**F5 [MED] — No defense against `s_pad == 0` / `n_in_group == 0` reaching the wrappers.**
Location: `launch_gather_group` (grid built from `grid_for(s_pad)` and `n_in_group`), `run_f2_gemms_group` (passes `s_pad` as the GEMM `k`), `launch_assemble_blocks_group`.
The backend guards `P<=0 || M<=0 || n_block<=0` at the top of `compute_f2_blocks` (cuda_backend.cu:130) and `max_blocks` is floored at 1, and `ceil_bucket` returns `>=1`, so `s_pad>=1` and `nb>=1` always hold *today*. But the wrappers themselves accept these as `int` with no contract. If `n_in_group==0`, `grid.z = 0` is an invalid launch config (a zero grid dimension), and `cublasGemmStridedBatchedEx` with `batchCount==0` is a degenerate call. A zero `s_pad` would make `Psp=0` and a `k=0` GEMM (no-op writing zeros only via `beta`). These are not reachable from the current sole caller, but the unit has no stated precondition.
Why it matters: edge-case/empty-input robustness (the review mandate's category 2). The single-responsibility wrappers should either no-op cleanly on empty or document the `>=1` precondition.
Fix: early-return when `n_in_group <= 0` (and document `s_pad >= 1`), mirroring the backend's own guard; or assert the precondition.
Severity: MED · Effort: S · before-M4.5? no.

**F6 [LOW] — `int twoP = 2 * P` in `run_f2_gemms_group` (line 197) is an `int` intermediate.**
Location: line 197 (`const int twoP = 2 * P;`), used as the GEMM `m` (line 219) and the lda for `dSg` / `dRg`.
`m`/lda are `int` parameters of `cublasGemmStridedBatchedEx`, so `twoP` *must* be `int` — that is correct. But the architecture states P can reach 4,266 (§12), giving `2*P = 8532`, comfortably within `int`. The risk only appears if P ever exceeds ~1.07e9, which is impossible for population count. The strides (`Psp`, `twoPsp`) are correctly cast to `long` first (lines 198-199, 206, 213, 221). Recorded as a verified non-issue, but worth a one-line `// P ≤ 4266 (§12) ⇒ 2P fits int` comment for the next reader, matching the sibling's `size_t indexing mandatory above P≈32k` density (f2_block_kernel.cu:131).
Severity: LOW · Effort: S · before-M4.5? no.

**F7 [LOW] — `block_offsets` is `long` but `gather_group_kernel` takes `const long* block_offsets`; `block_sizes` is `int`. Mixed widths are intentional but undocumented at the kernel.**
Location: kernel params lines 71-72; `block_offsets[id]` is `long` (offset into M, which can exceed 2^31), `block_sizes[id]` is `int` (a single block's SNP count, bounded by blgsize ≈ thousands). This is correct (M-scale offset vs small per-block size). The header doc-comment (`.cuh` lines 38-40) explains the layout but not *why* offsets are `long` and sizes `int`. Minor: a one-liner would prevent a future "make them consistent" mistake that would reintroduce 32-bit offset overflow at M > 2^31.
Severity: LOW · Effort: S · before-M4.5? no.

### (3) Numerical / precision vs §12

**F8 [HIGH-as-part-of-F1] — Emulated-FP64 determinism is not actually achieved (see F1).**
The numerics *design* is correct and faithful to §12: the GEMMs are precision-governed via the shared `f2_compute_type`, and the cancellation-prone numerator/divide stays NATIVE FP64 in `assemble_blocks_group_kernel` via the shared `assemble_f2_numerator`/`finalize_f2` (lines 154-160). The `2.0 * cross` is the true mathematical constant, not a magic number (consistent with f2_estimator.hpp:96). This is exactly right. The *only* numerical defect is that the workspace-reset (F1) means the emulated path is not run-to-run reproducible as §12 requires — a determinism defect, not a formula defect. Counted under F1, flagged here for the numerics category.

**F9 [LOW] — Per-block GEMM accumulation order differs from M0, by design and documented in the test, but not in this TU.**
Location: the whole grouped-GEMM design. The equivalence test (test_f2_blocks_equivalence.cu:255-257) notes the pad-to-pow2 width makes block 0's GEMM accumulate in a slightly different order than M0's exact-width GEMM (~1e-14). §12 explicitly marks native-FP64 GEMM accumulation order as `[UNCERTAIN]` and relies on the oracle diff, not source-order assumptions. The padding with zero columns is numerically inert in exact arithmetic (V=0 ⇒ zero contribution) and the FP perturbation is far below tolerance. No defect; noted because the `.cu` precision discussion (none beyond the assemble comment) could cite that padding is accuracy-neutral by the zero-fill argument — a half-line would close the loop for a reviewer worried about pad columns polluting the cancellation.
Severity: LOW · Effort: S · before-M4.5? no.

### (4) CUDA idioms / RAII / stream & async semantics / launch config / occupancy vs §7

**F10 [MED] — Stream is set inside the per-call wrapper, coupling stream management to the kernel wrapper and triggering F1.**
Location: `run_f2_gemms_group` line 190. Beyond the workspace-reset bug (F1), setting the stream inside the wrapper is a separation-of-concerns smell: the *backend* owns the stream (`stream_`, cuda_backend.cu:338) and the precision engagement is *already* hoisted to the backend (`engage_f2_precision` at cuda_backend.cu:214, per the explicit "engage ONCE" design). The stream binding should be hoisted the same way, so the wrapper is purely "issue these three GEMMs on the already-configured handle." Hoisting it also fixes F1 (one `cublasSetStream` + one `cublasSetWorkspace` together, before the loop).
Why it matters: §7 (stream/handle lifecycle), §2 (single responsibility) — the wrapper currently does handle *configuration* plus *issuance*.
Fix: move `cublasSetStream` to the backend next to `engage_f2_precision`+`cublasSetWorkspace`; drop the `stream` param's use for SetStream (the kernels still take `stream` for `<<<…,stream>>>`).
Severity: MED · Effort: M · before-M4.5? **yes** (same fix as F1).

**F11 [LOW] — `STEPPE_CUDA_CHECK_KERNEL()` after the gather/assemble launches is correct; no post-GEMM check after the strided-batched calls is also correct.**
Location: lines 182, 236 (kernel checks present); `run_f2_gemms_group` relies on `CUBLAS_CHECK` on each `cublasGemmStridedBatchedEx` (lines 203, 210, 218). The cuBLAS calls return a status synchronously checked by `CUBLAS_CHECK`; the launch-config check macro is for `<<<>>>` launches only. This matches the sibling and §7. Verified-correct, recorded as a positive.

**F12 [LOW] — Occupancy: 16×16 = 256-thread blocks for both kernels. For `gather_group_kernel` over `[P × s_pad]` and `assemble_blocks_group_kernel` over `[P × P]`, a 16×16 block is a reasonable, project-standard choice (`kCdivBlock`).**
Both kernels are memory-bound (a gather/scatter and a small elementwise assemble), so occupancy is bandwidth-limited, not block-shape-limited; 256 threads is fine and DRY-consistent with the single launch-geometry home. No change. One observation: the assemble kernel does 6 strided global loads (Gg, Vpairg, 4× Rg) per output element, with the Rg reads `Rg(j,i)` / `Rg(P+j,i)` being the *transpose* access — non-coalesced across the `j` (threadIdx.y) dimension. This is inherent to the symmetric formula (you need both triangles) and identical to the validated single-block kernel, so it is not a regression; flagged only as the one place a future perf pass (capability-tier "profiling" lever, TODO.md) would look. Not actionable now.
Severity: LOW (perf-note) · Effort: — · before-M4.5? no.

**F13 [LOW] — `<<<>>>` smem argument is `0` (no dynamic shared memory) — correct; no smem is needed for these gather/scatter kernels.** Verified-correct.

### (5) Magic numbers & hardcoded values vs §4 / ROADMAP §4

**F14 [LOW] — The literal `2` (in `2 * Pl`, `2 * Pp`, `2 * P`) is the true mathematical stacking factor for the `[2P × …]` S/R layout — a legitimate mathematical constant, not a magic number.**
Locations: lines 83, 140, 197, 198-199, 219, 221. ROADMAP §4 exempts "true mathematical constants." The `2` here is the height of the `[Qsq ; Hc]` stack (2 sub-blocks of P rows). This is the same `2` the sibling and the architecture S2 description use. Recorded as a verified non-issue — the unit has **no surviving magic numbers**; block edge comes from `kCdivBlock`, grid math from `grid_for`, bucket base from `kBlockGroupPadBase` (in the backend), compute type from the shared mapping. This is exactly the §4 standard met.

**F15 [LOW] — The `2P` stacking could be a named idea but probably should not be.** One could argue for `kSStackRows = 2` to name the stack height, but that would over-abstract a structural mathematical fact and *reduce* readability versus the `[2P × M]` notation used consistently throughout the codebase and architecture §5 S2. Considered and rejected — leave the `2` literal. (Listed to show the magic-number sweep was exhaustive, not skipped.)

### (6) Decomposition / single-responsibility / function size vs §2

**F16 [LOW] — Decomposition is excellent.** Three small, single-purpose kernels (each ~30-45 lines incl. comments) and three thin wrappers. Each wrapper does grid-math + launch + check, or set-config + 3 GEMMs. No function exceeds a screen. The only SoC nit is F10 (stream-set inside the GEMM wrapper). Otherwise this is a model of the §2 decomposition standard — the per-element numerics are delegated to the shared primitive, the index math is local, the launch geometry is the shared helper. No split needed.

### (7) Readability, naming, const-correctness, `[[nodiscard]]`/`noexcept`, comment density

**F17 [MED] — None of the `launch_*` / `run_*` declarations carry `[[nodiscard]]` — but more importantly, they are `void` and can throw; they should be documented `noexcept(false)` or the throwing contract stated.**
Location: header declarations lines 45-50, 64-68, 77-81; `.cu` definitions. The functions return `void`, so `[[nodiscard]]` is N/A (correctly). However the shared primitives in f2_estimator.hpp are all `[[nodiscard]] … noexcept` and the check.cuh accessors are `[[nodiscard]] … noexcept`, so the codebase clearly *does* annotate. These wrappers can throw (via `CUBLAS_CHECK`/`STEPPE_CUDA_CHECK` → `CublasError`/`CudaError`), which is the intended fail-fast behavior (architecture.md §7, §10) — but that throwing contract is *implicit*. Architecture §7 says "device functions are `noexcept`" — these are *host* launch wrappers, so they are correctly allowed to throw; but the asymmetry with the heavily-annotated neighbors deserves a one-line "throws CudaError/CublasError on failure" in each doc-comment so a caller wiring the public API's `steppe_status_t` translation (the §10 contract) knows the exception surface.
Why it matters: API contract clarity / const-correctness discipline (review category 7); the surrounding code annotates heavily so the silence here reads as an omission.
Fix: add "throws on CUDA/cuBLAS failure" to each header doc-comment.
Severity: MED · Effort: S · before-M4.5? no.

**F18 [LOW] — Const-correctness of pointer params is correct.** Inputs are `const double*` / `const int*` / `const long*`; outputs are non-const. `__restrict__` is applied to every kernel pointer param (lines 67-76, 127-133). The `run_f2_gemms_group` passes `dQg` as both A and B to the G GEMM (lines 203-207) — this is the standard syrk-via-gemm with two read-only aliased inputs and a distinct output `dGg`; cuBLAS permits aliasing read-only operands (no in-place restriction is violated since C ≠ A, C ≠ B). Verified-correct. No change.

**F19 [LOW] — Comment density is high and matches the surrounding files** — arguably *slightly* over-dense (the header and `.cu` repeat the same spike-verdict narrative, ~20 lines each). This is consistent with the project's documentation-heavy house style and the §13 "pin every parity caveat" mandate, so it is appropriate, not excessive. The one duplication worth trimming: the "WHY GROUPED" measured-timing block appears in both `f2_blocks_kernel.cuh` (lines 9-19) and `f2_blocks_kernel.cu` (lines 9-17) nearly verbatim. Single-home it (keep in the `.cuh`, reference from the `.cu`) per §2 DRY-for-prose. Very minor.
Severity: LOW · Effort: S · before-M4.5? no.

**F20 [LOW] — Naming: `Psp`/`twoPsp`/`dQidx`/`dSqsq`/`dShc`/`gSlab`/`rSlab`/`dstSlab` are terse but each is introduced with a comment; consistent with the sibling.** `dSqsq` (destination Σp² row) reads oddly next to `sSqsq` (source); acceptable given the comments. No change required.

### (8) Performance

**F21 [LOW] — `cudaStreamSynchronize` after every chunk (in the backend, not this TU) serializes gather→GEMM→assemble→free with no overlap.** This is a backend-orchestration property, not this unit's, but it is the throughput ceiling for the per-block path: the sync is needed for the one-chunk-resident VRAM discipline, and the per-call `cublasSetStream` (F1/F10) would in any case prevent multi-stream overlap (and §12 forbids concurrent statistic streams). For *this* TU the relevant perf observations are: (a) the gather kernel reads `dQ/dV/dS` with a stride-`Pl` column access that is coalesced across the `i` (threadIdx.x) row dimension — good; (b) the assemble kernel's transpose Rg reads (F12) are the one non-coalesced access. Neither is actionable at M4; both are deferred to the capability-tier "profiling" lever (TODO.md — `ncu --set full` on the PRO-6000). No change now.

**F22 [LOW] — `cublasGemmStridedBatchedEx` is the right primitive (vs the pointer-array `cublasGemmBatchedEx`) for these uniform-shape per-bucket batches.** §11.3 says "strided-batched GEMM/SYRK for homogeneous shapes, pointer-array batched only for non-uniform shapes" — the grouped design makes every slab in a bucket exactly `s_pad` wide, so the shapes *are* uniform and strided-batched is correct (and avoids the pointer-array indirection). Verified-correct, recorded as a positive. (Note `strideX` params are `long long int` per the cuBLAS signature ([cuBLAS strided-batched signature](https://developer.nvidia.com/blog/cublas-strided-batched-matrix-multiply/)); the code passes `long`, which on the LP64 Linux target is 64-bit and converts cleanly — correct, but see F23.)

**F23 [LOW] — `batchCount` is `int`; the code passes `n_in_group` (int) — correct and within the int domain (bounded by F2's 65535 grid-z guard anyway).** `cublasGemmStridedBatchedEx`'s `batchCount` parameter is `int` ([signature](https://developer.nvidia.com/blog/cublas-strided-batched-matrix-multiply/)). No overflow risk for `n_in_group`. Verified-correct.

### (9) Layering / API / ABI vs §4

**F24 [LOW] — Layering is clean.** The header is correctly PRIVATE (CUDA types exposed only to `steppe_device`), includes only `<cublas_v2.h>` + the CUDA-free `steppe/config.hpp`. The `.cu` includes the shared core primitive (`core/internal/f2_estimator.hpp`) and the sibling device header for the shared policy — both within the device layer's allowed dependency set (device→core internal primitive is the documented `__host__ __device__` sharing seam, architecture.md §13). No `core`/`api`/`io` reach in. No ABI concern (internal TU). Verified-correct.

**F25 [LOW] — The `run_f2_gemms_group` signature takes `const Precision&` but only uses `f2_compute_type(precision)`; it does *not* engage precision (correct — that's hoisted).** The doc-comment (header lines 60-63) correctly states the handle's math mode is engaged once by the caller and this routine sets only the per-call compute type. Good single-source-of-policy discipline. The only residual: passing the full `Precision&` when only the compute type is consumed is slightly wider than needed, but it keeps the signature parallel with `run_f2_gemms` (sibling) and future-proofs if a per-call knob is added. Acceptable; not a finding.

### (10) Testability vs §13

**F26 [MED] — The three wrappers have NO direct unit test; they are covered only transitively through `compute_f2_blocks` in the reference-equivalence test.**
The shared per-element numerics (`assemble_f2_numerator`, etc.) are CPU-unit-testable and exercised. `test_f2_blocks_equivalence.cu` is a strong *integration* gate (oracle diff on real AADR, Vpair recount, symmetry, single-block==M0). But the *gather* kernel's pad-column-zero contract and the *scatter* index (`i + P·j + P·P·id`) are only verified end-to-end. There is no test that directly asserts: (a) pad columns (`c >= sz`) are written as exactly 0.0 in Qg/Vg/Sg; (b) gathered real columns equal the feeder columns; (c) the scatter lands block `id`'s slab at the right `[P×P]` offset for a non-trivial `block_ids_in_group` permutation. The single-block==M0 test only exercises `block_ids_in_group = {0}` (identity), so a bug in the *scatter-to-arbitrary-id* path (e.g. transposed slab stride) could pass the n_block=1 case and only fail with ≥2 reordered blocks — which the real-AADR test would catch, but not pinpoint.
Why it matters: §13 testability; the gather/scatter index math is the M4-specific code (the GEMM and assemble formula are shared/validated), so it deserves a focused unit test rather than only integration coverage.
Fix: a small device unit test with a synthetic 2-3 block, 2-population layout asserting (a)-(c) above against hand-computed expected slabs — runnable on the box, no real data.
Severity: MED · Effort: M · before-M4.5? no (the integration gate suffices for M4 sign-off), but add before M5 generalizes the path.

**F27 [LOW] — The wrappers are not independently invocable without a `cublasHandle_t` / device buffers, which is inherent.** N/A as a defect — device wrappers are necessarily device-tested. The shared-primitive split (F16) is exactly what makes the *numerics* CPU-testable, satisfying the spirit of §13.

### (11) Capability tiers (PRO-6000-capable vs budget-5090)

**F28 [LOW] — This unit is on the pure-compute path; it touches the capability tiers only via the VRAM-budget chunking, which lives in the backend, not here.**
Per TODO.md the capability tiers are: P2P combine (M4.5), GDS ingest (M5/M7), profiling counters, nvCOMP. **None of these are in this TU** — the three kernels are on-device arithmetic with no host I/O, no peer transfer, no compression. The one tier-adjacent property is VRAM: the PRO-6000's 96 GB raises `max_blocks` (cuda_backend.cu:241) and thus `n_in_group`, which is precisely what makes the F2 grid-z=65535 limit reachable on the *capable* box but not the budget one. So the capability-tier contribution this unit *should* make is the F2 guard: the launch must respect 65535 on both boxes, with an explicit fallback (tile the z-axis) rather than a hard failure that only manifests on the high-VRAM box. That is the "build for capable, degrade with an explicit logged tag" mandate applied here — a per-launch precondition + a tiling fallback, tagged if it engages. Beyond F2, there is no further capability-tier hook this TU needs (it has no observability or data-movement lever).

## Considered & rejected

- **"`Q_all[sQ]` in the gather reads the *masked* Q, but pad columns should use the *raw* Q."** — Rejected. The gather copies the feeder *outputs* (`dQ` masked, `dV` mask, `dS` stacked), which is exactly the contract (the feeder already applied masking and zero-fill); pad columns are written 0 deliberately. The `.cuh` doc (lines 33-44) and the kernel comment (lines 57-65) describe this correctly. Verified consistent with the backend wiring (cuda_backend.cu:260, passes `dQ/dV/dS`).
- **"`finalize_f2` divides by `vpair` which is a GEMM output (Vg·Vgᵀ) — could be non-integer under emulation and break the `>0` guard."** — Rejected. `Vg` is exactly 0.0/1.0 (validity mask), so `Vpair = Σ 1·1` is an exact integer count; even under emulated FP64 the products are 0 or 1 and sums of small integers are exact in FP64 well past any realistic SNP count (2^53). The `>0.0` guard (f2_estimator.hpp:110-111) is the AT2 `Vpair==0⇒0` branch and is robust. The equivalence test independently recounts Vpair and asserts exact integer equality (test lines 137-165), which would catch any emulation perturbation. Note: this *also* means the Vpair GEMM does not actually need emulated precision (it's exact-integer), but running it under the same compute type as G/R is harmless and keeps the batch homogeneous — not worth a separate native call.
- **"Passing the same `dQg` pointer as both A and B to `cublasGemmStridedBatchedEx` is illegal aliasing."** — Rejected. A and B are read-only inputs; cuBLAS does not prohibit aliasing read-only operands, and C (`dGg`) is distinct. The docs document no in-place restriction violated here ([cuBLAS GEMM Ex](https://docs.nvidia.com/cuda/cublas/index.html)). This is the standard syrk-as-gemm idiom and matches the validated single-block path.
- **"`grid.x` from `grid_for(P)` could overflow `gridDim.x`."** — Rejected. `gridDim.x` max is 2^31−1; `grid_for(P) = ceil(P/16)` for P ≤ 4266 is ≤ 267. No risk. (The real grid limit is the y/z 65535 cap — F2.)
- **"`size_t` vs `long` mixing between the gather kernel (uses `long` indices) and the assemble kernel (uses `size_t`) is an inconsistency bug."** — Rejected. Both are 64-bit on the LP64 target; the gather uses signed `long` because it adds a signed `block_offsets` value, the assemble uses unsigned `size_t` for pure positive slab offsets — each is locally appropriate and matches its sibling's choice (the single-block assemble uses `size_t` too, f2_block_kernel.cu:143). Not a bug, though a project-wide "device index type" convention (e.g. always `cuda::std::int64_t`) would be tidier (a cross-cutting nit, not specific to this TU).
- **"The gather kernel should fuse into the feeder to avoid the extra global round-trip."** — Rejected. The feeder is block-agnostic (one sweep over all M SNPs) and deliberately reused verbatim (architecture.md §2 DRY, M4 comment lines 28-32); the gather is per-bucket and must run after bucketing. Fusing would re-introduce the second feeder the design explicitly avoids and couple the feeder to the bucket layout. The round-trip is bandwidth on the already-resident `dQ/dV/dS` and is part of the measured 7.2×/8.9× win.
- **"`cudaMemGetInfo`-based chunking is non-deterministic across machines and could make the f2 result depend on free VRAM."** — Rejected (and not this TU). The chunk boundary changes GEMM *batch grouping* but each block's f2 is computed identically regardless of which chunk it lands in (the zero-pad makes width-grouping inert; cross-chunk there is no reduction — each block writes its own slab). The accumulation order *within* a block is fixed by `s_pad` (bucket width), not by chunk size. So parity is chunk-count-independent. This is a backend concern anyway.

## Good patterns to keep

- **DRY across the M0 and M4 paths is exemplary:** the feeder, the precision engagement (`engage_f2_precision`), the compute-type mapping (`f2_compute_type`), the numerator/finalize primitives, and the launch-grid helpers are all single-sourced and reused — no second copy of the formula, the policy, or the grid math. This is the §2/§8 single-home standard genuinely achieved.
- **Shared `__host__ __device__` numerics seam:** the cancellation-prone assemble calls the *same* `assemble_f2_numerator`/`finalize_f2` the CPU oracle calls, so the formula cannot diverge (§13) — and it stays native FP64 in every precision mode (§12), correctly localizing the cancellation.
- **Correct primitive choice:** `cublasGemmStridedBatchedEx` (not pointer-array batched) for the uniform-shape per-bucket batches, with strides cast to 64-bit before multiplication — exactly the §11.3 guidance.
- **Zero magic numbers:** every constant is named (`kCdivBlock`, `kBlockGroupPadBase`, the compute-type mapping) or a true mathematical constant (the `2` stack factor) — the ROADMAP §4 bar is met.
- **Thin, single-responsibility kernels behind narrow `void launch_*` wrappers** with `STEPPE_CUDA_CHECK_KERNEL()` post-launch checks and `CUBLAS_CHECK` on every cuBLAS call (§7), `__restrict__` on all kernel pointers, and high, parity-caveat-pinning comment density that matches the house style and the §13 mandate.
- **The reference-equivalence test design** (oracle diff + independent Vpair recount + symmetry + single-block==M0 + explicit emu-engaged detection) is a model integration gate; the unit-level gather/scatter test (F26) is the one addition that would complete the §13 coverage.
