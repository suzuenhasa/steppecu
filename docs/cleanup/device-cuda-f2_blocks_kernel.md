## Role & layering

This TU owns the three on-device primitives that bracket the M4 grouped GEMMs:

1. `launch_gather_group` → `gather_group_kernel` — gather one power-of-2 size bucket's SNP columns out of the block-contiguous feeder outputs (`dQ/dV/dS` over all M SNPs) into a padded strided-batched slab layout; pad columns (`c ≥ sz`) zero-filled in Q/V/S so the masked GEMM is identical to the unpadded one (V=0 ⇒ zero contribution).
2. `run_f2_gemms_group` — the three f2 GEMMs as `cublasGemmStridedBatchedEx` over the bucket's slabs (`G = Qg·Qgᵀ`, `Vpair = Vg·Vgᵀ`, `R = Sg·Vgᵀ`), precision-governed by the SHARED `f2_compute_type`.
3. `launch_assemble_blocks_group` → `assemble_blocks_group_kernel` — fused numerator+divide (NATIVE FP64, via the SHARED `core::assemble_f2_numerator`/`finalize_f2`) scattered into the resident `[P×P×n_block]` f2 + Vpair tensors.

**Layering is correct and load-bearing.** The header includes `<cublas_v2.h>` and names `cublasHandle_t`/`cudaStream_t`, so it is correctly self-described as PRIVATE to `steppe_device` (architecture.md §4) — the device-internal seam, not the CUDA-free `ComputeBackend` seam (`device/backend.hpp`). Kernel bodies and `<<<>>>` live only in the `.cu`; the host calls narrow `void launch_* / run_*` wrappers (architecture.md §7). It reuses the SHARED `core::assemble_f2_numerator`/`finalize_f2` numerics (so the CPU oracle and GPU cannot diverge on the cancellation formula, §13), the SHARED `core::grid_for` launch helper, and the SHARED `engage_f2_precision`/`f2_compute_type` precision policy from the sibling TU — verified: the `.cu` includes `f2_block_kernel.cuh` (line 45) and calls `f2_compute_type` (line 194). The feeder is *not* duplicated — M4 reuses `launch_f2_feeder` verbatim (`cuda_backend.cu:203`). This is a genuinely well-factored unit; the structural DRY discipline is exemplary and most findings below are second-order or shared with the siblings.

## Score: 8/10

Strong, idiomatic, well-documented, with a correct happy path (the M4 reference-equivalence test is green on real AADR vs the CPU oracle). Held below the 9.5 bar by:

- **F1 (HIGH)** — the *shared* `cublasSetStream` → workspace-reset determinism bug: `run_f2_gemms_group` resets the §12 emulated-FP64 workspace to the default pool **on every chunk**, voiding the run-to-run bit-stability the ctor's `cublasSetWorkspace` is supposed to buy. The header/`.cu` comments *assert the opposite* (they claim the workspace is engaged once and honored), so they actively mis-describe the code. Confirmed verbatim against the cuBLAS docs.
- **F2 (MED→HIGH-latent)** — `gridDim.z = n_in_group` in both kernels is unclamped against the 65 535 z-axis hardware limit. Not hit at M4 shapes, but reachable on the high-VRAM PRO-6000 capability tier (96 GB ⇒ larger `max_blocks` ⇒ larger `nb`), which is exactly the "build for capable, degrade with a logged tag" mandate this unit must honor.
- A set of MED/LOW edge-guard, contract-doc, and testability gaps.

The draft scored this **8.5**; I am lowering to **8.0**. Rationale: the draft under-counted F2's severity (it is the budget-vs-capable launch-correctness landmine the capability-tier section explicitly calls out, and the sibling `cuda_backend.md` rates the same axis HIGH), and it missed two real findings (the math-mode/workspace *asymmetry* under the per-chunk `cublasSetStream` — N1 — and the unguarded `dGg`/`dRg` slab-stride `int`→`long` interplay being fine but the *output* GEMM stride being passed as bare `static_cast<long>(P)*P` rather than the named `Psp`/`twoPsp` style — N3 readability). These do not change the broad shape (the unit is good), but they keep it from the 8.5 the draft claimed.

## Findings

### (1) Correctness & bugs

**F1 [HIGH · M · before-M4.5: YES] `cublasSetStream` in `run_f2_gemms_group` silently discards the §12 determinism workspace — and the comments claim the opposite. — CONFIRMED verbatim.**
Location: `f2_blocks_kernel.cu`, `run_f2_gemms_group`, line 190 (`CUBLAS_CHECK(cublasSetStream(handle, stream));`); the misdescribing comments at `.cu` lines 191-193 and `.cuh` lines 61-63.
The `CudaBackend` ctor binds the determinism workspace exactly once (`cuda_backend.cu:51-52`, `cublasSetWorkspace(blas_.get(), workspace_.data(), workspace_.bytes())`, `kCublasWorkspaceBytes = 64 MiB`) and never again. **Verified verbatim this pass:** the cuBLAS 13.3 docs state for `cublasSetStream` — *"Additionally this function unconditionally resets the cuBLAS library workspace back to the default workspace pool"* ([cuBLAS 13.3](https://docs.nvidia.com/cuda/cublas/index.html)). `run_f2_gemms_group` is called once per bucket-chunk (`cuda_backend.cu:263`, inside the `for (start …)` loop), and *each call* issues `cublasSetStream`, which **resets the handle back to the default `cudaMalloc`/`cudaMallocAsync` workspace pool before every chunk's three GEMMs**. config.hpp:84-88 makes the explicit workspace load-bearing precisely *"for run-to-run reproducibility of emulated FP64"*, and architecture.md §12 lists *"an explicit `cublasSetWorkspace` workspace for the emulated-FP64 path"* as a hard determinism constraint. So in the default `EmulatedFp64` path the GEMMs run on the auto-allocated pool, and run-to-run bit-stability is **not** in force — the exact §12 property the ctor tried to buy.
Adversarial nuances re-checked:
- **Is the reset a no-op because `stream_ == nullptr` (same default stream)?** No — the docs say the reset is *unconditional*, independent of whether the stream actually changes (`cuda_backend.cu:338`, `stream_ = nullptr`, confirmed).
- **Does the ctor's `CublasHandle blas_{stream_}` even bind a stream?** No — `CublasHandle`'s ctor calls `cublasSetStream` only `if (s)` (architecture.md §7 reference shape; `handles.hpp`), and `stream_` is null, so the *first* `cublasSetStream` on the handle is the one in `run_f2_gemms_group` (or the sibling `run_f2_gemms`), which is the one that resets the just-bound workspace.
- **Shared bug:** the sibling `run_f2_gemms` (`f2_block_kernel.cu:242`) has the identical ordering, so M0 is affected too; the cross-file review `cuda_backend.md` rates this F1 HIGH.
Why it matters: §12 reproducibility/parity is a §6 done-criterion, and M4.5 layers multi-GPU parity directly on top of single-GPU bit-stability. The whole reason `kCublasWorkspaceBytes` exists is defeated. **Worse here than in M0:** the reset fires *per chunk* (M0 issues exactly three GEMMs in one `run_f2_gemms`; the M4 path issues a chunk's GEMMs after each reset), so a large bucket processed in many chunks resets repeatedly.
Fix: hoist the `cublasSetStream` out of the per-call wrapper entirely — bind the stream once in the backend alongside `engage_f2_precision`+`cublasSetWorkspace` (since `stream_` is fixed for the whole `compute_f2_blocks`), so the wrapper never touches stream/workspace; the kernels still take `stream` for `<<<…,stream>>>`. If a per-call `cublasSetStream` is ever genuinely required, pair it with an immediate `cublasSetWorkspace(handle, ws_ptr, ws_bytes)` re-bind (requires threading the workspace pointer/size into the wrapper). Either way, the header/`.cu` comments must be rewritten to describe the *true* invariant. This is the M4.5-blocking resource-ownership theme (TODO §A theme 1) and converges with F10.

**F2 [MED, latent-HIGH · M · before-M4.5: no, but document before M5] `gridDim.z = n_in_group` exceeds the 65 535 z-axis limit on the capable tier with no guard or tiling fallback. — CONFIRMED.**
Location: `launch_gather_group` line 178 (`grid.z = static_cast<unsigned>(n_in_group)`); `launch_assemble_blocks_group` line 233 (same).
**Verified this pass:** the maximum y- or z-dimension of a grid is **65 535** on all compute capabilities including Blackwell, while only x goes to 2³¹−1 ([CUDA C++ Programming Guide, Compute Capabilities](https://docs.nvidia.com/cuda/cuda-c-programming-guide/index.html)). Both kernels put the batch (block) count `nb` directly on `gridDim.z`. The backend chunks each bucket to `max_blocks` by VRAM (`cuda_backend.cu:241-243`); at P=768 the per-block footprint keeps `nb` far below 65 535, so this does *not* currently fire. But it is an unguarded launch-dimension limit that depends on an *external* invariant in a different TU, and the capability-tier section is explicit that **the PRO-6000's 96 GB raises `max_blocks` and thus `nb`** (TODO box-role split: single-GPU P_max ~3,331 @96 GB vs ~1,923 @32 GB; the chunk budget `0.80·free_b` is far larger). For a small-P / many-small-blocks regime on the capable box, `nb > 65 535` is reachable; the launch then returns `cudaErrorInvalidConfiguration`, which `STEPPE_CUDA_CHECK_KERNEL()` surfaces as a runtime throw — a hard failure that manifests *only on the high-VRAM box*, the exact "silent on budget, breaks on capable" trap the tier mandate forbids. (The gather kernel's `gridDim.y = grid_for(s_pad)` is also a 65 535-capped axis: the spike's largest bucket was s_pad=2048 → 128, safe today, same latent shape.)
Why it matters: §7 launch-config correctness + §2 fail-fast + the capability-tier "degrade with an explicit logged tag, never a hard failure on the capable box" contract (TODO). TODO backlog B literally lists "clamp `grid.z` to 65535." The sibling `cuda_backend.md` rates the same `grid.z = n_in_group` axis HIGH (latent silent launch failure on an unvalidated axis).
Fix: (a) cheapest — assert/document the precondition `n_in_group <= 65535` at the wrapper boundary plus a named `kMaxGridZ = 65535`; (b) better — tile the launch over z in ≤65 535 chunks (the batch axis already tiles by chunk in the backend, so this is the same idiom one level down), or fold the batch onto `gridDim.x` (map `(i, c, k)` so the unbounded count rides the 2³¹ axis). A tiling fallback should emit the capability-tagged log line when it engages.

**F3 [LOW · S · before-M4.5: no] `block_sizes[id]`/`block_offsets[id]` read with no in-range guard on a trusted `id`; the kernel states no precondition.**
Location: `gather_group_kernel` lines 87-88, 96 (`block_sizes[id]`, `block_offsets[id]` with `id = block_ids_in_group[k]`).
`id` is trusted to be in `[0, n_block)` and `block_offsets[id] + sz` to land in `[0, M)`. These invariants hold by construction (the host bucket scan `cuda_backend.cu:160-167` only ever pushes `b ∈ [0, n_block)`), but the kernel is a within-`steppe_device` entry point with no documented contract, and a malformed `block_ids_in_group` would be an out-of-bounds device read (UB; compute-sanitizer catches it, release would silently corrupt). The `.cuh` doc-comment describes the layout (lines 38-40) but not the in-range contract. Low because the only caller is internal and correct, and `n_block` is not passed in so a device-side assert is not even possible without a signature change.
Fix: state the `id ∈ [0,n_block)` and `block_offsets[id] + block_sizes[id] ≤ M` precondition in the `.cuh` doc-comment.

**F4 [verified-correct] The `R`-slab transpose indexing in `assemble_blocks_group_kernel` is right.**
Location: lines 149-152. Re-traced against the column-major `[2P×P]` slab (lda=2P): element `(row,col)` at `rSlab + row + col·2P`. `Rg[rSlab + si + sj·twoP]` = `R(i,j)` = `sumsq_i` ✓; `Rg[rSlab + sj + si·twoP]` = `R(j,i)` = `sumsq_j` ✓; `(Pp+si) + sj·twoP` = `R(P+i,j)` = `hsum_i` ✓; `(Pp+sj) + si·twoP` = `R(P+j,i)` = `hsum_j` ✓. Feeds `assemble_f2_numerator(sumsq_i,sumsq_j,Gij,hsum_i,hsum_j)` = `sumsq_i + sumsq_j − 2·cross − hsum_i − hsum_j` (f2_estimator.hpp:97-101), matching architecture.md §5 S2 line 221 and the validated single-block `assemble_f2_kernel` (f2_block_kernel.cu) with the added `gSlab`/`rSlab`/`dstSlab` offsets. No bug — recorded as positive verification of the load-bearing indexing.

**F5 [verified-correct, NEW positive] The gather kernel writes every row of the `2P`-tall `Sg` slab.**
Location: `gather_group_kernel` — grid.x = `grid_for(P)` (covers `i ∈ [0,P)`), yet `Sg` is `[2P × s_pad × n]`. I confirmed each thread `(i,c,k)` writes BOTH the Qsq row (`dSqsq` at row `i`) and the Hc row (`dShc` at row `P+i`), so the union over `i ∈ [0,P)` covers all `2P` rows with no gap and no double-write. Correct and complete; recorded because a "grid only covers P but slab is 2P" reader could wrongly suspect an uninitialized-row bug.

### (2) Edge cases & failure modes

**F6 [MED · S · before-M4.5: no] No defense against `n_in_group == 0` / `s_pad == 0` reaching the wrappers.**
Location: `launch_gather_group` (grid from `grid_for(s_pad)` and `n_in_group`), `run_f2_gemms_group` (passes `s_pad` as GEMM `k`, `n_in_group` as `batchCount`), `launch_assemble_blocks_group`.
The backend guards `P<=0||M<=0||n_block<=0` (`cuda_backend.cu:130`), `ceil_bucket` returns `≥1` so `s_pad≥1`, and a bucket always has `≥1` block so `nb≥1` — all hold *today*. But the wrappers accept these as bare `int` with no contract. `n_in_group==0` makes `grid.z=0`, an invalid launch config (a zero grid dimension is rejected with `cudaErrorInvalidConfiguration`); `batchCount==0` is a degenerate `cublasGemmStridedBatchedEx`; `s_pad==0` makes `Psp=0` and a `k=0` GEMM (beta-only). None reachable from the sole caller, but the unit states no precondition.
Why it matters: edge-case/empty-input robustness (review category 2) + consistency with the backend's own guards. M4.5 multi-GPU sharding can hand a device an empty SNP shard ⇒ an empty bucket.
Fix: early-return on `n_in_group <= 0` (and document `s_pad >= 1`) in each wrapper, mirroring the backend's guard; or assert the precondition.

**F7 [LOW · S · before-M4.5: no] `const int twoP = 2 * P` (line 197) is correct but undocumented as an `int`-by-necessity choice.**
Location: line 197, used as GEMM `m` (line 219) and lda for `dSg`/`dRg`. cuBLAS `m`/lda are `int`, so `twoP` *must* be `int` — correct. architecture.md §12 caps P at 4,266 ⇒ `2P ≤ 8,532` ≪ INT_MAX. The strides (`Psp`, `twoPsp`) are correctly widened to `long` first (lines 198-199, 206, 213, 221). But the sibling `f2_block_kernel.cu` keeps the same `int twoP` precisely because its `assemble_f2_kernel` flat index needs `size_t` above P≈32k — and this TU's `assemble_blocks_group_kernel` *does* use `size_t` for the flat index (lines 139-160). A one-line "P ≤ 4266 (§12) ⇒ 2P fits cuBLAS int m/lda; the [2P×P] flat index uses size_t in assemble_blocks_group_kernel" would prevent a future contributor from "optimizing" the kernel's `size_t` to `int`. Matches the sibling's E-2 finding.

**F8 [LOW · S · before-M4.5: no] Mixed `long block_offsets` / `int block_sizes` widths are correct but the *why* is undocumented at the kernel.**
Location: kernel params lines 71-72. `block_offsets[id]` is `long` (an offset into M, which can exceed 2³¹); `block_sizes[id]` is `int` (a single block's SNP count, bounded by `blgsize` ≈ thousands). This is correct (M-scale offset vs small per-block size) and matches the backend's `std::vector<long> block_offsets` / `out.block_sizes` (`int`). The `.cuh` doc explains the layout but not the width rationale; a one-liner prevents a future "make them consistent" change that would reintroduce 32-bit offset overflow at M > 2³¹.

### (3) Numerical / precision vs §12

**F9 [HIGH, counted under F1] Emulated-FP64 run-to-run determinism is not actually achieved (consequence of F1).**
The numerics *design* is faithful to §12: the three GEMMs are precision-governed via the shared `f2_compute_type` (line 194), and the catastrophic-cancellation numerator/divide stays NATIVE FP64 in `assemble_blocks_group_kernel` via the shared `assemble_f2_numerator`/`finalize_f2` (lines 154-160) with no precision branch — exactly §12's "the small `O(n_pop²)` numerator/divide stays native FP64." The `2.0 * cross` is the true mathematical constant (a²−2ab+b²), not a magic number (consistent with f2_estimator.hpp:96). The *only* numerical defect is that the per-chunk workspace reset (F1) means the emulated path is not run-to-run bit-reproducible as §12 requires — a determinism defect, not a formula or accuracy defect. The header/`.cu` "emulated-FP64 determinism" claims (`.cuh` 61-63, `.cu` 191-193) are therefore aspirational on the as-written code until F1 is fixed. Counted under F1; flagged here for the numerics category.

**F10-num [LOW · S · before-M4.5: no] Pad columns are accuracy-neutral by the zero-fill argument — worth a half-line.**
Location: the whole grouped-GEMM design. The pad-to-pow2 width makes block 0's GEMM accumulate in a slightly different *order* than M0's exact-width GEMM (~1e-14, far below the M0-vs-oracle ~1.1e-11; the equivalence test pins `kSingleBlockF2Tol`, lines 255-268). §12 marks native-FP64 GEMM accumulation order `[UNCERTAIN]` and relies on the oracle diff, not source-order. The padding with zero columns is numerically *inert in exact arithmetic* (V=0 ⇒ zero rows/cols ⇒ zero contribution), so it perturbs only the FP reduction order, not the value. No defect; the `.cu` could add a half-line that pad columns are accuracy-neutral by the zero-fill argument, to close the loop for a reviewer worried about pad columns polluting the cancellation. (The test already documents this; the source does not.)

### (4) CUDA idioms / RAII / stream & async / launch config / occupancy vs §7

**F11 [MED · M · before-M4.5: YES] Stream is set inside the per-call wrapper — a separation-of-concerns smell and the mechanism of F1.**
Location: `run_f2_gemms_group` line 190. Beyond the workspace-reset bug (F1), setting the stream inside the per-chunk wrapper couples handle *configuration* to kernel *issuance*. The backend owns the stream (`stream_`, `cuda_backend.cu:338`) and the precision engagement is *already* hoisted to the backend (`engage_f2_precision` at `cuda_backend.cu:214`, per the explicit "engage ONCE" design that this wrapper correctly relies on — it does *not* re-call `engage_f2_precision`, unlike the sibling `run_f2_gemms:243`). The stream binding should be hoisted the same way, so the wrapper is purely "issue these three GEMMs on the already-configured handle." Hoisting it fixes F1 (one `cublasSetStream` + one `cublasSetWorkspace` together, before the loop). architecture.md §7: handles are created once and bound once, never per-iteration.
Fix: move `cublasSetStream` to the backend next to `engage_f2_precision`+`cublasSetWorkspace`; the wrapper keeps only the `<<<…,stream>>>`-equivalent (the GEMMs already inherit the handle's stream). Same fix as F1.

**F12 [verified-correct] Error-check coverage is complete and idiomatic.**
`STEPPE_CUDA_CHECK_KERNEL()` follows each `<<<>>>` launch (lines 182, 236) — the §7 idiom (sync `cudaGetLastError` for bad-config + debug-only `cudaDeviceSynchronize`). Every `cublasGemmStridedBatchedEx` and the `cublasSetStream` are wrapped in `CUBLAS_CHECK` (lines 190, 203, 210, 218). No post-GEMM `STEPPE_CUDA_CHECK_KERNEL()` is needed (that macro is for `<<<>>>` launches; cuBLAS returns a status synchronously checked by `CUBLAS_CHECK`). Matches the sibling and §7/§8 single-home. Recorded as a positive.

**F13 [LOW, perf-note] Occupancy: 16×16 = 256-thread blocks; the assemble kernel has inherent transpose reads.**
Both kernels use `dim3(kCdivBlock, kCdivBlock)` (256 threads), DRY-consistent with the single launch-geometry home and reasonable for these bandwidth-bound (gather) / tiny (assemble) kernels — block shape is second-order. One observation: `assemble_blocks_group_kernel` does 6 strided global loads per output element (Gg, Vpairg, 4× Rg), of which the `Rg(j,i)`/`Rg(P+j,i)` reads are the *transpose* access — non-coalesced across the `j` (threadIdx.y) dimension. This is inherent to the symmetric formula (you need both triangles) and identical to the validated single-block kernel, so it is not a regression; it is the one place a future perf pass (the capability-tier "profiling" lever — `ncu --set full` on the PRO-6000, TODO) would look. Not actionable now. The gather kernel reads `Q_all/V_all/S_all` with a stride-`Pl` column access coalesced across `i` (threadIdx.x) — good. `<<<>>>` smem arg is `0` — correct, no dynamic shared memory needed.

### (5) Magic numbers & hardcoded values vs §4 / ROADMAP §4

**F14 [verified-correct] No surviving magic numbers; the literal `2` is the true mathematical stacking factor.**
Locations of `2`: lines 83 (`2 * Pl`), 140 (`2 * Pp`), 197 (`2 * P`), 198-199/206/213/221 (slab strides). ROADMAP §4 exempts true mathematical constants; the `2` is the height of the `[Qsq ; Hc]` stack (`S` is `[2P × M]`, architecture.md §5 S2), identical to the sibling and f2_estimator.hpp:96's a²−2ab+b² `2`. Block edge from `kCdivBlock`; grid math from `core::grid_for`; bucket base from `kBlockGroupPadBase` (backend); compute type from the shared `f2_compute_type`. This is exactly the §4 / ROADMAP §4 standard met. (Considered `kSStackRows = 2` for the stack height — rejected: it would over-abstract a structural mathematical fact and reduce readability vs the `[2P × …]` notation used consistently throughout the codebase and architecture §5 S2.)

### (6) Decomposition / single-responsibility / function size vs §2

Three small single-purpose kernels (~30-45 lines each incl. comments) and three thin wrappers (grid-math + launch + check, or set-config + 3 GEMMs). ### (7) Readability, naming, const-correctness, `[[nodiscard]]`/`noexcept`, comments

**F16 [MED · S · before-M4.5: no] The launch-wrapper doc-comments do not state the throwing contract.**
Location: header declarations (`.cuh` lines 45-50, 64-68, 77-81); `.cu` definitions. The functions return `void`, so `[[nodiscard]]` is correctly N/A. They are *not* `noexcept` and *can* throw (`CUBLAS_CHECK`/`STEPPE_CUDA_CHECK` → `CublasError`/`CudaError`) — which is *correct* (architecture.md §7's `noexcept` rule applies to `__device__` functions; host launch wrappers must throw, not `exit()`, so the §10 API layer can translate to `steppe_status_t`). But the neighbors annotate heavily — f2_estimator.hpp's primitives are `[[nodiscard]] … noexcept`, check.cuh's accessors are annotated — so the silence about the *throwing* surface reads as an omission. ## Considered & rejected

- **"`Q_all[sQ]` in the gather reads the *masked* Q, but pad columns should use the *raw* Q."** — Rejected. The gather copies the feeder *outputs* (`dQ` masked, `dV` mask, `dS` stacked), which is the contract — the feeder already applied masking/zero-fill; pad columns are written 0 deliberately. The `.cuh` doc (33-44) and the kernel comment (57-65) describe this correctly; consistent with the backend wiring (`cuda_backend.cu:260` passes `dQ/dV/dS`, the feeder outputs).
- **"`finalize_f2` divides by `vpair` (a GEMM output Vg·Vgᵀ) — could be non-integer under emulation and break the `>0` guard."** — Rejected. `Vg` is exactly 0.0/1.0 (validity mask, zero-filled in pad), so `Vpair = Σ 1·1` is an exact integer count; even under emulated FP64 the products are 0 or 1 and sums of small integers are exact in FP64 well past any realistic SNP count (2⁵³). The `>0.0` guard (f2_estimator.hpp:110-111) is the AT2 `Vpair==0⇒0` branch and is robust; the equivalence test independently recounts Vpair and asserts exact integer equality. (Note: the Vpair GEMM therefore does not *need* emulated precision — it is exact-integer — but running it under the same compute type as G/R is harmless and keeps the batch homogeneous; not worth a separate native call.)
- **"Passing the same `dQg`/`dVg` pointer as both A and B to `cublasGemmStridedBatchedEx` is illegal aliasing."** — Rejected. A and B are read-only inputs; cuBLAS does not prohibit aliasing read-only operands, and C (`dGg`/`dVpairg`) is distinct. Standard syrk-as-gemm idiom; matches the validated single-block path (F17).
- **"`grid.x` from `grid_for(P)` could overflow `gridDim.x`."** — Rejected. `gridDim.x` max is 2³¹−1; `grid_for(P) = ceil(P/16)` for P ≤ 4266 is ≤ 267. The real grid limit is the y/z 65 535 cap — F2.
- **"`size_t` (assemble kernel) vs `long` (gather kernel) index-type mixing is an inconsistency bug."** — Rejected. Both are 64-bit on the LP64 target; the gather uses signed `long` because it adds a signed `block_offsets` value, the assemble uses unsigned `size_t` for pure positive slab offsets — each locally appropriate and matching its sibling's choice (the single-block assemble uses `size_t` too). Not a bug, though a project-wide "device flat-index type" convention would be tidier (TODO backlog B: "standardize device flat-index on `size_t`" — a cross-cutting nit, not this TU's).
- **"The gather kernel should fuse into the feeder to avoid the extra global round-trip."** — Rejected. The feeder is block-agnostic (one sweep over all M) and deliberately reused verbatim (architecture.md §2 DRY, `.cu` comment 28-32); the gather is per-bucket and must run after bucketing. Fusing would re-introduce the second feeder the design explicitly avoids and couple the feeder to the bucket layout. The round-trip is bandwidth on the already-resident `dQ/dV/dS` and is part of the measured 7.2×/8.9× win.
- **"`cudaMemGetInfo`-based chunking makes the f2 result depend on free VRAM."** — Rejected (and not this TU). The chunk boundary changes only GEMM *batch grouping*; each block's f2 is computed identically regardless of which chunk it lands in (zero-pad makes width-grouping inert; cross-chunk there is no reduction — each block writes its own disjoint slab). Accumulation order *within* a block is fixed by `s_pad` (bucket width), not chunk size. Parity is chunk-count-independent. A backend concern anyway.
- **"`run_f2_gemms_group` should re-engage precision like the sibling `run_f2_gemms`."** — Rejected (the *opposite* is the correct design). The M4 path hoists `engage_f2_precision` to `cuda_backend.cu:214` (once, before the bucket loop) and this wrapper deliberately does *not* re-engage — the right "engage once, shared by all paths" discipline (TODO §C). The sibling's inline re-engage is the *sibling's* finding (`f2_block_kernel.md` P-1), not this unit's. This unit is the better-behaved of the two on precision engagement.
- **"`grid.z` clamp is the same finding as the sibling `f2_block_kernel.md`."** — Rejected as duplication: that sibling explicitly rejects grid-z for *its* unit (its kernels are 2-D, `grid.z` defaults to 1) and points the grid-z finding *here*. F2 is correctly this TU's.

## Good patterns to keep

- **DRY across the M0 and M4 paths is exemplary:** the feeder (`launch_f2_feeder`, reused verbatim), the precision engagement (`engage_f2_precision`), the compute-type mapping (`f2_compute_type`), the numerator/finalize primitives (`assemble_f2_numerator`/`finalize_f2`), and the launch-grid helper (`grid_for`) are all single-sourced and reused — no second copy of the formula, the policy, or the grid math (§2/§8 single-home genuinely achieved).

- **"Engage precision once, shared by all paths" — done right here:** `run_f2_gemms_group` correctly does *not* re-engage precision per call (it relies on the hoisted `engage_f2_precision`), the better-behaved of the two GEMM wrappers (TODO §C pattern to preserve).
