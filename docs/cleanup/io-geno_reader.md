## Role & layering

`GenoReader` is the `.geno`-side of the `io` LEAF (architecture.md §4, §5 S0). Its contract is deliberately narrow: own a path + parsed header, validate the on-disk size against the *derived* record stride, and serve a tiled raw-byte **gather** (`read_tile`) that copies the selected individuals' packed SNP-prefix bytes into a plain `GenotypeTile` — **no decode** (the device/CPU kernel decodes via the shared `core::genotype_code`/`finalize_af` primitive). It depends only on sibling `io`-leaf headers (`eigenstrat_format`, `genotype_tile`, `ind_reader`), pulls in nothing from `core`/`device`, includes no CUDA header, and surfaces failures as exceptions to the `app`/test caller — exactly the §4 leaf discipline. The CMake wiring (`steppe_io` links only `steppe::api` + `steppe::warnings` PRIVATE; deliberately NOT `steppe::core`/`steppe::device`) makes the layering compiler-enforced (§4 "link visibility is the mechanism"). **Layering is clean; no upward/CUDA dependency.** There is no `src/app` layer yet (verified: directory absent), so every contract concern below about an `app` caller is forward-looking, not a present-day break.

I re-verified the prefix-read consistency against the actual consumer: `cpu_backend.cpp::decode_af` decodes `s ∈ [0, tile.n_snp)` reading byte `g*tile.bytes_per_record + s/4` (line ~309). For `snp_begin==0` the reader sets `tile.n_snp = snp_end` and `tile.bytes_per_record = packed_bytes(snp_end)`, so every decoded byte index `s/4 < packed_bytes(snp_end) = bytes_per_rec` — the prefix read is exactly what the decoder consumes. The bridge narrows `n_pop → int` (`DecodeTileView::n_pop`, backend.hpp:74) and `M = (long)tile.n_snp` (`cpu_backend.cpp::decode_af`), confirmed (feeds 7.5). **Confirmed consistent.**

## Score: 7/10 — solid, well-documented, layering-clean leaf, held down by a silent integer-overflow heap-write path, an exception-type contract gap, an unenforced cross-object precondition, a misleading "owns an open stream" comment, and effectively zero dedicated test coverage

(Down half a point from the draft's 7.5: the newly-confirmed **1.5** is a *silent memory-corruption* path on a hostile/buggy `part`, which is a genuine `high`-severity robustness hole for a fail-fast leaf, not the merely-mis-typed-exception of 2.1.)

The header is excellent (precise doc comments, correct move-only + `[[nodiscard]]` + `noexcept`, derived-stride discipline, zero surviving magic numbers). The happy path is correct and matches the oracle bit-for-bit (M1 done — ROADMAP M1: max|Δ|=0 for Q/V/N on `derived_acc`). What keeps it below the 9.5–10 bar:

1. **Memory-safety on a hostile/stale partition.** `read_tile` accepts an arbitrary `IndPartition` and (a) never validates `row < records_present_` (1.2 / 9.2 — silent **wrong-individual** gather), and (b) computes `n_ind * bytes_per_rec` (1.5) as a `std::size_t` that can **wrap**, so `resize` allocates a too-small buffer and the gather then writes **past the heap allocation** — a silent buffer overflow, not even an exception. The fail-fast leaf must reject the partition at the boundary.
2. **Contract precision (exception type).** The unit's own header (`hpp` line 17–18) and `genotype_tile.hpp`/`ind_reader.hpp` promise "I/O failures surface as `std::runtime_error`." `tile.packed.resize(n_ind*bytes_per_rec)` can throw `std::bad_alloc` / `std::length_error` — **neither derives from `std::runtime_error`** — so an `app` caller written to the literal contract (`catch(std::runtime_error&)`) would miss it (2.1). The current tests catch `std::exception`, masking it.
3. **Unenforced precondition / misleading doc.** Correctness *assumes* `part` was built by `read_ind(…, records_present())` on the same file (1.2/9.2). And "Owns an open file stream … cheap to keep open across the tile loop" is **false** — the class stores only a path and reopens per call (7.2); this will mislead the M5/TODO⚡#2 author.
4. **No test.** The entire error surface is exercised only by two real-AADR happy-path `.cu` tests gated on the 4 GB gitignored dataset (10.1).

## Findings

### (1) Correctness & bugs

**1.1 — `read_tile` short-read guard checks `gcount` only, never stream state (`!in`); misses a `badbit` device error that extracts the full count.** `low→med` · effort S · before-M4.5: **yes**
`read_tile` lines 121–126. After `in.read(dst, bytes_per_rec)` the only test is `in.gcount() != bytes_per_rec`. Per `[istream.unformatted]` (verified), EOF before `n` chars calls `setstate(failbit|eofbit)` *and* leaves `gcount() < n`, so the EOF case **is** caught. The hole is the rare-but-real **`badbit`** path: a `basic_filebuf` underflow / device read error can `setstate(badbit)` after extracting some-or-all characters; if it extracted exactly `bytes_per_rec` before failing, `gcount() == bytes_per_rec` and the guard passes on corrupt/torn data. A failed `seekg` (line 119) past EOF can also leave `failbit`, after which the `read` extracts 0 — `gcount()==0 != bytes_per_rec` catches *that* subcase, but the message ("short read for individual row …") then misdiagnoses a *seek* failure. **Fix:** `if (!in || in.gcount() != static_cast<std::streamsize>(bytes_per_rec))`, and put `off`, the on-disk file size, and `records_present_` in the message so a partial-file / out-of-range mis-request is diagnosable. (Narrowed, confirmed version of the draft's 1.1 — see Considered & rejected for the multi-iteration carry-over mechanism that does *not* occur.)

**1.2 — `read_tile` can return STALE/wrong bytes for an out-of-`records_present_` row that lands inside a longer-than-expected valid region (silent mis-gather).** `med` · effort S · before-M4.5: **yes**
`read_tile` lines 114–119. Each `row` becomes `off = header_bytes + row*header_.bytes_per_record` and is seeked directly, with **no** check that `row < records_present_` (or even `< header_.n_records`). Correctness relies entirely on `part` being built by `read_ind(ind, sel, reader->records_present())` — which both `.cu` tests do (verified: `test_decode_equivalence.cu:182`, `test_filter_oracle.cu:210`), but `read_tile`'s signature accepts *any* `IndPartition`. A partition built against a stale/larger cap or a *different* `.geno` produces a `row` that, when the underlying file is longer than `records_present_*stride` (trailing junk, concatenation, or a header that under-reports), seeks to a fully-present-on-disk offset and reads a complete record of the **wrong individual** — `gcount() == bytes_per_rec`, guard passes, wrong data flows into the tile silently. **Fix:** before computing `off`, `throw std::runtime_error` if `row >= records_present_`, with `row`/`records_present_` in the message. Closes the silent-mis-gather path and documents the cross-object invariant in code, not just a comment. (API face is 9.2; same fix.)

**1.3 — Constructor `seekg`/`tellg` size validation is coupled to the stream state left by the header `read` (latent, currently masked).** `low` (latent) · effort S · before-M4.5: no
`ctor` lines 31–53. The header `read` (31) is guarded by `gcount() != kGenoHeaderBytes` (32) which throws *before* the seek — so the hazard is masked today. The latent coupling: per `[istream.unformatted]` (verified), `seekg(off,dir)` "first clears `eofbit`" then executes the seek **only if `fail() != true`**. So a residual `failbit` from a partial header read would make `seekg(0,end)` a no-op; `tellg()` then returns `pos_type(-1)` (fail()→-1). Line 52 stores `-1` into `std::streamoff fsize`; line 53's `fsize < header_bytes` (`-1 < 48`) throws "file smaller than header" — wrong message but not silent corruption. The defense is incidental (line 32 + the sign check), not designed. **Fix:** prefer `std::filesystem::file_size(path_)` (no stream-state coupling at all — see 8.2), or `in.clear()` before `seekg(0,end)` + an explicit `fsize >= 0` check with its own message. Bundles 1.3 + 2.6 + 2.7.

**1.4 — Wrong/overflowing header counts flow into a *mis-diagnosed* file-size error, not an "implausible header" error.** `low` · effort S · before-M4.5: no
The decimal-overflow site is `eigenstrat_format.cpp` line 57 (`v = v*10u + digit`, no guard, wraps silently for an absurd decimal count — `parse_geno_header` is `noexcept` and never validates magnitude). `GenoReader` is the consumer that turns the parsed counts into `bytes_per_record = packed_bytes(n_snp)`, `n_records`, and the size check. A corrupt/hostile header with a wrapped `n_snp` yields a wrapped stride and a `records_present_` computed against the wrong stride; lines 56–60 usually catch the gross mismatch, but the *message* ("no complete records" / size mismatch) misdiagnoses the real cause. The structural fix (overflow guard) belongs in `eigenstrat_format`; flagged here as the consumer's exposure. A cheap defense at this boundary: a plausibility bound (`header_.n_snp`/`n_ind` ≤ a configured `kMaxPlausibleCount`, or assert `packed_bytes` did not wrap) → fail-fast with the right cause. Med-leverage for hostile inputs; low for trusted AADR.

**1.5 — NEW: `n_ind * bytes_per_rec` (line 98) is a `std::size_t` multiply that can WRAP on a hostile/buggy partition, so `resize` allocates a too-small buffer and the gather then writes PAST the allocation — a silent heap-buffer-overflow, not an exception.** `high` (memory safety) · effort S · before-M4.5: **yes**
`read_tile` lines 91–98, 120. `n_ind = Σ g.rows.size()` over an *arbitrary* caller-supplied `part`; `bytes_per_rec = packed_bytes(tile_snps)`. The product `n_ind * bytes_per_rec` is computed in `std::size_t` (unsigned, modulo-2⁶⁴ wraparound is well-defined-but-silent — `[basic.fundamental]` unsigned modular arithmetic). With a partition whose `rows` are garbage/duplicated (or simply far larger than `records_present_`), the product can wrap to a *small* value; `tile.packed.resize(small)` then succeeds, and the gather loop (line 120) writes to `tile.packed.data() + out_ind * bytes_per_rec` for `out_ind` up to the true (un-wrapped) `n_ind` — overrunning the under-sized buffer. The `gcount`/seek checks do **not** save it: the out-of-bounds `dst` pointer is computed and handed to `in.read` *before* any state check, and `istream::read` writes to it. This is distinct from 2.1 (which is the allocator *throwing* on a large-but-non-wrapping request); 1.5 is the *non-throwing* path where the size silently becomes small. **The 1.2 fix (`row < records_present_`) bounds `n_ind ≤ records_present_` and therefore prevents the wrap in practice** — but the multiply should also be guarded directly (fail-fast, defense-in-depth): before `resize`, `throw` if `bytes_per_rec != 0 && n_ind > SIZE_MAX / bytes_per_rec` (the standard overflow-check idiom), or compute the size with a checked-multiply helper. Highest-severity item in the unit because the failure is *silent memory corruption*, exactly what a fail-fast leaf must never do (§2). New this pass.

**1.6 — `records_present_ > n_records` clamp is correct; the *under* case is silently trusted.** `low` · effort S · before-M4.5: no
`ctor` lines 58–64. `records_present_ = data_bytes / bytes_per_record`, clamped down to `n_records` if it exceeds (correct — a file padded past the header count must not over-report). When `records_present_ < n_records` (a genuine partial file) there is no warning — but the leaf has no logging facade by design (§10 logging lives downstream / in `core`), so this is acceptable; the `app` caller must notice the discrepancy. Fine for M1; note for M5 streaming. **Confirmed correct** (the oracle uses the same `(fsize-hdr)//bpi` cap; ROADMAP M1 / ind_reader doc).

### (2) Edge cases & failure modes

**2.1 — `tile.packed.resize(n_ind*bytes_per_rec)` can throw `std::bad_alloc` / `std::length_error`, violating the unit's own documented "failures surface as `std::runtime_error`" contract.** `med` (contract correctness) · effort S · before-M4.5: **yes**
`read_tile` line 98. `geno_reader.hpp` lines 17–18 promise "I/O failures surface as `std::runtime_error` to the `app`/test caller"; `genotype_tile.hpp`/`ind_reader.hpp` repeat it. But `std::vector::resize` may throw `std::length_error` (a `logic_error`, when the request exceeds the allocator `max_size()`) or `std::bad_alloc` (allocation failure) — **verified that neither derives from `std::runtime_error`** (`bad_alloc` derives directly from `std::exception`, `length_error` from `std::logic_error`). For AADR scale a full tile is ~4 GB; a malformed/hostile header (1.4) or a partition gathering more individuals than the device-tile budget makes the OOM path real, and the exception escapes typed as `std::bad_alloc`. The two `.cu` tests catch `std::exception` (verified: `test_decode_equivalence.cu:196`, `test_filter_oracle.cu`), so they mask it; a future `app` caller written to the *literal* contract (`catch(const std::runtime_error&)`) silently misses it → unhandled → `std::terminate`. **Fix (pick one):** (a) loosen the documented contract to "failures surface as `std::exception`" (honest, cheap) and have `app` catch `std::exception` at the I/O boundary; or (b) bound the tile size up front (validate `n_ind*bytes_per_rec ≤ a configured budget`) and `throw std::runtime_error` on over-budget *before* the `resize`, which also feeds the §11.2 VRAM-budget story and the 1.5 overflow guard. (b) is the architecturally aligned choice (fail-fast with file/line context, §2) and composes with 1.5.

**2.2 — Empty partition (`part.groups.empty()`) produces a degenerate-but-not-rejected zero-pop tile.** `low→med` · effort S · before-M4.5: no
`read_tile` lines 91–92, 110–130. If `part.groups` is empty: `n_ind == 0`, `tile.packed` empty, `pop_offsets == {0}`, `pop_labels == {}`, the loop never runs, and a `n_pop()==0` tile returns without error. `read_ind` *throws* on an empty selection (`ind_reader.cpp` line 115–116, verified), so a well-behaved caller can't reach this — but `read_tile` accepts an arbitrary `IndPartition`, and the leaf's fail-fast ethos (§2) argues for rejecting it at the boundary rather than emitting a zero-pop tile that the downstream `decode_af` short-circuits (`cpu_backend.cpp`: `if (P <= 0 || M <= 0) return out;`) into an empty Q/V/N a GEMM would then mishandle. **Fix:** early `throw` if `part.groups.empty()`. Also decide/document the **zero-`rows` population** case (`g.rows.empty()` → a zero-width pop segment `pop_offsets[p]==pop_offsets[p+1]`): the decoder's `finalize_af(an=0)` yields V=0 so it is *survivable* numerically (Q=0,V=0,N=0), but a population with no individuals is almost certainly a caller bug worth rejecting.

**2.3 — `snp_end <= snp_begin` is written generically though `snp_begin` is proven 0 — dead-but-forward-correct.** `low` · effort S · before-M4.5: no
`read_tile` lines 75–85. After `snp_begin != 0` is rejected (line 75), `snp_end <= snp_begin` reduces to `snp_end == 0`. Harmless and forward-correct (M5's nonzero-begin lift makes the generic form load-bearing); a one-line comment ("snp_begin proven 0 above; written generically for the M5 lift") removes a reader double-take. Cosmetic. Confirmed.

**2.4 — `n_snp == 0` is rejected only transitively via `bytes_per_record == 0`.** `low` · effort S · before-M4.5: no
`ctor` line 43 throws on `bytes_per_record == 0 || n_records == 0`. For TGENO `bytes_per_record == packed_bytes(n_snp)`, and `packed_bytes(0)==0`, so `n_snp==0` *is* rejected — but the coupling is implicit and rests on `packed_bytes(0)==0`. A one-line comment that the `bytes_per_record==0` arm *is* the `n_snp==0` guard for TGENO makes the invariant explicit. (GENO's `max(48,…)` floor means GENO's stride is never 0 — but a GENO file is rejected at `read_tile` line 70 anyway, and a degenerate GENO with `n_snp==0` ⇒ `n_records==0` is caught by the second arm.) Confirmed.

**2.5 — 64-bit offset arithmetic is handled correctly — a good pattern.** `n/a` (positive)
`read_tile` lines 115–118. `off = (streamoff)header_bytes + (streamoff)row * (streamoff)bytes_per_record` — each operand cast to `std::streamoff` (signed 64-bit) *before* the multiply, so the product is 64-bit, not a `row`-promoted 32-bit intermediate. AADR scale (27,594 ind × 146,033 bytes ≈ 4 GB, product ~4e9) is well within int64. **Confirmed correct.** Nit: `std::streamoff` is implementation-defined signed; a `static_assert(sizeof(std::streamoff) >= 8)` documents the assumption and costs nothing. *(Note: the offset multiply here is overflow-safe; it is the **separate** `n_ind * bytes_per_rec` size multiply in 1.5 that is **not** — the two are different expressions; do not conflate.)*

**2.6 — A negative `tellg` (`fsize == -1`) is filtered before the `size_t` cast — by ordering, not construction.** `low` (latent) · effort S · before-M4.5: no
`ctor` lines 52–57. `fsize` is signed; line 53's `fsize < header_bytes` throws on `-1` before `static_cast<std::size_t>(fsize)` at 57. Safe today *by statement ordering*. Pair with the 1.3 fix (`std::filesystem::file_size`, or `in.clear()` + explicit `fsize >= 0`) and it dissolves. Confirmed.

**2.7 — NEW: `in.tellg()` returns a `std::streampos` (`fpos`), narrowed to `std::streamoff` at line 52 without an explicit conversion comment.** `low` (latent) · effort S · before-M4.5: no
`ctor` line 52: `const std::streamoff fsize = in.tellg();`. `basic_istream::tellg()` returns `pos_type` (`std::fpos<mbstate_t>`), which has a defined conversion operator to `off_type` (`std::streamoff`) — so this is well-defined, and on failure yields `streamoff(-1)` (caught by 1.3/2.6). The point is only that an `fpos→streamoff` implicit conversion is happening; on a platform where `fpos` carries multibyte state the conversion drops it, harmless for a binary file but worth a one-word comment, *or* sidestepped entirely by `std::filesystem::file_size` (8.2), which returns a `std::uintmax_t` with no `fpos` semantics. Folds into the 1.3/8.2 fix. New this pass.

### (3) Numerical / precision (§12)

**3.1 — N/A by design — this unit performs no arithmetic on genotype values.** It copies packed *bytes*; all decode/precision happens downstream in `core::genotype_code`/`finalize_af` (one FP64 divide) and the f2 GEMMs (architecture.md §5 S0: "io reads/tiles; the device kernel decodes"). ### (4) CUDA idioms / RAII / streams / launch config (§7)

**4.1 — N/A for CUDA kernels/streams/launch config — host-pure leaf, no CUDA.** Correct per §4 (`io` is CUDA-free; CMake links no CUDA). Confirmed.

**4.2 — Host-level RAII is applied correctly (no leak), but the type does not hold the stream it claims to.** `n/a`→see 7.2/8.1. `ctor` and `read_tile` each use a local `std::ifstream` whose destructor closes the file — no leak, no manual `close()`. The RAII *gap* is not holding the stream as a member (the doc claims it does — 7.2/8.1); a missing opportunity, not a leak.

**4.3 — Defaulted move `noexcept` is honest today but will silently become a lie if an owning stream member is added.** `low` · effort S · before-M4.5: no
`hpp` lines 71–72: move-construct + move-assign `= default`, `noexcept`. With the current members (`std::string`, POD `GenoHeader`, `std::size_t`) the defaulted moves are correct and the `noexcept` is honest (`std::string`'s move is `noexcept`). **But** the day someone adds an owning `std::ifstream in_;` (8.1), the `= default` move adopts `std::ifstream`'s move — **verified to be declared without `noexcept`** in `[ifstream.cons]` (`basic_ifstream(basic_ifstream&& rhs);`). With a forced `noexcept` on the special member, a throwing move would `std::terminate`; if instead deduced, the special member silently becomes `noexcept(false)`, invalidating the explicit `noexcept` in the header — and `-Werror` does **not** flag a `noexcept`-mismatch on a defaulted special member. Flag now so the M5 owning-stream edit is done deliberately. Confirmed.

### (7) Readability, naming, const-correctness, `[[nodiscard]]`/`noexcept`, comments

**7.1 — `[[nodiscard]]`/`noexcept` discipline is exemplary.** `n/a` (positive). `header()`, `records_present()` are `[[nodiscard]] … const noexcept`; `read_tile` is `[[nodiscard]]` (dropping the returned tile is a bug) and correctly *not* `noexcept` (it throws by contract). Matches §7. Confirmed.

**7.2 — Header doc comment claims an open stream the class does not hold (actively misleading).** `med` (doc correctness) · effort S · before-M4.5: **yes**
`hpp` line 31 "Owns an open packed .geno file" and line 34 "Move-only (it owns a file stream); cheap to keep open across the tile loop." The class owns only `std::string path_`, `GenoHeader header_`, `std::size_t records_present_`. It opens a fresh `std::ifstream` in the ctor (closed at ctor exit) and **another per `read_tile` call** (line 102). The comment describes an intended design that was not implemented — and it will actively mislead the M5 author who reads "cheap to keep open across the tile loop" and assumes a persistent stream when planning the TODO⚡#2 ingest rewrite. **Fix:** make it true — hold an owning `std::ifstream in_;` opened in the ctor and reused by `read_tile` (8.1, the right pre-M5 foundation; decide move semantics per 4.3) — or, until then, correct the comment to "stores the path; opens the file per operation." Given TODO⚡#2 mandates the M5 ingest rewrite anyway, the owning-stream change is the better fix. Confirmed.

**7.3 — `reinterpret_cast<char*>(tile.packed.data() + …)` is correct but undocumented.** `low` · effort S · before-M4.5: no
`read_tile` line 120. `tile.packed` is `std::vector<std::uint8_t>`; `ifstream::read` wants `char*`. `uint8_t*→char*` is well-defined — `char`/`unsigned char` are narrow character types and accessing object representation through `char*`/`unsigned char*` is permitted by `[basic.lval]` (no strict-aliasing UB; `std::uint8_t` is `unsigned char` on all targeted platforms). Correct; a one-line `// uint8_t→char for istream::read; both character types, no aliasing UB` preempts a reviewer double-take. Cosmetic. Confirmed.

**7.4 — `const`-correctness: `read_tile` correctly non-`const` (forward-correct once it holds a stream member).** `n/a` (positive, caveat). Today `read_tile` mutates no member so it *could* be `const`, but `const` would be wrong once it owns a stream (8.1) — seeking mutates the stream position. Leaving it non-`const` is the forward-correct choice. Confirmed.

**7.5 — The tile this unit fills uses `std::size_t` throughout, but the `DecodeTileView` it bridges to narrows `n_pop`→`int` and `n_snp`→`long`.** `low` (cross-seam, not this unit's defect) · effort S · before-M4.5: no
`genotype_tile.hpp` types `n_snp`/`n_individuals`/`pop_offsets[i]` as `std::size_t`; `device/backend.hpp` `DecodeTileView` types `n_pop` as `int` (line 74) and `cpu_backend.cpp::decode_af` does `const long M = static_cast<long>(tile.n_snp);`. The narrowing happens at the `app`/test bridge (not in `geno_reader`), so it is **not this unit's bug** — but `GenoReader::read_tile` is the producer of the wide values, and the contract it documents (unbounded `std::size_t` populations/SNPs) is silently capped to `INT_MAX` populations / `LONG_MAX` SNPs at the seam. For AADR (P≤4,266, M≤584,131) this is moot, but the producer's doc could note "P fits `int`, M fits `long` at the decode seam." Flag against the seam, not this file. Confirmed.

### (8) Performance

**8.1 — Per-individual `seekg`+`read` random gather into pageable memory is the M5 ingest liability (TODO⚡#2 / §11.1).** `med` (M5; low for M1) · effort L · before-M4.5: no (M5)
`read_tile` lines 112–128 (+ the per-call reopen at 102). For each selected individual the code random-seeks to `header_bytes + row*stride` and reads into a pageable `std::vector<uint8_t>`. This is *exactly* the pattern TODO⚡#2 names: "replace `geno_reader`'s pageable `ifstream` per-individual gather with large sequential reads into `PinnedBuffer` ring slots + `cudaMemcpyAsync` on a dedicated copy stream … end-to-end → `max(read, gpu)` not their sum." Two costs: (a) random access — selected individuals are scattered TGENO rows, so the gather is inherently strided; (b) pageable destination forbids true H2D overlap (§11.1 needs pinned staging). **For M1 this is correct and fast enough** (67.6 ms end-to-end per ROADMAP M1). The forward fix is the M5 rewrite, internal because the **signature is already tile-shaped** (SNP range + partition). No action before M4.5; flag so the M5 author knows this function is the target. See 11.1 for the leaf-vs-CUDA tension the rewrite must resolve. Confirmed.

**8.2 — Ctor does an extra `seekg(end)`/`tellg` just to size the file.** `low` · effort S · before-M4.5: no
`ctor` lines 51–52. `std::filesystem::file_size(path_)` returns the size with no stream-state coupling and no `failbit`/`fpos` hazard (fixes 1.3/2.6/2.7). It throws `std::filesystem::filesystem_error` (derives from `std::system_error` → `std::runtime_error`), so it propagates *compatibly* with the documented contract — but wrap it to keep the uniform "io::GenoReader: …" message. Bundles 1.3/2.6/2.7. *(Note: this is one of the few stdlib failures that already throws a `runtime_error` subtype — contrast `resize` in 2.1.)* Confirmed.

**8.3 — `tile.packed.resize(n_ind*bytes_per_rec)` zero-fills then overwrites the whole buffer.** `low` · effort S · before-M4.5: no
`read_tile` line 98. `resize` value-initializes; the gather then overwrites every byte, so the memset is redundant (≈ tile-size). Negligible at M1; recurs if M5 calls per tile. `std::vector` has no uninitialized-resize, so this is unavoidable without changing the container — and the value-init *is* a cheap safety net that turns a torn-read residual into zeros. Acceptable; revisit alongside the 8.1 PinnedBuffer rewrite (which changes the container anyway). Confirmed.

### (10) Testability (§13)

**10.1 — There is NO dedicated unit/golden test for `GenoReader`; its entire error surface is untested.** `high` · effort M · before-M4.5: **yes**
**Re-confirmed by grep this pass:** the only coverage is two reference `.cu` tests (`test_decode_equivalence.cu:195`, `test_filter_oracle.cu:213`) that open the *real AADR `.geno`* and exercise **only the happy path** (`GenoReader(geno)` → `read_tile(part, 0, M)`), gated on the multi-GB gitignored AADR data being present (`STEPPE_AADR_ROOT`, default `/workspace/data/aadr`). There is **no** host-only `GenoReader` test in `tests/unit/` (which holds only `test_block_partition`, `test_f2`, `test_filters` — verified by `ls`). Per §13 ("Domain-outcome tests … assert the API returns [an error] as a *value*, not a crash") and fail-fast (§2), each error path should have a host-only GoogleTest writing a tiny synthetic `.geno` to a temp path and asserting the throw:
- bad magic (not TGENO/GENO) → throw "unrecognized .geno magic"
- file shorter than the 48-byte header → throw "shorter than the … header"
- degenerate header (`n_records==0`, or `n_snp==0` ⇒ `bytes_per_record==0`) → throw "degenerate header"
- valid magic but file smaller than header → throw "file smaller than header"
- header-only file (exactly 48 bytes) → throw "no complete records"
- partial data file (header says `n_records=10`, only 7 on disk) → `records_present()==7` (and `>n_records` padding → clamped to `n_records`)
- a GENO (SNP-major) file → `read_tile` throws "M1 decode targets TGENO"
- `snp_begin != 0` → throw "requires snp_begin == 0"
- `snp_end == 0`, `snp_end > n_snp`, `snp_end <= snp_begin` → throw "out of bounds"
- empty `part.groups` → throw (after the 2.2 fix)
- a `part` row `>= records_present_` → throw (after the 1.2 fix) — and a partition whose `n_ind*bytes_per_rec` would wrap → throw (after the 1.5 fix), *not* a heap overflow
- a synthetic TGENO with a known-bytes prefix → assert `pop_offsets`, `pop_labels`, and a few `packed` bytes exactly (the one structural check needing no 4 GB file)
- (after the 2.1 fix) an over-budget tile → throw a `std::runtime_error`, *not* a bare `std::bad_alloc`

A ~4-line `make_synthetic_tgeno(path, n_ind, n_snp, fill)` helper unlocks all of these GPU-free, fast, CI-able on the no-GPU PR runner (§14 stage 1; `tests/unit/test_filters.cpp` is the existing host-only `steppe::io`-linked model — verified it links `steppe::io` only). **The single highest-value gap:** the reader's robustness is *claimed* in comments but *unverified*, and findings 1.2, 1.5, 2.1, and 2.2 would each have been caught by such a test. Confirmed.

**10.2 — The reader is otherwise structured for testability (no globals, ctor-injected path, exceptions not `exit`).** `n/a` (positive). No global mutable state (§2), the path is ctor-injected (no hidden config), failures are exceptions. It just lacks the tests. Confirmed.

### (11) Capability tiers (TODO ⚡ / `wxz1fiiln`)

**11.1 — This unit is the M5/M7 INGEST capability-tier touch-point and must grow a runtime-detected, explicitly-tagged ingest-path selection (NOT yet) — and the rewrite collides with the CUDA-free-leaf rule.** `med` (M5) · effort L · before-M4.5: no
The TODO §⚡ capability-tier table (line 129) names ingest explicitly: capable path (PRO 6000 / full-host) = "true **GDS** cuFile DMA NVMe→VRAM (probe `DMA_BUF_SUPPORTED`/`cuFileBufRegister`, **not** `gdscheck`)"; budget-5090 fallback = "pinned double-buffer POSIX-pread", logged reason *"GDS unavailable (GeForce GPU-class / OverlayFS) → POSIX pread into pinned double-buffer"*. **Sharpened against the TODO this pass:** GDS is **explicitly ruled out on the routine vast 5090 box** (TODO lines 36, 112, 122 — "GeForce GSP-firmware lock survives every patch; no community GeForce-GDS exists") and is a **PRO 6000 / `.f64`-cache research bet only**; the *operative* baseline everywhere is the **pinned double-buffer POSIX-pread** (TODO line 36 M5 acceptance criterion, §11.1). The M5 design must: build the pinned double-buffer baseline; on the PRO 6000 only, runtime-probe the GDS lane (the TODO is explicit: **NOT** `gdscheck`) with the logged fallback; record which path the run took (the cross-cutting "capability-tagged results" slotted into `DeviceConfig`/`Resources`, TODO line 137); stay parity-neutral (the tier changes only *how bytes reach VRAM*, never a reported number — §12 / TODO line 133).

This is **not acknowledged anywhere in this unit** and should be an explicit M5 design note: either `io` emits a read-plan that a `device`-side pinned/GDS loader consumes, or the byte-mover migrates out of the leaf — the architecture's own §4 note ("the *app* layer is the only place that wires `io` output into compute") points at the read-plan answer. **No code change before M4.5; write the note.** Confirmed.

The `header().n_ind`/`n_snp`/`bytes_per_record` this ctor derives are the inputs §11.1's `bytes_per_tile(T)`/`T_max` and the §⚡ "`--dry-run` reports per-box P_max" line (TODO line 137) consume — a reason to keep the derived-stride accessor exactly as is. ## Considered & rejected

- **(carried from prior draft) "A short read leaves `failbit` set, the next iteration's `seekg` is a no-op, and the subsequent `read` returns the previous record's stale bytes silently."** Rejected as a live mechanism — within a single `read_tile` call, the **first** short read (`gcount() < bytes_per_rec`) triggers the throw at line 123 and **exits the function**, so control never reaches a "next iteration" with a residual `failbit`. The multi-iteration carry-over cannot occur inside one call. (Re-verified against `[istream.unformatted]`: EOF sets `failbit|eofbit` *and* `gcount() < n`, so the EOF case is always caught by the existing guard.) What survives is the *narrowed* 1.1 (missing `!in`/badbit check) and the *separate* 1.2 (wrong-record-of-a-valid-offset, an unenforced-precondition bug).
- **"The ctor's `read`+`gcount` check fails to detect a short header."** Rejected — verified: a short read into the 48-element `std::array<char>` sets `failbit|eofbit` and `gcount() < 48`; line 32 throws. The short-header case *is* caught; 1.3 is only the latent stream-state coupling.
- **"`seekg(0, end)` past a 48-byte file is undefined."** Rejected — seeking to end-of-file is well-defined; `tellg` returns the size. The only hazard is a residual `failbit` (1.3), which line 32 prevents today.
- **"`data_bytes / header_.bytes_per_record` (line 58) could divide by zero."** Rejected — line 43 throws on `bytes_per_record == 0` *before* the division is reached, so the divisor is provably nonzero at line 58. (Verified the statement ordering; the guard is the `bytes_per_record == 0` arm.) No bug; noted here for completeness because the mandate flags division/error paths.
- **"`std::isdigit` UB / decimal overflow in the header parse is `geno_reader`'s bug."** Rejected as *this unit's* defect — the `static_cast<unsigned char>` guard is correct in `eigenstrat_format.cpp` line 52, and the `v*10u` overflow (line 57) is *that* file's gap; `geno_reader` is only the **consumer** that mis-diagnoses the wrapped count (kept as 1.4, the consumer exposure). Also note `parse_geno_header` has no sign handling, so a `-`/`+`-prefixed count is silently treated as non-digit and skipped — not a `geno_reader` defect and not exploitable (counts stay non-negative).
- **"`reinterpret_cast<char*>` from `uint8_t*` is strict-aliasing UB."** Rejected — narrow character types; access through `char*`/`unsigned char*` is permitted by `[basic.lval]`. No UB (7.3 asks only for a comment).
- **"Return-by-value of `GenotypeTile` is a copy/perf bug."** Rejected — NRVO-eligible `return tile;` moves the vectors; no copy. (Re-confirmed.)
- **"`records_present_ > n_records` clamp is wrong; a too-large file should error."** Rejected — a file larger than `header + n_records*stride` (trailing junk / concatenation) is not necessarily corrupt; capping and ignoring the tail matches the oracle `(fsize-hdr)//bpi` and the comment says so (1.6 confirms this is *correct*).
- **"`GenoReader` should validate `.snp`/`.ind` counts against the header."** Rejected — cross-file consistency is the `app` layer's job (the only layer that sees all three readers — §4 "the *app* layer is the only place that wires `io` output"); pushing it into `GenoReader` couples the readers and breaks the leaf's single-file responsibility.
- **"`read_tile` should be `noexcept`."** Rejected — it throws by contract (out-of-range, short read, and per 2.1 a `resize` allocation failure); `noexcept` would `std::terminate` on a legitimate I/O error.
- **"`packed_bytes(tile_snps)` vs `header_.bytes_per_record` could mismatch and over/under-read."** Re-verified against the *actual consumer* this pass (`cpu_backend.cpp::decode_af` reads byte `s/4` for `s ∈ [0, tile.n_snp)`): for `snp_begin==0`, `bytes_per_rec = packed_bytes(snp_end) ≤ packed_bytes(n_snp) = header_.bytes_per_record`, every decoded `s/4 < bytes_per_rec`, reading `bytes_per_rec` from an offset computed with the full-record stride is a correct *prefix* read. Not a bug. M5's nonzero `snp_begin` will need within-record byte-offset handling — exactly why nonzero begin is rejected today.
- **"`g.rows` could be unsorted / contain duplicates, breaking the gather."** Rejected as a `geno_reader` defect — `ind_reader.cpp` builds `rows` by `push_back` in ascending file-row order (lines 64–68, verified) and never reorders within a group, so the contract holds; `read_tile` does not *depend* on sortedness for correctness (each row is an independent seek+read), only on `row < records_present_` (the 1.2 fix). *Duplicate* rows in a hostile `part` inflate `n_ind` and feed 1.5, but that is the partition-validation finding, not a sortedness bug. Out of this unit.
- **"The offset multiply (`row * bytes_per_record`, line 117) can overflow like the size multiply in 1.5."** Rejected — distinct expressions: the offset multiply casts each operand to signed 64-bit `std::streamoff` *before* multiplying (2.5, overflow-safe at AADR scale); the *size* multiply (1.5) is unsigned `std::size_t * std::size_t` with silent wraparound. Only the latter is the bug; do not conflate.
- **"`int ploidy` / decode missingness handling is wrong."** Rejected — out of this unit entirely; decode lives in `core::finalize_af`, which this reader never calls. The reader copies bytes only.

## Good patterns to keep

- **Derived-stride discipline (§4 / ROADMAP §4):** zero magic numbers — `packed_bytes(...)`, `header_.bytes_per_record`, `kGenoHeaderBytes` are the single sources; `read_tile` never re-spells a packing literal. Meets the "no literal survives except true mathematical constants" bar exactly.
- **Correct 64-bit OFFSET arithmetic** — casting each operand to `std::streamoff` *before* the multiply (2.5) is the overflow-safe idiom; replicate wherever record offsets are computed. (Contrast: extend the same care to the *size* multiply, 1.5.)
- **Prefix-read consistency with the decoder** — `tile.n_snp = tile_snps`, `tile.bytes_per_record = packed_bytes(tile_snps)` exactly matches what `cpu_backend.cpp::decode_af` reads (`byte g*bytes_per_record + s/4`, `s < n_snp`); the producer and consumer agree by construction. Keep this invariant when the M5 nonzero-begin lift arrives.
- **Doc comments that state the *why*** — the `read_tile` SNP-range/byte-prefix comment and the "nonzero begin is the M5 tile loop, rejected so a future caller doesn't get mis-aligned codes" rationale are exactly the forward-looking comment density the TODO "preserve" list calls for. (The one comment that *lies* — "owns an open file stream" — is the exception to fix, 7.2.)
- **Clean, compiler-enforced leaf layering** — stdlib + sibling `io` headers only; exceptions as the boundary type; no `core`/`device`/CUDA; CMake links no CUDA. Textbook §4. (Tighten only the *exact* exception subtype — 2.1.)
- **`[[nodiscard]] … const noexcept` accessors; `[[nodiscard]]` on the value-returning `read_tile`; throwing (not `noexcept`) where the contract can fail** — exactly §7.
- **Tile-shaped signature even though M1 calls it once** (`read_tile(part, snp_begin, snp_end)`) — §11.1 streaming readiness is baked into the API, so the M5 pinned-ingest rewrite (TODO⚡#2) is an internal change, not a signature break.
- **No global mutable state; ctor-injected path** — the §2 precondition for both thread-safety and the host-only unit tests 10.1 calls for.
