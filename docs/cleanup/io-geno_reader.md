# Review — `io/geno_reader` (.geno header parse + tiled raw-byte gather) — ADVERSARIAL FINAL

Unit under review: `src/io/geno_reader.hpp` (83 lines) + `src/io/geno_reader.cpp` (134 lines).
Scope: the packed-EIGENSTRAT `.geno` reader. The constructor parses the leading header (via `io::parse_geno_header`), classifies TGENO vs GENO, derives the record stride, and validates the on-disk size against `header_bytes + n_records·bytes_per_record`, recording `records_present_`. `read_tile` gathers the selected individuals' packed SNP-prefix bytes into a `GenotypeTile`, pop-contiguous in Q/V/N row order, for the SNP range `[snp_begin, snp_end)` (M1: `snp_begin == 0`). NO decode happens here — `io` reads/tiles packed bytes; `core::internal::decode_af` (host) and `decode_af_kernel.cu` (device) decode (architecture §5 S0).

This unit is the trust seam at the very front of the precompute pipeline: every byte that feeds the GPU/CPU decoder and ultimately `f2_blocks` passes through `read_tile`. A wrong offset, a silently truncated read, or a missing bounds check here corrupts every downstream statistic with no other guardrail in the `io` leaf. It is also the unit the TODO ⚡ section names as the **M5 rewrite target** ("replace `geno_reader`'s pageable `ifstream` per-individual gather with large sequential reads into `PinnedBuffer` ring slots").

**Verification status of every standard-library / OS claim used below (checked against the official cppreference mirror and CUDA/host docs, cited at the end):**
- `std::basic_istream::seekg` — on failure calls `setstate(failbit)`; it does **not** throw unless exceptions are enabled on the stream (they are not here). A `seekg` past EOF is **not** itself an error on most implementations; the subsequent `read` is what fails. [1]
- `std::basic_istream::tellg` — returns `pos_type(-1)` (which converts to `streamoff(-1)`) if `fail()` is set; it also calls `setstate(failbit)` on a failed `tellg`. [2]
- `std::basic_istream::read` — extracts up to `count` chars; on reaching EOF early it sets `eofbit` **and** `failbit` and stops; `gcount()` returns the number actually extracted (CharT, here `char` == 1 byte). [3]
- `std::isdigit` (used in `eigenstrat_format.cpp`, not here) — UB if the argument is not representable as `unsigned char` and is not `EOF`; the format TU already casts to `unsigned char`. [4]
- `ifstream` default open does **not** enable `std::ios::exceptions`, so all the above surface as state bits, not exceptions — the code's `if (!in)` / `gcount()` checks are the only line of defense. [1][3]

Cross-unit facts verified against the tree:
- The **only** callers of `GenoReader`/`read_tile` are `tests/reference/test_decode_equivalence.cu:165,195` and `tests/reference/test_filter_oracle.cu:195,213`. There is **no** `app` layer yet (planned P3), and **no dedicated unit test** for `geno_reader` (the two callers are end-to-end decode/filter equivalence tests that exercise only the clean AADR-v66 happy path with `snp_begin==0`, `snp_end==M`).
- `records_present_` is threaded into `read_ind(ind, sel, reader->records_present())` in both callers — i.e. the partial-file cap is honored *by the .ind reader*, which is what bounds `part.groups[].rows` to `< records_present_`. `read_tile` itself never re-validates the row indices it receives (Finding C2).
- `parse_geno_header` is `noexcept` and returns `format==Unknown` on a bad magic or unparsable counts; the constructor converts that to a throw — the documented `io`-leaf "probe doesn't throw across the boundary, the reader does" split (eigenstrat_format.hpp:108–111) is respected.
- `packed_bytes(n)` = `ceil(n/4)` lives once in `eigenstrat_format.hpp`; `read_tile` correctly calls it rather than re-spelling `ceil(tile_snps/4)` (ROADMAP §4 row "TGENO header 48 / ceil(nsnp/4)").

## Role & layering

`geno_reader` is a clean `io`-leaf unit per architecture §4: pure host C++20, includes only the standard library plus its own `io/` siblings (`eigenstrat_format.hpp`, `genotype_tile.hpp`, `ind_reader.hpp`), depends on nothing in `core`/`device`, contains no CUDA header, and surfaces I/O failure as `std::runtime_error` to the `app`/test caller — exactly the leaf error convention §4 prescribes ("the leaf does not use core/device error types"). It correctly does **no decode** (the §5 S0 split: `io` reads/tiles, the device/CPU kernel decodes) and **no block assignment** (that is `core::block_partition_rule`, consumed downstream). The record stride is *derived* from the parsed header, never hardcoded. Dependency direction is clean; no upward edge; move-only with `= default` move ops (it owns only a `std::string` path + a `GenoHeader` + a `std::size_t` — see Finding R3 on the now-stale "owns a file stream" comment). This skeleton is right.

The genuine tensions are (a) the unit's contract is **wider than its validation** — `read_tile` documents preconditions (`snp_begin==0`, rows in range) but only enforces some of them, leaning on a downstream short-read as the backstop; (b) it is the M5 capability-tier rewrite target and currently bakes in the pageable-`ifstream`-per-individual gather with **no capability hook, no NVTX/log tag, and a reopen-per-call** that the budget-vs-capable ingest ladder (TODO §⚡) will have to surgically replace; (c) several header comments have drifted out of sync with the code (the "owns a file stream" / "cheap to keep open across the tile loop" claims are now false). None of these bite the AADR-v66 happy path — which is why M1/M2 are green — but each is precisely the "silent corruption / fail-silent three stages later" §2 warns against, on the single trusted front-door reader.

## Score: 7/10 — correct, clean-layered, well-documented happy path, but a parity-bearing front-door reader with under-enforced preconditions, a reopen-per-tile that contradicts its own doc and the M5 design, stale comments, and zero dedicated tests

I assess **7/10**. The layering, derived-stride discipline, the deliberate no-decode/no-block-rule split, the explicit `Unknown`/`Geno`/`snp_begin!=0` rejections with informative messages, and the partial-file `records_present_` handling are all done at a senior level and match the sibling `snp_reader`/`ind_reader` units in tone. What holds it short of the 9.5 bar, for a unit on which *every downstream statistic depends bit-for-bit*: (a) `read_tile` **trusts the caller's row indices** with no `row < records_present_` (or `< n_records`) check, so a stale/oversized partition seeks past EOF and is caught only by a short-read whose message blames the wrong thing — and only if the OS happens not to return data (C2); (b) the constructor's size-validation path silently treats a **failed `tellg()` (== -1)** as "file smaller than header," a misleading diagnostic (C1); (c) `read_tile` **reopens the file on every call** and the header advertises "cheap to keep open across the tile loop" / "owns a file stream" — both now false; for the M5 per-tile loop this is an O(tiles) syscall + seek storm and directly contradicts the documented design (P1/R3); (d) there is a real but **latent** integer-overflow surface in `n_ind * bytes_per_rec` and `row * bytes_per_record` that is unguarded (E1/E2); (e) **no capability-tier seam** for the M5 GDS-vs-pinned ingest ladder the TODO mandates — no probe, no tagged log line, no config hook (CT1); (f) **zero dedicated unit tests** on a parity-bearing leaf (T1). Each is individually small; collectively they are the difference between "passes the one golden path we run" and "trustworthy front door."

## Findings

### (1) Correctness & bugs

**C1 — [med] A failed `tellg()` (returns -1) is silently misclassified as "file smaller than header."** `GenoReader::GenoReader`, lines 51–55.
```cpp
in.seekg(0, std::ios::end);
const std::streamoff fsize = in.tellg();
if (fsize < static_cast<std::streamoff>(header_.header_bytes)) {
    throw std::runtime_error("io::GenoReader: file smaller than header: " + path_);
}
```
Per cppreference, `tellg()` returns `pos_type(-1)` (→ `streamoff(-1)`) when the stream is in a failed state, and *itself* sets `failbit` on failure [2]. After the successful header `read`, `seekg(0, end)` should succeed on a regular file, so in practice `fsize >= 48`. But on a pipe / FIFO / unseekable fd, or a stream that entered `fail()` for any reason, `tellg()` returns -1, `-1 < 48` is true, and the user gets *"file smaller than header"* for a file that is not small at all — a misleading diagnostic on exactly the kind of degenerate input this constructor is supposed to fail loudly and *informatively* on (architecture §2 fail-fast: "with file/line context, not silent corruption"). **Why it matters:** the whole point of the size-validation block is honest diagnostics on a malformed/partial dataset; a -1 masquerading as "too short" sends the user down the wrong debugging path. **Fix:** check `if (!in || fsize < 0)` and emit a distinct *"cannot determine .geno size (unseekable or stream error)"* message; only then compare against `header_bytes`. Severity med (wrong message, not wrong acceptance — it still fails). Effort S. Before M4.5? No.

**C2 — [med→high] `read_tile` never validates the caller's row indices against `records_present_`/`n_records`; an out-of-range row seeks past EOF and is caught only by a short-read with a misleading message — and only if the OS returns short.** `read_tile`, lines 112–128.
The function trusts that every `row` in `part.groups[].rows` is `< header_.n_records` (and ideally `< records_present_`). Today that invariant is established *only* by the caller passing `reader->records_present()` to `read_ind` (both test callers do). But `read_tile` takes an arbitrary `IndPartition` and the header contract for it says nothing about the caller having capped rows — indeed `IndPartition` can legally be built with `n_records_present == SIZE_MAX` (ind_reader.hpp:81 "Use SIZE_MAX to use every .ind row"), in which case rows can exceed the *present* record count of a partial `.geno`. The offset math
```cpp
const std::streamoff off = header_bytes + row * header_.bytes_per_record;
in.seekg(off, std::ios::beg);
in.read(dst, bytes_per_rec);
if (in.gcount() != bytes_per_rec) throw ...short read for individual row...;
```
relies on the short-read check as the sole backstop. Two problems: (1) `seekg` past EOF is **not** an error on common implementations — it is the subsequent `read` that fails [1][3], so the guard is the `gcount()` check, which *does* fire here, but with the message *"short read for individual row N"* that frames a **caller bug** (a row index beyond the data) as an **I/O error** (a truncated file); the operator cannot tell "your `.ind`/`.geno` are mismatched" from "your disk read got cut short." (2) More subtly, an out-of-range row that happens to land within a physically larger-than-`n_records` file (e.g. trailing slack accepted by C4, or `records_present_` capped *below* `n_records` while the file is longer than `records_present_·stride`) would read **garbage bytes** as a valid record and corrupt the decode silently. **Why it matters:** §2 fail-fast + "no silent corruption"; this is the front door, and the `records_present_` cap the constructor computes is *not consulted by the very function that reads records*. **Fix:** at the top of the gather loop, validate `if (row >= records_present_) throw std::runtime_error("io::GenoReader::read_tile: individual row " + ... + " >= records present (" + ... + "); .ind/.geno mismatch or partial file");`. This makes the cap the constructor computes load-bearing and turns a misframed "short read" into a precise mismatch diagnostic. Severity med→high (corruption path is narrow but real; diagnostic is actively misleading). Effort S. Before M4.5? **Yes** — it is a one-line guard on the trust seam.

**C3 — [low] The `read_tile` SNP-range bounds check uses `n_snp`, while the per-record byte budget the seek strides over is the *full*-record `header_.bytes_per_record`; the two are consistent only because `snp_begin==0` is enforced — a correctness *trap* primed for the M5/M4.5 nonzero-begin lift.** `read_tile`, lines 80–88, 115–121.
The bounds check is `snp_end > header_.n_snp` (correct). The read length is `bytes_per_rec = packed_bytes(tile_snps)` and the stride is `header_.bytes_per_record == packed_bytes(n_snp)`. For `snp_begin==0` and `tile_snps <= n_snp`, `bytes_per_rec <= header_.bytes_per_record`, so each per-individual read is a genuine *prefix* of the record — correct. This is currently sound **only** because of the `snp_begin==0` rejection (lines 75–79). The header comment (hpp:55–61) flags the M5 hazard, so this is documented, not a bug today — but I flag it because the author who lifts the `snp_begin==0` guard (for M5 tiling *and* M4.5 SNP-range sharding, see CT2) must also handle **sub-byte misalignment**: `snp_begin % 4 != 0` puts the first wanted code mid-byte; reading from byte `snp_begin/4` then needs a bit-shift the "byte prefix" model does not do. The seek offset must become `header_bytes + row·bytes_per_record + snp_begin/4`, and the read length `packed_bytes(snp_end) - snp_begin/4`. None of this is wrong now; it is a trap. **Fix:** add a one-line TODO in the body next to the guard pointing at the offset/shift change the nonzero-begin path needs, so the constraint is co-located with the code, not only in the header prose. Severity low. Effort S. Before M4.5? See CT2.

**C4 — [low] A file *longer* than `header_bytes + n_records·stride` is silently accepted (surplus ignored), so a corrupt header count / concatenation accident is not surfaced.** Constructor, lines 56–64.
```cpp
records_present_ = data_bytes / header_.bytes_per_record;
if (records_present_ == 0) throw ...;
if (records_present_ > header_.n_records) records_present_ = header_.n_records;
```
The clamp is correct and the comment is right ("must never EXCEED the header count"). But note the asymmetry: a file *shorter* than the header claims is accepted as a partial dataset (good, matches the oracle's `n = (fsize-hdr)//bpi`), while a file *longer* than expected — usually a wrong `n_ind`/`n_snp` or a concatenation accident — is also silently accepted. A file 2× the expected size is a strong corruption signal. **Why it matters:** low — this is a "be liberal in what you accept" judgment call, and trailing slack is harmless to the gather *given the C2 guard* (rows beyond `n_records` are never read). But it interacts with C2 (slack widens the garbage-read window) and §2 fail-fast leans toward surfacing the surprising case. **Fix (optional):** when `data_bytes >= (n_records+1)·stride` treat it as suspicious (the `io` leaf has no logger, so at minimum document that surplus is intentionally ignored). Severity low. Effort S. Before M4.5? No.

### (2) Edge cases & failure modes

**E1 — [med] Unguarded `std::size_t` overflow in `tile.packed.resize(n_ind * bytes_per_rec)`.** `read_tile`, line 98.
`n_ind` is summed from `part.groups[].rows.size()` (≤ total individuals) and `bytes_per_rec = packed_bytes(tile_snps)`. For the real AADR v66 (`n_ind=27594`, `n_snp=584131` → `bytes_per_rec ≈ 146033`) the full product is ~4.0e9 — within 64-bit but a **multi-GB host vector allocation** with no budget check (architecture §11.1/§11.2: the whole point of tiling is to *not* materialize the dataset; the M1 "decode a small set in one shot" is the only reason this is acceptable today). On a 32-bit `std::size_t` platform (not the target, but the code asserts nothing) `n_ind * bytes_per_rec` wraps and `resize` under-allocates → the subsequent writes are a buffer overflow. **Why it matters:** the resize can throw `std::length_error`/`std::bad_alloc` with no context, and there is no fail-fast host-side budget guard — exactly the "OOM three stages later" §2/§11.2 call out. **Fix:** (a) a defensive overflow check before the multiply (`if (bytes_per_rec != 0 && n_ind > SIZE_MAX / bytes_per_rec) throw ...`); (b) longer-term the M5 tile loop replaces this whole-set materialization (the header says "M5 will call this per SNP tile"), so the resident host buffer becomes per-tile — but until M5, a guard + a comment that this is intentionally un-tiled in M1 is the senior move. Severity med (latent; the test path stays at M=100k so ≤ ~3 GB). Effort S. Before M4.5? No.

**E2 — [low] `row * header_.bytes_per_record` cast to `std::streamoff` can overflow for a pathological/corrupt header.** `read_tile`, lines 115–118.
`std::streamoff` is signed 64-bit on the target, so for realistic counts there is no overflow. But a hostile/corrupt header (huge `n_snp` → huge `bytes_per_record`) times a large `row` could exceed `INT64_MAX` → negative `off` → `seekg(neg, beg)` sets `failbit`, surfacing as the misleading "short read" of C2 rather than "offset overflow / corrupt header." Bounded by the C2 row guard (a valid `row < records_present_` keeps `off` within the validated file size). **Fix:** subsumed by C2's row guard + C1's size sanity; no separate change. Severity low. Effort S. Before M4.5? No.

**E3 — [low] An empty selection (`part.groups` empty, or all groups empty) yields `n_ind==0`, a zero-length `packed`, and `pop_offsets=={0}` — returned silently as a valid "no data" tile.** `read_tile`, lines 91–98, 110–130.
`read_ind` already throws on an empty selection (ind_reader.cpp:115), so the current pipeline never produces one. But `read_tile` accepts any `IndPartition`; a partition with zero rows produces a degenerate tile (`n_individuals==0`, possibly `n_pop>0` with empty segments) that the downstream decoder would then no-op or divide-by-zero over. **Why it matters:** low — defense-in-depth on the leaf's own invariants. **Fix (optional):** `if (n_ind == 0) throw "read_tile: empty selection"` to fail at the seam rather than emit a tile that silently means "no data." Severity low. Effort S. Before M4.5? No.

**E4 — [low] The `snp_end <= snp_begin` and `snp_end > n_snp` rejects share one "out of bounds" message phrasing that reads oddly for the empty-range case.** `read_tile`, lines 80–85. For TGENO the SNP axis is `n_snp` (a header count) and the per-record byte length is fixed regardless of how many *individuals* are present, so checking `snp_end > n_snp` is the right axis (no partial-file interaction on the SNP side) — correct. The only nit: the degenerate `[0,0)` case prints `[0, 0) out of bounds` which conflates "empty range" with "exceeds n_snp." Severity low (cosmetic). Effort S. Before M4.5? No.

**E5 — confirmed safe (no finding): `bytes_per_rec == 0` is impossible here.** `tile_snps = snp_end - snp_begin >= 1` (guaranteed by the `snp_end <= snp_begin` reject), so `packed_bytes(tile_snps) >= 1`; the constructor separately rejects `bytes_per_record == 0`. The genotype_tile.hpp divide-implied invariant (`packed.size() / bytes_per_record`) holds. N/A.

### (3) Numerical / precision vs §12

N/A for the arithmetic core — this unit does **no** floating-point work, no accumulation, no Ozaki/native-FP64 decision; it moves bytes. That is correct: §5 S0 explicitly places decode (and thus every precision concern) in `core::decode_af` / the device kernel, not in `io`. The one numerics-adjacent observation, recorded as a *positive*: the byte order this reader hands off (MSB-first 2-bit codes per `eigenstrat_format::code_in_byte`) is the single bit-order source the decoder consumes; `read_tile` correctly does *not* touch or reinterpret the bits, preserving that single-home invariant (§8). No finding.

### (4) CUDA idioms / RAII / streams vs §7

Mostly N/A — this is a host-pure `io` leaf with no CUDA, no kernels, no streams, no device allocation, by design (§4). The RAII surface is `std::ifstream` (auto-closed) and `std::string`/`std::vector` (the tile). Two observations:

**R1 — [low] The constructor opens an `ifstream`, reads + validates, then lets it close at scope end; `read_tile` opens a *second, independent* `ifstream` on every call.** Constructor line 25; `read_tile` line 102. This is the RAII-correct *local* idiom (each stream scoped + closed), but it means the class **does not actually own an open file handle** despite the header claiming it does (R3, P1). Not a leak; flagged under performance/decomposition. No correctness issue.

**R2 — [design note, no finding] The M5/M4.5 pinned-buffer / `cudaMemcpyAsync` / GDS ingest CANNOT live in this leaf — §4 forbids `io`→`device`.** TODO ⚡ item 2 says replace the per-individual gather with "large sequential reads into `PinnedBuffer` ring slots + `cudaMemcpyAsync` on a dedicated copy stream" — but `PinnedBuffer<T>` is a `device` RAII type (architecture §7) and `io` may not include a CUDA header. So the M5 redesign must keep `io` producing *plain bytes into a host buffer the app pins*, with the pinning/copy/transport owned by `app`/`device`. The current code respects this (returns a plain `GenotypeTile`), and the M5 author must preserve it. See CT1. No finding against current code.

### (5) Magic numbers & hardcoded values vs §4 / ROADMAP §4

**Clean — at the 9.5 bar.** Every format literal is sourced from `eigenstrat_format.hpp`: `kGenoHeaderBytes` (the 48), `packed_bytes()` (the `ceil(nsnp/4)`), `header_.header_bytes`, `header_.bytes_per_record`. There is **no** open-coded `48`, no `/4`, no `ceil`. This is exactly the ROADMAP §4 row ("TGENO header 48 / ceil(nsnp/4) belong in io format constants, derived from the header parse") satisfied. The only integer literals in the TU are `0` (sentinels / `seekg(0, end)` / `pop_offsets.push_back(0)`) and the `+1` in `pop_offsets.reserve(part.groups.size() + 1)` — both true structural constants (segment-count = pop-count + 1), the kind ROADMAP §4 explicitly exempts ("true mathematical constants"). No finding.

### (6) Decomposition / single-responsibility / function size vs §2

**D1 — [low] `read_tile` is one ~65-line function doing four things: validate (format/begin/range), size the output, open the stream, run the nested gather loop.** Lines 67–132. Readable and commented, so borderline. But the gather loop (110–130) — "for each pop, for each row, seek+read+check, advance, record offset" — is a self-contained operation that would read better as a private `gather_record(in, row, dst_span)` helper, which is *also* the natural home for the C2 row-bounds guard and makes the offset math testable in isolation (T1/T2). Severity low. Effort S. Before M4.5? No.

**D2 — [low] The constructor mixes header-read, format/degeneracy validation, and size-validation in one body.** Lines 24–65. Readable; at 41 lines under any hard threshold. The size-validation block (47–64) holds the only subtle logic (the partial-file cap) and would be a candidate `validate_size(in)` helper for unit-testing the cap math against synthetic file sizes with no filesystem (T1/T2). Severity low. Effort S. Before M4.5? No.

### (7) Readability, naming, const-correctness, [[nodiscard]]/noexcept, comments

**R3 — [med] Stale header comments: the class no longer "owns a file stream" and `read_tile` is NOT "cheap to keep open across the tile loop."** hpp:34 ("Move-only (it owns a file stream); cheap to keep open across the tile loop") and hpp:31–33. The class owns a `std::string path_`, a `GenoHeader header_`, and a `std::size_t records_present_` — **no stream**. The move ops are `= default` precisely because there is no stream to transfer (hpp:71–72). `read_tile` *reopens* the file every call (cpp:102), so "cheap to keep open across the tile loop" is the opposite of what the code does. **Why it matters:** §2/§7 "comment density matching surrounding code" and the sibling-review bar — a comment asserting an ownership model the code abandoned is worse than no comment; it will mislead the M5 author into thinking open-handle reuse already exists. **Fix:** change to "Move-only and cheaply movable (owns only the path + parsed header; `read_tile` opens the file per call — M5 rewrite target)" and drop the "owns a file stream" clause. Severity med (actively misleading on the exact unit slated for the M5 rewrite). Effort S. Before M4.5? No, but cheap.

**R4 — [low] `read_tile` is not `const` though it mutates nothing in `*this`.** It is presumably left non-const to leave room for an M5 cached-stream member — a defensible forward-looking choice — but given R3's stale comment it currently reads like an oversight. **Fix:** either make it `const` now (it touches no member mutably) and note that M5 will introduce a `mutable` cached stream, or add a one-line comment that non-const is deliberate for the upcoming handle cache. The `[[nodiscard]]` on `read_tile` and `[[nodiscard]] ... noexcept` on the accessors are correct; the throwing constructor correctly is not `noexcept`. Severity low. Effort S. Before M4.5? No.

**R5 — [low] `reinterpret_cast<char*>(tile.packed.data() + out_ind * bytes_per_rec)` (uint8_t*→char* for `istream::read`) is correct but uncommented.** cpp:120. This is the standard, well-defined idiom (`char`/`unsigned char`/`std::uint8_t` are inter-accessible for object representation; `read` requires `char*`). A brief `// uint8_t* -> char* for istream::read (object-representation aliasing, well-defined)` would match the surrounding comment density and pre-empt a reviewer double-take. Severity low. Effort S. Before M4.5? No.

**R6 — [low] In-loop raw pointer offset where a `std::span` subspan would be clearer and debug-bounds-checked.** cpp:120. §7's "span/mdspan views not raw pointers" targets *kernel* interfaces, so it does not strictly bind a host leaf — but `std::span{tile.packed}.subspan(out_ind*bytes_per_rec, bytes_per_rec)` reads more clearly and bounds-checks in debug. Minor. Severity low. Effort S. Before M4.5? No.

### (8) Performance

**P1 — [med] `read_tile` reopens the `.geno` on every call, then issues one `seekg`+`read` per selected individual (a random-access scatter-gather).** cpp:102 (reopen) and the loop cpp:114–127.
  - For M1 (one `read_tile` call) the reopen is one `open()` syscall — negligible. But the header (hpp:34) *sells the class as "cheap to keep open across the tile loop,"* and the M5 design (TODO ⚡) is an out-of-core tile loop calling this per SNP tile → **reopening per tile** is an O(n_tiles) `open`/`close` storm plus loss of OS read-ahead, on top of the per-individual random seeks.
  - The per-individual gather is inherently random-access (individual rows scattered across pop-contiguous output); for the AADR file each record is ~146 KB apart, so a 50-pop / thousands-of-individuals gather is thousands of seeks. The TODO explicitly names this: *"replace `geno_reader`'s pageable `ifstream` per-individual gather with large sequential reads."*

**Why it matters:** this is the documented M5 bottleneck and the unit's reason-for-rewrite. For M1 it is fine; I flag **med** because the header comment *claims the optimized behavior already exists* (R3) and the reopen+scatter structure is what M5 must dismantle — leaving it without a marker invites a future maintainer to "optimize" it ad hoc rather than per the planned ingest ladder. **Fix (now):** add the M5 marker comment; optionally hold the constructor's `ifstream` as a member and reuse it in `read_tile` (cheap win, makes the "owns a file stream" comment true) — but only if the M5 design wants a single persistent fd (the pinned-ring design may prefer raw-fd `pread`, so coordinate). Severity med (perf latent at M1; the misleading comment is current). Effort S now / L for the full M5 rewrite. Before M4.5? No (M5).

**P2 — [positive, no finding] The `reserve` calls (cpp:99–100) correctly pre-size `pop_offsets`/`pop_labels`, avoiding reallocation.** The per-call `pop_labels.push_back(g.label)` string copies are negligible (P ≤ ~50 short strings). Keep.

### (9) Layering / API / ABI vs §4

**Clean and a model for the leaf.** No CUDA, no `core`/`device` include, no upward dependency; emits a plain `GenotypeTile` (std::vector/POD) that crosses to `core::decode_af` with no `io`→core/device edge (genotype_tile.hpp:11–13). Errors are `std::runtime_error`, the documented leaf convention. One API observation:

**A1 — [low, folded into C2] `read_tile` takes `const IndPartition&` but has an implicit precondition (rows ≤ records_present) that is neither in the type nor enforced.** This is an API-contract gap: the function's safety depends on an invariant established by a *different* function (`read_ind` with the right `n_records_present`). A leaf API should enforce the precondition (C2's runtime guard — the proportionate fix) or encode it (a typed `BoundedPartition` — overkill here). Severity low. Effort S. Before M4.5? See C2.

### (10) Testability vs §13

**T1 — [med] Zero dedicated unit tests for `geno_reader`; the only coverage is two end-to-end `tests/reference/*.cu` decode/filter equivalence tests that exercise *one* path (real AADR, `snp_begin==0`, `snp_end==M`, auto-top-50, full file).** Verified: `grep` finds `GenoReader`/`read_tile` only in `test_decode_equivalence.cu` and `test_filter_oracle.cu`, both happy-path. Architecture §13 wants "pure host numerics tested in plain `.cpp` on the host — no GPU"; this is a *pure host* unit yet has **no host unit test**, while siblings (`eigenstrat_format`, the filter predicates) do (per the M2 DONE note). The untested surfaces are exactly the finding list: the partial-file cap (C4), the `Unknown`/`Geno` rejections (cpp:39–45, 70–74), the `snp_begin!=0` reject (75–79), the out-of-range SNP range (80–85), and — once added — the C2 row guard and C1 size-failure path. **Why it matters:** §13/§18 ("every statistic-bearing path has a reference test"); this is the front-door reader and a regression here silently poisons every downstream golden. None of these need a GPU or the 4 GB AADR file — a handful of tiny synthetic `.geno` buffers (a 48-byte header + a few records, written to a temp file) fully exercise parse + gather + partial-file + every rejection. **Fix:** add `tests/unit/test_geno_reader.cpp` (GoogleTest) covering: valid TGENO open + stride derivation; partial file → `records_present_` cap; over-long file (C4); bad magic → throw; GENO file → `read_tile` throws; `snp_begin!=0` → throw; `snp_end>n_snp` → throw; gather order/pop_offsets correctness on a 2-pop synthetic; and the C2 row-OOB throw once added. Severity med. Effort M. Before M4.5? **Yes** — cheapest highest-leverage item; it also locks in the C1/C2 fixes.

**T2 — [positive, no finding] The unit is testable as-is (no DI needed) — path-based, no globals, no CUDA.** The only friction is that it opens by path, so a test writes a temp file rather than feeding an in-memory buffer — acceptable for an `io` reader. Extracting the D1 `gather_record` / D2 `validate_size` helpers would push the offset/cap math to *pure-memory* testability. Positive note; the extraction would raise the ceiling.

### (11) Capability tiers (PRO-6000-capable vs budget-5090) — where THIS unit touches them

**CT1 — [med] This unit is the M5 ingest capability-tier seam, and it currently has no probe, no tagged log line, and no config hook for the GDS-vs-pinned-double-buffer ladder the TODO mandates.** TODO §⚡ "Capability tiers" row **M5/M7 ingest**: *capable path = true GDS cuFile DMA NVMe→VRAM (probe `DMA_BUF_SUPPORTED`/`cuFileBufRegister`); budget-5090 fallback = pinned double-buffer POSIX-pread, with the logged reason "GDS unavailable (GeForce GPU-class / OverlayFS) → POSIX pread into pinned double-buffer."* The cross-cutting requirement is *"a capability probe + capability-tagged results (every run records which path it took + why it degraded), slotting into `DeviceConfig`/`Resources`."* `geno_reader` is the byte-ingest unit those tiers act on; today it is a single pageable-`ifstream` path with **no awareness of which tier it is on and no tag emitted**.

This is correct *for M1* (M5 unstarted; the budget box is the only target; and the leaf cannot depend on CUDA so the GDS path can't live here — R2). But the seam should be anticipated:
  - The leaf must stay CUDA-free, so the **GDS/pinned decision and the tag live in `app`/`device`, not in `geno_reader`** — the leaf exposes a *plain* byte-producing interface (it does) and the caller chooses transport. The M5 author must NOT push `cuFile`/`PinnedBuffer` into this TU (breaks §4 and the "io is a leaf, no CUDA" invariant — R2).
  - What this unit *should* expose for M5 is a lower-level "read raw record bytes for a SNP-tile × individual-set into a **caller-provided** host span" entry point, so `app` can hand it a pinned slot to fill — keeping pinning in `device`/`app` and byte-layout knowledge in `io`. The current `read_tile` allocates its *own* `std::vector` (cpp:98), which a pinned-ring caller cannot use; that allocation is exactly what M5 must invert.

**Why it matters:** the TODO makes "build for capable, degrade with an explicit logged tag, parity-neutral" a cross-cutting standard, and this is the unit the M5 ingest tier physically operates on. Getting the *seam shape* right now (plain bytes out, caller owns buffer/transport/tag) avoids a layering violation later. **Fix (now, doc-only):** add a header note that the M5 ingest tiers (GDS vs pinned-pread) are owned by `app`/`device` and this leaf will gain a "fill a caller-provided host span" overload — keeping the GDS/pinned/tag decision out of the leaf. No code change at M1. Severity med (forward-looking design correctness on the named M5 unit). Effort S (doc) now / L (M5). Before M4.5? No (M4.5 is the multi-GPU combine, which does not touch this reader; M5 is ingest). The note is cheap to add now.

**CT2 — [low] M4.5 (multi-GPU SNP-shard + parity combine) shards by SNP range and therefore needs the nonzero-`snp_begin` path — the same guard M5 lifts — but the header attributes the nonzero-begin need solely to M5.** Each GPU streams its own SNP range (ROADMAP M4.5 "its SNP range/tiles"), which is precisely the `[snp_begin, snp_end)` interface — *except* the `snp_begin != 0` reject (cpp:75) forbids the nonzero begin a sharded stream needs. So M4.5's per-device SNP-range sharding is gated behind the same `snp_begin==0` lift (and the C3 sub-byte-alignment work) as M5. The header (hpp:59–61) calls nonzero begin "the M5 tile loop"; M4.5 shares it. **Fix:** broaden the hpp:59–61 comment to "nonzero begin is the M5 tile loop / M4.5 SNP-range shard." Severity low (doc accuracy). Effort S. Before M4.5? **Yes if M4.5 shards by SNP range through this reader** — confirm the shard granularity; if M4.5 shards whole-file SNP ranges it hits this guard.

## Considered & rejected

- **"The seek stride should use `bytes_per_rec` (the tile prefix), not `header_.bytes_per_record`."** Rejected — the stride between *individual records on disk* is the full record length `header_.bytes_per_record = packed_bytes(n_snp)`; we read only a `bytes_per_rec`-long prefix of each. Striding by `bytes_per_rec` would land mid-record. The code is correct (cpp:117–118 uses the full stride; cpp:121 reads the prefix). This is the crux of why the gather is right, and it is right.
- **"`packed_bytes` could overflow computing `n_codes + 3`."** Rejected — `n_codes = tile_snps <= n_snp` (header count, realistically < 2^31); `+3` cannot overflow `std::size_t`. The overflow surface is the *product* `n_ind * bytes_per_rec` (E1), not the stride formula.
- **"`std::isdigit` UB in the header parse."** Rejected for *this* unit — the digit parsing is in `eigenstrat_format.cpp` (separate unit), which correctly casts `static_cast<unsigned char>(...)` (verified eigenstrat_format.cpp:52,56) [4]. `geno_reader.cpp` does no character classification.
- **"The constructor leaks the `ifstream` if `parse_geno_header` throws."** Rejected — `parse_geno_header` is `noexcept` (eigenstrat_format.hpp:111) and the `ifstream` is a stack RAII object closed on any scope exit regardless. No leak.
- **"Move ops should be deleted because the class owns a file."** Rejected — the class owns **no** file handle (R3); `= default` move on `string`+`GenoHeader`+`size_t` is correct and the `noexcept` is justified (all members nothrow-movable). The *comment* claiming stream ownership is the bug (R3), not the move ops.
- **"`read_tile` should simply be `const`."** Partially rejected / folded into R4 — it touches no member mutably so `const` would compile, but the deliberate non-const leaves room for the M5 cached-stream member; flagged as a doc gap (R4), not a defect.
- **"The §12 parity caveats (`Vpair==0⇒0`, het-correction, N-convention) apply here."** Rejected — those live in `core::f2_estimator`/`decode_af`; `geno_reader` produces only packed bytes + population segments, no statistic. Correctly out of scope (category 3 N/A).
- **"`records_present_ > n_records` clamp hides a real error."** Considered, demoted to C4 (low) — the oracle tolerates short files; the long-file case is a judgment call, not a clear bug, so it is a low-severity "surface the surprising case" suggestion.
- **"`pop_offsets`/`pop_labels` could desync from `groups`."** Rejected — they are filled in lockstep in the single loop (push label, then push offset per group; cpp:113,129) with `pop_offsets[0]=0` seeded (cpp:111). The genotype_tile.hpp invariant `pop_labels.size() == pop_offsets.size()-1` holds by construction. Correct.
- **"`seekg` past EOF should be detected immediately."** Rejected as a *separate* bug — per [1] `seekg` past EOF is not a guaranteed error; the `read`+`gcount()` check is the correct, portable detection point. The real gap is the *missing semantic guard* (C2), not the seek itself.

## Good patterns to keep

- **Derived-stride discipline (no magic numbers).** Every format constant comes from `eigenstrat_format.hpp` (`kGenoHeaderBytes`, `packed_bytes`, `header_.bytes_per_record`); not a single open-coded `48` or `/4`. ROADMAP §4 satisfied exactly; this category is already at the 9.5 bar.
- **Correct disk-stride-vs-read-prefix separation.** Striding by the full `header_.bytes_per_record` while reading only the `packed_bytes(tile_snps)` prefix is the subtle thing the gather has to get right, and it does.
- **Explicit, informative, layered rejections.** `Unknown` magic, `Geno`-into-`read_tile`, `snp_begin!=0`, and out-of-range SNP range each throw a `std::runtime_error` with a *specific* message naming the file and the reason (and, for format/begin, the M1-vs-M5 rationale). §2 fail-fast + §4 leaf-error convention done well.
- **Honest partial-file handling matching the oracle.** `records_present_ = data_bytes / bytes_per_record`, clamped to `n_records`, mirrors the oracle's `n = (fsize-hdr)//bpi` and threads through to `read_ind` — the right way to tolerate a partial dataset without inventing data.
- **Clean leaf layering + plain-data output.** No CUDA, no upward dependency; returns a POD/vector `GenotypeTile`; the no-decode / no-block-rule split (§5 S0, §8) respected to the letter.
- **`reserve` before the push loops** on `pop_offsets`/`pop_labels` — small, correct, avoids reallocation.
- **`[[nodiscard]] ... noexcept` accessors and `[[nodiscard]]` on `read_tile`** — const-correct getters; the only non-`noexcept` member is the throwing constructor, which is correct.
