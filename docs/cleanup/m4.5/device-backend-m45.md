### (8) Performance (first-class this pass)

**8.1 — [N/A in the delta, audited and cleared] The `BackendCapabilities` POD and the `capabilities()` base default contain ZERO performance surface; the probe that fills it is O(G) host calls off every hot path. (lines 144–185, 304–306; override `cuda_backend.cu:470–547`)**
The mandate makes performance first-class, so I audited the full probe + consumer chain for the five named anti-patterns and found none this header owns:
- **Data bouncing** — none. The probe issues no `cudaMemcpy`/`cudaMemcpyPeer`; it reads scalars (`cudaGetDeviceCount`, `cudaGetDevice`, `cudaSetDevice`, `cudaGetDeviceProperties`, `cudaMemGetInfo`, `cudaDeviceCanAccessPeer`). `BackendCapabilities` is 40 bytes (3×`int` = 12, +4 pad, +2×`size_t` = 16, +2×`bool` = 2, +6 tail-pad ⇒ 40), returned by value (NRVO/register-stack move). `PerGpuResources::caps = backend->capabilities()` (`resources.cpp:92`) is a 40-byte copy, once per device.
- **Grid-stride / launch config** — N/A; no kernel in the delta or the probe.
- **Sequential transfers that should overlap** — N/A; the probe moves no bulk bytes. (The *combine* that reads `can_access_peer` does — but `cuda/p2p_combine.cu`'s sequential `cudaMemcpyPeer` fan-in is a `p2p_combine.cu` finding, not a `backend.hpp` finding; the header only carries the bit that selects it.)
- **Type-casting noise** — none in the POD. `device_count`/`compute_major`/`compute_minor` are `int` mirroring `cudaGetDeviceCount`/`cudaDeviceProp::major,minor` out-params (the override assigns int→int, no cast: `cuda_backend.cu:479, 492–493`); `*_vram_bytes` are `std::size_t` mirroring `cudaMemGetInfo` out-params (size_t→size_t: `:502–503`). Widths match the CUDA APIs exactly, so the override copies with zero conversion. The only cast in the consumer chain (`resources.cpp:60` `static_cast<std::size_t>(visible)` for `reserve`) is correct and not on a hot path.
- **Needless sync / default-stream / hot-path alloc** — the probe does a `cudaSetDevice`+restore (2.3/4.2), no stream sync, no allocation. It is called once per device at `build_resources` construction, never on the streaming path.

Expected benefit of "fixing" anything here: zero. Risk of touching it: introducing a parity-relevant change to a parity-neutral struct. **Severity N/A. Effort N/A. PARITY-SAFE? yes (nothing to change).** Recorded so the perf axis is explicitly discharged.

**8.2 — [low, perf-adjacent, override/builder not header] The CUDA `capabilities()` override re-enumerates + re-runs the full P2P peer loop on EVERY call, and `build_resources` auto-enumeration constructs a THROWAWAY device-0 backend solely to read `device_count`. (`resources.cpp:52–53`; override `cuda_backend.cu:477–528`)**
Not a `backend.hpp` defect — the header correctly leaves the probe `const` and side-effect-free — but the unit-focus names probe efficiency, so I record it where the cost lives. `resolve_device_order` (`resources.cpp:52–53`) does `auto probe = make_cuda_backend(0); const int visible = probe->capabilities().device_count;` purely to learn the device count, then destroys the backend (constructing + tearing down a cuBLAS handle + `kCublasWorkspaceBytes` emulation workspace on device 0 — confirmed against `cuda_backend.cu:596–598`) before constructing the real per-device backends. For a G-device auto-enumerate run this is one extra full backend construct/destruct. `device_count` *is* the right field to expose (it keeps `resources.cpp` CUDA-free), but a lighter CUDA-free "count-only" probe (e.g. a free `device::visible_device_count()` in `backend_factory.hpp`) would avoid the throwaway cuBLAS context. One-time at build, negligible vs the run ⇒ cleanliness not throughput. **Severity low. Effort S. PARITY-SAFE? yes.** Out-of-strict-scope (a `resources.cpp`/factory finding); recorded for completeness because the unit-focus names the probe. (See also the `device-resources.md` review.)

### (1) Correctness & bugs

**1.1 — [low, CONFIRMED] The `total_vram_bytes`/`free_vram_bytes` field doc encodes a `cudaMemGetInfo` ORDERING fact ("writes free then total", line 162) on the CUDA-free POD — VERIFIED CORRECT against the NVIDIA Runtime API, but the API-ordering rationale belongs in the override, not the header. (lines 159–164)**
Verified against the published CUDA Runtime API that `cudaMemGetInfo(size_t* free, size_t* total)` writes `free` first, `total` second (Sources). The override (`cuda_backend.cu:500–503`) calls `cudaMemGetInfo(&free_b, &total_b)` and assigns `caps.free_vram_bytes = free_b; caps.total_vram_bytes = total_b;` — correct. So both the *fact* and the *code* are right. The only nit: the header field doc puts the CUDA-API ordering detail ("CUDA Runtime API: cudaMemGetInfo writes free then total") into a CUDA-free POD's prose; the field doc should say *what the field means* (total physical VRAM for the §11.2 budget / live free headroom), and the ordering rationale belongs in the override (where `cuda_backend.cu:498–499` already repeats it verbatim). U5 asserts the *runtime* invariant `free <= total` (`test_backend_capabilities_probe.cu:155–156`) — the real guard against a swapped assignment — so the ordering note in the header earns its keep only as redundant commentary. Fix: trim the field doc to the semantic; leave the `cudaMemGetInfo`-ordering note solely in the override. **Severity low (doc-placement). Effort S. PARITY-SAFE? yes.**

**1.2 — [N/A, verified positive] The value-initialized "unknown" defaults are internally consistent and match the base `capabilities()` return. (lines 144–185, 304–306)**
Every member has a `= 0`/`= false` default member initializer, so `BackendCapabilities{}` is all-zero/false, and the base `capabilities()` returns exactly `BackendCapabilities{}`. U2 proves both with one `constexpr defaults_all_unknown(...)` asserted on `BackendCapabilities{}` (line 81, compile-time `static_assert`) and on `base.capabilities()` (line 120, runtime). Crucially, **every default is the SAFE/DEGRADE side of its lever**: `device_count==0` (no CUDA device — CPU backend's honest answer), `{0,0}` compute cap (unknown, never a real sm), `0` VRAM (unknown), `can_access_peer==false` (host-staged baseline), `emulated_fp64_honorable==false` (native FP64). An unprobed/CPU backend therefore reports the host-staged, native-FP64, no-peer tier — never accidentally claiming a capable fast-path. Correct fail-safe direction. Retained as positive.

**1.3 — [N/A, verified positive] No field can encode a self-inconsistent capability a consumer would misuse.** The only cross-field invariant a consumer relies on is `free <= total` (runtime, asserted in U5). The consumer gate reads a single field (`gpus[0].caps.can_access_peer`); there is no field whose value the header could constrain at compile time beyond the types. (NB: the draft's claim that the gate "also checks `G >= 2`" is corrected here — see 11.5: the actual gate has no `G` term.) Not a defect.

### (2) Edge cases & failure modes

**2.1 — [low, CONFIRMED] The header does not document what `capabilities()` returns on a SINGLE-device process, an EMPTY-`devices` auto-enumerate, or an ISOLATED device — all of which collapse `can_access_peer` to false (correct, but undocumented at the seam). (lines 167–175, 285–306)**
Override behavior (`cuda_backend.cu:515–529`): `can_access_peer` is set true iff the bound device reaches *at least one* other visible device; a lone device (`count==1`, the loop body never runs) or a device whose every peer answers "no" both yield false. Consumers depend on this: the gate (`f2_blocks_multigpu.cpp:171–172`) reads `gpus[0].caps.can_access_peer`, and the `G==1` fast-path (`:88`) returns before the gate so a 1-device run never reads it for combine selection — but `Resources` still *probes and stores* it (`resources.cpp:92`). U5 (`test_backend_capabilities_probe.cu:201–202`) asserts `can_access_peer == false` on a single visible device. Correct and tested, but the **header field doc** describes only the two-device PRO-vs-GeForce dichotomy + the value-init default; it does not state `count==1 ⇒ false`. Fix: add to the `can_access_peer` doc: "On a single visible device (or a device with no reachable peer) this is false by construction — no peer to pull from, so the host-staged baseline is the only meaningful combine; the value-initialized default agrees." **Severity low (doc-completeness on the gate-critical field). Effort S. PARITY-SAFE? yes.**

**2.2 — [low, CONFIRMED] The probe is documented NON-throwing only for the P2P answer; the header does not state that the OTHER queries DO throw — so the exception contract of `capabilities()` is half-specified at the seam, which reads as "fully non-throwing." (lines 296–303)**
The header (lines 300–303) says "The probe is NON-throwing: a 'no peer access' answer … is a tagged degrade, not a fault." Verified true for `can_access_peer`: the override routes `cudaDeviceCanAccessPeer` through `STEPPE_CUDA_WARN` (`cuda_backend.cu:522–523`), the non-throwing sibling (`check.cuh:181`). But the override's other queries — `cudaGetDeviceCount` (`:478`), `cudaGetDevice` (`:484`), `cudaSetDevice` (`:485, 545`), `cudaGetDeviceProperties` (`:491`), `cudaMemGetInfo` (`:501`) — all go through the **throwing** `STEPPE_CUDA_CHECK` (`check.cuh:168`). A backend that cannot read its own device IS a fault (correct, fail-fast §2). The override's own doc states this split correctly (`cuda_backend.cu:461–463`), and `resources.cpp:17–22` documents it. But the *header* (the contract source) states only the non-throwing half. Fix: one clause — "the peer-access query is non-throwing (tagged degrade); the device-count/properties/mem-info queries throw `CudaError` on a genuine fault (a backend that cannot read its own device is an error, §2)." This interacts with the prior review's 7.2 (the whole interface's exception contract is undocumented) — the M4.5 `capabilities()` makes it worse by documenting *half*. **Severity low. Effort S. PARITY-SAFE? yes.**

**2.3 — [low, CONFIRMED] The `const`-ness + DEVICE-NEUTRALITY contract of `capabilities()` (the override save/restores the ambient device) is a real, load-bearing property NOT stated on the interface. (line 304 `const`; override `cuda_backend.cu:481–485, 544–545`)**
The override is device-neutral: `cudaGetDevice(&entry_device)` (`:484`), `cudaSetDevice(device_id_)` (`:485`), queries, then `cudaSetDevice(entry_device)` (`:545`) to leave no side effect — so `Resources` can probe each per-device backend in turn from any ambient context (`resources.cpp:88–94` loops). The interface declares `capabilities() const` (logical constness), but device-neutrality (no observable `cudaSetDevice` side effect) is an *additional* contract a future override must honor and a caller relies on, documented only in the override (`cuda_backend.cu:465–469`), not the interface. Fix: add to the interface doc "MUST be device-neutral: an implementation that switches the current device to probe it MUST restore the ambient device, so calling `capabilities()` never leaks a `cudaSetDevice` (the per-device `Resources` probe relies on this)." **Severity low (an unstated implementer obligation on the seam). Effort S. PARITY-SAFE? yes.**

**2.4 — [N/A, verified positive] No integer-width / overflow edge in the POD.** `int` device_count/major/minor mirror the CUDA out-params (a real machine has 1–8 devices; sm major/minor are single/low-double digits — never near `INT_MAX`). `std::size_t` `*_vram_bytes` matches `cudaMemGetInfo` and holds the §0 top-end ~96 GB without overflow. No cast in the override. Not a defect.

**2.5 — [low, CONFIRMED] The header is silent that the strict probe assertions are TIER-KEYED (PRO ⇒ `can_access_peer==true` mandatory; GeForce ⇒ false is a PASS), so a reader of `backend.hpp` cannot tell that `true` is *expected-mandatory* on one tier and *expected-absent* on another. (lines 167–175; tests `test_backend_capabilities_probe.cu:174–203`, `test_resources_build.cu` header)**
The field doc (lines 167–175) describes "true is the PRO 6000 … case … false is EXPECTED on consumer GeForce." This is correct prose, but U5 and I1 turn it into a hard assertion keyed on `strstr(prop.name, "PRO")` (`test_backend_capabilities_probe.cu:105–108, 174–194`). The header neither names the keying mechanism nor warns that the *probe value itself* is not invariant across tiers — only `compute_major >= 12`, `total > 0`, `free <= total`, `device_count >= 1`, and `honorable == library predicate` are tier-invariant. A future maintainer reading only the header could write a portable assertion `caps.can_access_peer == true` and it would correctly fail on the 5090 box. The header's existing prose mostly covers this; the gap is that it does not explicitly enumerate the *tier-invariant* vs *tier-keyed* fields. Fix (optional): one line — "NB: among these fields only compute cap (>=12), total VRAM (>0), free<=total, device_count, and honorability are tier-INVARIANT; `can_access_peer` is tier-KEYED (mandatory-true on datacenter/PRO Blackwell, expected-false on consumer GeForce) — never assert it portably." **Severity low (doc-completeness; the tests carry the real keying). Effort S. PARITY-SAFE? yes.**

**4.2 — [low, override not header] The probe's `cudaSetDevice`/restore is hand-rolled save/restore, not a scoped RAII guard — on a throw between set and restore the ambient device is left as `device_id_`. (`cuda_backend.cu:485, 544–545`)**

PARITY-SAFE? yes.** (See the `device-cuda-cuda_backend-m45.md` review for the override-side treatment.)

### (11) Capability-tier coherence (the unit-focus item)

**11.1 — [med, CONFIRMED — THE unit-focus G>2 finding] `can_access_peer` is a SINGLE BOOLEAN that (a) is "true iff the bound device can reach AT LEAST ONE peer" and (b) is only ever read for the ROOT device (`gpus[0]`); for G>2 with a PARTIALLY-connected fabric this bit cannot prove the combine's actual transport (root→EVERY peer) is reachable — correct for the shipped G==2 gate, but the header does not state the invariant. (lines 167–175; consumer `f2_blocks_multigpu.cpp:171–188`; override `cuda_backend.cu:516–529`; P2P contract `p2p_combine.hpp:64–76`)**
The heart of the unit-focus. End to end:
- The override (`cuda_backend.cu:516–528`) sets `can_access_peer = true` iff the bound device reaches **at least one** other visible device (it `break`s on the first peer that answers yes, `:524–527`). The doc (lines 510–512 of `cuda_backend.cu`) is explicit: "we report can_access_peer == true iff the bound device can reach AT LEAST ONE peer."
- The consumer gate (`f2_blocks_multigpu.cpp:171–172`) reads **only `gpus[0].caps.can_access_peer`** — it never inspects `gpus[1..G-1].caps.can_access_peer`.
- The P2P combine (`p2p_combine.cu`, declared `p2p_combine.hpp:42–108`) then has the ROOT pull EACH peer's partial via `cudaMemcpyPeer` — i.e. it needs the root to reach **every** non-root owning device, not just one.

For **G==2**: "root can reach at least one peer" == "root can reach the one peer" == "root can reach every peer." So the single bit is exactly sufficient, and the proven parity test (G==2, PRO box) exercises precisely this. **No bug today.**

For **G>2 with a partial fabric** (root reaches device 1 but NOT device 2): the root's `can_access_peer` is `true` (it reached device 1), the gate selects P2P, and the combine then attempts `cudaMemcpyPeer(root←device2)` — which would fail-fast at the throwing `STEPPE_CUDA_CHECK` on the `cudaMemcpyPeer` itself (`p2p_combine.hpp:72–74` documents exactly this: "a GENUINE peer-enable failure on a device the caller PROMISED is peer-reachable is a fault — the DMA below would fail anyway — so the routine fails-fast"). So the *outcome* is a fail-fast throw, not silent corruption (good) — but it is a **throw on a configuration the gate accepted**, where the cleaner §11.4 contract is: "the root must reach EVERY non-root device, else degrade to host-staged (which always works)." The single root-only "any-peer" bit cannot express "all-peers," so for G>2 the gate is over-optimistic.

Adversarial self-check: is this actually wrong, or fine-by-parity? It does NOT break parity (a throw is not a wrong number, and G==2 — the only tested/shipped case — is exactly covered). So it is a **contract-precision + graceful-degradation gap for an untested G>2 topology**, not a parity bug. But the header asserting a clean per-device boolean while the consumer reads only the root and the combine needs all-peers is the kind of latent mismatch that calcifies. The header is where the invariant belongs because `can_access_peer` is the seam datum.

Fix (two parts; header-side is the cheap, actionable one):
1. **Header (now, S):** state the invariant on `can_access_peer` — "true iff the bound device can reach AT LEAST ONE visible peer (exactly sufficient for G==2; the combine root reads only `gpus[0]`'s value). For G>2 on a PARTIALLY-connected fabric this bit does NOT prove the root reaches EVERY peer — the P2P combine requires all-peers reachability and fails-fast (never silently wrong) if a promised peer is unreachable; a future all-peers tier (or a per-pair reachability matrix) belongs here, not in the consumer."
2. **Consumer (when a >2-GPU box exists, M):** either make the gate require the root reach every non-root `device_id` (degrade to host-staged otherwise), or have the combine catch the unreachable-peer case and degrade with the tagged WARN rather than throw. Untestable on the current 2-GPU boxes ⇒ the header doc is the actionable part now.

**Severity med (the unit-focus contract-precision item; benign at G==2/shipped, latent for G>2). Effort S (header doc) / M (consumer all-peers gate). PARITY-SAFE? yes — neither part changes any reported number; both make the degrade cleaner.**

(lines 147–149; consumer `resources.cpp:53`)**
The doc (lines 147–149) is correct: process-visible count, "the upper bound the probe saw." `resources.cpp:53` uses it exactly that way (auto-enumerate reads `device_count` to build `0..count-1`; an explicit `config.devices` ignores it, `:45–46`). ## Considered & rejected (incl. rejected-for-parity)

- **"The first-pass claim that the P2P gate checks `G >= 2`."** REJECTED as a false verification — corrected to 11.5. The gate (`f2_blocks_multigpu.cpp:171–172`) has no `G` term; `G >= 2` is a structural precondition from the `:88` early-return, documented (incorrectly) as part of the gate in five sibling files. This is the single substantive correction this pass makes to the first-pass draft.

- **"Make `capabilities()` `noexcept`."** REJECTED. The override can throw `CudaError` on a genuine device-query fault (`STEPPE_CUDA_CHECK`), the correct fail-fast §2 behavior — `noexcept` would `std::terminate` instead of propagating a catchable typed error. The base default is noexcept-in-fact but cannot be annotated so without lying for the override. The doc, not the signature, carries the exception contract (2.2/7.1).

- **"Report a per-peer reachability MATRIX instead of a single `can_access_peer` bool, so G>2 partial fabrics are expressible."** Tempting for 11.1, REJECTED as over-engineering for the shipped reality: only 2-GPU boxes exist (rtxbox spun down, budget 5090 has P2P disabled entirely), G==2 makes the single bit exactly sufficient, and a matrix is untestable. The right fix is the header-doc invariant (11.1 part 1); the matrix is a "when a >2-GPU partial-fabric box exists" item. Noted, not adopted.

- **"Probe `can_access_peer` BIDIRECTIONALLY (root↔peer both directions), since the combine needs root←peer pull."** REJECTED. `cudaDeviceCanAccessPeer(canAccessPeer, device, peerDevice)` returns whether `device` can access `peerDevice`'s memory (verified — Sources). The combine has the root PULL each peer's partial, i.e. root accesses peer memory, i.e. `cudaDeviceCanAccessPeer(root, peer)` — exactly the direction the probe queries (`device_id_` as `device`, `peer` as `peerDevice`, `cuda_backend.cu:523`). The directionality is already correct; a bidirectional probe is redundant for the pull-combine.

- **"`emulated_fp64_honorable` should also carry the mantissa-bit count it was probed with."** REJECTED — verified against the predicate body. `emulation_honorable(precision)` (`f2_block_kernel.cu:197–204`) checks only `kind != EmulatedFp64` then `STEPPE_HAVE_EMU_TUNING` — it NEVER reads `mantissa_bits`, so it returns the same answer for `EmulatedFp64{32}` and `EmulatedFp64{40}`. Probing with the default `kDefaultMantissaBits` is sufficient; adding the count would imply a bit-count dependence that does not exist.

- **"Move `BackendCapabilities` into a public `include/steppe/` header so public API consumers can read the tier."** REJECTED for this delta. The POD is correctly internal (it crosses the within-tree `core ↔ device` seam, like `MatView`); the public which-path/tier surface is the future run-record metadata (§12), not the raw probe. Promoting it now over-exposes a device-internal probe. (Same internal/public-type question the prior review flagged at 9.3 for `F2Result`; not re-scored here.)

- **"The throwaway device-0 backend in auto-enumeration (`resources.cpp:52`) should be avoided by exposing a count-only probe."** Partially rejected as a `backend.hpp` finding — `device_count` IS the right field, and using it keeps `resources.cpp` CUDA-free; the throwaway cuBLAS context is a `resources.cpp`/factory cost (8.2), not a header defect.

- **"`capabilities()` doing a `cudaSetDevice`/restore is a hidden side effect on a `const` method — make it non-`const` or `mutable`-guard."** REJECTED. The device-current is process-global runtime state, not part of the backend's logical value; the override restores it (device-neutral, `cuda_backend.cu:545`), so `const` is honest (no observable change to the object OR the ambient device). The real fix is RAII-restore on the throw path (4.2, override) + documenting the device-neutrality obligation (2.3, header) — not dropping `const`.

- **"PARITY RISK: does probing `can_access_peer` and selecting P2P ever change a reported number?"** REJECTED-as-not-a-risk (verified). Both transports sum the same fixed `g=0..G-1` order onto a zero-initialized full tensor (`p2p_combine.hpp:24–31`); `cudaMemcpyPeer` is a byte-exact DMA; the proven parity test shows P2P == host-staged == single-GPU bit-for-bit. The probe selects transport, never arithmetic. **REJECTED-FOR-PARITY preemptive flag:** if a future "fix" tried to make the combine pick a *different* fan-in order based on fabric topology to optimize bandwidth, THAT would break the fixed `g=0..G-1` order and the §12 bit-identity — it must be rejected for parity. Flagged here so it cannot be slipped in under the "optimize the combine" banner.

## Good patterns to keep

- **The CUDA-free capability seam is real and compile-time-proven.** `BackendCapabilities` adds no include, no CUDA type; U2 is a plain `.cpp` that links no CUDA, so the §4 layering holds *by construction* and a regression that dragged a CUDA header in would fail to compile, not just at runtime. The load-bearing win of the whole DI design, extended cleanly.
- **POD-by-value + value-initialized "unknown" defaults on the SAFE/degrade side.** Trivially-copyable, standard-layout, every default the host-staged / native-FP64 / no-peer tier — so an unprobed or CPU backend can never accidentally claim a capable fast-path. Fail-safe by default member initializer, pinned by `static_assert`.
- **Tag off the numeric payload (§12), every field parity-neutral.** The probe POD carries no statistic; the which-path tag is a separate `CombinePath` enum on `Resources`; `F2BlockTensor` stays pure numeric storage. The probe → `caps` → §4 gate → `last_combine_path` chain is textbook out-of-band observability, and the "PARITY-NEUTRAL BY CONSTRUCTION" block is an accurate, re-derivable statement of why the locked bit-identity is untouched.
- **The non-pure virtual base default.** Returning `BackendCapabilities{}` lets `CpuBackend` not override (no device tier) and keeps `backend.hpp` CUDA-free-compilable standalone — and makes test doubles (U2's `DefaultCapsBackend`) cheap. The right way to add a probe to a multi-impl interface.
- **Single-source honorability.** `emulated_fp64_honorable` defers to the ONE `emulation_honorable` predicate the GEMM path also consults (verified: the predicate reads only the kind + `STEPPE_HAVE_EMU_TUNING`, not the mantissa bits), and U5 asserts probe == predicate — so the probe and the compute path can never disagree on §12 honorability.
- **Three-altitude testing of a tiny delta.** Compile-time POD-shape `static_assert` (U2) + live-device probe values (U5) + consumer integration (I1), with the host-only TU doubling as the CUDA-free-seam proof. Exemplary §13 coverage for ~70 lines.
- **Probe is a pure query, not an action.** `capabilities()` never calls `cudaDeviceEnablePeerAccess` — enabling is the combine's job, gated by the probed bit. The MAY (`enable_peer_access`) / WHICH (`prefer_p2p_combine`) / CAN (`can_access_peer`) three-knob split lives in the right three homes (`DeviceConfig` intent vs `Resources` discovered state), and the probe touches none of the intent knobs.
