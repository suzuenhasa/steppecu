## Role & layering

`GenotypeTile` is the plain-data output of the `io` decode front-end (architecture.md §4 "`io` LEAF emits plain data structs"; §5 S0 "`io` reads/tiles; the device kernel decodes"; ROADMAP §2; the §4 repo-layout line "src/io/genotype_tile.hpp [BUILT, M1] plain decoded-tile struct (the leaf's output to app)"). It carries the **packed** genotype bytes for the gathered, population-contiguous individuals (TGENO individual-major), the per-record stride `bytes_per_record`, the tile SNP count `n_snp`, the gathered count `n_individuals`, and the population partition over the individual axis (`pop_offsets`, size P+1) plus parallel `pop_labels` (size P). It does **no decode** — that is the device/CPU backend's job through `ComputeBackend::decode_af` (verified at `decode_af_kernel.cu:58` and `cpu_backend.cpp:287`).

Layering is exactly right and is the chief virtue here:
- It includes only `<cstddef> <cstdint> <string> <vector>` — all four are used (IWYU-clean, F19), no CUDA header, no `core`/`device` dependency. It is host-pure POD-with-`std::vector` that crosses the CUDA-PRIVATE seam by value (architecture.md §4 "core/cli physically cannot include a CUDA header"; §7).
- Dependency direction is clean and verified: `geno_reader.hpp` (io) depends *down* on it (`#include "io/genotype_tile.hpp"`, `geno_reader.hpp:26`); `device/backend.hpp` deliberately does **not** include it — it re-declares the shape as `DecodeTileView` (`backend.hpp:54–81`) so the device layer stays io-independent and CUDA stays PRIVATE; `app`/the test is the only place that bridges the two (architecture.md §4 "the *app* layer is the only place that wires `io` output into compute" — realized in `test_decode_equivalence.cu:205–212`).
- Include guard `STEPPE_IO_GENOTYPE_TILE_HPP` matches the path; namespace `steppe::io` is correct.

A faithful, disciplined realization of the §4 spec line. The findings below concern the *contract around* the struct (validity, the seam mirror, forward-fit for M4.5/M5/M6), not its architectural placement, which is correct.

## Score: 8.5/10 — an exemplary, near-perfect leaf data struct, held back only by the two-way shape duplication across the io→device seam (a runtime-checked, not compiler-checked, contract) and the lack of any validity/consistency affordance the next three milestones will reach for

Rationale: For a 77-line POD seam type the bar is "right layer, right shape, zero magic, self-documenting, no defect, forward-fit for the milestones that consume it." This file hits the first five squarely (§4 layering compiler-clean; ROADMAP §4 zero surviving literals; doc density matching the surrounding io headers; rule-of-zero value type). It loses ~1.5 points on two real, substantiated, *forward-fit* issues that are cheap now and calcify later: (1) the packing+partition contract is declared twice and hand-copied field-by-field across the layer seam, and the two have already diverged (`ploidy`/`int n_pop` only on the consumer view; `n_pop()` returns `size_t`) — X-10; (2) there is no `valid()`/`empty()`/`bytes()` affordance and no underflow-safe statement of the cross-field invariant, so the half-populated and empty states are silently representable and the documented invariant `pop_offsets.size() - 1` unsigned-underflows on the empty tile. None is a live bug on the single audited construction path (`read_tile` returns a complete tile by value), which is why this is high-8s, not lower. It is not 9.5+ because a central seam type at this quality bar is expected to make its own contract machine-checkable (a `valid()` predicate + an `as_view` adapter that turns a missed field into a one-TU compile error) rather than relying on a single equivalence test to catch divergence — and M4.5/M5/M6 (incremental tile construction, streaming reuse, multi-source merge) are precisely the paths that will reach the unchecked states.

## Findings

### (1) Correctness & bugs

**F1 [info — no defect in the struct itself; CONFIRMED] `GenotypeTile` is pure storage; it has no executable correctness bug.** The only member function `n_pop()` is a one-line `pop_labels.size()`. There is no arithmetic, no resource ownership beyond `std::vector`/`std::string` (rule-of-zero), no invariant *enforced* in code. Every "bug" below is therefore a contract/edge-case/forward-fit issue, not a miscomputation. Recorded explicitly so the absence of a §1 bug list is a verified result, not an omission.

**F2 [low-med; CONFIRMED — feeds X-10] `n_pop()` reads `pop_labels.size()`, but every decoder indexes the partition through `pop_offsets` (size P+1); when the two disagree, `n_pop()` is the value that sizes the device upload of `pop_offsets` and the kernel's `i+1` subscript, driving an out-of-bounds device read.** The hazard chain is concrete: the bridge fills `DecodeTileView::n_pop = static_cast<int>(tile.n_pop())` (`test_decode_equivalence.cu:211`); the CUDA backend uploads `(P+1)` offsets sized from that `P` (`cuda_backend.cu:288`, `tile.n_pop`); the kernel reads `pop_offsets[i]` and `pop_offsets[i+1]` for `i ∈ [0,P)` (`decode_af_kernel.cu:69–70`). If a tile is built with `pop_labels.size() != pop_offsets.size()-1`, `i+1` runs past the uploaded buffer. The single audited producer (`geno_reader.cpp:111–130`) keeps them in lockstep (one `push_back(label)` per `push_back(offset)`), so this is not reachable *today* — but the struct makes both members independently public and offers no check.
- Why it matters: architecture.md §2 "fail-fast … not silent corruption … OOB"; a cross-field invariant the device read depends on bit-for-bit should be assertable. Cat-1/Cat-2.
- Concrete fix: (a) document `n_pop()` as authoritative *only when the invariant holds*; (b) add `[[nodiscard]] bool valid() const noexcept` (F5) and have the bridge/`as_view` (F10) assert it before the upload. Deriving `n_pop()` from `pop_offsets` (`pop_offsets.empty() ? 0 : pop_offsets.size()-1`) merely *moves* the ambiguity to whichever member a future builder forgets — the real fix is the consistency check.
- Severity: low-med. Effort: S. Before-M4.5? Optional but recommended — M4.5/M5/M6 construct tiles by paths other than the single audited `read_tile`, which is exactly when an unchecked cross-field invariant becomes reachable; cheap to land with F5.

### (2) Edge cases & failure modes

**F3 [low-med; CONFIRMED] The empty/no-population and half-populated states are silently representable, and `n_pop()` cannot distinguish "0 populations" from "labels not yet filled in."** A default-constructed `GenotypeTile{}` has `n_pop()==0`, `packed.empty()`, all sizes 0, `pop_offsets.empty()`. The genuinely reachable hazard is the *partial* tile: `read_tile` does `tile.pop_offsets.push_back(0)` (`geno_reader.cpp:111`) **before** the per-group loop fills labels/bytes, and the gather loop can `throw` mid-fill (`geno_reader.cpp:123` short-read) — so an object observed mid-construction has `pop_offsets == {0}`, `pop_labels` empty, `packed` sized for the full gather but only partly written. `read_tile` returns by value, so a *completed* tile is always whole and the partial is destroyed on throw — but the type offers no way to assert "fully populated," and the empty/partial state is indistinguishable from a legitimately tiny tile.
- Why it matters: Cat-2 (degenerate/partial inputs). The incremental-build milestones (M4.5 shard partials, M6 merge plan) construct tiles outside the single audited path.
- Concrete fix: add `[[nodiscard]] bool valid() const noexcept` (F5); the bridge asserts it before building `DecodeTileView`.
- Severity: low-med. Effort: S. Before-M4.5? Optional (pairs with F2/F5).

**F4 [low — non-finding, type choice CONFIRMED correct] Integer width: the `n_individuals * bytes_per_record` product is `size_t`-wide in both producer and consumer; `std::size_t` is the right type and not a bug-site.** Producer: `tile.packed.resize(n_ind * bytes_per_rec)` (`geno_reader.cpp:98`), both `std::size_t`. Consumer: `packed_bytes = tile.n_individuals * tile.bytes_per_record` (`cuda_backend.cu:305`), both `std::size_t`. For real AADR (`n_ind ≤ 27,594`, `bytes_per_record = packed_bytes(584,131) = 146,033`) the product is ≈4.03e9 bytes — exceeds `UINT32_MAX`, which is precisely why `std::size_t` is mandatory (ROADMAP §4 "keep `size_t` … mandatory above P≈32k"). No overflow is physically reachable (a tile overflowing 2^64 cannot be allocated). The struct gets this right; preserve it. No fix.

**F5 [low; CONFIRMED] No documented machine-checkable invariant ties `packed.size()`, `bytes_per_record`, `n_individuals`, `n_snp`, `pop_offsets`, `pop_labels`; the struct can be left half-populated and nothing notices.** The doc-comment *states* the relationships (`packed` length `== n_individuals * bytes_per_record`, line 41; `n_individuals == packed.size()/bytes_per_record`, line 60; `pop_offsets[0]==0`, `pop_offsets[P]==n_individuals`, line 65; `P == pop_labels.size() == pop_offsets.size()-1`, line 46) but none is enforced.
- Why it matters: §2 testability + fail-fast; §13 wants host-pure unit tests of leaf data types, and a `valid()` predicate is the cheapest possible such test.
- Concrete fix: add
  ```cpp
  [[nodiscard]] bool valid() const noexcept {
      const std::size_t P = pop_labels.size();
      return pop_offsets.size() == P + 1                 // addition, never subtraction (F20)
          && (P == 0 || (pop_offsets.front() == 0
                         && pop_offsets.back() == n_individuals))
          && (bytes_per_record == 0
              ? (n_individuals == 0 && packed.empty() && n_snp == 0)   // legal empty
              : packed.size() == n_individuals * bytes_per_record);
  }
  ```
  Unit-test it in `tests/unit/` (good tile → `valid()`, perturb each field → `!valid()`). Have `read_tile` assert its output is `valid()` (one line) and the bridge assert it before `DecodeTileView` (F2/F3).
- Severity: low. Effort: S. Before-M4.5? Optional but cheap; pairs with F2/F3.

**F6 [low; CONFIRMED] The doc identity `n_individuals == packed.size() / bytes_per_record` (line 60) divides by `bytes_per_record`, undefined on the documented default state (`bytes_per_record == 0`).** A default-constructed tile has `bytes_per_record == 0`, so the stated identity divides by zero. This is a documentation/contract defect, not a runtime one (no code executes the division), but a deserializer or `assert` author copying the comment hits it.
- Concrete fix: qualify line 60 to "`== packed.size() / bytes_per_record` **when `bytes_per_record > 0`**; both are 0 for an empty tile," and let `valid()` (F5) treat the all-zero case as the legal empty tile.
- Severity: low. Effort: S. Before-M4.5? Optional (pairs with F5).

**F20 [low; CONFIRMED] The documented invariant itself (`P == pop_labels.size() == pop_offsets.size() - 1`, line 46) unsigned-underflows on the empty case: `pop_offsets.size() - 1` with an empty `pop_offsets` is `SIZE_MAX`, not `-1`.** For a default `GenotypeTile{}`, `pop_offsets.size()` is `0` (`std::size_t`), so `pop_offsets.size() - 1` wraps to `0xFFFF…FFFF`, and the documented equality reads `0 == SIZE_MAX` (false). The invariant is silently only-true-for-populated-tiles, and any consumer that *codes* it as written gets a wraparound bug on the empty tile. Distinct from F11's "the `pop_offsets[0]` subscript doesn't exist" — this is the unsigned-width form of the same edge (Cat-2: integer wraparound).
- Concrete fix: state the invariant addition-side: `pop_offsets.size() == pop_labels.size() + 1` (never subtraction), and note `pop_offsets.front()==0` / `pop_offsets.back()==n_individuals` hold *only for a non-empty `pop_offsets`*. F5's `valid()` already encodes the safe form.
- Severity: low. Effort: S. Before-M4.5? Optional (pairs with F5).

**F21 [low; CONFIRMED] The trailing partial byte of each record holds undefined codes for SNP positions `>= n_snp` when `n_snp % 4 != 0`; the struct does not document that those padding positions exist or that consumers must bound by `n_snp`.** `bytes_per_record == ceil(n_snp/4)` (`packed_bytes`, `eigenstrat_format.hpp:67`), so when `n_snp` is not a multiple of 4 the last byte of each record contains `4 - (n_snp % 4)` two-bit positions whose contents are whatever the source file held past the tile's SNP range. Decoders correctly read only `s ∈ [0, n_snp)` (the `s >= M` early return at `decode_af_kernel.cu:67`; the `s < M` loop bound at `cpu_backend.cpp:308`), so this is not a live miscompute. But the doc-comment (lines 41–44) describes the byte/position addressing without stating that positions `>= n_snp` in the final byte are *not* valid genotypes — a future consumer iterating `4 * bytes_per_record` positions instead of `n_snp` would decode garbage. Interacts with F23 (addressing is correct only for `snp_begin == 0`).
- Why it matters: Cat-2 (misaligned/degenerate input boundary).
- Concrete fix: one sentence on the `packed`/`bytes_per_record` doc — "the final byte of each record may contain up to 3 padding code positions for SNP indices `>= n_snp` when `n_snp % 4 != 0`; consumers MUST bound iteration by `n_snp`, never by `4 * bytes_per_record`."
- Severity: low. Effort: S. Before-M4.5? No (M5 cosmetic).

**F23 [low — NEW; CONFIRMED] The byte-addressing documented on `packed` ("SNP `s` … is the 2-bit code at byte `s/4`, position `s%4`", lines 43–44) is correct ONLY because the tile is a `snp_begin == 0` prefix; the struct does not state that precondition, though its producer enforces it (`geno_reader.cpp:75`) and the M5 tile loop will violate it.** `bytes_per_record` is `packed_bytes(tile_snps)` over the *tile's* SNP count, and the decoders index `byte_in_rec = s/4` with `s` the *tile-local* SNP index (`decode_af_kernel.cu:71`, `cpu_backend.cpp:309`). This is only self-consistent when SNP 0 of the tile is byte-aligned, i.e. `snp_begin % 4 == 0` (M1 requires `snp_begin == 0`, rejected otherwise at `geno_reader.cpp:75–79` "nonzero begin is the M5 tile loop"). The `GenotypeTile` doc describes the addressing as if absolute, which is fine today but is exactly the assumption M5's streaming tiles will break — a tile carrying mid-genome SNPs at a non-multiple-of-4 begin would silently mis-decode if a consumer trusts the comment.
- Why it matters: Cat-2 (misaligned input) + forward-fit. The struct is the seam M5 streams through; the byte-alignment precondition is part of its contract and is currently implicit.
- Concrete fix: document on `n_snp`/`packed` that "this tile covers a SNP **prefix** (`snp_begin == 0`); the `s/4, s%4` addressing assumes byte-aligned SNP 0. The M5 tile loop must keep tiles 4-SNP-aligned or carry the intra-byte offset." Optionally add a `std::size_t snp_begin = 0;` field when M5 lands.
- Severity: low. Effort: S. Before-M4.5? No — flag for M5 (composes with M4.5 streaming).

### (3) Numerical / precision vs §12

The §12 native-FP64-vs-Ozaki / determinism / `q(1-q)/max(N-1,1)` concerns belong to the decode primitive (`core/internal/decode_af.hpp` — integer-accumulate AC/AN then one FP64 divide, confirmed at `decode_af_kernel.cu:85` and `cpu_backend.cpp:324`) and the f2 kernels downstream, never to this raw-byte carrier. The struct correctly stops at *packing* and defers all arithmetic to the decoder (`backend.hpp:171–186`) — the right §12 separation: cancellation-sensitive math lives where the arithmetic is, not in the transport struct. Skipped only because genuinely N/A.

### (6) Decomposition / single-responsibility / function size vs §2

**F10 [med; CONFIRMED — cross-cutting X-10] The packing+partition contract is declared TWICE in fully-mirrored form across the layer seam (`io::GenotypeTile` ↔ `device::DecodeTileView`), hand-copied field-by-field at the call sites, and the two have already DIVERGED.** Side-by-side:
| field | `io::GenotypeTile` | `device::DecodeTileView` |
|---|---|---|
| packed bytes | `std::vector<std::uint8_t> packed` (owning) | `const std::uint8_t* packed` (borrowed) |
| record stride | `std::size_t bytes_per_record` | `std::size_t bytes_per_record` |
| SNP count | `std::size_t n_snp` | `std::size_t n_snp` |
| gathered count | `std::size_t n_individuals` | `std::size_t n_individuals` |
| partition | `std::vector<std::size_t> pop_offsets` (owning) | `const std::size_t* pop_offsets` (borrowed) |
| P | `std::size_t n_pop()` (derived from `pop_labels`) | `int n_pop` (a field) |
| labels | `std::vector<std::string> pop_labels` | — (absent) |
| ploidy | — (absent) | `int ploidy = 2` (a field) |

The bridge hand-copies six fields and supplies `ploidy`/`n_pop` separately (`test_decode_equivalence.cu:205–212`, identically in `test_filter_oracle.cu:233–237`). The divergence is already real: `ploidy` and an `int n_pop` field live only on the consumer view; `n_pop()` returns `size_t` and is derived from a *different* member (`pop_labels`) than the kernel indexes (`pop_offsets`) — see F2. The §4 layering *requires* two structs (the device layer cannot include the io header, and ownership differs — owning vs borrowed). So the fix is **not** a merged type; it is an `as_view(const GenotypeTile&, int ploidy) -> DecodeTileView` adapter in the wiring (`app`/test) layer, where a missed or mistyped field becomes a one-TU compile error instead of a silent hand-copy.
- Why it matters: §2 DRY/single-source + §4 separation; the seam's correctness currently rests on a single equivalence test, not the compiler (F16). X-10 in the overview.
- Concrete fix: add a free function `DecodeTileView as_view(const io::GenotypeTile& t, int ploidy)` in the bridge TU; the three call sites (2 tests + future M4.5 `Resources` wiring) call it; optionally a `static_assert`/golden that the field sets line up. Document on `GenotypeTile` that `ploidy` is a *metadata* parameter supplied at bridge time, deliberately not stored here (a per-sample property from the `.ind`/metadata, not of the packed bytes — consistent with `decode_af.hpp`'s "never inferred from genotypes").
- Severity: med. Effort: S–M. Before-M4.5? Recommended — M4.5 adds a third bridge site (`PerGpuResources` wiring); landing `as_view` first prevents replicating the hand-copy. (Note: TODO §A does *not* enumerate this item; it is not a documented "pay-first" gate, so "before-M4.5" here is a recommendation, not a requirement — correcting the first pass's claim, see F26.)

**F11 [low; CONFIRMED] Single-responsibility is otherwise correct — the struct carries data only, with one trivial derived accessor.** `GenotypeTile` does exactly one thing (hold a gathered packed tile + its partition); `n_pop()` is a one-line derivation, not behavior. No god-struct, no responsibility creep — the right decomposition for a leaf data type. (Nuance: line 65 states `pop_offsets[0] == 0` as an invariant, but a default-constructed tile has `pop_offsets.empty()`, so the subscript does not exist — see F20 for the related unsigned-underflow in the line-46 invariant; fold both into F5's `valid()` + a one-line doc clarification.)

### (8) Performance

**F14 [low — non-finding, CONFIRMED] The struct is moved, not copied, across the `read_tile` return.** `read_tile` returns `GenotypeTile` by value (`geno_reader.cpp:131`). Verified against the standard: this is NRVO-*eligible* but NRVO is an optional (permitted-not-required) optimization for a named local — mandatory copy elision applies only to prvalues (C++17 [class.copy.elision]). So the *worst case* is a move of the three containers, never a deep copy of `packed`. The test assigns it (`tile = reader->read_tile(...)`, `test_decode_equivalence.cu:195`) — move-assignment of the containers, cheap. No accidental copy. Forward note (info, M5): reusing one `GenotypeTile` and `packed.resize()`-ing per tile reuses capacity and avoids re-allocation; the struct already supports this (plain mutable aggregate), so no change now — a doc note that it is reuse-friendly for the M5 loop (with the F22 affordances) would echo §11.1. ## Considered & rejected

- **"`n_pop()` should be derived from `pop_offsets` (size − 1) instead of `pop_labels.size()`."** Rejected as the *fix* (kept as a doc note). Switching the source member only relocates the ambiguity to whichever member a future builder forgets to fill; the labels are needed anyway for f4/qpAdm output naming, so neither member is removable. The real fix is the cross-field `valid()` check (F5), not changing the accessor's source.

- **"Merge `GenotypeTile` and `DecodeTileView` into one type."** Rejected — it would break §4 layering. The device layer cannot include the io header (CUDA must stay PRIVATE to `steppe_device`), and the two have genuinely different ownership (owning `std::vector` vs borrowed `const T*`) and different lifetimes. Two structs + an `as_view` adapter (F10) is the layering-legal answer.

- **"Store `ploidy` on `GenotypeTile`."** Rejected. Ploidy is a per-sample *metadata* property (from the `.ind`/declared metadata), explicitly "never inferred from genotypes" (`decode_af.hpp:24–25`, ROADMAP §2). It is not a property of the packed bytes, so it correctly enters at bridge time on `DecodeTileView` (F10), not on the byte-transport struct. The fix is to *document* the deliberate split, not to add the field.

- **"`ploidy<=0` (X-11) is a defect of this struct."** Rejected for this file. The `ploidy` field and its `finalize_af` NaN/Inf behavior live in `decode_af.hpp`/`DecodeTileView`/`snp_filter.cpp` (X-11), not in `GenotypeTile`, which has no `ploidy` member at all. The cross-cutting X-11 fix touches those files; `genotype_tile.hpp` is only adjacent (via the bridge that supplies `ploidy`). Not a finding here.

- **"Make `GenotypeTile` move-only (delete the copy)."** Rejected. It is a value type with rule-of-zero semantics; copyability is harmless (P-sized metadata + a `std::vector<uint8_t>` that is moved on the hot `read_tile` return, F14) and aggregate-ness is preserved by *not* declaring special members. Adding `= delete` copy ctors would make it non-aggregate (losing brace-init the tests use) and buy nothing — there is no resource that double-frees. Leave it rule-of-zero.

- **"Add a `static_assert` inside `genotype_tile.hpp` checking it matches `DecodeTileView`."** Rejected *here* (belongs in the bridge). `genotype_tile.hpp` must not include `device/backend.hpp` (that would invert the layering and pull a device header into the io leaf). The cross-struct check belongs in the `app`/test bridge TU that includes both — exactly where `as_view` (F10) lives.

- **"`n_pop()`'s `noexcept` is a latent `std::terminate` risk."** Rejected. `std::vector::size()` is `noexcept` per the standard ([vector.capacity]); `n_pop()` cannot throw.

- **"`packed` should be `PinnedBuffer`/page-locked now."** Rejected for M1/M4 (premature; ingest is not yet the measured bottleneck — TODO ⚡ requires the Nsight gate first). The pageable `std::vector` is the correct M1 shape; the pinned path is M5 and is decoupled by `as_view` (F18). Pinning the 4 GB `.geno` is explicitly *wrong* per TODO ⚡ (RLIMIT_MEMLOCK cap; pin only small slots).

- **"The `s/4`,`s%4` literals in the doc-comment are surviving magic numbers (ROADMAP §4)."** Rejected — they are true mathematical layout constants of 2-bit packing, which ROADMAP §4 explicitly exempts. The *operational* `ceil(n_snp/4)` is correctly routed through `packed_bytes`/`kCodesPerByte` (F9).

## Good patterns to keep

- **Textbook io-leaf layering** (§4): host-pure, stdlib-only includes (all four used, IWYU-clean — F19), no CUDA, no upward dependency, crosses the seam by value. The §C "thin CUDA-free orchestration seam" / "predicate-as-shared-primitive in the io leaf" discipline applied to a data type.
- **Zero surviving magic numbers** (ROADMAP §4): the one derived quantity (`bytes_per_record`) is explicitly "DERIVED … never hardcoded" and routes through the single format-constant home (`packed_bytes`/`kCodesPerByte`); only genuine mathematical layout constants appear, and only in comments (F9).
- **Doc density matching the surrounding io headers** (§C "every constant doc-commented w/ arch+ROADMAP cite + measured rationale"): each member is tied to architecture.md §/ROADMAP § with verified-against-the-oracle provenance.
- **Rule-of-zero value semantics** (F8/F14): owning `std::vector`/`std::string` (host RAII), implicitly movable, aggregate-initializable, no hand-written special members — the right shape for a transport struct, returned by value with a worst-case move (no deep copy).
- **Correct `[[nodiscard]] const noexcept` on the single accessor** (F13), with the `noexcept` honestly backed by `std::vector::size()`'s `noexcept`.
- **The deliberate two-struct seam** (`GenotypeTile` owning / `DecodeTileView` borrowing): even though it costs the F10 mirror, the *direction* (device re-declares rather than including io) is the correct way to keep CUDA PRIVATE — preserve the direction, just add the `as_view` adapter to make the agreement compiler-checked.
