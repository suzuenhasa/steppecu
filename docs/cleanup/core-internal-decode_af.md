## Role & layering

This is one of the two "single-source per-element numerics" headers (`f2_estimator.hpp` is the other) that hold a domain formula exactly once so the CPU oracle (`CpuBackend::decode_af`) and the GPU kernel (`decode_af_kernel`) **cannot diverge** — the explicit mandate of architecture.md §2 (DRY/single-source), §8 (DRY table: one primitive shared by CPU-ref and GPU), §13 (oracle validates results; the per-element primitive is shared so they "can't diverge on the formula"). It carries the S0 (format decode → 2-bit unpack) and S1 (allele-freq reduction → Q/V/N) per-element math from the §5 pipeline table.

Layering is **correct and load-bearing**: the header is in `core/internal/`, host-pure, CUDA-free (includes only `<cstdint>`; the CUDA qualifiers come from a macro guarded on `__CUDACC__`). Verified by grep across the whole tree: the only two TUs that actually `#include` it are `decode_af_kernel.cu` (device, nvcc — `__CUDACC__` defined) and `cpu_backend.cpp` (host, `.cpp` — `__CUDACC__` undefined); `backend.hpp` and `mind_prepass.hpp` *name* it only in comments, nothing in `io` includes it, and nothing in `include/steppe/` re-exports it. It deliberately re-states `kMissingGenotypeCode` rather than `#include`ing `io::kMissingCode`, because that would invert the dependency direction (`core` must not depend on the `io` leaf — §4). That is the *right* call and the header documents it. Symmetrically, `io`-side `mind_prepass.hpp` deliberately does **not** include this header (it uses `io::code_in_byte`) for the same reason, and says so in its own comment. The two share a bit order *by construction*, pinned only at runtime by end-to-end tests — a real DRY tension consciously accepted (Finding L-1).

The `genotype_code`/`genotype_valid`/`finalize_af` triple plus `AfResult` is exactly the set re-exported by both consumers (`using core::...` in the kernel, lines 40-43; `core::...` in the CPU backend). I confirmed the call shape is identical on both paths. **But** the *accumulation convention itself* (`if valid: ac += code; ++an`) is hand-duplicated in both loops and is **not** in the shared header — a genuine residual divergence surface the first pass missed (Finding A-1).

## Findings

### (1) Correctness & bugs

**C-1 [low, S, before-M4.5: no] CONFIRMED — `genotype_code` silently masks `k` outside [0,3]; documented as a position but the precondition is unstated.**
Location: `genotype_code`, lines 66-70. `shift = (3 - (k & 3)) * 2`. The `& 3` masks `k` to its low 2 bits, so `k = 4` silently extracts position 0, `k = 5` → position 1. **Verified both callers pre-mask**: kernel line 72 `pos_in_byte = static_cast<int>(s & 3)`, cpu line 310 identical. So in-range today, harmless. But the parameter is named/documented as a position-in-byte; a future caller passing a raw SNP index `s` would get a silently wrong code with no diagnostic. Why it matters: this is the single shared bit-extraction primitive (§8) — a silent mis-decode corrupts every downstream statistic, and the §13 oracle would *agree* with the GPU (both wrong) because they share this function, so the equivalence test would not catch it. `io::code_in_byte` (eigenstrat:77) uses `k % kCodesPerByte`, the identical wraparound — consistent, not a divergence. Fix: (a) sharpen the doc-comment to "`k` is reduced modulo 4; callers pass `s & 3`" so the masking is intentional contract; or (b) a debug `assert(k>=0 && k<4)` — but `assert` in a `__host__ __device__` function pulls in `<cassert>` and is awkward on the device side, so (a) is the proportionate fix. Adversarial check: is this a bug? No — well-defined, matches the sibling; contract-clarity gap, hence low.

**C-2 [low, S, before-M4.5: no] CONFIRMED — negative `k` is well-defined (not UB), but the "0-based" wording undersells it.** `k & 3` on a negative `int`: C++20 mandates two's-complement for signed integers (P0907R4, ISO C++20 [basic.fundamental]/p3; verified the paper makes two's-complement the *mandatory* representation, eliminating ones'-complement/sign-magnitude), so `(-1) & 3 == 3` — defined, no UB. **Empirically confirmed on the box**: `g++ -std=c++20` prints `(-1)&3=3, 5&3=1, 4&3=0`. Same masking story as C-1; no fix beyond C-1's comment. Project compiles at C++20 (`CMAKE_CXX_STANDARD 20`, `CMAKE_CUDA_STANDARD 20`), so the guarantee applies to both host and device builds. Listed separately to record the negative path was checked.

**C-3 [verify — not a bug; CONFIRMED, dangling cross-ref CORRECTED] `finalize_af` integer→double conversions are exact for the real data and the divide is the single rounding.** Lines 99-101: `n = (double)ploidy * (double)an; q = (double)ac / n`. For AADR `ploidy = 2`, `an ≤ n_individuals` (≈ 27 594), `ac ≤ 2·an` — all exactly representable in `double` (< 2^53), so the comment's "single FP64 divide … keeps Q exact" holds: `ac` and `n` convert exactly and `q` is the single correctly-rounded IEEE division. The test asserts `max|Δ| == 0` against the numpy oracle (which also integer-accumulates then divides once in `float64`). Verified correct; no action. **Correction to the first pass:** its C-3 said "see C-5 below" / "residual risk in C-5" — there is **no C-5 finding** in either draft; that is a dangling reference. The genuine overflow-headroom concern it gestured at lives in **E-3** (the `long` accumulator width); folded there, and the dead "C-5" pointer is removed.

### (2) Edge cases & failure modes

**E-1 [med, S, before-M4.5: yes] CONFIRMED, SHARPENED, and empirically pinned — `finalize_af` does not defend against `ploidy <= 0`; `ploidy == 0` produces a silent non-finite `q` that defeats the `an > 0` guard, AND this contradicts an existing clamp in `snp_filter.cpp`.**
Location: `finalize_af`, lines 96-108. The guard is `if (an > 0)`, then `n = ploidy * an; q = ac / n`. If a caller passes `ploidy == 0` (uninitialized metadata, or a future haploid/mixed path that forgot to set it), then with `an > 0` we get `n == 0.0` and `q = ac / 0.0`. On both targets this is **well-defined IEEE-754 behavior** — `std::numeric_limits<double>::is_iec559` is `true` (compiled `static_assert` passes on the box; the CUDA device is IEEE-754): `ac > 0 ⇒ +inf`, `ac == 0 ⇒ NaN` (not UB; division by zero is well-defined only *because* `is_iec559` holds — see the rejected objection below). The `an > 0` branch is *supposed* to mean "we have data, the result is finite," but `ploidy == 0` slips a non-finite `q` and `n == 0` past it while still reporting `v = 1.0` (valid). That NaN/Inf poisons every f2 GEMM touching this column, and `V = 1` means it is *not* masked out. Negative `ploidy` is worse: `n < 0`, `q` wrong-signed, `v = 1`. Why it matters: architecture.md §2 "fail-fast … not silent corruption"; ROADMAP §6 DoD. **The sharpening the first pass missed:** the codebase already takes a side on this exact bad input — `io/filter/snp_filter.cpp:30` does `static_cast<double>(in.ploidy > 0 ? in.ploidy : 1)`, i.e. it clamps `ploidy <= 0` to 1. So today the *filter* path silently treats a bad ploidy as 1 while the *decode finalize* path silently produces a NaN-with-validity — two different silent behaviors for the same illegal input, in primitives that are both supposed to be the single source of the Q/V/N contract. A live inconsistency, not a hypothetical. Adversarial check: "caller's job to validate at `build()`"? No — `ploidy` lives on `DecodeTileView` (backend.hpp:80, a per-call view filled by `app`/tests from a `GenotypeTile`), **not** on `RunConfig`, so there is no `ConfigBuilder::build()`-time validation path for it (§9 frozen-config does not cover it; re-verified by reading backend.hpp). The header's own contract says ploidy is "never auto-detected" but never says "always validated upstream." Fix: fold ploidy into the validity test — `if (an > 0 && ploidy > 0)` — so a bad ploidy degrades to `{q:0, n:0, v:0}` (masked out), total and pure, one extra comparison; document the choice and align the semantics with `snp_filter.cpp` (mask-out is the more defensible "no trustworthy N" answer; the two should agree and say which). Before-M4.5 because M4.5 introduces the pseudo-haploid (`ploidy == 1`) path and multi-GPU, exactly when a misset ploidy is most likely.

**E-2 [low, S, before-M4.5: no] CONFIRMED — the `an == 0` path is correct and tested.** Lines 103-106 return `{q:0, n:0, v:0}` — the "no non-missing individuals" case (whole population missing at a SNP). Correct: `V = 0` masks the column out of the GEMM; `Q = 0` is the zero-fill the masked GEMM needs (architecture.md §5 S2: "the zero is what makes the masked GEMM correct"). The equivalence test's `inv_v`/`inv_n` invariants (test 266-277, `V==(N>0)`) cover it. No action — listed to record the empty path was checked.

**E-3 [med, S (was M — DOWNGRADED, empirically confirmed), before-M4.5: yes] CONFIRMED with corrected effort — `ac`/`an` are bare `long`; correct on the LP64 target but under-typed for a header otherwise meticulous about fixed widths, and the overflow headroom is undocumented.**
Location: `finalize_af` signature `(long ac, long an, int ploidy)`, line 96; the accumulators in both consumers are also `long` (kernel 74-75, cpu 311-312). On the box (Linux x86-64, LP64) `long` is 64-bit, so `ac ≤ 2·27 594` and `an ≤ 27 594` never overflow — fine; the project is Linux-only (architecture.md §3, CUDA 13 / sm_120), so not a live bug. But `long` is 32-bit on LLP64 (Windows) and LP32, and a header that uses `std::uint8_t` for codes should not silently rely on the platform `long` width for an accumulator. Why it matters: ROADMAP §4 "say what you mean" on widths — the §4 inventory explicitly kept `(size_t)i + (size_t)j*P` as a *deliberate* 64-bit choice, so the codebase reasons about widths elsewhere; the bare `long` here is the inconsistency. **Correction to the first pass, now empirically pinned:** I compiled `static_assert(std::is_same_v<std::int64_t, long>)` and `static_assert(sizeof(long)==8)` with `g++ -std=c++20` on the box — **both pass**. So on the LP64 target `std::int64_t` **is** `long`, and changing the `finalize_af` parameters to `std::int64_t` binds the consumers' existing `long ac, an` with **zero conversion and no `-Werror`/`--Werror;all-warnings` warning** — no consumer edit is forced. Effort is **S**, purely intent-signalling on the target. Fix: use `std::int64_t` for the `finalize_af` parameters (and, for symmetry, the consumers' accumulators in the same sweep) OR add a one-line comment: "`long` is the AC/AN accumulator; ≥ 64-bit on the LP64 target, headroom ≫ max AC = ploidy·n_individuals." Cross-reference: TODO backlog B's "standardize device flat-index on `size_t`" sweep is the natural home.

**E-4 [low, S, before-M4.5: no] CONFIRMED — `genotype_code`'s `packed_byte` cannot fail (total function).** `packed_byte` is `std::uint8_t`; every value 0-255 is valid input, `& 0x3u` always isolates exactly 2 bits. No edge case. Listed to record the input domain was checked and fully defended.

### (3) Numerical / precision vs §12

**N-1 [verify — compliant] CONFIRMED — the decode is exactly where §12 says native-FP64-only must live.** §12: "Allele-frequency accumulation … stays native FP64 regardless of `precision`." This header does integer accumulation (`ac`/`an`) and a single native-`double` divide; there is no `Precision` knob here at all, no Ozaki/TF32/emulation. Correct — the decode is bandwidth-bound (§11.3) and reduced precision "buys nothing and would break the bit-for-bit oracle match" (kernel comment 15-18). The single divide is the textbook minimal-rounding form. Compliant; no action.

**N-2 [verify — compliant] CONFIRMED — determinism via integer accumulation; order-independent.** Because AC/AN are integer sums (associative/commutative), the per-(pop,SNP) result is independent of visitation order — the GPU's single-thread-per-(pop,SNP) loop and the CPU's nested loop produce bit-identical `ac`/`an`, hence bit-identical `q` after the one FP64 divide. This is *why* the test asserts `max|Δ| == 0` (test 237, 256-258) rather than a tolerance. The cleanest possible determinism story (no §12 `run_to_run` reduction needed because the reduction never enters FP). Subtlety worth a comment (see D-2): the exactness holds *only* because accumulation is integer and the divide is single — if M4.5's "fuse the filter into decode as a validity-mask multiply" (TODO ⚡ item 3) moves any of this into FP before the divide, the `== 0` gate silently loosens. Compliant today; doc-only protective note recommended.

### (4) CUDA idioms / RAII / stream & async semantics / launch config / occupancy vs §7

**Largely N/A — this is a CUDA-free `core` header by design; it owns no resources, no streams, no launches.** Correct per §4 (core must not include a CUDA header) and §7 (the per-element primitive is the `__host__ __device__` scalar called *inside* the thin kernel shell). **U-3 [verify — compliant] CONFIRMED — no `--expt-relaxed-constexpr` reliance, which is exactly right for this header today.** The header marks nothing `constexpr` (only `inline`), so the nvcc constexpr-dialect question never arises for the current code — correct, because the build passes **no** `--expt-relaxed-constexpr` (verified across `CMakeLists.txt` and all `cmake/*.cmake`). **R-3a [med, S, before-M4.5: yes] NEW / CORRECTION — the first pass's stated justification for R-3 is WRONG, and acting on R-3 naively could cause a device build break.** The first-pass draft wrote: *"does `__host__ __device__ constexpr` compile under nvcc? Per the NVCC/CUDA toolchain rules (corroborated by the NVCC driver doc and the Clang-CUDA implementation): an **un-annotated** `constexpr` function is treated as **`__host__`-only** by default, and `--expt-relaxed-constexpr` is precisely the (experimental, nvcc-only) flag that "implicitly converts all constexpr functions to `__host__ __device__` constexpr functions." **The steppe build passes NO such flag** (verified across `CMakeLists.txt`, `cmake/SteppeOptions.cmake`, `cmake/SteppeWarnings.cmake`, `cmake/CUDAArch.cmake`, `src/device/CMakeLists.txt`), and runs `--Werror;all-warnings` for CUDA. Consequences:
- The reason the siblings work is **NOT** "constexpr is implicitly device-callable" — it is that `cdiv`/`grid_for` carry the **explicit** `STEPPE_HD` (`= __host__ __device__ constexpr` under nvcc). And `io::code_in_byte` is bare `constexpr` *only* because `eigenstrat_format.hpp` is an `io`-leaf header that nvcc never compiles and device code never calls — its host-only `constexpr` is harmless there. Two different reasons, neither of them "implicit host+device."
- Therefore the correct fix for R-3 is to add `constexpr` to the existing **`STEPPE_HD inline`** functions, yielding `__host__ __device__ constexpr` under nvcc — explicitly device-callable, no flag needed. That is safe. Writing bare `constexpr` (dropping/forgetting `STEPPE_HD`) on a function the kernel calls would make it `__host__`-only and the device call would be ill-formed (or silently host-only with the relaxed flag, which the project doesn't set).
Why it matters: this is a "fix the wrong fix" item per the mandate — the draft's recommendation is right but its rationale would mislead the implementer into the one variant that breaks the sm_120 build. Fix: replace R-3's adversarial-check sentence with: "add `constexpr` to the `STEPPE_HD inline` functions → `__host__ __device__ constexpr`; do NOT rely on bare `constexpr` being device-callable (the project sets no `--expt-relaxed-constexpr`)." ### (NEW this pass) A — items the first pass missed

**A-1 [med, S, before-M4.5: yes] NEW — the AC/AN *accumulation convention* (`if valid: ac += code; ++an`) is hand-duplicated in BOTH backends and is NOT in the shared primitive, leaving a residual divergence surface the header was supposed to eliminate.**
Locations: kernel `decode_af_kernel.cu:78-82` and cpu `cpu_backend.cpp:317-321` contain the byte-for-byte identical inner step (re-read both this pass):
```
const std::uint8_t code = genotype_code(byte, pos_in_byte);
if (genotype_valid(code)) { ac += static_cast<long>(code); ++an; }
```
The shared header gives both backends the unpack (`genotype_code`), the missing-test (`genotype_valid`), and the finalize (`finalize_af`) — but the *semantic that AC is the sum of raw codes and AN is the count of valid individuals* is re-implemented in each loop, by hand, outside the single-source header. This is precisely the kind of formula the §2/§8/§13 divergence-prevention thesis says must live in exactly one place. Why it matters: the whole argument for this header is "the CPU oracle and the GPU kernel cannot diverge on the unpack/missing/divide." But the AC/AN *accumulation* is the fourth piece of that math and it is *not* shared — a future change to the AC convention (e.g. switching to "AC = Σ(ploidy − code)" for an alt-allele frequency, or adding dosage weighting) must be made in two places and a one-sided edit would silently diverge, with the §13 equivalence test the only catch (and only if the real data happens to exercise the difference). Adversarial check: is hand-duplication unavoidable here? The *loop structure* differs (GPU thread-per-(pop,SNP) vs CPU nested host loops), so the loops legitimately differ — but the *per-individual step* does not, and is exactly the granularity this header already operates at (`genotype_code`/`genotype_valid` are per-individual). Fix: add a fourth tiny shared primitive, e.g.
```
STEPPE_HD inline void accumulate_genotype(std::uint8_t code, long& ac, long& an) noexcept {
    if (genotype_valid(code)) { ac += static_cast<long>(code); ++an; }
}
```
and have both loops call it. That moves the last per-element decode semantic into the single home, completing the divergence-prevention story (and giving T-1 one more thing to pin). Severity med (a real single-source gap in the *one header whose entire purpose is single-sourcing*), effort S, before-M4.5 because M4.5's filter-fusion (TODO ⚡ item 3) will touch exactly this inner step and is the moment a one-sided edit is most likely. (Adversarial counter-check: should the signature take `std::int64_t&` per E-3? Yes — fold the two: `accumulate_genotype(code, std::int64_t& ac, std::int64_t& an)`, which binds the consumers' `long` accumulators with zero conversion on the LP64 target.)

**A-2 [low, S, before-M4.5: no] NEW — `genotype_valid` reports out-of-range codes (>3) as VALID, the symmetric contract gap to C-1.** Location: `genotype_valid`, lines 74-76: `return code != kMissingGenotypeCode;`. The parameter is `std::uint8_t`, so codes 4..255 are representable, and for any such value the function returns `true` (valid). ## Considered & rejected

- **"`genotype_code` should mask the input byte / handle >2-bit codes" — rejected.** `packed_byte` is `std::uint8_t`; `& 0x3u` already isolates exactly 2 bits; every input is valid. No defect (the *output*-side analog, `genotype_valid` accepting >3, is A-2 — a separate, real contract-clarity item).
- **"The `q` divide should use a guarded/`fma` form for accuracy" — rejected.** A single IEEE division of two exactly-representable integers is correctly-rounded and is the minimal-rounding form; `fma` is irrelevant (no multiply-add). §12 wants native FP64 here, which it is. Anything added would reduce clarity without improving accuracy.
- **"`AfResult` should use `cuda::std::` types or be `__align__`ed" — rejected.** Three `double`s, `is_trivially_copyable`+`is_aggregate` (compiled-confirmed), 24 bytes naturally 8-aligned; must stay CUDA-free (`core` header). `cuda::std::` would *break* the CUDA-free layering (§4). Correctly plain.
- **"Promote `kMissingGenotypeCode` to the public `include/steppe/config.hpp` to single-source with io" — rejected.** Re-verified config.hpp holds only `kHetCorrDenomFloor` and the like — no decode constants, by design. Promoting would make a packing-internal sentinel part of the public API/ABI (§16) and still wouldn't be reachable by `io` without an upward dep. The local duplication pinned by a test (L-1) is the correct resolution; the right *additional* step is the exhaustive bit-order unit test, not promotion.
- **"`finalize_af` should return `std::optional<AfResult>` / an error on `an==0`" — rejected.** `an==0` is a normal, expected domain outcome (whole-population-missing SNP), not an error; the `{0,0,0}` masked-out result is the §5-S2-correct representation and is what the GEMM consumes. An optional would force every caller into a branch for the common case and break the flat-array fill. Correct as a total function.
- **"Make the functions `static` to give internal linkage" — rejected.** `inline` in a header included by multiple TUs across two libraries is the correct ODR story (external linkage, merged); `static` would give each TU its own copy. `inline` is right.
- **"`int ploidy` should be `std::uint8_t` to match the code type" — rejected (mostly).** `ploidy` is a small multiplier; `int` is the natural arithmetic type; narrowing to `uint8_t` buys nothing and risks an implicit-promotion surprise. The real ploidy concern is the *value* check (E-1: `ploidy > 0`), not its width.
- **"Negative `k` / out-of-range `k` is UB" — rejected as UB, kept as contract-clarity (C-1/C-2).** Verified `k & 3` on signed `int` is well-defined under C++20's mandatory two's-complement (P0907R4) and empirically (`(-1)&3==3` on the box). Only the contract clarity is a (low) issue.
- **"The `q = ac/0.0` NaN claim depends on unspecified FP behavior" — rejected as an objection; the claim is sound but must be stated precisely.** `double` division by zero is *not* guaranteed by the C++ core language (it is well-defined only when `std::numeric_limits<double>::is_iec559`), but `is_iec559` is `true` on both targets (compiled `static_assert` passes on the x86-64 box; the CUDA device is IEEE-754), so `+inf`/`NaN` (not UB) is exactly what happens — E-1 is correct, and now states the condition.
- **"Should there be an NVTX range / log line here" — rejected.** §10 NVTX ranges belong on pipeline-phase boundaries in the backend, not inside a per-element device function called billions of times. Correctly absent.
- **"`finalize_af` should clamp `q` to [0,1]" — rejected.** With `0 ≤ ac ≤ ploidy·an = n` and a single divide, `q ∈ [0,1]` is guaranteed exactly (the test's `inv_q` invariant, line 269, confirms it on real data); a clamp would be dead code and could mask the very ploidy/overflow bug E-1/E-3 are about. Correctly absent — the right defense is E-1's input guard, not an output clamp.
- **"Just add bare `constexpr` to all three functions (per the first pass's R-3 rationale)" — rejected as written; corrected into R-3a.** The first pass justified R-3 with "constexpr functions are implicitly `__host__ __device__`, no `--expt-relaxed-constexpr` needed." That generalization is false for nvcc: un-annotated `constexpr` is host-only by default, and the project sets no relaxed-constexpr flag (verified across all cmake files). The functions already carry `STEPPE_HD`, so adding `constexpr` to *those* yields `__host__ __device__ constexpr` (device-callable, no flag) — the recommendation is kept, but the mechanism is corrected so the implementer does not drop `STEPPE_HD` and break the device build.
