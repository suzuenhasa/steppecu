## Role & layering

Layering is **correct and load-bearing**. The `.cuh` names `cublasHandle_t`/`cudaStream_t` and is correctly PRIVATE to `steppe_device` (architecture.md §4) — the device-internal seam between `cuda_backend.cu` and the kernel TU, not the CUDA-free public `ComputeBackend` seam (`device/backend.hpp`). The `.cu` includes only `f2_block_kernel.cuh`, `<cublas_v2.h>`, `<cuda_runtime.h>`, `<library_types.h>`, the shared `f2_estimator.hpp` primitive, and `check.cuh` — no upward dependency, no `io`, no `core` orchestration. The `<<<>>>` and kernel bodies live only in the `.cu`; the host meets the path through narrow `void launch_*` / `run_*` wrappers (architecture.md §7 line 491 "host code never includes kernel bodies or `<<<>>>`"). This realizes the §5 S2 "two custom kernels only" mandate: a fused elementwise pre-pass and a fused numerator+divide, the heavy arithmetic delegated to cuBLAS — **not** a hand-rolled outer-product kernel. The per-element numerics route through the SHARED `het_correction` / `assemble_f2_numerator` / `finalize_f2` primitives so the CPU oracle and GPU cannot diverge (§13). `engage_f2_precision` and `f2_compute_type` are deliberately exposed so the M4 grouped path in `f2_blocks_kernel.cu` shares the single source of the precision policy (§2 DRY) — verified: `f2_blocks_kernel.cu` includes this header and calls both (line 45, 190-194).

This is **not** a trivial unit. It owns the precision engagement that architecture.md §12 / ROADMAP §0 call "the law", and the index math of two kernels whose outputs feed every downstream statistic. I held it to the substantive senior bar.

## Score: 7.5/10 — strong structure and a correct happy path, but two confirmed high-severity §12 defects (one already flagged in TODO.md) and an `int`-narrowing landmine hold it below the 9.5 bar

The structure, layering, DRY sharing, and native-FP64-cancellation discipline are excellent and must be preserved. The first-pass score of 7.5 is **upheld** — the two disqualifying defects are real and verified against the official docs:

1. **C-1**: when `STEPPE_HAVE_EMU_TUNING` is off (the **default** per the `#ifndef … 0` in this very file, lines 64-66), `engage_f2_precision` sets `CUBLAS_FP64_EMULATED_FIXEDPOINT_MATH` but compiles out the FIXED-slice pin, so cuBLAS falls back to its **documented DYNAMIC default** (~60-bit on real data, 79-bit cap) — the exact REJECTED trap (ROADMAP §0; architecture.md §12 line 723), while still reporting `EmulatedFp64`. (Accuracy is fine; the speedup and §12 bit-pinning are voided.)
2. **C-3**: `run_f2_gemms` calls `cublasSetStream` on every invocation, and the cuBLAS docs state this "unconditionally resets the cuBLAS library workspace back to the default workspace pool" — so the explicit `kCublasWorkspaceBytes` workspace bound in the `CudaBackend` ctor (which §9 `build()` validation and §12 require for the emulated-FP64 path) is discarded before the GEMMs run.

Both touch §12; both are confirmed, not asserted-from-memory. Below the high-sev items the unit is clean. Net change from the first pass: **+5 findings added, −0 findings rejected as false** (every first-pass finding survived re-verification), with three findings *sharpened* by official-doc citations and one (T-1) *partially downgraded* because the M4 test already detects the trap (warn, not fail).

## Findings

### (1) Correctness & bugs

**C-1 [high, M, before-M4.5: yes] `STEPPE_HAVE_EMU_TUNING=0` (the default in this file) silently runs the REJECTED dynamic-mantissa Ozaki trap while still reporting `EmulatedFp64`. — CONFIRMED against official docs.**
Location: `engage_f2_precision`, `.cu` lines 201-217, with `#ifndef STEPPE_HAVE_EMU_TUNING / #define … 0` at lines 64-66. When `precision.kind == EmulatedFp64`, the code unconditionally sets `cublasSetMathMode(handle, CUBLAS_FP64_EMULATED_FIXEDPOINT_MATH)` (line 203), but `cublasSetEmulationStrategy(EAGER)` + `cublasSetFixedPointEmulationMantissaControl(FIXED)` + `cublasSetFixedPointEmulationMaxMantissaBitCount(bits)` are gated behind `#if STEPPE_HAVE_EMU_TUNING` (lines 204-209). **Verified in the cuBLAS docs this pass:** the default fixed-point mantissa control is `CUDA_EMULATION_MANTISSA_CONTROL_DYNAMIC` ("Dynamic mantissa control represents the cuBLAS library default mantissa control"), and the default emulation strategy is `CUBLAS_EMULATION_STRATEGY_DEFAULT` (= PERFORMANT). So with tuning off, the handle engages emulation under DYNAMIC control — which architecture.md §12 line 723 and ROADMAP §0 measured to "auto-select ~60 bits on real data's wide dynamic range … collapsing to parity with native (no win)". The result still threads through `f2_compute_type` returning `CUBLAS_COMPUTE_64F_EMULATED_FIXEDPOINT`, so the reported precision tag says "EmulatedFp64{40}" while the silicon runs dynamic mantissa.

Why it matters: this is the central §12 / ROADMAP §0 law, and **architecture.md §9 (line 620) already mandates the correct behavior** — `build()` must reject/fall-back "a `precision` the selected backend cannot honor (e.g. `EmulatedFp64` … on a toolkit/arch where the emulation math mode is unavailable — fall back to native `Fp64` or error)". So C-1 is not merely a foot-gun; it is the *non-implementation of an already-specified §9 contract*. The TU's own file comment (lines 36-40) understates it as "the tuning only pins the FIXED slice count" — but the consequence is "the explicitly-rejected dynamic mode."

Adversarial re-check (the precise blast radius): is the *number wrong*? **No** — per the cuBLAS docs, DYNAMIC control "automatically computes required fixed-point mantissa bits to maintain equal or better FP64 accuracy," dispatching to native FP64 if the needed bits exceed the cap. So the f2 values are still accurate (≈ native). C-1 is therefore a **performance + reproducibility-tag** defect, not an accuracy defect — the headline harm is (a) no measured speedup and (b) the path is no longer the FIXED-slice, bit-pinnable mode §12 requires, yet it is *labeled* EmulatedFp64. Is it silent on the production path? On the **default build, yes** (the math-mode call returns SUCCESS, no diagnostic). The remote box passes `-DSTEPPE_HAVE_EMU_TUNING=1` so the *measured* path is safe, but a stock-toolkit build (a contributor laptop, a CI lane without the flag) silently degrades while claiming the tag. This is logged as TODO.md line 84.

Fix (matching TODO.md line 84 and architecture.md §9): in the `#else` branch, do **not** call `cublasSetMathMode(…FIXEDPOINT…)` and proceed. Instead one of: (a) `#error "EmulatedFp64 requires STEPPE_HAVE_EMU_TUNING"`; (b) fall back to native `CUBLAS_PEDANTIC_MATH` + `f2_compute_type → CUBLAS_COMPUTE_64F` + a logged, capability-tagged warning ("EmulatedFp64 requested but FIXED-slice tuning unavailable → native Fp64 [tag: emu_tuning_unavailable]") so the reported tag matches the arithmetic; or (c) throw a typed `INVALID_CONFIG`. The foot-gun must close observably. Note (b) requires `f2_compute_type` to also know the tuning state — see C-2.

**C-2 [high, M, before-M4.5: yes] The precision policy is split across `engage_f2_precision` and `f2_compute_type`, which can disagree under `STEPPE_HAVE_EMU_TUNING=0`. — CONFIRMED.**
Location: `f2_compute_type` (`.cu` lines 175-184) returns `CUBLAS_COMPUTE_64F_EMULATED_FIXEDPOINT` for `EmulatedFp64` unconditionally, with no awareness of `STEPPE_HAVE_EMU_TUNING`; `engage_f2_precision` (lines 201-217) is the only function that sees the flag. If C-1 is fixed by falling back to native math mode when tuning is off, the compute type handed to `cublasGemmEx` (line 245 here; line 194 in `run_f2_gemms_group`) would *still* be the emulated compute type — a math-mode (PEDANTIC native) / compute-type (emulated fixedpoint) mismatch. The compute type is what selects the algorithm, so the naive C-1 fix that touches only `engage_f2_precision` leaves the compute-type half wrong. Why it matters: §2 single-source — the header (`.cuh` lines 48-51) advertises both as "the single source," but they are two sources of one decision (whether EmulatedFp64 is honorable) and the build flag is visible to only one. Fix: route the honorability decision through ONE predicate (e.g. `bool emulation_honorable()` or a `resolve_precision(precision) -> {math_mode, compute_type}` consumed by both the engage step and the GEMM calls), so math mode and compute type are always derived together and the `STEPPE_HAVE_EMU_TUNING` fallback is applied in exactly one place. Re-checked: this is a *distinct* structural finding from C-1 (it is why C-1's fix is harder than it looks), worth calling out separately.

**C-3 [high, S, before-M4.5: yes] Per-call `cublasSetStream` inside `run_f2_gemms` resets the user workspace, defeating the §9/§12 reproducibility workspace bound in the backend ctor. — CONFIRMED verbatim against cuBLAS docs.**
Location: `run_f2_gemms`, `.cu` line 242. The `CudaBackend` ctor (`cuda_backend.cu` lines 48-53) calls `cublasSetWorkspace(blas_.get(), workspace_.data(), workspace_.bytes())` once, "for emulated-FP64 bit-stability", with `kCublasWorkspaceBytes = 64 MiB`. **Verified verbatim this pass:** the cuBLAS docs state for `cublasSetStream` — *"Additionally this function unconditionally resets the cuBLAS library workspace back to the default workspace pool (see cublasSetWorkspace())."* So every `compute_f2`: ctor binds workspace → `run_f2_gemms` calls `cublasSetStream` → the 64 MiB workspace is reset to cuBLAS's default pool → the three GEMMs run on the default pool, not the §12-mandated explicit workspace.

Architecture.md §9 (`build()` validation, line 620) explicitly rejects "`deterministic == true` with `precision == EmulatedFp64` unless an explicit `cublasSetWorkspace` workspace is configured"; §12 (line 31, line 732) lists the explicit workspace as a hard emulated-FP64 constraint ("Per-stream `cublasSetWorkspace` prevents *corruption*"); and this TU's own `.cuh` lines 62-63 say "the handle must already be created … and have its workspace set for emulated-FP64 determinism." The `cublasSetStream` call silently undoes that precondition.

Adversarial nuances re-checked this pass:
- **Does the ctor even bind a stream?** `CublasHandle` (handles.hpp lines 44-47) calls `cublasSetStream` *only* `if (stream)`. In `cuda_backend.cu` `stream_ == nullptr`, so `CublasHandle blas_{stream_}` does **not** call `cublasSetStream` at construction. The first and every subsequent `cublasSetStream` is the one in `run_f2_gemms` (`cublasSetStream(handle, nullptr)`). So the ctor binds the workspace, then the first GEMM call resets it. The reset is real on the very first `compute_f2`.
- **Is it a no-op because the stream is the same null default stream?** No — the docs say the workspace reset is *unconditional*, regardless of whether the stream actually changes.
- **M4 path:** `compute_f2_blocks` engages precision once (`cuda_backend.cu` line 214) and then `run_f2_gemms_group` calls `cublasSetStream` once per chunk (`f2_blocks_kernel.cu` line 190) — same workspace reset, **per chunk**. So the defect is live on both M0 and M4; the M0 path is only "incidentally less harmful" because it issues exactly three GEMMs in one call.

The failure is invisible (numbers are *close*, just not bit-reproducible run-to-run, and emulated-FP64 may pick a different internal kernel without its workspace). Fix: do **not** call `cublasSetStream` inside `run_f2_gemms` / `run_f2_gemms_group`; bind the stream once at handle creation (the §7 idiom; the M4.5 owning-`Stream` cleanup, TODO line 74, will make the handle stream-bound at construction). If a per-call stream is ever genuinely needed, pair every `cublasSetStream` with an immediate re-bind of the explicit workspace (`cublasSetWorkspace`). Not in TODO.md as a standalone item, though TODO line 76 ("drop the redundant re-engage in `run_f2_gemms`") is adjacent.

**C-4 [resolved-correct — re-verified] The GEMM operand transposes, leading dims, and the `R` row split match the §5 S2 / `assemble_f2_numerator` contract.**
Re-traced all three this pass: G = Q·Qᵀ (OP_N/OP_T, m=n=P, k=M, lda=ldb=ldc=P) ⇒ `G(i,j)=Σ_s Q(i,s)Q(j,s)` ✓; Vpair = V·Vᵀ likewise ✓; R = S·Vᵀ with S `[2P×M]` lda=2P, V OP_T ldb=P, R `[2P×P]` ldc=2P, top P rows `Σ_s Qsq(i,s)V(j,s)` and bottom P rows `Σ_s Hc(i,s)V(j,s)` ✓. In `assemble_f2_kernel`: `sumsq_i=R(i,j)`, `sumsq_j=R(j,i)`, `hsum_i=R(P+i,j)`, `hsum_j=R(P+j,i)` (lines 150-153), feeding `assemble_f2_numerator(sumsq_i,sumsq_j,Gij,hsum_i,hsum_j)` = `sumsq_i+sumsq_j−2·cross−hsum_i−hsum_j` (f2_estimator.hpp lines 97-101) ✓. The masked-GEMM zero-fill (`Q=0`, `V=0` on invalid) makes `R(i,j)` sum only over SNPs valid in both i and j. Verified correct; no action.

**C-5 [low, S, before-M4.5: no] `f2_feeder_kernel` reads `Q_raw[idx]` / `N_raw[idx]` for `het_correction` even when `valid` is false.**
Location: `.cu` line 111. When `valid==false`, `het_correction` returns 0 early (f2_estimator.hpp line 60) and the operands are discarded; the loads are in-bounds (`idx ∈ [0, P·M)`) and the backend uploads fully-populated `[P×M]` arrays (`cuda_backend.cu` lines 76-81), so every `idx` holds a defined value. No NaN risk (value discarded). Re-verified: passing `Q_raw[idx]` (raw) rather than the masked `q` to `het_correction` is equivalent — when valid, `q == Q_raw[idx]`; when invalid, `het_correction` returns 0 regardless. Correct; no action (the redundant *load* of `Q_raw[idx]` is the only nit — see P-2).

### (2) Edge cases & failure modes

**E-1 [high, M, before-M4.5: yes] `const int Mi = static_cast<int>(M)` in `run_f2_gemms` is an unchecked narrowing of a `long` SNP count into `int`. — CONFIRMED.**
Location: `.cu` line 248. `M` is `long` (the `.cuh` line 33 explicitly says "`M` is `long` (SNP-count scale)"); **verified this pass** that `cublasGemmEx` takes `int k` while the `_64` variants take `int64_t`. `run_f2_gemms` is the M0 *single-block, all-SNPs* path (`cuda_backend.cu` `compute_f2` passes the full `M`, line 86), **not** a per-tile call — so the cast comment "M fits a tile" is aspirational and misleading (R-3). For M ≥ 2^31 the conversion is implementation-defined; on the usual lowering it yields a negative/wrapped `k` → cuBLAS either rejects (`CUBLAS_STATUS_INVALID_VALUE`) or silently reduces over fewer SNPs → a wrong-but-plausible f2. Why it matters: architecture.md §11.1/§11.2 push toward streaming at AADR scale (10^6–10^7 SNPs); ROADMAP §4 treats index/size widths as a "say what you mean" discipline (it kept `size_t` indexing "mandatory above P≈32k"). Adversarial re-check: is M genuinely bounded < 2^31? `compute_f2` does not tile; it uploads all M, so the only bound is VRAM for `dQ_raw` (P·M·8 bytes), which at P=768 caps M around ~10^9 — *above* 2^31. The bound is not enforced anywhere. Fix: (a) `if (M > std::numeric_limits<int>::max()) throw`/typed error before the cast (fail-fast, §2), or (b) move the M0 path to `cublasGemmEx_64` (int64_t k). Given the M0 role is "one big GEMM over all SNPs," (b) is more future-proof; (a) is the minimum. Does **not** apply to the M4 `cublasGemmStridedBatchedEx` (k = `s_pad`, small).

**E-2 [med→low, S, before-M4.5: yes] `twoP = 2 * P` computes in `int`; a documentation gap on the supported P range, not a live overflow. — CONFIRMED (downgraded, as the first pass had it).**
Location: `run_f2_gemms` line 249 `const int twoP = 2 * P;`, passed as `int m`/`lda`/`ldc` at lines 266-268. ROADMAP §0 tables go to P=4,266 (TODO projects single-GPU `P_max ~3,331` on the PRO 6000), so `2*P ≤ ~8,532` — nowhere near `INT_MAX`; cuBLAS m/lda *must* be `int` here, so `twoP` correctly stays `int`. Worth a finding only because `assemble_f2_kernel` (lines 143-144) computes `twoP` in `size_t` precisely because the flat index `R[(Pp+si)+sj*twoP]` overflows `int` above P≈32k — the file already knows scaled products need width, yet `run_f2_gemms` keeps it `int` (correctly) without stating the assumption. Fix: a one-line comment "P bounded by cuBLAS int lda (2·P ≤ INT_MAX); the [P×P]/[2P×P] flat indices use size_t in assemble_f2_kernel for P>32k." Low/med — undocumented, this invites a future contributor to "optimize" the assemble kernel's `size_t` back to `int`.

**E-3 [med, S, before-M4.5: yes] No `P<=0 || M<=0` guard on the M0 path; a zero/negative extent throws (not corrupts) but inconsistently with the sibling entry points. — CONFIRMED, with a sticky-error nuance added.**
Location: `launch_f2_feeder` (lines 223-235), `run_f2_gemms` (237-269), `launch_assemble_f2` (271-280); caller `compute_f2` (`cuda_backend.cu` lines 55-103). Re-verified: `compute_f2` does **not** guard `P<=0||M<=0`, while `compute_f2_blocks` (line 130) and `decode_af` (line 298) *do* early-return. With M=0: `core::cdiv(0, 16) = 0` → a zero-y-extent grid. **Verified this pass:** CUDA rejects a zero grid dimension with `cudaErrorInvalidConfiguration`, and the error is *sticky* — but `STEPPE_CUDA_CHECK_KERNEL()` calls `cudaGetLastError()` immediately, which **clears and throws** before the sticky state can corrupt the next call. So the feeder launch throws cleanly. (If the M0 path ever issued a second kernel before the check, the sticky error would mis-attribute — but here the check is immediate, so it is safe.) Then `run_f2_gemms` would pass `Mi=0` as `k` to `cublasGemmEx` → `CUBLAS_STATUS_INVALID_VALUE` (caught by `CUBLAS_CHECK`). Net: it throws from deep in the runtime/cuBLAS rather than from a clear precondition. Why it matters: §13 testability and consistency with the sibling guards; M4.5 multi-GPU sharding can hand a device an empty SNP shard. Fix: add the `P<=0||M<=0` early-return to `compute_f2` (matching `compute_f2_blocks`/`decode_af`), and/or document `P>=1, M>=1` on each launch wrapper.

**E-4 [med, S, before-M4.5: yes] `precision.mantissa_bits` forwarded to `cublasSetFixedPointEmulationMaxMantissaBitCount` with no range validation. — CONFIRMED, sharpened with the doc-confirmed FIXED-mode cap.**
Location: `engage_f2_precision` lines 208-209. `precision.mantissa_bits` is an `int` (config.hpp line 156, default 40); nothing in this TU clamps it. **Verified this pass:** the cuBLAS FIXED-mode default max mantissa bit count is **55**; config.hpp documents meaningful values {32, 40, 48}. A caller building `Precision{EmulatedFp64, 0}` / `{-1}` / `{200}` forwards that verbatim; cuBLAS may reject (`INVALID_VALUE`, caught) or silently clamp (undocumented behavior at out-of-range). Why it matters: §9 "typed immutable config validated once" — the value is *typed* but not *validated*, and the validation home (`build()`) is not in this TU; but this TU is the only place the value crosses into cuBLAS, and the `.cuh` line 41 advertises "FIXED mantissa control at `precision.mantissa_bits`" as a contract. Fix: document the precondition "`mantissa_bits` validated to a sane range [≈10, 55] at `build()`" OR add a debug assert / typed throw for out-of-range. Low cost.

**E-5 [low, S, before-M4.5: no] `f2_feeder_kernel` index-width path is correct (`long` throughout the SNP axis).**
Re-verified lines 95-101. `i,s` are `long`; compared to `P` (int promotes to long); `sidx = i + (2*Pl)*s` and `idx = i + Pl*s` are long arithmetic, no overflow for any feasible P/M; all in-bounds. This is the one kernel here that *correctly* uses `long` for the SNP axis (in contrast to `run_f2_gemms`'s int cast, E-1). No action.

### (4) CUDA idioms / RAII / stream & async semantics / launch config vs §7

**I-1 [high — same root as C-3, S, before-M4.5: yes] Per-call `cublasSetStream` violates the "create once / bind once" handle idiom (§7) and is the mechanism of the C-3 workspace reset. — CONFIRMED.**
Location: `run_f2_gemms` line 242 (and `run_f2_gemms_group` line 190). Architecture.md §7 line 489/423 says library handles are created once and never re-bound per-iteration. The handle is constructed via `CublasHandle blas_{stream_}` (`cuda_backend.cu` line 339) — though, re-verified, with `stream_ == nullptr` the ctor does *not* call `cublasSetStream`, so the per-call `cublasSetStream(handle, nullptr)` in the wrapper is both the *only* stream bind and (per C-3) destructive of the workspace. 1. **Fix C-1 / C-2 / T-CAP-1 (the dynamic-mantissa trap) — the single most important item.** When `STEPPE_HAVE_EMU_TUNING` is off and EmulatedFp64 is requested, do not silently engage DYNAMIC emulation: route the honorability decision through ONE predicate driving *both* the math mode and the compute type, and either `#error`, or fall back to native Fp64 with a **capability-tagged logged warning** + recorded tag, or throw `INVALID_CONFIG`. ## Good patterns to keep

- **The two-kernel + three-GEMM reformulation** exactly as §5 S2 mandates — never materializing `[SNP×pop×pop]`, heavy arithmetic in cuBLAS, only the bandwidth-bound feeder and the cancellation-prone numerator/divide as custom kernels.
- **Native FP64 held for the catastrophic-cancellation numerator/divide in every precision mode** (the assemble kernel has no precision branch) — the §12 law, correctly realized.
- **The shared `__host__ __device__` per-element primitives** (`het_correction`, `assemble_f2_numerator`, `finalize_f2`) called from thin kernel shells — the oracle-cannot-diverge discipline (§13) and the exemplar of the §7 "thin shell around a scalar primitive" model.
- **Exposing `engage_f2_precision` / `f2_compute_type` as the single shared precision-policy source** consumed by both M0 and M4 (DRY §2) — the *intent* is exactly right (the C-2 fix preserves it, just makes the two impossible to disagree).
- **Complete error-check coverage** — `CUBLAS_CHECK` on every cuBLAS call, `STEPPE_CUDA_CHECK_KERNEL()` after every launch, routed through the single check.cuh home (§7/§8).
- **All spike literals promoted to named config/constants** — launch geometry via `kCdivBlock`, grid math via `grid_for`/`cdiv`, per-SNP `N` (no `/198.0`), mantissa bits via `precision.mantissa_bits` (§4 exemplary).

- **The M4 blocks test's bit-identity engagement guard** (`emu != native` ⇒ engaged) — a clever, header-independent silent-fallback detector; the fix is to make it FAIL (not skip) when EmulatedFp64 is the declared default, and to extend it to the M0 production path.
