## Role & layering

The unit is the GPU implementation of architecture.md §5 stage **S0 (format decode)** fused with **S1 (allele-freq reduction)**: a packed individual-major TGENO tile → the column-major `[P × M]` Q/V/N contract. It is structured exactly as §7 prescribes for a hot-path feeder kernel — a thin `__global__` shell doing index math + bounds, calling the shared `__host__ __device__` per-element primitive (`genotype_code` / `genotype_valid` / `finalize_af`) that is *also* the CPU oracle's per-element numerics. I confirmed the CPU oracle (`cpu_backend.cpp:304-326`) and this kernel (`.cu:65-90`) are structurally identical (same `i + P·s` offset, same `s/4` / `s&3` split, same accumulate-then-`finalize_af`) — the divergence-proofing is genuine. The header correctly documents itself as CUDA-private to `steppe_device` (it names `cudaStream_t`), and the launch wrapper is the narrow `void launch_decode_af(...)` with no `<<<>>>` leaking to host code (architecture.md §4, §7). The `STEPPE_CUDA_CHECK_KERNEL()` post-launch check is present and correctly placed (verified against `check.cuh:135-146`: release does only `cudaGetLastError()`, debug adds `cudaDeviceSynchronize`).

Layering is clean: the `.cu` includes only the shared core primitive (CUDA-free header) and the device check header — no `io`, no upward dependency. **The price of "core must not depend on io" (architecture.md §4) is paid as a documented duplication — see F-MAG-3, where a neighbouring io comment overstates that duplication as a "single site."**

## Score: 8.5/10 — correct, well-layered, oracle-shared; held back by (1) the launch-config DRY violation (the lone kernel that re-picks block geometry + re-implements `cdiv`), (2) a *factually wrong* code comment here and a *factually false single-source claim* in the io header about this primitive, (3) the deeper but legitimate core↔io bit-order/missing-code duplication, (4) a few missing precondition guards the wrapper delegates to its sole caller, and (5) the lone shared core primitive without a GPU-free unit test when the sibling pattern already exists.

The debt is concentrated in DRY/consistency/comment items. **The only one of these the project plan files as `before-M4.5` is the launch-config refactor (`TODO.md:87`, section A); the `cdiv` deletion, the `/4,&3` literal, the stale "16×16" comment, and the grid-clamp are all in section B ("Later / M5-timed", `TODO.md:91-92`) and are correctly `before-M4.5: no` — the first review's `YES` on these is a mis-read, corrected here.**

## Findings

### (1) Correctness & bugs

**F-COR-1 — `grid.y` (population axis) is never guarded against the 65 535 grid-dimension limit. Severity: low. Effort: S. Before M4.5: no (corrected back from the first review's erroneous "yes").**
`launch_decode_af` (`.cu:106-108`) computes `grid.y = cdiv_i(P, kDecodeBlockY)` with `kDecodeBlockY = 8`. I verified against the CUDA C++ Programming Guide compute-capability technical-specifications table that the maximum grid dimensions are **(x, y, z) = (2 147 483 647, 65 535, 65 535)** — only `gridDim.x` may reach 2³¹−1; `.y`/`.z` are capped at 65 535, and these limits are unchanged through compute capability 12.0 ([CUDA C++ Programming Guide, Features and Technical Specifications](https://docs.nvidia.com/cuda/cuda-c-programming-guide/index.html); corroborated [NVIDIA forums maxGridSize](https://forums.developer.nvidia.com/t/maximum-grid-size-2147483647/35211)). A launch exceeding a grid limit returns `cudaErrorInvalidConfiguration` ([CUDA Runtime API, Execution Control](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__EXECUTION.html)). The kernel correctly puts the *large* axis (SNP count `M`) on `x` and the small axis (`P`) on `y`, so this only bites at `P > 65 535·8 = 524 280` — far beyond `P_max ≈ 3 331` (TODO capability tiers; the PRO-6000 96 GB single-GPU ceiling). **Not a live bug.** *Re-classification correction:* the first review moved this to `before-M4.5: yes`, asserting `TODO.md:92` lists "clamp `grid.z` to 65535" as a "committed pre-M4.5 cleanup item." That is wrong twice over: (a) `TODO.md:92` is in section **B "Later (real but cosmetic / M5-timed)"** (`TODO.md:91`), explicitly NOT before M4.5; (b) the TODO item is about the f2 `grid.z`, and — verified — `f2_blocks_kernel.cu:176-178,231-233` set `grid.z = n_in_group` by a **direct** `static_cast<unsigned>(n_in_group)`, *not* through `grid_for`, so folding a clamp into `grid_for` (the draft's proposed fix) would NOT catch the f2 `grid.z` it cites. The decode `grid.y`, by contrast, *would* be covered by a `grid_for` clamp once F-CUDA-1 routes it through `grid_for`. *Fix:* when the wrapper adopts the launch-config home (F-CUDA-1), have the grid helper assert (debug) / clamp the y/z extents ≤ 65 535; separately ensure the f2 `grid.z` (which bypasses `grid_for`) is covered by the same guard or its own. Grep confirms NO `65535`/`maxGridSize` guard exists anywhere in `src` today — a project-wide latent gap, not decode-specific, and correctly deferred.

**F-COR-2 — no validation that `ceil(M/4) <= bytes_per_record`; only the `ceil(M/4)` half is checkable in the wrapper, the `pop_offsets[P]==n_individuals` half is not. Severity: low. Effort: S. Before M4.5: no.**
The kernel reads `packed[g * bytes_per_record + byte_in_rec]`, `byte_in_rec = s/4` for `s ∈ [0,M)`. Correctness depends on `bytes_per_record >= ceil(M/4)` (the requested SNP byte is present in the record). I verified the wrapper signature (`.cuh:37-42`) takes `bytes_per_record` and `M` but **not** `n_individuals` — so the related invariant `pop_offsets[P] == n_individuals` genuinely *cannot* be checked inside `launch_decode_af`; only `CudaBackend::decode_af` (which has the full `DecodeTileView`, `backend.hpp:68-81`, with `n_individuals` and `pop_offsets`) can check it. The first review correctly disentangled these two halves. *Fix:* (a) debug-assert `static_cast<std::size_t>((M + 3) / 4) <= bytes_per_record` in the wrapper from its own parameters; (b) push the `pop_offsets[P] == n_individuals` check up to `CudaBackend::decode_af`, where `n_individuals` is in scope. Out-of-contract reads otherwise produce silently-wrong Q.

**F-COR-3 — `int i` vs `std::size_t` offset indexing is internally consistent (verified, not a bug).** `i` is `int` (matches `P`); every memory index casts to `std::size_t` first (`pop_offsets[static_cast<std::size_t>(i)]`, and `off = i + P·s` is computed with all three operands cast to `std::size_t`, `.cu:86-87`). Cannot overflow at `P·M` scale; matches the column-major contract (`backend.hpp:64-67`) and the oracle's `i + P·s` (`cpu_backend.cpp:325-326`) exactly. No action.

### (2) Edge cases & failure modes

**F-EDGE-1 — empty tile (`P<=0 || M<=0`) guard lives in the caller, not the wrapper. Severity: low. Effort: S. Before M4.5: no.**
Verified: `CudaBackend::decode_af` (`cuda_backend.cu:298`) — *and* the CPU oracle (`cpu_backend.cpp:300`) — early-return on `P<=0 || M<=0`, so the kernel never launches empty in practice. But `launch_decode_af` is a header-declared seam other device TUs can call; with `M==0` it computes `grid.x = cdiv_l(0,32) = 0`, a zero-extent grid. I verified with the official docs that a grid dimension of zero is an invalid configuration: `blockDim`/`gridDim` components cannot be zero, yielding `cudaErrorInvalidConfiguration` ([CUDA Runtime API, Execution Control](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__EXECUTION.html); behavior corroborated [NVIDIA forums on invalid configuration](https://forums.developer.nvidia.com/t/getting-error-cudaerrorinvalidconfiguration/110033)) — which `STEPPE_CUDA_CHECK_KERNEL()`'s `cudaGetLastError()` surfaces as an opaque `CudaError` throw rather than a clean no-op. Separation of concerns (§2): the wrapper should own its own launch validity. *Fix:* `if (P <= 0 || M <= 0) return;` at the top of `launch_decode_af`.

**F-EDGE-2 — empty population segment (`seg_begin == seg_end`) is correct (verified).** The `for (g…)` body never executes, `ac=an=0`, `finalize_af(0,0,ploidy)` returns `{0,0,0}` via its `an>0` else-branch (`decode_af.hpp:103-107`). Matches the contract's "0 where invalid" and the oracle. No action.

**F-EDGE-3 — `ploidy == 0` (or negative) is silently mishandled; the home for the guard is the ingest boundary. Severity: low. Effort: S. Before M4.5: no.**
Verified against `finalize_af` (`decode_af.hpp:96-109`): for `ploidy == 0` with `an>0`, it computes `n = 0.0·an = 0`, then `q = ac/0 = ±inf/nan`, and `v = 1.0` — violating the `V==(N>0)` invariant the equivalence test asserts (`test_decode_equivalence.cu:272`, `(v != 0.0) != (n > 0.0)`). For negative `ploidy`, negative `N` and out-of-range `Q`. `DecodeTileView::ploidy` defaults to 2 and is documented "never inferred" (`backend.hpp:80`); the test hard-codes `view.ploidy = 2` (`test_decode_equivalence.cu:212`). So this is "garbage in," not a live bug — but nothing on the path (kernel, wrapper, `finalize_af`, or the ingest) rejects it. *Fix:* validate `ploidy >= 1` at the config/ingest boundary that sets `DecodeTileView::ploidy`; optionally a debug assert in the wrapper. This is also the branch the F-TEST-1 unit test would pin.

**F-EDGE-4 — integer width of accumulators is safe (verified).** `ac`/`an` are `long`; `an ≤ n_individuals` (≈27 594 for AADR v66), `ac ≤ 2·an`, both far below `LONG_MAX`. The byte index `g*bytes_per_record + byte_in_rec` is `size_t×size_t`, 64-bit. The `finalize_af` product `ploidy*an` ≈ 55 188 — no FP64 representability loss. No wraparound at any adversarial tile size. No action.

**F-EDGE-5 — `cdiv_l`/`cdiv_i` overflow is theoretical only. Severity: low. Effort: S. Before M4.5: no (folds into F-CUDA-1).** `cdiv_l(M,32)` overflows only for `M > LONG_MAX-31`, impossible for a SNP tile. The shared `core::cdiv` (`f2_estimator.hpp:125-132`) has the identical `(n+b-1)/b` form, so adopting it neither helps nor hurts overflow — it just removes the duplicate. No correctness action beyond F-CUDA-1.

### (3) Numerical / precision vs §12

**F-NUM-1 — precision strategy is exactly right (verified, keep).** Integer accumulation (`ac`/`an` are `long`) + a single native FP64 divide in `finalize_af` (`decode_af.hpp:99-101`). This is precisely §12's mandate for the bandwidth-bound decode and what makes Q bit-exact (`max|Δ|=0`) against the CPU oracle and the numpy golden (the test's `EXACT` gate, `test_decode_equivalence.cu:23-24,259-261`). No cross-thread floating accumulation ⇒ no accumulation-order concern. N/A for Ozaki/emulation — integer + one-divide by construction.

**F-NUM-2 — determinism is structurally guaranteed (verified).** One thread owns one `(i,s)` output, reducing serially in fixed index order; the result is independent of warp scheduling, block count, and stream — run-to-run and GPU-to-GPU bit-identical, which the parity gate needs. A one-line "determinism is free here; do not convert to a cross-thread/atomic reduce" comment would pin the property against a future F-PERF-2-style change. Low value.

### (4) CUDA idioms / RAII / stream & async / launch config / occupancy vs §7

**F-CUDA-1 — re-picks block geometry and re-implements `cdiv`, violating the single launch-config home. Severity: med (high on the DRY-standard axis). Effort: S. Before M4.5: YES.**
The headline issue, re-verified. The unit defines its own `kDecodeBlockX = 32`, `kDecodeBlockY = 8` (`.cu:50-51`) and its own `cdiv_l`/`cdiv_i` (`.cu:95-96`). Confirmed against the tree:
- `config.hpp:50-51` documents `kCdivBlock = 16` and states verbatim *"The single source of launch geometry lives in `core/internal/f2_estimator.hpp` / `launch_config.hpp`; kernels never re-pick a block size."*
- Both sibling kernels use the shared home: `f2_block_kernel.cu:229-231` (`dim3 block(steppe::kCdivBlock, steppe::kCdivBlock)` + `core::grid_for(P)` + `core::cdiv(M, static_cast<long>(steppe::kCdivBlock))`) and `f2_blocks_kernel.cu:175-178` (same pattern). **This TU is the lone divergence — verified by reading all three wrappers.**
- `TODO.md:87` (section **A "Before M4.5"**) lists the phantom `launch_config.hpp` with the fix verbatim: *"create `core/internal/launch_config.hpp`, move `cdiv`+`grid_for` there, route the decode kernel through `core::cdiv`."* This is the project's own before-M4.5 plan, so this finding is correctly `before-M4.5: yes`.

There is a *legitimate* technical reason a decode-specific geometry differs: `kCdivBlock=16` gives `x=16` = half a warp, so SNP-axis coalescing would be worse than the warp-aligned `x=32` chosen here. The 32×8 choice is defensible *on the merits*. But the project rule is that such a decision lives in the launch-config home as a **named, warp-alignment-justified constant**, not as bare literals re-picked in a kernel TU. *Fix (corrected from the first review's imprecise wording):* (a) add a decode-specific 2-D block constant to the launch-config home alongside `kCdivBlock`, with the warp-alignment rationale; (b) delete the local `cdiv_l`/`cdiv_i` and consume `core::cdiv`. **Note the width subtlety the first review's "consume `core::grid_for`/`core::cdiv`" glossed over:** `core::grid_for` is `int`-only and returns `int` (`f2_estimator.hpp:138`), so it cannot cover the SNP/x axis where `M` is `long`. The correct mapping mirrors the sibling f2 feeder exactly — `grid.x = core::cdiv(M, static_cast<long>(kDecodeBlockX))` (the `long` overload, `f2_estimator.hpp:130`) and `grid.y = core::grid_for(P, kDecodeBlockY)` or `core::cdiv(P, kDecodeBlockY)` (the `int` overload). The "kept here so this kernel TU has no dependency beyond the decode primitive" rationale (`.cu:93-94`) is weak — both sibling kernels already include `f2_estimator.hpp` for exactly these helpers with **zero layering harm** (it's a CUDA-free core header), proving the dependency is fine.

**F-CUDA-2 — the doc comment references a file that does not exist. Severity: low. Effort: S. Before M4.5: yes.**
Re-verified by `find`: there is **no** `core/internal/launch_config.hpp`. Three sites reference this phantom: `.cu:93` (this TU's `cdiv_*` comment), `config.hpp:50`, and `f2_estimator.hpp:117` ("These mirror core/internal/launch_config.hpp's `cdiv`/`grid_for`"). The helpers actually live in `f2_estimator.hpp:114-140`. `TODO.md:87` (section A) explicitly tracks creating this file. *Fix:* create the long-promised `launch_config.hpp` and migrate `cdiv`/`grid_for`/the new decode geometry there (cleanest — kills three phantom references at once and gives F-CUDA-1 a clean home), or correct all three references to `f2_estimator.hpp`. This is a cross-TU comment defect, but this TU is one of the three carriers, so it is in-scope.

**F-CUDA-3 — kernel takes raw pointers, not `cuda::std::span`/`mdspan`. Severity: low. Effort: M. Before M4.5: no.**
§7 states the interface currency is `cuda::std::span`/`mdspan` and "kernels accept only views, never owners or bare pointers." This kernel (and both sibling f2 kernels) take bare `const std::uint8_t* __restrict__` etc. — a project-wide pattern, not decode-specific. I verified `__restrict__` is *sound* here: the three output buffers `dQ/dV/dN` are distinct `DeviceBuffer<double>`s (`cuda_backend.cu:309`) and never alias `packed`/`pop_offsets`, so the compiler's no-alias promise — which lets it eliminate redundant loads, reorder loads/stores, and (with `const`) use the read-only data cache — holds ([CUDA Pro Tip: Optimize for Pointer Aliasing](https://developer.nvidia.com/blog/cuda-pro-tip-optimize-pointer-aliasing/): "any data written to through that pointer is not read by any other pointer with the restrict property"). `TODO.md:92` (section B "Later") lists "restore `DeviceBuffer::view()` + new M4.5 wrappers span-first" — so the span port is a tracked M5-timed/M4.5-era sweep, correctly deferred. Rate low, not-before-M4.5: a uniform consistency debt across the device kernels.

**F-CUDA-4 — stream usage and post-launch check are correct (verified, keep).** Launched on the caller's `stream` with `0` dynamic shared memory; `STEPPE_CUDA_CHECK_KERNEL()` immediately follows (`.cu:109-112`). Verified against `check.cuh:135-146`: synchronous `cudaGetLastError()` in release, debug-only `cudaDeviceSynchronize()` to attribute async faults under compute-sanitizer. The wrapper does NOT synchronize the stream — the caller `CudaBackend::decode_af` owns the single `cudaStreamSynchronize` after the D2H copies (`cuda_backend.cu:330`). Correct separation: feeder stays async, orchestrator owns ordering. No action.

**F-CUDA-5 — occupancy is not reasoned about anywhere. Severity: low. Effort: M. Before M4.5: no.**
256 threads/block (32×8) is a reasonable default, but §7's launch-config home is supposed to carry "occupancy-aware block sizing," and `grid_for` (`f2_estimator.hpp:138-140`) is pure `cdiv` — no occupancy computation exists project-wide. The decode kernel's per-thread serial reduction means register pressure and segment-length trip count dominate, not block size — a fixed 32×8 is fine until profiled (the Nsight/`ncu` gate runs on the PRO-6000 box per the capability plan, `TODO.md:99-138`). Flag only so the "occupancy-aware" §7 promise is acknowledged as not-yet-delivered. No `__launch_bounds__` is fine (measure-first).

### (5) Magic numbers & hardcoded values vs §4 / ROADMAP §4

**F-MAG-1 — `kDecodeBlockX=32`, `kDecodeBlockY=8` are kernel-local literals where ROADMAP §4 wants the launch-config home. Severity: med. Effort: S. Before M4.5: yes (rides on the section-A `launch_config.hpp` item).** Same root as F-CUDA-1. ROADMAP §4 inventories `dim3 block(16,16)` as a magic number whose home is the launch-config header; §5 says "No magic numbers." These are *named* TU-local constants (better than bare literals) but are launch geometry the project decreed lives in one home. Move with the warp-alignment justification per F-CUDA-1. Because the home file is the section-A `launch_config.hpp` task, this is `before-M4.5: yes`.

**F-MAG-2 — the `s/4` + `s & 3` byte/position split is duplicated host↔device; `kCodesPerByte=4` already exists in io. Severity: low-med. Effort: S. Before M4.5: no (corrected back from the first review's erroneous "yes" — it is a section-B "Later" item, `TODO.md:91-92`).**
`.cu:71-72` compute `byte_in_rec = s/4u` and `pos_in_byte = s & 3`. I verified this *exact pair* is open-coded identically in the CPU oracle (`cpu_backend.cpp:309-310`, modulo `static_cast<std::size_t>(s)/4u` vs `s/4u` — numerically identical for `s≥0`). io *already* owns `kCodesPerByte = 4` (`eigenstrat_format.hpp:54`) and `packed_bytes()` (ceil-div, `eigenstrat_format.hpp:67`). The bit *extraction* is correctly delegated to the shared `genotype_code`, but the byte/position arithmetic is bare. Because `core` cannot include `io` (§4 layering), the clean fix is a tiny shared `code_byte_index(long s)` / `code_bit_position(long s)` (or `decode_locate(s) -> {byte, pos}`) in `decode_af.hpp` itself, consumed by both the kernel and the oracle. *Re-classification correction:* the first review moved this to `before-M4.5: yes`, citing `TODO.md:92` as a "committed pre-M4.5 cleanup item." But `TODO.md:91` is the header **"### B. Later (real but cosmetic / M5-timed)"** and the verbatim entry on line 92 is *"`/4`,`&3` packing literals in 3 TUs → `code_byte_index`/`kCodesPerByte`"* — section B is by definition NOT before M4.5. The original `before-M4.5: no` was correct; restored.

**F-MAG-3 — the io header's "SINGLE bit-order site" claim is FALSE; core re-implements io's extraction and missing-code. Severity: low (correctness-of-claim/DRY). Effort: M. Before M4.5: no (design tension, not a quick fix).**
`eigenstrat_format.hpp:74-75` asserts `code_in_byte` is *"the SINGLE bit-order site shared by the host reader and (via the decode primitive) the GPU/CPU decoders."* I verified this is **not true**: `decode_af.hpp::genotype_code` (`decode_af.hpp:66-70`) does **not** call `code_in_byte` — it re-implements the identical MSB-first extraction independently (`shift = (3 - (k&3))*2`; `(byte >> shift) & 0x3u`), because `core` must not depend on the `io` leaf (§4). Likewise `io::kMissingCode = 3` (`eigenstrat_format.hpp:62`) and `core::kMissingGenotypeCode = 3` (`decode_af.hpp:59`) are duplicate constants, and `io::packed_bytes()` vs the bare `s/4`/`(M+3)/4` (core/device). So there are **two** independent homes for the 2-bit packing math, reconciled *only* by the equivalence test — a deliberate, layering-driven mirror that `decode_af.hpp:55-58,62-65` honestly documents ("mirrors io::kMissingCode … SAME bit order as io::code_in_byte … pinned by the equivalence test"). The defect is two-fold: (a) the *io* comment overstates ("SINGLE … site") what is actually a two-site mirror, which will mislead a maintainer into thinking editing `code_in_byte` alone is sufficient (note: the `decode_af.hpp` comment is honest — it says "SAME bit order … kept here … with no io dependency," not "single"); (b) there is no compile-time `static_assert` tying the two missing-code constants together. *Fix:* (1) correct `eigenstrat_format.hpp:74-75` to say the bit order is *mirrored* in `decode_af.hpp` and pinned by the test (not "single site"); (2) add a `static_assert(io::kMissingCode == core::kMissingGenotypeCode)` at a TU that legitimately sees both (e.g. `eigenstrat_format.cpp` or a test), or — cleaner — define the packing constants once in `decode_af.hpp` (core, the lower layer io is *allowed* to depend on) and have io re-export them via `using`/`inline constexpr` aliases, collapsing the two homes with the dependency arrow pointing the legal direction (io → core). That is the only structural fix that makes the "single site" claim true. This finding straddles two files; it stays in-scope because the false claim is *about* this TU's primitive.

### (6) Decomposition / single-responsibility / function size vs §2

**F-DEC-1 — decomposition is exemplary (verified, keep).** `decode_af_kernel` is a ~26-line thin shell: index math, bounds, serial integer reduction calling the shared scalar primitives, one finalize, one strided write. `launch_decode_af` is a ~14-line pure launch wrapper. Numerics live in `decode_af.hpp`, shared with the oracle. Textbook §7 "thin `__global__` shell around a `__host__ __device__` per-element primitive." No action.

### (7) Readability, naming, const-correctness, `[[nodiscard]]`/`noexcept`, comment density

**F-RD-1 — the "mirroring kCdivBlock … 16×16 = 256" comment is factually wrong. Severity: low (actively misleading). Effort: S. ### (8) Performance

**F-PERF-1 — per-thread serial reduction is the documented S1 design and is appropriate (verified, keep).** §5 S1 specifies "segmented reduction over samples within pop partition"; the kernel does one output per `(i,s)` thread reducing serially over `[seg_begin, seg_end)`. For the AADR shape (P≈50–3 300, individuals-per-pop modest, M huge) the parallelism is the enormous `P×M` output grid; the device is saturated and the kernel is bandwidth-bound on the packed reads (§11.3). No change for M1–M4.

**F-PERF-2 — the packed byte for a (g, byte) is read by 4 lanes; no cross-population redundancy. Severity: low. Effort: M. Before M4.5: no.** Verified: each `(i,s)` thread loads `packed[g·stride + s/4]` independently. Because `pop_offsets` partitions individuals, an individual `g` belongs to exactly one population, so there is **no** cross-population re-read of the same byte. The only redundancy is the 4-codes-per-byte fan-out, which L1/L2 absorbs (4 lanes hit the same sector — the 4-way broadcast of F-RD-2). Shared-mem staging or vectorized `uint4`/`char4` loads is a *possible* later optimization but would complicate the kernel and *risk the determinism property* (F-NUM-2) if it tempted a cross-thread reduce. Measure-first (Nsight gate). Explicitly defer.

**F-PERF-3 — `pop_offsets` is re-read per output column. Severity: low. Effort: S. Before M4.5: no.** Each thread reads `pop_offsets[i]` and `[i+1]` (`.cu:69-70`). Within a block, the same `i` reloads its pair across all 32 x-lanes, all hitting one cached L1 line; 8 distinct lines per block (one per `y`). Negligible; not worth shared-mem caching at this bandwidth profile. Note only.

## Considered & rejected

- **"`int i = blockIdx.y * blockDim.y + threadIdx.y` could overflow `int`."** Rejected: `blockIdx.y·blockDim.y ≤ gridDim.y·blockDim.y ≤ 65 535·8 ≈ 524 k`, within `int`. The real exposure is the un-guarded `gridDim.y` limit (F-COR-1), not an `int` overflow. The SNP axis correctly does `static_cast<long>(blockIdx.x)*blockDim.x + threadIdx.x` so the multiply is in `long`. Verified.
- **"`s & 3` is wrong / should be `s % 4` (sign concern)."** Rejected: `s = blockIdx.x*blockDim.x + threadIdx.x ≥ 0` always, so `s & 3 == s % 4`, and `&3` matches `genotype_code`'s own `k & 3` (`decode_af.hpp:68`) and the io `% kCodesPerByte` (`eigenstrat_format.hpp:77`) numerically. Consistent. Not an issue (but the *duplication* of the split is F-MAG-2).
- **"`s/4u` (mixing signed `long` with unsigned literal) is a sign-conversion hazard."** Rejected: `s` is non-negative `long`; in `s/4u` the `unsigned int` operand widens to `long` on LP64 (long can represent all unsigned-int values), result `long`, assigned to `std::size_t` — value-preserving for `s≥0`. The CPU oracle's `static_cast<std::size_t>(s)/4u` is numerically identical. A cosmetic style mismatch, not a bug.
- **"`finalize_af` divides by zero when `an==0`."** Rejected: guarded `if (an > 0)`, returns `{0,0,0}` otherwise (`decode_af.hpp:98-107`). The only residual zero-divide is `ploidy==0` with `an>0` → F-EDGE-3 (config-validity, not a kernel bug).
- **"Missing `__launch_bounds__` hurts occupancy."** Rejected: bandwidth-bound, simple integer inner loop; `__launch_bounds__` is a measure-first knob (the `ncu` gate, on the PRO-6000 box per `TODO.md`). A speculative add would violate "don't optimize against a guess."
- **"Use `cg::reduce`/`__shfl` for the segment reduction."** Rejected: would convert the embarrassingly-parallel one-thread-per-output design into a cross-thread reduce, adding complexity and *risking determinism* (F-NUM-2) for a kernel already saturated by the `P×M` grid and bandwidth-bound. §7 prefers `cg::reduce` *when an in-kernel reduction is needed* — here the decomposition makes one unnecessary.
- **"`STEPPE_CUDA_CHECK_KERNEL()` forces a serializing `cudaDeviceSynchronize`."** Rejected: that sync is `#if !defined(NDEBUG)` only (`check.cuh:135-146`); release does only `cudaGetLastError()`. Verified, correct per §7.
- **"`pop_offsets` typed `std::size_t` wastes bandwidth vs `int`."** Rejected: two cached loads per column (F-PERF-3); `std::size_t` is correct for an individual-axis offset that can exceed 2³¹ at scale and matches the io tile + oracle. Width-correctness > a non-measurable nit.
- **"Header includes `<cuda_runtime.h>` — forward-declare to lighten the surface?"** Rejected: the header is deliberately CUDA-private (names `cudaStream_t`); including `<cuda_runtime.h>` is correct. The CUDA-free seam to protect is `backend.hpp`, which is clean.
- **"The wrapper should check `pop_offsets[P] == n_individuals` itself."** Rejected for the *wrapper*: `launch_decode_af`'s signature is not given `n_individuals` (`.cuh:37-42`), so it structurally *cannot*; that half of F-COR-2 must live in `CudaBackend::decode_af` (which has the full `DecodeTileView`). Only the `ceil(M/4) <= bytes_per_record` half is checkable in the wrapper.
- **"`finalize_af` returning a 24-byte `AfResult` by value into the kernel is wasteful."** Rejected: it is a POD of three doubles, `STEPPE_HD inline`; the compiler keeps `q/v/n` in registers and there is no call overhead. Verified the kernel writes the three fields straight to global (`.cu:88-90`). Non-issue.
- **NEW — "These five comment/cdiv/literal/clamp findings are on the committed pre-M4.5 list (`TODO.md:92`), so they are before-M4.5."** Rejected (this was the first review's own claim, now overturned): `TODO.md:91` is the section header **"### B. Later (real but cosmetic / M5-timed)"** and line 92 is its body. Section B is by definition NOT before M4.5. Only `TODO.md:87` (section **A "Before M4.5"**, the phantom `launch_config.hpp`) is a genuine before-M4.5 decode item, covering F-CUDA-1/F-CUDA-2/F-MAG-1. F-COR-1, F-RD-1, F-RD-2, F-MAG-2 and the `cdiv` deletion are correctly `before-M4.5: no`.
- **NEW — "fold the grid-limit clamp into `grid_for` to close both decode `grid.y` and the f2 `grid.z`."** Rejected as written (the first review's F-COR-1 fix): verified `f2_blocks_kernel.cu:177-178,232-233` set `grid.z = static_cast<unsigned>(n_in_group)` *directly*, bypassing `grid_for` — so a clamp inside `grid_for` would NOT catch the f2 `grid.z` it claims to.
