# Review — `device-cpu-cpu_backend` (`src/device/cpu/cpu_backend.cpp`)

Unit under review: the CPU **reference / oracle** backend (`CpuBackend`), the scalar/long-double implementation of `ComputeBackend` that the GPU 3-GEMM path is continuously diffed against. This is the **adversarial second pass**: every first-pass finding was re-verified line-by-line against the actual source and against the official CUDA / C++ docs; false positives are demoted to *Considered & rejected* with the reason; new issues were hunted across all mandate categories. The first pass was unusually thorough and accurate — every load-bearing claim survived independent verification, so this pass **confirms** the existing findings, **adds** a small cluster of genuinely-new minor items (2.4b dead `n_individuals` field; 2.6 `size_t→long` narrowing in decode; 8.1 `terms` over-alloc for P==1), tightens the doc-citations, and re-confirms the score.

Re-read for this pass: `cpu_backend.cpp` (the unit), `device/backend.hpp` (interface + contract), `core/internal/{views,f2_estimator,decode_af}.hpp`, `include/steppe/{config,fstats}.hpp`, `core/domain/block_partition_rule.hpp`, `core/fstats/f2_from_blocks.{hpp,cpp}` (the orchestration seam), the GPU peer `device/cuda/{cuda_backend.cu,f2_block_kernel.cu,f2_blocks_kernel.cu,decode_af_kernel.cu}`, `io/filter/mind_prepass.cpp` (third packing TU), `cmake/SteppeWarnings.cmake` (the `-Werror` regime), and the equivalence tests `tests/reference/test_f2{,_blocks}_equivalence.cu` + `test_decode_equivalence.cu`. Standards: `architecture.md` §2/§4/§5/§8/§9/§11/§12/§13, `ROADMAP.md` §4/§5/§6, `TODO.md` (cleanup backlog + capability tiers). Two external-doc claims were verified against authoritative sources (cited inline at 2.3 and 3.4).

## Role & layering

This file is the correctness anchor of the whole project (`architecture.md` §2 "Testability"/"Correctness before speed"; §4 line 150 names it "REFERENCE backend: long-double cancellation-free f2 (the correctness oracle)"; §13 makes "identical inputs through `CpuBackend` and `CudaBackend`, asserted equal" the central trust seam). It is the *oracle*, not a structural template: its triple-loop scalar walk must agree with the GPU's three-GEMM reformulation at the `f2`/`Vpair` seam within the tight tolerance tier (§12, §13), and it does so because both consume the **same** `core/internal/f2_estimator.hpp` primitives (`het_correction`, `f2_term`, `finalize_f2`) and the same `decode_af.hpp` primitives. Verified at the call sites on both sides: the GPU `assemble_f2_kernel` (`f2_block_kernel.cu` line ~155) calls `assemble_f2_numerator`/`finalize_f2`; the CPU side calls `f2_term`/`finalize_f2`; the GPU `decode_af_kernel` (`decode_af_kernel.cu` lines 71-88) and this oracle (lines 309-324) call the identical `genotype_code`/`genotype_valid`/`finalize_af`. The formula is shared, not copied.

Layering is correct and exemplary: pure host C++20, **no CUDA header included** (verified: includes at lines 48-57 are `<cstddef>/<cstdint>/<memory>/<vector>` plus four CUDA-free project headers). It satisfies the §4 rule that the reference backend "compiles into `steppe_device` … but pulls in no CUDA toolkit headers" (CMake `target_sources(steppe_device PRIVATE … cpu/cpu_backend.cpp)`, §4 line 503-504) and is what makes `import steppe` work GPU-free. RAII/ownership is delegated to the base class's deleted copy/move (`backend.hpp` lines 108-113) and `std::unique_ptr` via the factory (lines 342-344) — it owns no raw resources, so the §7 owning-wrapper findings are genuinely N/A for this unit.

The file is small (347 lines, four methods) and disciplined — the `TODO.md` "Good patterns to PRESERVE" verdict ("Shared `__host__ __device__` per-element primitives (oracle≡GPU) … every constant doc-commented w/ arch+ROADMAP cite") applies directly. But it is **not** near-perfect: there is one genuine latent correctness divergence (the M0 diagonal), one DRY violation the backlog already tracks (the block-range scan, verbatim-duplicated with the GPU backend), an absent fail-fast cluster around `block_id`/`DecodeTileView`, two near-duplicate pair-loop bodies, and a set of smaller numerical/readability/testability items. The review below is exhaustive per the mandate.

## Score: 8.5/10 — strong oracle; one latent M0 parity divergence + a DRY/fail-fast cluster keep it off 9.5

Re-verified at 8.5 (unchanged from the first pass — the score is correct after independent confirmation). A high-quality, well-documented reference backend whose numerics are right and whose layering is clean. It loses ground on, all confirmed against source: (1) a real but uncaught M0 `compute_f2` diagonal divergence from its own GPU peer and its own sibling `compute_f2_blocks` (1.1); (2) the production `CpuBackend::compute_f2` has **zero** test coverage — the M0 test uses an inline private oracle, which is *why* 1.1 went unnoticed (10.1); (3) absent fail-fast on the `block_id` contract in a function that indexes vectors by `block_id[s]` — a malformed partition is silent OOB / silent SNP-dropping, not a fast failure, contra §2 "Fail-fast" (2.1); (4) the block-range scan duplicated verbatim with `cuda_backend.cu` and re-deriving a §8 single-home domain rule (6.2, backlog line 81); (5) two ~25-line near-duplicate pair-loop bodies (6.1, backlog); (6) capability-tier blindness — the oracle is the parity gate for **both** the PRO-6000 and budget-5090 tiers (`TODO.md`: "the AT2-golden + native-FP64 oracle gate both boxes") yet carries no acknowledgement of that role and no hook for the M4.5 tagged-fallback validation it underwrites (11.1). These are concentrated, not pervasive; none is a shipped wrong number today, but (1)+(10.1) together are a live latent-bug incubator. The adversarial pass did not find a new high-severity issue and did not reject any existing finding — it confirms the existing six pressure points and the 8.5.

## Findings

### (1) Correctness & bugs

**1.1 [HIGH, before-M4.5: yes] `compute_f2` (M0) leaves the diagonal at 0; the GPU `compute_f2` peer fills it with `−2·mean_het_i` — a real, latent, untested divergence between the two production M0 backends. CONFIRMED — arithmetic re-derived.**
Location: `compute_f2`, loop bound line 123 (`for (int j = i + 1; j < P; ++j)`) and zero-init lines 107-109.
The CPU `compute_f2` walks only the strict upper triangle (`j = i+1`), mirrors to `(j,i)`, and **leaves the diagonal at its 0 default** (comment line 107: "f2(i,i) = 0"). The GPU `compute_f2` computes **every** `(i,j)` including `i==j`: `launch_assemble_f2` (`f2_block_kernel.cu` lines ~277-285) launches `grid_for(P) × grid_for(P)`, and `assemble_f2_kernel` (lines ~138-160) guards only `if (i >= P || j >= P) return;` — so the diagonal thread runs, and `cuda_backend.cu` line 89-96 copies the whole `[P×P]` `dF2` (the full `pp` block) back. Working the diagonal numerator through `assemble_f2_numerator(sumsq_i, sumsq_j, cross, hsum_i, hsum_j)` with `sumsq_i = R(i,i) = Σp_i²`, `sumsq_j = R(i,i) = Σp_i²` (because `j==i`), `cross = G(i,i) = Σp_i²`, `hsum_i = hsum_j = Σhc_i` gives `Σp_i² + Σp_i² − 2·Σp_i² − Σhc_i − Σhc_i = −2·Σhc_i`, divided by `Vpair(i,i)=count_i` ⇒ **`−2·mean_het_i`, generally nonzero**. (Re-derived this pass against the actual `assemble_f2_numerator` body in `f2_estimator.hpp` line 100 and the `R`/`G`/`Vpair` index map in `assemble_f2_kernel` — the arithmetic holds.) So `CpuBackend::compute_f2` and `CudaBackend::compute_f2` produce **different diagonals** on identical input.
Why it matters: §13 makes the CPU↔GPU agreement the central trust gate; §2 "Testability" says the two "must agree at the seams." The diagonal convention is **pinned to `−2·mean_het`, not 0**, in three independent places: (a) `fstats.hpp` lines 36-39 ("The DIAGONAL carries the full (i,i) computation (= -2·mean within-pop het correction, generally nonzero — **the M0 compute_f2 convention**), NOT a forced 0"); (b) this file's own sibling `compute_f2_blocks` loops `for (int j = i; …)` (line 245), including the diagonal, with an explicit comment (lines 232-238) that it MATCHES the M0 GPU path; (c) the M4 test's single-block==M0 check (`test_f2_blocks_equivalence.cu` lines 249-253) asserts that path bit-for-bit. `CpuBackend::compute_f2` is the lone hold-out using the old "diagonal = 0" convention — directly contradicting its own documented convention and its sibling.
Why it is currently *uncaught*: the M0 test `test_f2_equivalence.cu` does **not** call `CpuBackend::compute_f2` (see 10.1) — it uses an inline oracle `cpu_reference_f2` (lines 468-505) that loops over **all** `(i,j)` including the diagonal (line 479-480 `for j / for i`, no `j=i+1`), so its diagonal IS `−2·mean_het`, matching the GPU; and `rel_error_stats` (line 525) skips `i==j`. The production `CpuBackend::compute_f2` diagonal is therefore diffed against nothing.
Scope note (verified, prevents over-claiming): this is M0-only. Both M4 paths agree — the CPU `compute_f2_blocks` loops `j=i` and the GPU M4 assemble (`f2_blocks_kernel.cu` line 137, `if (i >= P || j >= P || k >= n_in_group) return;`) computes the full diagonal too. The bug is isolated to `compute_f2`.
Fix: change line 123 to `for (int j = i; j < P; ++j)` (the mirror self-write on the diagonal is harmless — see 1.2), matching `compute_f2_blocks` and the documented `−2·mean_het` convention; update the header comment at lines 106-107 and 118-121. Land alongside 10.1 (wire the production method into the M0 test with a full-matrix, diagonal-inclusive assertion) so the convention is gated.
Severity: high. Effort: S. Before-M4.5: yes.

**1.2 [LOW, before-M4.5: no] Diagonal `(i,i)` is written twice in `compute_f2_blocks` (the mirror is redundant on the diagonal). CONFIRMED — harmless.**
Location: `compute_f2_blocks`, lines 264-271. With `j` starting at `i` (line 245), when `i==j` the code computes `ij == ji` (lines 264-267 give the same flat index) and writes `out.f2[ij]`/`out.f2[ji]` and `out.vpair[ij]`/`out.vpair[ji]` to the same slot twice with the same value. Dead work, not a bug. After 1.1 lands, `compute_f2` would have the same shape and the same harmless self-write.
Fix: optional `if (i != j)` guard on the two mirror writes, or a one-line comment that the diagonal self-write is intentional. Severity: low. Effort: S. Before-M4.5: no.

**1.3 [INFO / verified-correct] `het_correction` is called with `valid=true` unconditionally — correct here. CONFIRMED.**
Lines 139-140, 253-254. Both populations are established valid by the `if (!vi || !vj) continue;` guard (lines 129/250) before the call, so `true` is right. It matches the GEMM feeder result: `f2_feeder_kernel` (`f2_block_kernel.cu` line ~111) passes the real `valid`, producing `hc=0` for invalid entries, but those columns are masked out of the `R = S·Vᵀ` GEMM by `V=0`, so only jointly-valid SNPs contribute — identical to the CPU's `continue`. No divergence. Not a defect.

### (2) Edge cases & failure modes

**2.1 [HIGH, before-M4.5: yes] No validation of the `block_id` contract in `compute_f2_blocks`; a malformed partition is silent out-of-bounds UB or silent SNP-dropping, not a fast failure. CONFIRMED, with three distinct failure sub-modes verified, and reachability re-confirmed.**
Location: the block-range scan, lines 214-226, and the per-block loop, lines 240-243.
The code reads `const int b = block_id[s];` (line 217) and immediately indexes `begin[b]`, `end[b]`, `out.block_sizes[b]` (lines 220-222) with **no check** that `0 ≤ b < n_block`. The interface contract (`backend.hpp` lines 146-160) says `block_id[s] ∈ 0..n_block-1`, non-decreasing — and `assign_blocks` (verified `block_partition_rule.hpp` lines 79-81) *does* guarantee dense + non-decreasing — but this is the *reference oracle*, the obviously-correct anchor (§2 "Correctness before speed", §13), and the method signature takes a **raw `const int*` + raw `int n_block`**, so any caller can supply an off-contract pointer. Reachability is not hypothetical: `test_f2_blocks_equivalence.cu` line 251 **hand-builds** a partition (`BlockPartition one; one.block_id.assign(M, 0); one.n_block = 1;`) rather than going through `assign_blocks`; a future caller's off-by-one in `n_block` or a hand-built partition is exactly the off-contract input this method does not defend against. §2 "Fail-fast" (line 67): invalid input must "surface immediately with file/line context, not as silent corruption." Verified failure sub-modes:
  - *OOB write* — `n_block` too small for the max id, or a negative `block_id[s]` from an upstream sign bug ⇒ `begin[b]`/`end[b]`/`out.block_sizes[b]` write at an out-of-range (or negative) index. Silent heap corruption.
  - *Silent SNP-dropping* (non-monotone `block_id`) — verified by hand on `[0,1,0]`: the inner `while (block_id[e]==b)` ends each run early; a later re-occurrence of `b` **overwrites** `begin[b]/end[b]` with the *last* run, so the earlier SNPs of that block silently vanish from the f2 sum and `block_sizes[b]` reflects only the last run. No error; a wrong-but-plausible number that can pass a loose tolerance gate.
  - *Degenerate (non-dense) sub-case* — `n_block=3` but no SNP has id 1: `begin[1]=end[1]=0` stay zero, block 1's slab is all-zero (`finalize_f2(0,0)=0`), `block_sizes[1]=0`. Not a crash, but a silent shape lie about the partition. (`assign_blocks` cannot produce this, but a hand-built partition can.) Folds into the same validation.
Why it matters: this feeds the M4 deliverable; a wrong-but-quiet result is worse than a crash. The GPU peer has the *identical* gap (`cuda_backend.cu` lines 136-147), so validating in one shared place fixes both.
Fix: validate `0 ≤ b < n_block` and the non-decreasing property before indexing; ideally fold the scan into the sanctioned `core/domain/block_partition_rule.hpp` `block_ranges(...)` helper (6.2 / backlog line 81) and validate the contract there **once**, for both backends. Since `compute_f2_blocks` returns `F2BlockTensor` (no `Status`), an `assert` + documented precondition is the minimum (a debug-build check is consistent with §7's "forced sync in debug").
Severity: high. Effort: S (assert) / M (extract+validate helper). Before-M4.5: yes.

**2.2 [LOW, before-M4.5: no] `int b = block_id[s]` truncation — contractually bounded, documented non-issue. CONFIRMED N/A as a risk.**
Line 217. `block_id` is `const int*` (interface type, `backend.hpp` line 167); block counts are O(1e3) on a whole genome (`block_partition_rule.hpp` line 84: "block counts are O(1e3)"). Contractually bounded — the `int` block axis is a deliberate, documented choice (`fstats.hpp` lines 63-64), not an oversight. No fix.

**2.3 [INFO / verified-correct] Empty/degenerate inputs are handled correctly in all four methods. CONFIRMED against the C++ standard (authoritative source).**
`compute_f2` (line 111) and `decode_af` (line 300) early-return a correctly-shaped zero-filled result on `P<=0 || M<=0`; `compute_f2_blocks` guards `P<=0 || M<=0 || n_block<=0` (line 207) *after* sizing `block_sizes`/`f2`/`vpair` with a `n_block < 0 ? 0 : n_block` clamp (lines 202-204), so a negative `n_block` cannot produce a negative allocation. `pairwise_sum(terms.data(), 0)` is safe: the `if (n == 0) return 0.0L;` (line 73) fires before any dereference, and the C++ standard guarantees the range `[data(), data()+size())` "is always a valid range, even if the container is empty," while "if the container is empty, `data()` is not dereferenceable" (verified against `std::vector::data` semantics — cppreference, confirmed via web lookup this pass: the empty-vector `data()` pointer is non-dereferenceable but `[data(), data()+0)` is a valid empty range). `compute_f2`/`compute_f2_blocks` additionally filter `M<=0` upstream (lines 111/207), so `terms.data()` is never even passed `n==0` in the M-pair-loop path. Good defensive coding; not a defect.

**2.4 [MED, before-M4.5: no] `decode_af` does not validate `tile.pop_offsets`/`tile.packed` non-null, `pop_offsets` monotonic, or `byte_in_rec < bytes_per_record`. CONFIRMED; strengthened with sub-point 2.4b.**
Location: `decode_af`, lines 305-322. It dereferences `tile.pop_offsets[i]`, `tile.pop_offsets[i+1]`, and indexes `tile.packed[g * tile.bytes_per_record + byte_in_rec]` with no check that the pointers are non-null, that `seg_end >= seg_begin`, or that `byte_in_rec < bytes_per_record`. The `byte_in_rec = s/4u` with `s` up to `M-1`; the `DecodeTileView` contract (`backend.hpp` lines 63-71) implies `bytes_per_record ≥ ceil(M/4)`, but it is unchecked — a contract violation (`M > 4·bytes_per_record`) is an OOB **read** of `packed`, not a fast failure. A `seg_end < seg_begin` (off-contract offsets) makes the inner loop not execute (`AN==0` ⇒ invalid entry), masking the upstream bug silently. The GPU `decode_af_kernel` (`decode_af_kernel.cu` lines 69-77) has the *identical* unguarded structure, so the parity holds but neither side fails fast. The same fail-fast argument as 2.1 applies (the oracle must not silently corrupt/mis-read on off-contract input), though decode is integer-exact and an OOB read is more likely to crash than to produce a quiet wrong number.

**2.4b [LOW, before-M4.5: no] `tile.n_individuals` is a documented field that the oracle never reads — the iteration is governed entirely by `pop_offsets`, and `n_individuals` is never cross-checked against `pop_offsets[P]`. NEW.**
Verified this pass: `grep n_individuals` in `cpu_backend.cpp` returns no use; the GPU `decode_af_kernel` likewise never uses it. The `DecodeTileView` documents `n_individuals` as "gathered individuals across all pops" (`backend.hpp` line 72), but the decode loop bounds come from `pop_offsets[i]`/`pop_offsets[i+1]` only. So `n_individuals` is a struct field the consumer ignores, *and* the obvious sanity check it enables (`pop_offsets[n_pop] == n_individuals`) is the cheapest possible contract validation for 2.4 — yet it is absent. Either the field is dead weight (remove it from the view, or document it as informational-only) or it should be the basis of the 2.4 fail-fast check.
Fix: in `decode_af`, debug-assert `packed`/`pop_offsets` non-null, `pop_offsets` non-decreasing, `pop_offsets[n_pop] == n_individuals`, and `(M-1)/4 < bytes_per_record`, at the top. Severity: med (potential OOB read on the contract; plus a dead-field smell). Effort: S. Before-M4.5: no.

**2.5 [INFO / verified-correct] No integer overflow in the decode accumulators or the flat-index products. CONFIRMED.**
`ac` accumulates `code ∈ {0,1,2}` over a population segment; `ac ≤ 2·n_individuals`. `n_individuals` is O(1e4) (AADR), so `ac` is O(1e4) — nowhere near `LONG_MAX` even genome-wide. `count`/`an` likewise bounded by M ≤ ~584k. The flat-index products are formed in `std::size_t`: `compute_f2`/`compute_f2_blocks` use `(size_t)P·(size_t)P` and `slab·(size_t)n_block` (lines 108-109, 203-204); `decode_af` uses `(size_t)P·(size_t)M` (lines 294-295). With P up to ~3,331 (PRO 6000) and `n_block` ~1e3, `P²·n_block ≈ 1.1e10` exceeds 2^31 but fits 64-bit `size_t` — both operands are `size_t`, so the multiply is 64-bit. No overflow. Listed for completeness so the category is not skipped.

**2.6 [LOW, before-M4.5: no] `decode_af` narrows `tile.n_snp` (`std::size_t`) to `long M` unchecked. NEW — low; defensively safe by accident.**
Line 289: `const long M = static_cast<long>(tile.n_snp);`. `n_snp` is `std::size_t` (unsigned 64-bit on LP64); `M` is `long` (signed 64-bit). For realistic M (~584k) this is value-preserving. An absurd `n_snp > LONG_MAX` would wrap to a negative `M`, but the immediately-following `if (P <= 0 || M <= 0) return out;` (line 300) catches a negative `M` and returns an empty result — so the narrowing is *accidentally* safe rather than checked. Cosmetic; the only real cost is that a future reader cannot tell the safety is intentional.
Fix: none required; optionally `assert(tile.n_snp <= static_cast<std::size_t>(std::numeric_limits<long>::max()))` for documentation, or a one-word comment. Severity: low. Effort: S. Before-M4.5: no.

### (3) Numerical / precision vs §12

**3.1 [INFO / verified-correct, keep] The cancellation-free long-double accumulation is exactly the §12 oracle property and is implemented correctly. CONFIRMED.**
`f2_term` forms `(p_i−p_j)` and squares it directly (`f2_estimator.hpp` lines 78-79: `const double d = p_i - p_j; return d*d - hc_i - hc_j;`), each summand is cast to `long double` (lines 147-148, 255-256), and the cross-SNP sum is pairwise/cascade (`pairwise_sum`, lines 71-81). This sidesteps the `Σp_i²+Σp_j²−2Σp_i p_j` cancellation the GEMM path localizes and keeps in native FP64 (§12; `assemble_f2_numerator` `f2_estimator.hpp` line 100 is the FP64 home; the file header lines 23-32 articulate this precisely and correctly). The 128-element base case matches the spike (line 72) and the inline test oracle (`test_f2_equivalence.cu` line 449). This is the load-bearing good pattern — preserve it.

**3.2 [LOW, before-M4.5: no] The `static_cast<double>(numerator)` narrow-before-divide point is correct but under-documented at the M4 site. CONFIRMED.**
Lines 160-161 and 262-263. The long-double `numerator` is narrowed to `double` *before* the divide in `finalize_f2` (signature `finalize_f2(double numerator, double vpair)`, `f2_estimator.hpp` line 110), i.e. `f2 = double(numerator) / vpair`. This matches the GPU's native-FP64 numerator/divide (§12 — the small numerator/divide step stays native FP64) and is the right call. The M0 site is commented (lines 152-156); the M4 site (lines 259-263) carries no such comment, and a future reader could "optimize" to `double(numerator / vpair)`, silently changing the oracle's last bits and risking the tight-tier diff.
Fix: comment the M4 narrowing site; consider a shared `finalize_f2_ld(long double num, double vpair)` wrapper in `f2_estimator.hpp` that both sites call so they cannot drift. Severity: low. Effort: S. Before-M4.5: no.

**3.3 [INFO / verified-correct] `V.element(...) != 0.0` validity test matches the GPU feeder bit-for-bit. CONFIRMED.**
Lines 127-128, 248-249 use `!= 0.0`; the GPU feeder uses the same (`f2_block_kernel.cu` line ~103 `V_raw[idx] != 0.0`). The producer guarantees `V != 0 ⇔ N > 0` (`views.hpp` line 33), so the validity test and the het denominator are consistent. Not a defect.

**3.4 [MED, before-M4.5: no] The oracle's "better-than-native" accuracy silently depends on `long double` being wider than `double`, which is implementation-defined; no `static_assert` guards it. CONFIRMED via authoritative source.**
The file header (lines 5, 36-37) and PRECISION POLICY block claim the oracle computes "better than native: long-double" — the §12 / `TODO.md` *native-FP64 oracle* the whole project's parity rests on. Verified against the standard this pass (web lookup, MSVC docs + Wikipedia "Long double"): "The language standard only requires long double to be at least as precise as double" — i.e. *equal* precision is conforming; and concretely "Microsoft Visual C++ for x86 … makes long double a synonym for double" (64-bit), whereas x86-64 gcc/clang give 80-bit extended. So on the box (Linux/gcc/clang) the property holds, but on an MSVC-x86 port the "better than native" guarantee silently collapses to native, and the tight-tier diff against the GPU could narrow or shift. No `static_assert` for this exists anywhere in `src` (grepped this pass — only `long double` *uses* appear, no width assertion). The box is fine today, but the oracle's defining accuracy property is silently platform-dependent and undocumented — and an oracle that silently loses its defining property on a port is a §12 hazard, not merely cosmetic.
Fix: `static_assert(std::numeric_limits<long double>::digits > std::numeric_limits<double>::digits, "the CPU oracle assumes extended-precision long double; on a 64-bit-long-double ABI it loses its 'better-than-native' property (architecture.md §12)");` near the top of the file (add `<limits>`). Severity: med (it guards the load-bearing oracle property). Effort: S. Before-M4.5: no.

**3.5 [INFO / verified-correct] Determinism of the long-double accumulation is order-fixed. CONFIRMED.**
`pairwise_sum`'s tree shape is fixed by `n` and the 128 base case (lines 71-81), and the per-block/per-pair iteration order is fixed (`b`, then `i`, then `j`, then `s`), so the oracle is run-to-run deterministic on a given host — the host-side analogue of the §12 reproducibility argument. (The 3.4 caveat is about *magnitude* of the wide type, not order.) Not a defect.

### (5) Magic numbers & hardcoded values vs §4 / ROADMAP §4

**5.1 [MED, before-M4.5: yes] `kPairwiseBaseCase = 128` is a local tuning literal duplicated verbatim in the test, not in the single config home. CONFIRMED.**

## Considered & rejected

- **"`pairwise_sum` is recursive — stack-overflow risk on large M."** Rejected. Depth is `O(log2(M/128))`; for M=584k that is ~13 frames. No risk. The recursion is also the clearest expression of the cascade; an explicit-stack rewrite would hurt readability for no benefit.
- **"Passing `data()` of a possibly-empty `terms` to `pairwise_sum` is UB."** Rejected. `pairwise_sum` checks `n==0` first (line 73) and never dereferences in that case; per the C++ standard `[data(), data()+0)` is a valid empty range and `data()` need not be dereferenceable when empty (verified against `std::vector::data` semantics via web lookup this pass — see 2.3). And `M<=0` is filtered upstream (lines 111/207). Safe.
- **"`terms[count]` could overflow the buffer."** Rejected. `count` is bounded by the jointly-valid SNPs in the range ≤ range length ≤ M; `terms` is sized to M. No overflow (also 2.5).
- **"`het_correction(p_i, …, true)` should pass the real validity, like the GPU feeder."** Rejected — see 1.3; both pops are provably valid at the call site (the `continue` guard), and the GEMM result after the `V`-mask is identical. Passing a runtime bool would be dead code.
- **"`decode_af`'s `ac`/`an` (`long`) could overflow on whole-genome data."** Rejected — see 2.5; `ac ≤ 2·n_individuals` is O(1e4), far from `LONG_MAX`.
- **"The `total = slab * n_block` flat-element count could overflow."** Rejected — both operands are `std::size_t`; with P up to ~3,331 and `n_block` ~1e3 the product (~1.1e10) exceeds 2^31 but is well within 64-bit `size_t`. The `n_block < 0 ? 0 : n_block` clamp prevents a negative-to-size_t wrap (2.5).
- **"The `(size_t)i + (size_t)P*j` index casts are cast churn that should be removed."** Rejected. ROADMAP §4's magic-number table explicitly says to **keep** the `size_t` index cast (free below P≈32k, mandatory above). These are correct width promotions, not noise. (Distinct from the `count` cast in 7.2, which *is* churn.)
- **"The backend should take `RunConfig`/`Resources`, not a bare `Precision`."** Rejected for this layer — the `ComputeBackend` interface (`backend.hpp` lines 135-169) deliberately takes `Precision` (the only knob that could matter), and the oracle ignores it by design (§12). Injecting `Resources` into a CPU oracle that owns nothing would be over-engineering and would break the shared interface.
- **"Mirror writes are a data race / need synchronization."** Rejected — single-threaded host code; no concurrency. (Even if parallelized over pairs, the upper-triangle structure makes writes disjoint — but it is not, and should not be, per §12.)
- **"`(void)precision` should be `[[maybe_unused]]`."** Rejected as a finding — purely stylistic; the project's chosen idiom is `(void)` with an explanatory comment, mirrored in the GPU backend (7.3); `-Werror` makes *some* suppression necessary (verified `SteppeWarnings.cmake`).
- **"1.1 also affects the M4 path."** Rejected — verified the M4 paths agree: CPU `compute_f2_blocks` loops `j=i` (line 245) and the GPU M4 assemble computes the full diagonal (`f2_blocks_kernel.cu` line 137). The divergence is M0-`compute_f2`-only; scoping 1.1 to M4 would be a false claim.
- **"`compute_f2_blocks` should separately validate that block columns are actually contiguous."** Folded into 2.1 (the non-monotone case is one of its verified sub-modes); not listed separately.
- **"`STEPPE_HD` is doubly-defined across the two shared headers this file includes — a macro-clash risk in this TU."** Rejected for *this* unit. `cpu_backend.cpp` includes both `f2_estimator.hpp` and `decode_af.hpp`, each of which `#define STEPPE_HD` (to nothing, since `__CUDACC__` is undefined on the host compiler) with no `#undef`. Both expand to the *same* empty token here, so there is no clash in this TU and no defect in `cpu_backend.cpp`. The double-definition-without-a-shared-home is a real `TODO.md` item ("`STEPPE_HD` macro doubly-defined … Fix: one `core/internal/host_device.hpp`"), but it lives in the two headers, not in this file — out of scope for this unit's review.
- **"The score should drop to ~7 given a HIGH correctness divergence."** Rejected. 1.1 is latent (never consumed downstream — f3/f4 read off-diagonal f2 only, per `fstats.hpp` line 38), the numerics that *are* consumed are correct and tight-tier, and the layering/precision discipline is exemplary. 8.5 is the right level: real concentrated debt, no shipped wrong number.
- **"`n_individuals` being unused is a correctness bug."** Rejected as a correctness finding — the decode is correct without it (iteration is `pop_offsets`-driven and the GPU does the same). Re-scoped to a readability/contract smell + a free validation opportunity (2.4b), not a bug.

## Good patterns to keep

- **Shared `__host__ __device__` per-element primitives** (`het_correction`, `f2_term`, `finalize_f2`; `genotype_code`/`genotype_valid`/`finalize_af`) consumed identically by oracle and GPU — the structural reason the two cannot diverge on the formula (§13, verified at the call sites on both sides this pass). The single best decision in the file.
- **Cancellation-free per-SNP difference + long-double pairwise accumulation** (3.1) — the load-bearing oracle property, correctly and clearly implemented; the narrow-before-divide step matches the GPU's native-FP64 divide (§12).
- **CUDA-free purity** — no CUDA header, plain host types across the seam, factory returns the base interface; this is what makes the GPU-free import and the DI seam work (§2/§4/§8).
- **Documentation discipline** — every method and constant cites its arch/ROADMAP section and explains *why*; the diagonal-convention comment (lines 232-238) is exemplary and is exactly the kind of note whose *absence* on `compute_f2` produced 1.1.
- **Defensive empty/degenerate handling** — early returns with correctly-shaped zero results, the `n==0` guard in `pairwise_sum`, the `n_block < 0 ? 0 : n_block` allocation clamp (2.3, verified against the standard).
- **O(M) scratch, never the `[P×P×M]` intermediate** (§11.1) — the streaming-friendly space discipline holds even in the reference.
