## Role & layering

`error.hpp` lives under `include/steppe/` alongside `config.hpp` and `fstats.hpp`. It is wired into the `steppe_api` `INTERFACE` target (`include/CMakeLists.txt`, `target_compile_features … cxx_std_20`), propagated by include-dir to every downstream layer (`src/core/CMakeLists.txt:33` links `steppe::api` with the comment `# config.hpp, error.hpp`). Layering-wise it is the *lowest* public artifact: **zero `#include`s**, depends on nothing, cannot create an upward edge, cannot drag in the toolkit. That property is correct, clean, and is its single biggest strength.

The file defines one C++ `enum class Status` with five enumerators: `Ok`, `DeviceOom`, `RankDeficient`, `NonSpdCovariance`, `InvalidConfig`. Its own banner calls it "the trimmed §10 set" and says the full ABI surface is "a richer C enum `steppe_status_t` plus an internal `Error` view" that this header does **not** carry.

**Facts established by direct verification (these drive the review):**

1. **No source file in the repository consumes `steppe::Status` or any of its enumerators.** Re-run greps for `steppe::Status`, `Status::`, `#include "steppe/error.hpp"` return only this file's own definition plus *comment-only* mentions: two CMake comments (`include/CMakeLists.txt:6`, `src/core/CMakeLists.txt:33`) and `src/device/cuda/check.cuh:15,37` referencing the **C-ABI** name `steppe_status_t`. I read the actual M0/M4 producers: `ComputeBackend::compute_f2`/`compute_f2_blocks`/`decode_af` (`src/device/backend.hpp:135,164,186`) return value structs (`F2Result`, `F2BlockTensor`, `DecodeResult`); the `core` orchestrators `compute_f2_block`/`compute_f2_blocks` (`src/core/fstats/f2_from_blocks.hpp:26,38`) likewise return `F2Result`/`F2BlockTensor`. **None returns `Status` or takes a `Status&` out-param.** The CUDA fault path throws `CudaError`/`CublasError` (`check.cuh`). So `error.hpp` is a *declared-but-unwired* contract on this branch — confirmed, not inferred.

2. **The §17 scaffold manifest (architecture.md:869) names *this file* as the home of the C enum `steppe_status_t`** — "`error.hpp` (`steppe_status_t` taxonomy, §10)" — yet the file defines a C++-only `enum class Status`. `include/CMakeLists.txt:6` likewise calls the surface "C-ABI-shaped." A C translation unit cannot name `steppe::Status` or its scoped enumerators, so the file does not satisfy the manifest's stated intent.

3. **`qpadm.hpp`, `version.hpp`, `io/genotype.hpp`, `steppe.hpp`, `internal/expected.hpp`, and any `Error` type do not exist on this branch.** `ls include/steppe/` shows exactly `config.hpp`, `error.hpp`, `fstats.hpp`. Grep for `struct Error`/`class Error` (excluding `CudaError`/`CublasError`) returns nothing; grep for `expected.hpp`/`STEPPE_TRY`/`std::expected` returns only `docs/architecture.md`. So the banner's forward-references describe an end-state, not the branch.

These facts drive the findings: real divergence from the canonical §10/§16/§17 ABI contract, a taxonomy missing 3 of §10's 8 values, a doc-comment that overstates current reality, a self-contradictory "three… (two listed)" parenthetical, and — new this pass — a per-value comment that misattributes an §10 status category. The code that *is* present is correct; the problems are spec-fidelity and accuracy-of-documentation, not runtime behavior.

## Findings

### (1) Correctness & bugs

**1.1 — Banner says "three DOMAIN-OUTCOME values" then parenthesizes two. [HIGH, S, before-M4.5: yes] — CONFIRMED.**
Location: lines 11–14 (block comment).
The comment reads *"The three DOMAIN-OUTCOME values (RankDeficient, NonSpdCovariance) are *expected* results…"* — the count says **three**, the list names **two**. Verified against §10 (architecture.md:635–637): the three domain outcomes are `STEPPE_ERR_NON_SPD_COVARIANCE`, `STEPPE_ERR_RANK_DEFICIENT`, **and** `STEPPE_ERR_CHISQ_UNDEFINED` ("dof ≤ 0 or χ² not computable"). The prose count (3) matches §10; both the parenthetical and the enum body omit `ChisqUndefined`. The per-value `///` comments are internally consistent — exactly two values are marked "Domain outcome (recoverable)" — so the block comment is the inconsistent one. Why it matters: §10 is the canonical taxonomy and §18 DoD (architecture.md:897) explicitly enumerates "non-SPD, rank-deficient, **χ²-undefined**" as the domain outcomes that must be "returned as values and covered by a test." A reader trusting "three (RankDeficient, NonSpdCovariance)" is doubly misled. Fix: reconcile with 1.2 — add `ChisqUndefined` and list all three; or, if deferring the value, change the count to "two."

**1.2 — `ChisqUndefined` / `STEPPE_ERR_CHISQ_UNDEFINED` is absent from the taxonomy. [HIGH, S, before-M4.5: decide now] — CONFIRMED.**
Location: enum body, lines 21–41.
§10 (architecture.md:637), §13 (the domain-outcome test, architecture.md:761) and §18 DoD (architecture.md:897) all list χ²-undefined as a first-class recoverable domain outcome. It is not in the enum. Adversarial check: is omission defensible as "the trimmed M0 set"? The banner does scope the file to "the small strongly-typed C++ subset the early layers need," and the f2 precompute slice (M0–M4) never computes a χ² (qpAdm/χ² is P2). So deferring the *value* is arguable — **but only if 1.1 is fixed and the deferral is recorded.** Recommendation: add it now. Cost is one enumerator; §10 is canonical; the public enum is meant to be additive-only under §16 SemVer (additive C API ⇒ MINOR, architecture.md:857), so adding it later is a deliberate MINOR-bump churn that a one-line addition today avoids. Fix: add `ChisqUndefined` in the domain-outcome cluster with `/// Unused until S7 (qpAdm); kept for §10 completeness.`

**1.3 — Missing taxonomy value `IoFormat` / `STEPPE_ERR_IO_FORMAT`. [MED→conditional on 9.1, S, before-M4.5: no] — CONFIRMED, reframed.**
Location: enum body.
§10 (architecture.md:632) lists `STEPPE_ERR_IO_FORMAT` ("malformed/unsupported `.bed`/`.geno`/`.snp`/`.ind`; bad magic bytes"). The `io` leaf is built (`src/io/{snp,geno,ind}_reader.hpp` etc.) and parses magic bytes, so this is a *live* failure mode today. Adversarial check: `io` is an isolated leaf that produces plain-data structs and `app` (P3) wires it; `io` may report format errors via exception/its own type and never touch `Status`. Plausible — but §10 assigns `IoFormat` a `steppe_status_t` category and §18 DoD requires "new failure modes map to a `steppe_status_t` category." Resolution: **subsumed by the 9.1 policy decision.** Option A (file becomes the C `steppe_status_t`) ⇒ it must carry all 8 §10 values including `IoFormat`. Option B (stays the C++ subset) ⇒ the header must state `IoFormat` is a deliberate exclusion and where it *is* surfaced. The unacceptable state is silent omission.

**1.4 — Missing fault value `CudaRuntime` / `STEPPE_ERR_CUDA_RUNTIME`, and asymmetry with `DeviceOom`. [MED→conditional on 9.1, S, before-M4.5: no] — CONFIRMED.**
Location: enum body.
§10 (architecture.md:634) lists `STEPPE_ERR_CUDA_RUNTIME` (sticky/async fault, bad launch). `check.cuh:37–39` explicitly says the public API maps `CudaError` to `STEPPE_ERR_CUDA_RUNTIME` and distinguishes `cudaErrorMemoryAllocation → STEPPE_ERR_DEVICE_OOM` via `status()`. So both `DeviceOom` *and* `CudaRuntime` come out of the *same* `CudaError` translation — yet only `DeviceOom` is in the enum. That asymmetry is the real defect: the header picked one of the two `CudaError`-derived codes without saying why. Adversarial check: per §10/§16 CUDA faults are fail-fast and never cross as a recoverable value (architecture.md:639), so a *recoverable* C++ `Status` arguably needn't carry `CudaRuntime` (only the C ABI's `steppe_status_t` must, since the C shim returns *all* codes). That is a legitimate reading — but then `DeviceOom`'s presence needs a one-line justification (it is the one resource code that is "maybe recoverable with a smaller chunk/budget," §10 line 633) and `CudaRuntime`'s absence must be stated, not left implicit. Fix: add `CudaRuntime` for symmetry, OR document that faults other than OOM are not surfaced as a recoverable `Status` and why OOM is the exception. Folds into 9.1.

**1.5 — No explicit fixed underlying type; relies on the implicit default. [MED, S, before-M4.5: yes] — CONFIRMED; first-pass rejection-reasoning corrected (see Considered & rejected).**
Location: `enum class Status {` (line 21).
This is *correct but underspecified*, not UB. Verified against the C++ standard: **[dcl.enum]/5 — "For a scoped enumeration type, the underlying type is int if it is not explicitly specified."** (eel.is/c++draft/dcl.enum, fetched 2026-06). So the layout is well-defined and stable today; any "enum-class size is implementation-defined / UB" claim is wrong for the no-specifier case. The real point is ABI/cross-language clarity: §16 (architecture.md:853,857) makes the installed boundary a C ABI and the SemVer promise, and the banner (line 7) ties `Status` to the C enum `steppe_status_t`. A C `enum`'s underlying type *is* implementation-defined, so writing the width explicitly is where it earns its keep — it documents the ABI width and lets a `static_assert` confirm the C++ mirror and the C enum agree. House-style data point: **neither sibling header pins an underlying type** — `config.hpp`'s `Precision::Kind` (config.hpp:123) is a bare `enum class`. So pinning `: int` here is *stricter than the current siblings*; it is a spec-driven improvement, not a consistency fix. Fix: `enum class Status : int { … };` (or `std::int32_t` if the C ABI must guarantee a fixed 32-bit width — see Considered & rejected) with a one-line comment that the width matches the C `steppe_status_t`.

**1.6 — Header claims to be `steppe_status_t`'s C++ subset but no single-source / no `static_assert` lockstep. [MED, M, before-M4.5: yes] — CONFIRMED.**
Location: lines 6–9 vs §10/§16/§2 (DRY).
The banner says the header "carries the small strongly-typed C++ subset" of the C enum `steppe_status_t`. But the two enums (if/when both exist) have different *spelling* (`STEPPE_OK`/`STEPPE_ERR_*` vs `Ok`/`DeviceOom`/…) and there is no mapping table, no `static_assert`, and no generated single source. Per §2 ("every cross-cutting concern implemented exactly once") an error taxonomy living in two parallel hand-maintained forms is a drift-prone DRY violation — precisely what §16 (architecture.md:851) warns against. Adversarial check: §16 (architecture.md:855) explicitly permits an *optional same-toolchain C++ convenience layer* wrapping the C ABI, so a C++ mirror is legitimate — but the legitimate pattern keeps them in lockstep with `static_assert` per value. As written there is no such guard. Fix: when `steppe_status_t` lands, pin values and add `static_assert(static_cast<int>(Status::DeviceOom) == STEPPE_ERR_DEVICE_OOM, …)` one per value in the convenience layer. For M0, at minimum stop describing `Status` as "the subset of `steppe_status_t`" while that enum does not exist — say "the M0 strongly-typed status set; the C ABI enum (§16/§17) is added at the public-boundary milestone and kept in lockstep."

**1.6b — NEW (sharpening of 1.6): the C++ enum ORDER already diverges from §10's numeric order, so a future positional/`static_cast` mapping cannot round-trip. [MED, S, before-M4.5: yes] — NEW.**
Location: enum body, lines 21–41, vs §10 table (architecture.md:630–637).
§10's table fixes an implicit numeric order: `OK(0), INVALID_CONFIG, IO_FORMAT, DEVICE_OOM, CUDA_RUNTIME, NON_SPD, RANK_DEFICIENT, CHISQ_UNDEFINED`. The header's order is `Ok, DeviceOom, RankDeficient, NonSpdCovariance, InvalidConfig` — i.e. with the implicit values, `DeviceOom == 1`, `RankDeficient == 2`, `NonSpdCovariance == 3`, `InvalidConfig == 4`. These do **not** match §10's positions (`DEVICE_OOM == 3`, `NON_SPD == 5`, `RANK_DEFICIENT == 6`, `INVALID_CONFIG == 1`). The first pass said the lockstep "keeps matching value *order* so a `static_cast` round-trips" — but it did not flag that **the current order already breaks that property**. Why it matters: the cheapest, hardest-to-drift mapping is `static_cast<steppe_status_t>(status)` with identical numeric values; the present ordering forecloses it, forcing a hand-maintained `switch` (more drift surface) unless the values are explicitly pinned to §10's numbers. This is the concrete, today-visible reason 1.5/1.6 must pin *values*, not just a width. Fix: when the C enum lands, pin the C++ mirror's values to §10's numeric assignment (`Ok = 0, InvalidConfig = STEPPE_ERR_INVALID_CONFIG, DeviceOom = STEPPE_ERR_DEVICE_OOM, …`), not declaration order, and `static_assert` each.

**1.7 — NEW (correctness of the doc-contract): `InvalidConfig`'s per-value comment misattributes "over-budget VRAM," contradicting §10/§11.2 AND the file's own `DeviceOom` comment. [MED, S, before-M4.5: yes] — NEW.**
Location: `InvalidConfig` comment, lines 38–40, vs `DeviceOom` comment, lines 25–27.
The `InvalidConfig` comment reads: *"Configuration failed `ConfigBuilder::build()` validation (bad arch list, conflicting flags, **over-budget VRAM**, unhonorable precision). Fail-fast."* But §10 (architecture.md:633) categorizes "device allocation or **budget check failed**" as `STEPPE_ERR_DEVICE_OOM` (resource), and §11.2 is explicit that the **`build()`-time** over-budget rejection returns `DEVICE_OOM`, not `INVALID_CONFIG`: architecture.md:695 — *"`ConfigBuilder::build()` … rejects over-budget configs up front (`STEPPE_ERR_DEVICE_OOM`)"* — and architecture.md:679 — *"`build()` returns `STEPPE_ERR_DEVICE_OOM` with the shortfall."* So an over-budget config at `build()` is a `DeviceOom`, **not** an `InvalidConfig`. The header's own `DeviceOom` comment even agrees — line 25–26 says `DeviceOom` = "VRAM-budget check failed … architecture.md §11.2." So the file lists over-budget VRAM under **both** `InvalidConfig` and `DeviceOom`, and §10/§11.2 resolve it to `DeviceOom`. This is a genuine spec-contradiction in the doc-comment (the kind §18 "Errors classified" guards against: every failure mode maps to *one* `steppe_status_t` category). Adversarial check: is there a reading where a *structurally invalid* budget request (e.g. `T_max < 1` from a pathological sample count) is `InvalidConfig`? No — architecture.md:679 explicitly assigns even that pathological single-row case `STEPPE_ERR_DEVICE_OOM`. The §9 validation list (architecture.md:620) folds the budget check in among `build()` rejections but §11.2/§10 pin the *category* to OOM. Fix: delete "over-budget VRAM" from the `InvalidConfig` comment; it already lives (correctly) on `DeviceOom`. Keep `InvalidConfig` = bad arch list, conflicting flags, unhonorable-mandatory precision.

**7.2 — Type-level `[[nodiscard]]` is well-justified and missing. [LOW→MED, S, before-M4.5: no] — CONFIRMED.**
Location: type definition.
A `Status` is exactly the value that must not be silently dropped — dropping a `RankDeficient`/`NonSpdCovariance` silently accepts a garbage fit, defeating §10's "the search records the outcome and moves on" (architecture.md:639). C++17+ allows marking the *enumeration type* `[[nodiscard]]` (`enum class [[nodiscard]] Status { … };`). Verified that this makes **every** function returning `Status` by value warn on a discarded-value expression: cppreference — *"If a function declared nodiscard or a function returning an enumeration or class declared nodiscard by value is called from a discarded-value expression other than a cast to void, the compiler is encouraged to issue a warning"* (cppreference, nodiscard attribute; confirmed via search 2026-06). This is strictly stronger and DRYer than per-function `[[nodiscard]]`. House-style data point: `fstats.hpp:67`'s `size()` and `check.cuh`'s `what()`/`status()` already use `[[nodiscard]]`, and `backend.hpp` marks every `compute_*`/`decode_af` `[[nodiscard]]` — the attribute is idiomatic here. Adversarial check: type-level `[[nodiscard]]` also fires on `Status::Ok`-returning helpers where discarding is benign and can be noisy — but in the warnings-as-errors codebase (ROADMAP §6, architecture.md:131) that pressure is *desirable*: it forces an explicit `(void)` at the rare intentional-discard site. Fix: `enum class [[nodiscard]] Status : int { … };` (combine with 1.5). Bumped LOW→MED because §10's entire premise is that the caller must inspect the value and ROADMAP §6 makes the warning hard-fail.

### (9) Layering / API / ABI vs §4 / §16 / §17

**9.1 — The central ABI tension: a C++ `enum class` where §17 names the C `steppe_status_t`. [HIGH, M, before-M4.5: yes — decide the policy] — CONFIRMED; this is the top finding.**
Location: whole file vs §16 (architecture.md:851–855), §17 step 5 (architecture.md:869: "`error.hpp` (`steppe_status_t` taxonomy, §10)").
§17's scaffold manifest explicitly designates *this file* the home of the **C** enum `steppe_status_t`; §16 makes the installed boundary a C ABI exposing "functions returning `steppe_status_t`." The file instead defines a C++-only `enum class Status`. These are not the same artifact:
 - A `steppe_status_t` usable from C must be a C-compatible `typedef enum {…} steppe_status_t;`, dual-inclusion-safe for C and C++, with `STEPPE_*` enumerators.
 - `enum class Status` is C++-only; C translation units cannot name `steppe::Status` or its scoped enumerators.
So the file does **not** satisfy §17 step 5 / §16, and the divergence is *replicated in the build files*: `include/CMakeLists.txt:6` calls the surface "C-ABI-shaped" while it is C++, and `src/core/CMakeLists.txt:33` links it as `steppe::api`. Adversarial check — defensible as staging? Yes *if* the intent is that the C ABI enum is added later and `Status` is the optional C++ convenience mirror §16 (architecture.md:855) permits (the banner hints at this). ## Considered & rejected

- **"`enum class` size is implementation-defined / UB-adjacent at a C boundary."** Rejected as stated. Verified: [dcl.enum]/5 — *"For a scoped enumeration type, the underlying type is int if it is not explicitly specified"* (eel.is/c++draft/dcl.enum, fetched 2026-06). Size and value set are well-defined, not UB. The real (weaker) point is documentation/cross-language clarity — captured as 1.5 at MED, not a correctness bug.
- **First-pass auditor's claim that `int` is "guaranteed ≥ 32 bits, needs no include" — CORRECTED.** The C++ standard guarantees `int` is **only ≥ 16 bits** (minimum width 16, range −32767..+32767; eel.is/c++draft/basic.fundamental, Table 14, fetched 2026-06). The original rejection of `<cstdint>`/`std::int32_t` rested on a false premise. The *conclusion* (prefer `: int` over `<cstdint>` to preserve the zero-include property) is still defensible for the C++ mirror — 5 enumerators fit any conforming `int`, and ABI agreement is enforced by a `static_assert` (1.6), not by the spelling. **But** if the C `steppe_status_t` (Option A, 9.1) must guarantee a *fixed 32-bit width across toolchains*, that boundary should use a fixed-width base, and the zero-include property is then not the deciding factor. Net: keep `: int` on the C++ mirror with a `static_assert`; pin a fixed width on the C ABI enum if cross-toolchain width is contractual.
- **"Add a `Count`/`Last` sentinel for iteration."** Rejected — for an ABI/taxonomy enum a public `Count` is an anti-pattern (pins itself last forever, pollutes `static_assert` mappings). See 2.2.
- **"Mark every API function `[[nodiscard]]` instead of the type."** Rejected as the primary recommendation — type-level `[[nodiscard]]` (7.2) is strictly stronger and DRYer; verified it fires on every by-value return of the type (cppreference). Per-function would be redundant. (The producers already use per-function `[[nodiscard]]`, so a type-level mark is additive, not conflicting.)
- **"Add an `Unknown`/`Internal` catch-all status."** Rejected — §10's taxonomy is deliberately closed and the fault path is `CudaError`/fail-fast (architecture.md:639); a catch-all weakens the "every failure maps to a category" DoD (architecture.md:897). An unmappable case is a bug to fix, not a status to invent.
- **"Move `Status` into a `steppe::api::` sub-namespace to mirror `core`/`device`."** Rejected — the sibling public headers (`config.hpp`, `fstats.hpp`) put public types directly in `steppe::`; matching them is the right consistency call (9.3).
- **"A capability-degradation `Status` value is missing."** Rejected — capability degrade is parity-neutral and surfaces as a logged tag + capability-tagged `DeviceConfig`/`Resources` field (TODO:125,137), not an error. Adding one would conflate success with failure. See 11.1.
- **"`DeviceOom` overlaps `CudaError`'s `cudaErrorMemoryAllocation` and is redundant."** Rejected — §10 deliberately separates resource-OOM (maybe-recoverable, architecture.md:633) from `CUDA_RUNTIME` faults (not recoverable, architecture.md:634); `check.cuh:37–39` branches on `status()` to pick OOM. The distinction is intentional and correct. (The *asymmetry* with the absent `CudaRuntime` is the real point — see 1.4.)
- **"The file banner should be a Doxygen `/** @file */` block, not `//`."** Rejected — both siblings (`config.hpp`, `fstats.hpp`) use the same `//` banner style; per-symbol Doxygen `///` is what feeds the §16 Doxygen/Sphinx build and is present and (mostly) excellent here. Consistent; not a finding.
- **"`Ok = 0` should be explicit even now."** Considered, kept as 2.1 at LOW: it is *already* 0 by [dcl.enum]/2 and matches `STEPPE_OK == 0`, so there is no bug today; the explicit-pin + `static_assert` is only load-bearing once the C mapping exists, so it is correctly future-tied rather than a present defect.
- **"`InvalidConfig` listing 'conflicting flags' / 'bad arch list' is also wrong."** Rejected — those ARE `INVALID_CONFIG` per §10 (architecture.md:631: "bad arch, conflicting flags") and §9 (architecture.md:620). Only "over-budget VRAM" is misattributed (1.7); the rest of the `InvalidConfig` comment is correct.
- **"`InvalidConfig` should be `ConfigInvalid` / reordered first to match §10."** Rejected as a naming change — naming is fine; the *value* ordering vs §10 is the real issue (1.6b), to be fixed by pinning values, not by reordering the C++ enumerators (which would be a gratuitous source diff with no behavioral gain until the C enum lands).
