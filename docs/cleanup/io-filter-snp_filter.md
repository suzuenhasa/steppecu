## Role & layering

`snp_filter` is the M2 **per-SNP keep-mask producer**. It owns two host-pure functions in `steppe::io::filter`:

1. `derive_per_snp_summary(in)` — reduces the decoded per-pop column-major `[P × M]` Q/V/N into a length-`M` array of SNP-global scalars (`pooled_ref_af`, `pooled_minor_af`, `missing_frac`, `pooled_allele_count`). The pooling convention is the single one pinned in `filter_decision.hpp` (`Σ Q·N / Σ N`).
2. `build_snp_keep_mask(in, snps, cfg, mem)` — applies the unconditional drop-not-flip class drops (multiallelic / strand-ambiguous), then the `cfg`-thresholded MAF/geno predicates, then the flag-gated monomorphic / transversions / autosome filters, then include/exclude membership, into one `std::vector<bool>` of length `M`.

**Layering is correct and is the unit's strongest property.** This is an `io`-leaf TU (architecture §4): pure host C++20, no CUDA, no `core`/`device` dependency. It deliberately consumes *plain `const double*`* Q/V/N rather than `core::MatView` (which lives in `core/internal`), so the `io` leaf does not reach up into `core` — the header comment calls this out explicitly and it is the right call given the dependency-direction rule (§4: "`io` an isolated leaf"). Every threshold comes from `FilterConfig`; every decision predicate is delegated to the single-source `filter_decision.hpp`. The function returns a SNP-global `std::vector<bool>` — one bool per SNP — which is exactly the shape the §1/§5-S2 "SNP-GLOBAL, never per-(pop, SNP)" invariant requires, and which `filter_plan.hpp::FilterPlan::snp_keep` consumes verbatim.

The unit is small and disciplined. Most of what follows is low/medium-severity polish, a couple of genuine robustness/edge-case gaps, and one design observation (parallelizability of the mask loop vs the `std::vector<bool>` return type) that touches the capability-tier story. None of it is a parity bug — the oracle test (`test_filter_oracle.cu` check c) proves integer-exact agreement with an independent scalar oracle across 7 configs.

## Findings

### (1) Correctness & bugs

**1.1 — `pop_individuals` shorter than `P` silently corrupts the missing fraction (and is undetected). [CONFIRMED] [med, S, before-M4.5: yes]**
`derive_per_snp_summary`, lines 25–28:
```cpp
for (int p = 0; p < P && p < static_cast<int>(in.pop_individuals.size()); ++p)
    total_indiv += in.pop_individuals[static_cast<std::size_t>(p)];
```
The `total_indiv` (missing-fraction *denominator*) loop is bounded by `min(P, pop_individuals.size())`, but the per-SNP `nonmissing_indiv` *numerator* loop (lines 39–48) is bounded by `P` alone. If a caller passes `pop_individuals.size() < P` (a contract violation, but an easy one — the field is a separate `std::vector` the caller fills by hand, exactly as the oracle test does at lines 225–229), the denominator counts only the first `pop_individuals.size()` pops while the numerator counts all `P`. Result: `missing_frac = 1 - nonmissing/total` with `nonmissing` summed over *more* pops than `total`, so `nonmissing/total > 1` and `missing_frac` can go **negative** — which then *passes* `snp_passes_geno` (`<= geno_max_missing`) for any non-negative cap, silently keeping SNPs that should fail. No exception, no log, no assert — a textbook fail-fast violation (§2: "fail-fast … not silent corruption"). The header documents `pop_individuals` as "length P" but nothing enforces it; the `&& p < size()` guard makes the *short* case look handled when it actually produces a corrupt scalar.
*Adversarial check:* is the asymmetry real, or does the guard protect both loops? Verified — only the `total_indiv` loop has the `&& p < size()` clause; the per-SNP loop on line 39 is `for (int p = 0; p < P; ++p)` with no size clause, and it indexes `in.n[off]`/`in.q[off]` (the Q/V/N arrays, length `P·M`), *not* `pop_individuals`, so the short-`pop_individuals` case does not segfault the numerator — it just makes the two reductions span different pop counts. So the bug is a silent wrong-value, exactly as stated. Confirmed real and latent (the only current caller, the oracle, builds `pop_individuals` of exactly size `P`).
**Fix (corrected from the first pass):** validate `in.pop_individuals.size() == static_cast<std::size_t>(P)` once at the top of `derive_per_snp_summary` and `throw std::invalid_argument` — this is precisely what every sibling `io` reader does (`snp_reader.cpp`, `ind_reader.cpp`, `geno_reader.cpp`, `include_exclude.cpp` all `throw std::runtime_error`), so it matches the established `io`-leaf "surface failures as exceptions" convention (§4). The first pass's alternative suggestion "`STEPPE_LOG_WARN` + treat missing as 0" is **not currently available**: `internal/log.hpp` does not exist in this tree (it is referenced only as a future facility), so `throw` is the right and only mechanism today.

**1.2 — `in.q`/`in.n` null pointers are dereferenced without a guard when `P>0 && M>0`. [CONFIRMED] [low, S, before-M4.5: no]**
`derive_per_snp_summary` reads `in.n[off]` and `in.q[off]` (lines 42, 45) with no null check; the struct default-initializes all three to `nullptr`. The early-return at line 21 guards only `P<=0 || M<=0`, so a caller who sets `P>0, M>0` but forgets one of the arrays gets a segfault rather than a diagnostic. Note the *direct subsystem inconsistency*: the sibling `mind_prepass.cpp` line 32 **does** guard its packed pointer (`if (in.packed != nullptr && n_snp > 0)`), so within the same M2 filter module one input struct null-checks and the other does not. A one-line `if (!in.q || !in.n) throw std::invalid_argument(...)` makes them consistent and is cheap insurance for a raw-pointer public API. (`v` is documented but never read — see 7.4.) Severity low because the documented contract makes it a misuse, not a normal input.

**1.3 — `snps.count` vs `in.M` consistency is documented as a precondition but only half-enforced. [CONFIRMED] [low, S, before-M4.5: no]**
The header (line 99) states "`snps.count` must be >= `in.M`". `build_snp_keep_mask` then defensively bounds every `SnpTable` access with `(si < snps.ref.size())` / `.alt.size()` / `.chrom.size()` / `.id.size()` and falls back to `'N'` / `0` / `kEmptyId` when short (lines 76–77, 94, 101). This is *graceful* but it silently substitutes sentinel data: an `'N'` ref makes the SNP look multiallelic (via `is_multiallelic`) and be dropped; a `0` chrom makes it a non-autosome. So a too-short `SnpTable` quietly drops the tail SNPs instead of failing. That is half-and-half: a precondition in the docs and a silent fallback in the code. **Fix:** pick one. Either assert/throw the precondition once (`if (snps.ref.size() < static_cast<std::size_t>(M) || snps.alt.size() < ... ) throw ...`) and *delete the per-element guards from the hot loop* (faster and clearer — see 8.x), or document the sentinel-fallback as intentional defensive behavior. The first is more in keeping with the §2 fail-fast bar and with what the sibling readers do.
*Adversarial check:* is the sentinel fallback maybe deliberate (drop-not-flip is conservative, so dropping unknown-allele tail SNPs is safe)? Partly — but it conflicts with the stated `>=` precondition, and "safe-but-silent" still masks a genuine producer bug (a truncated `.snp`). Confirmed as a real consistency defect, not a false positive.

**1.4 — `ploidy <= 0` is clamped to 1 silently rather than rejected. [CONFIRMED] [low, S, before-M4.5: no]**
Line 30: `const double ploidy_d = static_cast<double>(in.ploidy > 0 ? in.ploidy : 1);`. A non-positive ploidy is a misconfiguration (the only valid values are 1 pseudo-haploid / 2 diploid per `views.hpp`), yet it is silently coerced to 1, producing a `nonmissing_indiv` of `N` (haploids counted as individuals) — a wrong missing fraction with no signal. Same fail-fast concern as 1.1/1.4. Prefer rejecting `ploidy <= 0` by `throw`. Defensible counter-argument: the clamp guarantees no divide-by-zero and ploidy comes from a validated `DecodeTileView` upstream, so a bad value should be impossible here — but "impossible by construction" is exactly where a cheap throw documents the invariant for free, and it costs one branch on a per-tile (not per-SNP) path.

### (2) Edge cases & failure modes

**2.1 — `build_snp_keep_mask` does not early-return on `in.P <= 0`; a zero-population tile keeps clean-ACGT SNPs with no data behind them. [CONFIRMED] [low, S, before-M4.5: no]**
`derive_per_snp_summary` returns early on `P <= 0` (line 21), so its offset math `static_cast<std::size_t>(P) * ...` (line 41) is only reached with `P > 0` — safe. `build_snp_keep_mask`, however, guards only `M <= 0` (line 67), then calls `derive_per_snp_summary` which on `P<=0` returns a zero-summary vector of length `M`. With `P<=0`, every `PerSnpSummary` is default (`pooled_minor_af=0`, `missing_frac=1`, `pooled_allele_count=0`), so MAF≥0 passes and `missing_frac=1 <= geno_max_missing` (default 1) passes; a clean-ACGT SNP is *kept* with zero data behind it. Degenerate input; arguably should keep nothing. **Fix:** `if (M <= 0 || in.P <= 0) return keep;` makes the empty-population case explicitly keep-nothing.

**2.2 — Integer overflow of the flat offset `p + P*s` is well-guarded; no defect. [N/A — verified safe]**
Line 41 promotes through `std::size_t` *before* multiplying (`static_cast<std::size_t>(P) * static_cast<std::size_t>(s)`), matching the `views.hpp` rule (`MatView::element` promotes `i + P·s` to `long` first). For the M2 cap (`P≈768`, `M≤100k`) the product is ~7.7e7; at full-dataset scale (`P~768`, `M~584k → ~4.5e8`) it is still far below `SIZE_MAX`. Good. (Contrast: in `int`, `P*s` would overflow at `s ≈ 2.8M` for `P=768` — but it never multiplies in `int`.)

**2.3 — All-missing SNP (every pop missing at SNP `s`) is handled correctly. [N/A — verified]**
When every `(p,s)` cell is missing, `N=0` there, so `pooled_allele_count == 0` → `pooled_ref_af = 0` (guarded denom, line 53) → `pooled_minor_af = 0`, and `nonmissing_indiv = 0` → `missing_frac = 1` (if `total_indiv>0`). At default, MAF 0 ≥ 0 passes and geno `1 <= 1` passes, so an all-missing clean-ACGT SNP is *kept at default* — the documented no-op-keeps-everything behavior, matching the oracle. With any biting `geno_max_missing < 1` it is correctly dropped. The "zero-fill makes the branch unnecessary" comment (lines 43–44) is accurate and relies on the documented `V≠0 ⟺ N>0` producer invariant (`views.hpp` line 33). Correct, nicely reasoned.

**2.4 — `total_indiv == 0` (no individuals) yields `missing_frac = 1.0`. [N/A — verified, sensible]**
Line 56 guards the denominator: `(total_indiv_d > 0.0) ? ... : 1.0`. A SNP with no individuals behind it reports fully-missing — correct and consistent with the all-missing case.

**2.5 — The negative-`M` clamp `static_cast<std::size_t>(M < 0 ? 0 : M)` is duplicated across both functions. [CONFIRMED] [low, S, before-M4.5: no]**
The exact expression appears at `snp_filter.cpp:20` and `:66`. A tiny DRY smell and a sign that "M is a count that cannot be negative" wants to be expressed once (a validation at entry — see 6.3). Not a bug.

**2.6 — `pooled_ref_af` is provably in `[0,1]` and `pooled_minor_af` in `[0,0.5]`; no clamp needed, no NaN path. [N/A — verified]**
Since `Q ∈ [0,1]` (the producer contract, `views.hpp` line 21) and `N ≥ 0`, we have `0 ≤ Σ Q·N ≤ Σ N`, so `pooled_ref_af = (Σ Q·N)/(Σ N) ∈ [0,1]` whenever the denom is positive (and is forced to 0 otherwise). `folded_maf` therefore always receives a valid argument and `pooled_minor_af ∈ [0,0.5]`. There is no division-by-near-zero amplification (the denom is a sum of non-negative haploid counts, guarded `> 0`), and no NaN can arise from these arrays. This is why finding 3.3's double-fold is provably harmless. Verified.

### (3) Numerical / precision vs §12

**3.1 — Pooling accumulation order matches the oracle exactly; parity holds. [N/A — verified, correct by design]**
`derive_per_snp_summary` accumulates `pooled_ref_count`, `pooled_allele_count`, `nonmissing_indiv` in ascending pop order `p = 0..P-1` (lines 39–48). The independent oracle in `test_filter_oracle.cu::oracle_keep_mask` (lines 148–154) accumulates in the *same* ascending order with the *same* `+= q*n` / `+= n` / `+= n/ploidy` operations. §12's determinism story for this host-side derivation rests on a fixed combine order; the match is exactly what the architecture requires, and the oracle asserts integer-exact mask equality (check c), which transitively pins the FP accumulation closely enough that no keep/drop flips. The filter decisions are threshold comparisons, not cancellation-prone differences, so the Ozaki-vs-native concern of §12 does not apply — this is pure host FP64 (the oracle path). One of the unit's best properties.

**3.2 — `missing_frac = 1.0 - nonmissing_indiv / total_indiv_d` can produce a tiny negative near 0 from rounding; the per-pop-divide form is parity-load-bearing. [CONFIRMED] [low, S, before-M4.5: no]**
For a fully-non-missing SNP, `nonmissing_indiv` (a sum of `N_p/ploidy` terms) should equal `total_indiv`, but the per-pop division `npn / ploidy_d` (line 47) and the summation can leave it a few ULPs above `total_indiv_d`, making `missing_frac` a tiny negative (e.g. `-1e-16`). That still passes `snp_passes_geno` and never flips a decision at any realistic threshold, and the oracle computes it the *same* way (line 153 `n / ploidy`, line 157), so parity is unaffected. Subtler point worth pinning: summing per-pop `npn/ploidy_d` is *not* FP-identical to `(Σ npn)/ploidy_d`; the code chose the per-pop-divide form and the oracle mirrors it. If anyone "optimizes" to divide the pooled sum once, parity could shift by ULPs. **Fix:** a one-line comment at line 47 marking the per-pop divide as *parity-load-bearing* (must match `oracle_keep_mask` line 153). Comment only.

**3.3 — `is_monomorphic(sm.pooled_minor_af)` re-folds an already-folded value (harmless, confusing). [CONFIRMED] [low, S, before-M4.5: no]** (Cross-file; manifests here.)
`build_snp_keep_mask` line 91 calls `is_monomorphic(sm.pooled_minor_af)`. `is_monomorphic` (`filter_decision.hpp:110`) is `folded_maf(x) == 0.0` — it folds its argument again. `sm.pooled_minor_af` is *already* `folded_maf(pooled_ref_af)` (line 54), so this double-folds. It is provably harmless (`folded_maf(folded_maf(q)) == folded_maf(q)` because a folded value is in `[0,0.5]` where `min(x,1-x) == x`; see 2.6), but it reads oddly: `is_monomorphic` documents its parameter as `pooled_minor_af` yet folds it, which only makes sense if the parameter were a *ref* af. A naming/contract smell in `filter_decision.hpp` surfacing at this call site. **Fix belongs in the predicate file** (drop the inner `folded_maf` in `is_monomorphic`, or rename its parameter to `pooled_ref_af`); noted here because the caller is here. Exact-zero result unaffected either way.

### (6) Decomposition / single-responsibility / function size vs §2

**6.1 — Functions are appropriately small and single-purpose. [N/A — good]**
`derive_per_snp_summary` (~44 lines) does one thing (Q/V/N → per-SNP scalars). `build_snp_keep_mask` (~48 lines) does one thing (per-SNP filter chain). Neither is overlong; both read top-to-bottom. **6.3 — Signed `int P` / `long M` counts with `static_cast` clamps; and the sibling `MindPrepassInput` uses unsigned for the same conceptual counts. [CONFIRMED + expanded] [low, M, before-M4.5: no]**
`P` is `int`, `M` is `long`, `pop_individuals` is `std::vector<std::size_t>`. The mix forces casts on lines 26, 27, 41 and the `M<0?0:M` clamps. `M` being `long` matches `MatView::M` (deliberate, to avoid 32-bit SNP-count overflow), and `P` being `int` matches `MatView::P` — so these types are *consistent with the established Q/V/N contract*, which is the right call; diverging here would be worse. **New observation the first pass missed:** the sibling M2 input struct `MindPrepassInput` (`mind_prepass.hpp:39–44`) models the *same* conceptual counts (`n_snp`, `n_individuals`) as **unsigned `std::size_t`**, matching `GenotypeTile`. So inside the one M2 filter module, `DecodedTileSummaryInput` is signed-count (mirrors `MatView`) and `MindPrepassInput` is unsigned-count (mirrors `GenotypeTile`). Both choices are individually defensible (each mirrors the struct it is fed from), but a reader moving between the two filter inputs meets opposite conventions — a §2 single-source/consistency wrinkle, not a bug. **I would NOT change the types** (each correctly matches its upstream contract; unifying would force a worse mismatch with either `MatView` or `GenotypeTile`). I would instead **validate `P >= 0 && M >= 0` once at entry** (turning the silent clamp into explicit fail-fast) and add a one-line note that the signed/unsigned split is intentional (each mirrors its source). Low priority.

### (7) Readability, naming, const-correctness, `[[nodiscard]]`/`noexcept`, comments

**7.1 — `[[nodiscard]]` present on both public functions. [N/A — good]**
Both `derive_per_snp_summary` and `build_snp_keep_mask` carry `[[nodiscard]]` (header lines 80, 101) — appropriate, since discarding a computed mask/summary is always a bug. Consistent with the rest of the filter headers.

**7.2 — Neither function is (or can be) `noexcept`, and that is correct. [N/A — verified]**
Both allocate `std::vector`s, so they can throw `bad_alloc`; marking them `noexcept` would `std::terminate` on OOM instead of propagating, contradicting the `io`-leaf "surface failures as exceptions" convention. Leaving them non-`noexcept` is right — and it is also why the 1.1/1.2/1.4 fixes can safely `throw std::invalid_argument` (the throw propagates correctly through `build_snp_keep_mask`, which calls `derive_per_snp_summary`).

**7.3 — `static const std::string kEmptyId;` inside the loop is oddly scoped; the first pass's *performance* framing is overstated. [CONFIRMED but DOWNGRADED] [low, S, before-M4.5: no]**
Line 100: `static const std::string kEmptyId;` declared inside the per-SNP loop body, inside the `if (!mem_noop)` block. The first pass said it "incurs a one-time guarded-initialization check … on every entry to the block." That is *technically* true but materially misleading: per the C++11 block-scope-static rule (`[stmt.dcl]`, confirmed in Sources), initialization happens once when control first passes the declaration, guarded by `__cxa_guard_acquire`/`release`; **after first init the fast path is a single relaxed atomic-load of the guard byte, not a lock**, so the per-iteration cost is one predictable load — negligible, and the variable is correctly never re-created. So this is **not a performance finding**; the real issue is purely *readability/scope*: declaring a `static` deep inside a loop body reads as if it is recreated per iteration (it is not). **Fix:** hoist it to namespace/file scope (next to where other file-scope constants would live) or to a `const std::string` declared once before the loop. Truly minor; downgraded from the first pass's implied perf concern to a pure clarity nit.

**7.4 — `in.v` (validity mask) is documented and accepted but never read. [CONFIRMED] [low, S, before-M4.5: no]**
`DecodedTileSummaryInput::v` (header line 42) is part of the struct and documented, but `derive_per_snp_summary` reads only `q` and `n` (it relies on `V≠0 ⟺ N>0`, so `N=0` already zeroes missing cells — comment lines 43–44). So `v` is dead *input*. Defensible (it keeps the struct a faithful mirror of `DecodeResult{q,v,n}` so the caller fills it from one source), but a reader sees a member taken and ignored. **Fix:** a one-line note on the `v` member ("accepted for symmetry with `DecodeResult`; the derivation uses only N since V≠0 ⟺ N>0, so V is not read here") prevents a future maintainer "fixing" a phantom missing use of `v`. See 7.7 for the output-side twin.

**7.5 — Comment density is excellent and matches the surrounding files. [N/A — good, keep]**
Both files carry the house style: a file-header rationale tying to architecture §§, inline comments explaining *why* (the zero-fill-no-branch reasoning, the SNP-global invariant, the drop-not-flip ordering). At or above the density of `filter_decision.hpp`. Do not trim.

**7.6 — `const`-correctness is thorough. [N/A — good]**
`P`, `M`, `total_indiv_d`, `ploidy_d`, the per-SNP `off`, `npn`, `ref`, `alt`, `si`, `mem_noop` are all `const` where appropriate; the `PerSnpSummary& sm` (line 50) and `keep[si]` writes are the only mutations and are clearly the outputs. The `summary` vector is bound `const` and indexed by `const` reference. No `const` is missing.

**7.7 — NEW: `PerSnpSummary::pooled_ref_af` and `pooled_allele_count` are computed but read by NO consumer (the output-side twin of the dead `v` input). [NEW] [low, S, before-M4.5: no]**
Grepping every consumer of `PerSnpSummary` across `src/` and `tests/`: `build_snp_keep_mask` reads only `sm.pooled_minor_af` (lines 87, 91) and `sm.missing_frac` (line 88). The oracle (`oracle_keep_mask`) recomputes pooled ref-af / missing-frac inline and **never touches the `PerSnpSummary` struct at all**. So `pooled_ref_af` (line 52) and `pooled_allele_count` (line 51) are *written but never read* by any current caller or test. They are reasonable to expose for future diagnostics / a finer-grained host test (the header sells the struct as "exposed so the oracle test can recompute and compare it exactly"), but right now they are unconsumed output — the structural analogue of 7.4 on the output side. **Fix:** either add the host test that actually asserts these fields (which is finding 10.1's natural content — assert `pooled_ref_af`/`pooled_allele_count` on a hand-built tile), giving the fields a reader; or note on the two members that they are diagnostic-only and currently unread. The former is preferred (it kills two findings at once). Not a bug — but a senior reviewer flags computed-and-discarded fields.

### (8) Performance

**8.1 — `build_snp_keep_mask` always recomputes the full per-SNP summary, even when no threshold filter is active. [CONFIRMED] [med, M, before-M4.5: no]**
Line 69 unconditionally calls `derive_per_snp_summary(in)` — an `O(P·M)` reduction over the whole tile — before the per-SNP loop. But when `cfg.maf_min == 0 && cfg.geno_max_missing >= 1 && !cfg.drop_monomorphic` (the common no-op / parity default), the summary is *never consulted*: only `is_multiallelic`/`is_strand_ambiguous`/`is_transversion`/`is_autosome` (char/chrom-only) and membership matter. On the parity path the entire `O(P·M)` pooling pass is wasted — and the parity path is the dominant streaming path (§11.1, every tile, every run that does not request a MAF/geno filter). **Fix:** gate the derivation — `const bool needs_summary = cfg.maf_min > 0.0 || cfg.geno_max_missing < 1.0 || cfg.drop_monomorphic;` compute `derive_per_snp_summary` only when `needs_summary`, and guard the summary-using predicates (lines 87, 88, 91) behind it. On a 768×100k tile this saves ~7.7e7 multiply-adds per tile on the default path. The standalone public `derive_per_snp_summary` stays (the oracle calls it directly), so the optimization lives *only inside* `build_snp_keep_mask`. Medium severity: pure hot-path overhead, no correctness impact; before-M4.5 "no" since M2 is functionally done, but a clean win for the streaming integration.
*Adversarial check:* does gating risk a subtle behavior change (e.g. a SNP whose `missing_frac` would have flipped it)? No — when `needs_summary` is false, none of the three summary-consuming predicates can fire (MAF≥0 always passes, geno≤1 always passes, monomorphic flag off), so the mask is bit-identical with or without the summary. Safe.

**8.2 — The per-SNP loop is embarrassingly parallel but writes a `std::vector<bool>` (cannot be safely parallelized as-is). [CONFIRMED — standard verified] [low→med, M, before-M4.5: no]**
The loop over `s` (lines 72–106) is independent per SNP (the summary is precomputed, membership is read-only, the only write is `keep[si] = true`). A natural candidate for host parallelism on a large tile. But the output is `std::vector<bool>`, and the C++ standard **explicitly exempts `std::vector<bool>` from the data-race-freedom guarantee for concurrent modification of distinct elements** — because it is bit-packed, distinct logical elements can share an underlying word. Verified against the working draft: `[container.requirements.dataraces]/2`: "implementations are required to avoid data races when the contents of the contained object in different elements in the same … sequence, **excepting `vector<bool>`**, are modified concurrently" (Sources — `eel.is`/`timsong` working draft and LWG issue 1329). So `y[0]=true` may race with `y[1]=true` for `vector<bool> y`. Today the loop is single-threaded, so there is *no bug now*; the trap is latent for a future parallelization. The return type `std::vector<bool>` is the established subsystem contract (`FilterPlan::snp_keep` is `std::vector<bool>`), so changing it would ripple and is not warranted. **Fix:** leave the public type; add a comment at the loop documenting the `vector<bool>` no-parallel-write hazard, and if/when the loop is parallelized, write into a `std::vector<char>`/`std::uint8_t` scratch and convert once. The one place this unit intersects the §11.1 streaming-throughput story.

**8.3 — `mem_noop` short-circuit and membership-last ordering are good performance choices. [N/A — good, keep]**
`build_snp_keep_mask` caches `mem.is_noop()` once (line 70) and does the per-SNP string lookup only when membership is active and the SNP already passed everything else. Avoids both a per-SNP virtual-ish call and the string hash on the parity path. Good.

**8.4 — `derive_per_snp_summary` takes `const DecodedTileSummaryInput&` — no needless copy. [N/A — verified]**
Pass-by-const-ref throughout; `pop_individuals` is not copied. The returned `std::vector<PerSnpSummary>` is moved (NRVO). `summary` is bound as a `const` value and indexed by reference. Fine.

**8.5 — NEW (minor): the per-element `SnpTable` bounds checks run on every SNP in the hot loop. [NEW] [low, S, before-M4.5: no]**
`build_snp_keep_mask` performs `(si < snps.ref.size())`, `(si < snps.alt.size())`, and conditionally `(si < snps.chrom.size())` / `(si < snps.id.size())` *per SNP* (lines 76–77, 94, 101). Each is one cheap comparison, so the cost is negligible — but it is redundant work that exists only because the `>=` precondition (1.3) is checked per-element instead of once. If 1.3 is resolved by asserting/throwing the precondition once at entry, all four per-element guards disappear from the loop, making the body a flat sequence of unchecked reads — both faster and clearer. Bundled with 1.3; flagged separately so the perf benefit of fixing 1.3 is explicit. Low.

### (9) Layering / API / ABI vs §4

**9.1 — Layering is exemplary. [N/A — the unit's headline strength]**
No CUDA include; no `core`/`device` include; consumes plain `const double*` precisely to avoid depending on `core::MatView`. The header comment (lines 12–16, 32–39) states the rule and the reason. Exactly the §4 "`io` an isolated leaf" discipline. Keep.

**9.2 — The plain-pointer `DecodedTileSummaryInput` vs `core::MatView` is a deliberate, justified duplication. [N/A — verified, acceptable]**
One could argue `DecodedTileSummaryInput` re-declares the Q/V/N layout `MatView` owns (DRY, §2/§8). But pulling in `MatView` would force `io` to depend on `core/internal`, violating §4 — and the layout rule (`i + P·s`) is re-implemented identically on line 41, matching `MatView::element`. The rare case where DRY and layering conflict; the unit correctly prioritizes layering, with a comment. Cost: the `i + P·s` indexing now lives in two host places (`MatView::element` and `snp_filter:41`, plus the oracle). Acceptable for a leaf; the future CUDA-free `span_view.hpp` (mentioned in `views.hpp`) could unify it. No action now.

**9.3 — Public API is stable and minimal. [N/A — good]**
Two free functions + two structs, all in `steppe::io::filter`. No hidden globals, no statics-with-state (the lone `kEmptyId` static is a `const` empty string, not mutable state). Header is self-contained (`<cstddef>`, `<vector>` + the three project headers it genuinely needs). Matches §9 "typed immutable config / injected resources" — everything arrives through `cfg`/`mem`/`in`. Good.

### (10) Testability vs §13

**10.1 — There is NO direct unit test of `derive_per_snp_summary` or `build_snp_keep_mask`. [CONFIRMED] [med, M, before-M4.5: no]**
Verified by reading `tests/unit/test_filters.cpp` end to end: its ten cases exercise every *predicate* in `filter_decision.hpp` (MAF/geno/mind boundaries, monomorphic, strand-ambiguous, multiallelic, ts/tv, autosome), plus `SnpMembership` and `run_mind_prepass` — but **never** call `derive_per_snp_summary` or `build_snp_keep_mask` (grep confirms zero references). Those two functions are covered *only* by `tests/reference/test_filter_oracle.cu`, which requires the real AADR `.geno/.snp/.ind` at `/workspace/data/aadr` and a CUDA toolchain (a `.cu` linking `steppe::device`). So on a box without the data (or a fast host-only CI lane), the unit's own logic — the pooling math, the `total_indiv`/`nonmissing` reduction, the filter-chain ordering, and every edge case in §(1)/§(2) above (short `pop_individuals` (1.1), all-missing (2.3), `total_indiv==0` (2.4), `M<=0`/`P<=0`) — is **untested in isolation**. This contradicts §13's "the whole pipeline is exercisable GPU-free" and breaks the M2 pattern where the predicates *are* host-unit-tested. **Fix:** add host-only cases to `test_filters.cpp` (it already has zero GPU/data deps) that hand-build a tiny `DecodedTileSummaryInput` (e.g. 2 pops × 3 SNPs with a known missing pattern) and assert `derive_per_snp_summary` returns the expected `pooled_ref_af`/`pooled_minor_af`/`missing_frac`/`pooled_allele_count` (this *also* gives 7.7's two unread fields a reader) and that `build_snp_keep_mask` keeps/drops the expected SNPs under a few `FilterConfig`s — including the `pop_individuals.size() < P` (1.1) and `total_indiv == 0` (2.4) edge cases. Straightforward (plain doubles, no GPU). The single clearest "what it takes to reach 10/10" item.

**10.2 — The derive/decide split is itself a testability win. [N/A — good]**
Exposing `derive_per_snp_summary` separately (header lines 62–81) is explicitly "so the oracle test can recompute and compare it exactly." Good design-for-test; 10.1 is only that the cheap host test does not yet take advantage of it.

### (11) Capability tiers (PRO-6000 capable vs budget-5090) — where this unit touches them

**11.1 — A host-pure leaf; no direct device-capability branch, and that is correct. [N/A — with a forward note]**
`snp_filter` runs entirely on the host and produces a small plan (`std::vector<bool>` of length `M`) consumed downstream (`filter_plan.hpp`) to gate which SNP *columns* feed the masked GEMM on whatever GPU tier is active. It does **not** itself need a GDS / ncu / P2P capability probe — the capability tiers (TODO "Keeping it GPU-dominant") live in the device/streaming layers, not here. The right behavior is exactly what it does: produce a tier-independent plan; parity holds on both tiers because the mask is computed identically host-side regardless of GPU. No tagged log line is owed by this file. (And note: even if it wanted one, `STEPPE_LOG_*` does not exist yet in this tree.)

**11.2 — Forward note: the mask-build is the host step in the §11.1 per-tile streaming loop, so its cost and parallelizability feed the GPU-dominant goal. [CONFIRMED] [low, M, before-M4.5: no]**
Per §11.1, every tile runs decode → reduce → f2-accumulate, with the SNP filter in S0' producing `snp_keep`. If mask-building is serialized on the host while the GPU waits, it eats into the "GPU-dominant" budget the TODO tier-work optimizes. Two concrete, tier-relevant improvements: (a) skip the summary pass on the no-op/parity path (8.1) so the host step is `O(M)` char checks not `O(P·M)` reductions; (b) make the loop parallelizable (8.2) for large tiles on the many-core full-host box, with a `std::vector<char>` scratch to dodge the `vector<bool>` race. Neither is a runtime capability *probe* (there is no GPU capability to detect in this file) — they are host-throughput hygiene so this step never becomes the bottleneck that makes the run host-bound. The capability tag belongs at the streaming orchestrator. This is the honest answer to "where does this unit touch the tiers": indirectly, via host-step cost, not via a device probe.

## Considered & rejected

- **"`folded_maf` uses `<` so 0.5 maps oddly?"** — Rejected. `folded_maf(0.5) = (0.5 < 0.5) ? 0.5 : 0.5 = 0.5`; both branches yield 0.5, and the test pins it (`test_filters.cpp:46`). No defect. (Lives in `filter_decision.hpp` anyway.)
- **"The offset `p + P*s` could overflow `int`."** — Rejected; computed in `std::size_t` after explicit casts (line 41), verified in 2.2. The casts precede the multiply.
- **"`std::vector<bool> keep(...)` then setting `true` is slow vs a bitset."** — Rejected. `M ≤ 100k`; the allocation is negligible next to decode, and the type matches the downstream `FilterPlan::snp_keep` contract. Changing it for speed is premature and would ripple.
- **"Pass `summary` by value into the loop copies it."** — Rejected; bound as `const std::vector<PerSnpSummary> summary = ...` (line 69, move-constructed) and indexed by `const PerSnpSummary&` (line 74). No copy.
- **"`derive_per_snp_summary` should fuse into `build_snp_keep_mask` to avoid the separate vector."** — Rejected. The split is a deliberate testability seam (the oracle re-derives independently). Fusing would couple derivation to decision and lose the comparison point. The real win is 8.1 (skip the derive when no numeric filter is active), not fusing.
- **"`is_strand_ambiguous` checked after `is_multiallelic` — order matters for non-ACGT."** — Rejected as a bug; verified fine. `is_multiallelic('A','N')` is `true` (N normalizes to `'\0'`), so the `||` short-circuits and the SNP drops before `is_strand_ambiguous` runs; and `is_strand_ambiguous` independently returns `false` on non-ACGT anyway (`filter_decision.hpp:164`). Both orders give the same result. Harmless.
- **"`missing_frac` default `1.0` in `PerSnpSummary` could survive uncomputed."** — Rejected. Every `s` in `[0,M)` is written (line 55); the default survives only for never-written entries, which do not exist (the loop covers all `M`). The default is a safe initializer.
- **"`pooled_ref_af` could exceed 1.0 and make `folded_maf` return a negative MAF."** — Rejected; provably impossible. `Q ∈ [0,1]` and `N ≥ 0` ⇒ `Σ Q·N ≤ Σ N` ⇒ `pooled_ref_af ∈ [0,1]`, so `pooled_minor_af ∈ [0,0.5]`. No clamp needed (2.6). This also makes 3.3's double-fold provably a no-op.
- **"`total_indiv` (a `std::size_t` sum) could overflow."** — Rejected. It sums per-pop individual counts (~thousands total); nowhere near `SIZE_MAX`. The Q/V/N *element* count (`P·M`) is the only large index and is handled in 2.2.
- **"`is_monomorphic` double-fold could change the exact-zero result."** — Rejected; `folded_maf(folded_maf(q)) == folded_maf(q)` for a folded value in `[0,0.5]`, so `== 0.0` is identical either way. The double-fold is a clarity smell (3.3), not a correctness one.
- **"`SnpMembership::passes` is `noexcept` but does a hash lookup that could throw."** — Rejected as out of scope (that is `include_exclude`'s concern). `passes` only does `find` on already-constructed sets (no allocation), so `noexcept` is sound there. Not this unit.
- **"`kEmptyId` function-local static is a per-iteration performance cost."** — *Downgraded, not rejected* (7.3). The first pass framed it as perf; per the C++11 guard fast-path it is one relaxed atomic load after first init — negligible. The genuine issue is readability/scope, so the finding survives as a clarity nit, not a perf one.
- **"`build_snp_keep_mask` does not early-return on `in.P <= 0`."** — *Kept* as a real (low) finding (2.1): the `derive` call masks it into a zero-summary, but keeping a clean-ACGT SNP on a zero-population tile is arguably wrong.
- **"Signedness mismatch between `DecodedTileSummaryInput` (signed) and `MindPrepassInput` (unsigned) is a bug."** — Rejected *as a bug*, kept as a consistency observation (6.3). Each struct correctly mirrors its upstream (`MatView` vs `GenotypeTile`); changing either would create a worse mismatch. Worth a note, not a change.

4. **Resolve the `SnpTable`-too-short ambiguity (1.3 + 8.5):** assert/throw the documented `>=` precondition once and drop the four per-element `si < .size()` guards from the hot loop (faster and clearer), OR document the sentinel-fallback as intentional. 5. **Reject (don't silently clamp) `ploidy <= 0`, `M < 0`, `P < 0` (1.4, 2.1, 6.3)** by `throw` at entry; turn the defensive clamps into explicit validation. [S]

## Good patterns to keep

- **Strict `io`-leaf layering** (plain `const double*`, no `core::MatView`, no CUDA) — the unit's headline strength; do not "DRY it up" by pulling in `MatView` (9.1, 9.2).
- **The derive/decide split** exposing `derive_per_snp_summary` as a pinnable seam for the oracle (6.1, 10.2).
- **Zero-fill-makes-the-branch-unnecessary** reduction relying on the documented `V≠0 ⟺ N>0` invariant — branch-free and correct (2.3, comment lines 43–44), and `pooled_ref_af`/`pooled_minor_af` provably stay in range so no clamp is needed (2.6).
- **Fixed ascending-pop accumulation order matching the oracle exactly** — the parity-determinism property §12 wants (3.1).
- **`mem.is_noop()` cached once + membership evaluated last** — short-circuits the only O(hash) test off the parity path (8.3).
- **Every threshold sourced from `FilterConfig`, every decision delegated to `filter_decision.hpp`** — no duplicated rule, no magic threshold (§2/§8 DRY).
- **High, *explanatory* comment density** matching the surrounding filter files — comments say *why*, not *what* (7.5).
- **`[[nodiscard]]` on both functions; the correct decision to leave them non-`noexcept`** (7.1, 7.2) — which also lets the validation fixes throw cleanly.

**Sources consulted (documented behavior, cited per the brief):**
- `std::vector<bool>` is **explicitly excepted** from the C++ standard's container data-race-avoidance guarantee (`[container.requirements.dataraces]/2`: "implementations are required to avoid data races when the contents of the contained object in different elements in the same … sequence, *excepting `vector<bool>`*, are modified concurrently"), because the common implementation is an unsynchronized bit-field — `y[0]=true` may race with `y[1]=true`. Confirms finding 8.2. (C++ working draft [container.requirements.dataraces] via eel.is / timsong-cpp; LWG issue 1329 "Data races on vector<bool>".)
- C++11+ requires **thread-safe one-time initialization of block-scope statics**: initialization occurs when control first passes the declaration, guarded by `__cxa_guard_acquire`/`__cxa_guard_release`; concurrent entry blocks until init completes; after init the guard is a cheap relaxed-load fast path (not a lock). This is why finding 7.3's *performance* framing is overstated and is downgraded to a readability nit. (C++ `[stmt.dcl]` static-init guard semantics; corroborated by MSVC `/Zc:threadSafeInit` docs.)
- Verified in-tree that **`STEPPE_LOG_*` / `internal/log.hpp` does not yet exist** (referenced only as a future facility in `device/cuda/{device_buffer.cuh,handles.hpp,stream.hpp}` — "swap … once log.hpp lands"), and that every sibling `io` reader (`snp_reader.cpp`, `ind_reader.cpp`, `geno_reader.cpp`, `include_exclude.cpp`) surfaces bad input via `throw std::runtime_error`.
