## Findings

### (1) Correctness & bugs

**1.1 [med] `FilterConfig` thresholds (`maf_min`, `geno_max_missing`, `mind_max_missing`) and `Precision::mantissa_bits` have no documented valid range, and `build()` (the §9 validation choke point) does not exist yet.** — CONFIRMED.
Location: lines 208, 216, 222 (filter fractions); 151-156 (`mantissa_bits`).
Issue: MAF is a folded frequency in `[0, 0.5]`; a missing-fraction is in `[0, 1]`. The contract states no domain, and `grep` confirms there is no `ConfigBuilder`/`RunConfig` definition anywhere in the tree (§9 names them but they are unbuilt — M0–M4 stop at the device slice). So `maf_min = 0.7` silently drops every SNP, `geno_max_missing = 2.0` is a vacuous no-op, `geno_max_missing = -0.1` drops everything, `maf_min = -1.0` keeps everything. These are the config foot-guns §2/§9 fail-fast is meant to catch.
Why it matters: §9 makes `build()` the single validation home; until it lands, the *contract* should document the legal range so the eventual validator and current consumers agree on one definition. (`filter_decision.hpp` already pins the *boundary side* — inclusive `>=`/`<=` — but not the legal *range*.) architecture.md §2 fail-fast; §9 validation list.
Fix: add a one-line range to each comment (`maf_min ∈ [0, 0.5]`; the two fractions `∈ [0, 1]`) and enumerate them in the §9 `build()` validation when it lands.
Severity: med. Effort: S. Before-M4.5? No.

**1.2 [low] `Precision::mantissa_bits` is a bare `int` with no bounds, and an out-of-range value reaches the cuBLAS driver.** — CONFIRMED, with the driver path and the API signature verified against the live docs.
Location: lines 151-156; consumer `f2_block_kernel.cu:208-209`.
Issue: the comment enumerates 32/40/48 but the field is unbounded. `engage_f2_precision` passes it straight into `cublasSetFixedPointEmulationMaxMantissaBitCount(handle, precision.mantissa_bits)` under `STEPPE_HAVE_EMU_TUNING` (verified `f2_block_kernel.cu:208-209`). I verified the API takes a **signed `int`** — `cublasStatus_t cublasSetFixedPointEmulationMaxMantissaBitCount(cublasHandle_t handle, int maxMantissaBitCount)` (cuBLAS 13.3 reference; see Considered & Rejected) — so a `≤0` value is a malformed argument at the driver call and `>53` is meaningless for FP64 (52 stored mantissa bits + implicit 1). `mantissa_bits = kDefaultMantissaBits` (line 156) correctly ties the *default* to the named constant — good — but nothing bounds a caller-set value.
Why it matters: §9 says `build()` rejects "a precision the selected backend cannot honor"; a nonsensical `mantissa_bits` is exactly that. ROADMAP §4/§0 pin 32/40/48.
Fix: state the legal range in the comment (`// VALID: (0, 53]; meaningful 32/40/48`) and have `build()` reject out-of-range. The *type* should stay `int` (matches the driver signature — see rejected item), so a documented range + a build-time check is the fix, not an unsigned type.
Severity: low. Effort: S. Before-M4.5? No.

### (2) Edge cases & failure modes

**2.1 [med] `DeviceConfig::devices` ordering+uniqueness is the parity guarantee but the contract does not name the invariant, and the type is an unguarded `std::vector<int>`.** — CONFIRMED.
Location: lines 164-170.
Issue: the comment correctly makes `devices` the "single source of truth" and the "fixed … host-side combine order" (load-bearing for §11.4/§12 parity — §12 states the combine "sums them in fixed device order (`g=0..G−1`, the `DeviceConfig::devices` order)"). But nothing prevents `{0,0,1}` (duplicate ⇒ a device's partial double-counted ⇒ parity violation) or `{-1}`/`{99}` (absent device). §9 explicitly lists "any device id … that is absent or duplicated" as a `build()` rejection — but `build()` does not exist yet, and the field comment does not flag these as illegal. For the single most parity-critical field, that is the least-guarded one.
Why it matters: §12 parity rests on this exact order and uniqueness; a duplicate silently breaks "bit-identical across G." Note that today's `cuda_backend.cu` is single-GPU only (no `cudaSetDevice`/peer/multi-device code exists — verified, and TODO §A flags "No `device_id` / `cudaSetDevice`" as an open med item), so the hazard is latent until M4.5 wires the multi-GPU combine — which is precisely why the contract should name the invariant *before* that code lands.
Fix: add to the comment: "INVARIANT (enforced by `build()`): ids are distinct and each is a visible CUDA device; empty ⇒ auto-enumerate. A duplicate or absent id is `InvalidConfig`."
Severity: med. Effort: S. Before-M4.5? **Yes** — the duplicate-id parity hazard becomes live the moment `devices.size() > 1`, which is M4.5.

**2.2 [low] `stream_count` and `search_streams` are unbounded `std::size_t` with no `≥1` floor, and the `deterministic`-relationship §9 describes is inexpressible (no such field).** — CONFIRMED.
Location: lines 179, 183.
Issue: `stream_count = 0` is representable and meaningless; `search_streams = 0` silently disables the S8 search lanes. §9 says `build()` forces `stream_count == 1` when `deterministic` — but there is no `deterministic` field in this struct at all (finding 9.1), so the contract cannot even express the constraint §12 hinges on.
Why it matters: §12 determinism requires `stream_count == 1` on the statistic path; a `0` or unvalidated large value is a latent foot-gun, and the missing `deterministic` field means the relationship can't be stated.
Fix: document `stream_count ≥ 1` (forced to 1 on the deterministic statistic path) and `search_streams ≥ 1`. Add `deterministic` (see 9.1).
Severity: low. Effort: S. Before-M4.5? No.

**2.3 [low — DOWNGRADED from the first pass's framing] The include/exclude/prune set-algebra is *not* ambiguous — it is unambiguously single-homed in `include_exclude.hpp`; the only residual is one word in the `include_snp_ids` field comment.**

It is not. `include_exclude.hpp:13-17` pins it exactly and in one place (the §8 DRY single home), verbatim: `keep-set = include_snp_ids ∪ prune.in ids`; `drop-set = exclude_snp_ids`; *"A SNP passes membership iff: (keep-set empty OR id ∈ keep-set) AND id ∉ drop-set."* So include and prune compose by **union** (matching the `prune_in_path` comment "composes … as an additional keep set"), and the membership keep-set is then **intersected with the other filters** (MAF/geno/…). ### (3) Numerical / precision vs §12

**3.1 [low — was first-pass `high`; PART (a) REJECTED, PART (b) REFRAMED] `kCublasWorkspaceBytes` comment.**
Location: lines 84-88.

**(a) Reproducibility claim — REJECTED as a false positive (now doc-verified).** The comment says "an explicit workspace is REQUIRED for run-to-run reproducibility of emulated FP64." The first pass called this "stale and wrong vs CUDA 13.x," citing the general §2.1.4 paragraph ("This guarantee no longer holds when … fixed-point emulation is used"). That is only half the vendor's statement. I re-fetched the cuBLAS 13.3 docs this pass and confirmed the two sections:
  - §1.5.2.5 Fixed-Point Performance Guide (verbatim): *"Providing workspace via cublasSetWorkspace() avoids both the allocation cost and the fallback path, and **is required to guarantee reproducible results**."*
  - §1.5.2.4 Fixed-Point Workspace Requirements (verbatim): *"The amount of memory required depends on problem size, mantissa bit count, GPU architecture, and library version, but never exceeds 8 GB."* — i.e. emulation is non-reproducible *by default* because an insufficient per-handle workspace triggers a dynamic allocation / fallback path.
  So NVIDIA's actual position is: emulation falls back / dynamically allocates *by default*, and supplying an adequate workspace is **exactly what restores reproducibility**. The config.hpp comment is therefore *correct* and matches the vendor; architecture.md §0/§12 ("fixed-point emulation voids that guarantee unless an adequate workspace is supplied") is also vindicated, not "stronger than the vendor wording" as the first pass claimed. No change needed for the reproducibility sentence.

**(b) Sizing claim "Ample" — survives, but reframed from `stale/wrong` to `missing-MEASURED-tag`.** The comment asserts 64 MiB is "Ample for the reduce-to-[P×P] GEMMs." Re-checked against the cuBLAS 13.3 §2.4.8 `cublasSetWorkspace()` table (re-fetched, verbatim): *"NVIDIA Hopper Architecture (sm90): 32 MiB; NVIDIA Blackwell Architecture (sm10x): 32 MiB; Other: 4 MiB."* So 64 MiB is **2× the Blackwell recommendation** — a reasonable "more than recommended" margin (the docs note exceeding the recommendation can improve performance and guarantee reproducibility). For the M4 per-block reductions (small P×P, P≤768) 64 MiB is very likely sufficient. The real defect is narrow: unlike *every other constant in this file*, `kCublasWorkspaceBytes` carries **no MEASURED tag and no per-arch citation** — it states "Ample" as a categorical fact, violating this file's own ROADMAP §0/§4 "measured, attributed" standard. And if a future per-block GEMM grows, 64 MiB could drop below the per-handle requirement → dynamic allocation/fallback → potential non-determinism (the very thing the workspace exists to prevent), so the bound matters. (TODO §A separately notes the *budget* arithmetic in `cuda_backend.cu` "doesn't subtract the 64 MiB cuBLAS workspace before applying the fraction" — that is a consumer concern, partly already handled by allocation ordering; see 5.1.)
Why it matters: §12 is the law; a precision-policy constant should carry the same measured-attributed rationale its siblings do.
Fix: keep the (correct) reproducibility sentence; replace "Ample" with a measured/attributed statement, e.g. "64 MiB = 2× the cuBLAS-13.3 Blackwell/Hopper recommended 32 MiB (§2.4.8; 4 MiB others); the FP64-emulation requirement grows with problem size and never exceeds 8 GB (§1.5.2.4), so re-measure if the per-block GEMM grows. MEASURED OK for the M4 P×P reductions at P≤768."
Severity: low (down from high). Effort: S. Before-M4.5? No.

**3.2 [med] The `EmulatedFp64` comment promises FIXED slices "is not representable by this struct," but the device build can silently run the REJECTED dynamic mantissa when `STEPPE_HAVE_EMU_TUNING=OFF`, and the public contract gives no hint of that coupling.** — CONFIRMED, with the foot-gun re-verified line-by-line in the device source, the cmake option, and TODO.
Location: lines 16-22 (banner), 40-44 (`kDefaultMantissaBits`), 131-139, 151-156 (`mantissa_bits`).
Issue (the substantive part): config.hpp asserts dynamic mantissa "is not representable by this struct" (line 154) — true at the *type* level. But the *runtime* engagement of the FIXED pin is compile-time-gated: `engage_f2_precision` (verified `f2_block_kernel.cu:201-217`) always calls `cublasSetMathMode(…FIXEDPOINT_MATH)` for EmulatedFp64, but `cublasSetEmulationStrategy(EAGER)` + `cublasSetFixedPointEmulationMantissaControl(FIXED)` + `…MaxMantissaBitCount(mantissa_bits)` are inside `#if STEPPE_HAVE_EMU_TUNING`. When that macro is OFF, cuBLAS falls back to its *dynamic* default (~60-bit) — the exact rejected trap (ROADMAP §0/§12). The kernel's own comment confirms this (`f2_block_kernel.cu:196-198`: "without it the emulated math mode still engages emulation (just not the pinned slice count), which is why the build flag is required for the measured speedup"), as does TODO §A verbatim ("EmulatedFp64 silently runs the REJECTED dynamic-mantissa trap when STEPPE_HAVE_EMU_TUNING=OFF … voids §12 bit-stability while still reporting EmulatedFp64"). Mitigating fact (re-verified): `SteppeOptions.cmake:34-36` defaults the macro **ON** and is `PRIVATE` to `steppe_device`, so the trap is *not* the default build — which is why this is med, not high, *for config.hpp*. Note the public header cannot even *see* `STEPPE_HAVE_EMU_TUNING` (it is a device-private define), so config.hpp cannot enforce the coupling — only point at it.
Issue (the minor part): the speedup figures drift across the file vs ROADMAP §0. Banner lines 18-19: "7–13×"; enum line 134: "MEASURED 7–17×"; ROADMAP §0 table: "8.5–17.5×" (32-bit) and "7.2–13.3×" (40-bit); architecture.md §0 itself says "8–17×". "7–17×" conflates the rows. This file is the single source of truth and should quote one attributed pair.
Why it matters: ROADMAP §0's dynamic-mantissa cautionary tale is the most-emphasized precision lesson; a reader of the public header would reasonably believe selecting `EmulatedFp64` guarantees fixed slices — true only when the device build engages the pin.
Fix: (a) add a one-line caveat to the `EmulatedFp64` comment: "The FIXED-slice pin is engaged in the device layer only when built with `STEPPE_HAVE_EMU_TUNING` (default ON); a build with it OFF degrades to the rejected dynamic mantissa — the device build must assert/fall back rather than silently run dynamic (see device `engage_f2_precision`; TODO precision item)." This keeps config.hpp the authoritative cross-reference without leaking CUDA in. (b) Reconcile to one attributed pair, e.g. "MEASURED 7.2–13.3× at 40-bit, 8.5–17.5× at 32-bit on real AADR; ROADMAP §0/§12."
Severity: med. Effort: S. Before-M4.5? No (the *enforcement* fix is the device layer; here it is an accurate cross-reference + figure reconciliation).

**3.3 [low] `kRelFloor` and `kAbsFloor` comments cross-describe each other's role.** — CONFIRMED against the test usage.

Issue: `kRelFloor = 1e-12` is documented as both "pairs whose reference |f2| is below this are scored by absolute error" *and* "the divide-by-zero guard in relative-error comparisons." `kAbsFloor = 1e-300` is "guard denominators … `max(|ref|, kAbsFloor)`."
