## Role & layering

This header is the single home of the f2 numerics: four scalar functions (`het_correction`, `f2_term`, `assemble_f2_numerator`, `finalize_f2`) plus three launch-grid helpers (`cdiv` ×2, `grid_for`). It is the literal embodiment of the architecture's most-emphasized DRY rule — §2 line 55 "one `cdiv()`… one `block_assignment()` rule… every domain rule shared between layers is implemented exactly once"; §5 S2 caveat (b) (line 225: "all of which live in the shared `__host__ __device__` feeder primitive so the oracle and the GEMM path cannot diverge on the formula"); §12 line 723; §13; ROADMAP §5; ROADMAP §4's magic-number table line 94. It lives in `core/internal` (host-pure C++20), is CUDA-free-compilable via the `STEPPE_HD` macro, and is consumed by exactly the right set: the CPU oracle (`cpu_backend.cpp:139,140,148,158,161,253,254,256,260,263`), both GPU feeder/assemble TUs (`f2_block_kernel.cu:111,156,157`; `f2_blocks_kernel.cu:155,159`), the host unit test (`test_f2.cpp`), and the GPU equivalence test (`test_f2_equivalence.cu:237,279,280,490,491,493,500`). Layering is clean: it includes only `steppe/config.hpp` (also CUDA-free), introduces no upward dependency, and `kHetCorrDenomFloor`/`kCdivBlock` resolve via enclosing-namespace lookup (`steppe::core` sees `steppe::`). The dependency-direction rule (§4) is respected; nothing here forces a CUDA include into `core`.

This is a small, mature, heavily-commented header that has clearly already been through the M0 cleanup pass (the spike's `/198.0`, `N=100`, `dim3(16,16)`, open-coded `(P+b-1)/b` are all gone and named). The bar here is therefore high, and most findings are low/medium severity. There is, however, one **real correctness/contract bug** (the GPU feeder *and the GPU equivalence test* call `het_correction` on *raw, unmasked* `Q`/`N` and are correct only by short-circuit-evaluation order — the pattern is replicated in three sites, raising the stakes), one **single-source-of-truth lie that should not survive** (the cited `launch_config.hpp` does not exist — independently confirmed in TODO §A as "`launch_config.hpp` is a phantom"), one **genuine DRY violation in the immediate neighborhood** that this header was meant to prevent (`decode_af_kernel.cu` re-rolls `cdiv`), and a cluster of edge-case / precision / const-correctness / API items. The review is long because the categories must be covered exhaustively, not because the header is bad — it is one of the better files in the tree.

## Score: 8.5/10 — strong, near-canonical primitive with a contract ambiguity, a false single-source claim, and a missed DRY home it was meant to own

A 9.5–10 header would (a) have its launch-config story actually be single-source (the `launch_config.hpp` it cites must exist, or the citation be corrected, **and** config.hpp:49–50's twin dangling reference fixed, **and** `decode_af_kernel.cu`'s `cdiv_l`/`cdiv_i` must not be a parallel re-derivation — all already on the pre-M4.5 backlog); (b) state, and make non-relocatable, the `het_correction` validity contract so the GPU feeder's `het_correction(Q_raw[idx], N_raw[idx], valid)` cannot read an out-of-contract `q`/`N`; (c) be `constexpr` throughout (the four numeric functions are not, blocking `static_assert` compile-time tests and matching the grid helpers); (d) document its NaN/Inf and integer-width edge behavior; (e) harden the wide same-type signatures against silent argument transposition AND give the `cdiv` overload set a mixed `(long,int)` form (or a single template) so its only mixed call site is not forced to cast around an ambiguity; and (f) carry the precision-tag-neutrality note the capability-tier work will want. ## Findings

### (1) Correctness & bugs

**1.1 — `het_correction` is called on out-of-contract operands in the GPU feeder AND the GPU equivalence test; the primitive's "returns 0 when invalid" promise hides a real precondition that holds only by evaluation order. [MED, effort S, before-M4.5: yes] — CONFIRMED, verified at three call sites.**
Location: `het_correction` lines 59–63, in light of `f2_block_kernel.cu:111`, `test_f2_equivalence.cu:237`, and the contrasting `cpu_backend.cpp:139,253`.
Verified: the GPU feeder (`f2_block_kernel.cu`) computes `const double q = valid ? Q_raw[idx] : 0.0;` (line 107) for the masked `Q_masked`/`S` outputs, but then passes the **raw** values to the correction: `const double hc = het_correction(Q_raw[idx], N_raw[idx], valid);` (line 111). The GPU equivalence test's host-side feeder does the *identical* thing — `const double q = valid ? praw : 0.0;` (line 236) for the mask, then `het_correction(praw, n, valid)` with the raw `praw`/`n` (line 237). The CPU oracle, by contrast, passes the *already-validated* `p_i = Q.element(i,s)` together with `valid=true` (`cpu_backend.cpp:139`, only reached after `if (!vi || !vj) continue;` at line 129). All three are equivalent **only** because `het_correction`'s first statement is `if (!valid) return 0.0;` (line 60), short-circuiting before `q`/`n` are read. For a missing entry the producer guarantees `V!=0 ⟺ N>0` (views.hpp:33), so `N_raw[idx]==0` and `Q_raw[idx]==0` (zero-filled, views.hpp:21) — but the function never looks. That equivalence is load-bearing for §13 parity and is **entirely implicit**: nothing in the signature or contract says "when `valid==false`, `q` and `n` are unconstrained and MUST NOT be read." If a future edit reorders the body for "readability" (e.g. hoists `const double denom = …` above the branch, or rewrites to `return valid ? q*(1-q)/denom : 0.0`), the two GPU paths would evaluate `q*(1-q)/max(N_raw-1,floor)` on raw missing data; with the current zero-fill it would *still* give `0*(1)/max(-1,1)=0` **by luck** (and only because of that zero-fill invariant), not by design. That the *same* mask-then-pass-raw idiom is duplicated across the feeder and the equivalence-test feeder makes the implicit precondition more, not less, dangerous: a single reorder would break parity on both the production path and the very test that is supposed to catch it. Why it matters: §5 S2 caveat (b, line 225) and §13 make "oracle and feeder cannot diverge on the formula" the entire purpose of this primitive; a precondition that survives only by statement-ordering is exactly the divergence the design forbids. Fix: (i) state the contract in the doc comment — "when `valid == false`, `q` and `n` are unconstrained and not read; the early return MUST remain the first statement"; (ii) keep the early return first with a `// do not relocate` note; (iii) optionally, behind a host-only `STEPPE_DEBUG` macro, `assert(!valid || (q >= 0.0 && q <= 1.0 && n >= 0.0))`. This remains the single most important substantive finding.

**1.2 — `finalize_f2` per-block divide composes with the S4 jackknife `Vpair` weight; the double-normalize hazard §5 caveat (a) warns of is now concrete in the M4 grouped path (`vpair_all` written alongside the divided f2), but nothing in or near this primitive encodes the compose-not-double-normalize invariant. [LOW→MED, effort M, before-M4.5: no] — CONFIRMED and sharpened.**
Location: `finalize_f2` lines 110–112; cross-ref `f2_blocks_kernel.cu:159–160` (`f2_all[dstSlab+…] = finalize_f2(num, vp);` immediately followed by `vpair_all[dstSlab+…] = vp;`) and `f2_block_kernel.cu:157`/`cpu_backend.cpp:161,263` (and `run_f2_gemms`:259–261 retains `Vpair` as "the S4 jackknife weight").
Verified: the M4 grouped assemble kernel writes the *already-per-block-mean* f2 into `f2_all` and the *same* `vp` into `vpair_all` in adjacent statements — so the S4 weighted jackknife will see both a normalized f2 and the weight that produced the normalization. The header comment (lines 107–109) correctly flags the hazard ("the per-block divide here and the S4 jackknife weighting by `vpair` must COMPOSE … not double-normalize"). The bug risk is not in `finalize_f2` (correct in isolation), but the file that is the single source of the f2 *result* offers no companion expressing "f2_blocks is already per-block-mean; S4 weights by Vpair, it does not re-divide." S4 is not built yet (Phase 2, TODO), so the hazard is live but unrealized. Why it matters: §12 / §5 S2 caveat (a, line 225) name this a parity caveat "pinned to a golden." Fix: when S4 lands, add a `f2_jackknife_weight(vpair)` / composition helper *here* so the rule lives in the same single-source file as the divide; for now the comment is the right amount of defense — flagging so it is not forgotten when S4 is wired.

**1.3 — `assemble_f2_numerator` on the diagonal (`i==j`) yields exactly `−2·hsum_i`, the `−2·mean_het` value three call sites rely on, but no test pins this algebraic identity. [LOW, effort S, before-M4.5: no] — CONFIRMED.**
Location: lines 97–101; consumers `cpu_backend.cpp:244–245` (walks `j=i..P-1`, including the diagonal — confirmed) with the explicit comment at `cpu_backend.cpp:232–238` ("the diagonal `f2(i,i) = -hc_i - hc_i ⇒ -2·mean_het`… it must not spuriously differ across paths"), and the GPU `assemble_f2_kernel`/`assemble_blocks_group_kernel` which compute every `(i,j)` including the diagonal. On the diagonal `sumsq_i==sumsq_j`, `cross==sumsq_i`, `hsum_i==hsum_j`, so `assemble_f2_numerator = 2·sumsq − 2·sumsq − 2·hsum = −2·hsum_i`. Correct, but a non-obvious identity that the oracle, the M0 GPU path, and the M4 grouped path all depend on and the diagonal-consistency check (cpu_backend comment) leans on. Fix: add a unit case `assemble_f2_numerator(a,a,a,h,h) == -2*h` (and as a `static_assert` once the function is `constexpr`, see 7.6) so a refactor of the GEMM index math can't silently break the (unconsumed but cross-path-checked) diagonal.

### (2) Edge cases & failure modes

**2.1 — NaN / Inf propagation is undocumented; `finalize_f2` silently masks a NaN numerator when `vpair<=0`; and the `het_correction` ternary suppresses NaN in `n` (but not `q`). [MED, effort S, before-M4.5: yes] — CONFIRMED.**
Location: all four numeric functions.
Verified by tracing: `het_correction`: `q==NaN ⇒ NaN`; `n==+Inf ⇒ denom==+Inf ⇒ result 0` (or NaN if `q` is NaN). The denominator ternary `(n - 1.0 > kHetCorrDenomFloor) ? (n - 1.0) : kHetCorrDenomFloor` *suppresses* a NaN `n` — `NaN > 1.0` is `false`, so a NaN `n` returns `kHetCorrDenomFloor` rather than propagating NaN. This happens to agree with `fmax(n-1.0, floor)` (which also returns `floor` on NaN), but an `std::max(n-1.0, floor)` would *not* (it returns the first arg, `NaN`, on a NaN compare) — so the choice is behaviorally load-bearing and undocumented. `f2_term`: `d*d` with `d==Inf ⇒ Inf`. `assemble_f2_numerator`: NaN in ⇒ NaN out. `finalize_f2(NaN, 0.0)` returns `0.0` — the `vpair>0.0` guard short-circuits before touching the numerator, so a NaN numerator at `vpair==0` is masked to 0 (arguably good: matches AT2's `Vpair==0 ⇒ 0`, but undocumented). Why it matters: §12 (line 723) says `EmulatedFp64` "is not IEEE-754-compliant on special values," and the emulated-GEMM outputs flow straight into `assemble_f2_numerator` — this primitive is the boundary where a special value would enter the native-FP64 numerator. Fix: one contract line per function stating special-value behavior (forward NaN/Inf; `het_correction` returns `floor`-denominated for NaN `n`; `finalize_f2` returns 0 when `vpair<=0` regardless of numerator). No code change required; this is a contract-completeness gap. (A unit case `finalize_f2(NaN, 0.0)==0.0` is cheap, see 10.1.)

**2.2 — `het_correction` accepts `n < 0` and `q` outside `[0,1]` silently. [LOW, effort S, before-M4.5: no] — CONFIRMED.**
Location: lines 59–63. `q` is documented "in `[0,1]`" and `n` a non-negative count, but neither is checked. With `n==0` (contract-violating valid entry; producer guarantees `V!=0 ⟺ N>0`, views.hpp:33, so it cannot happen on the happy path), `denom = max(-1,1) = 1`, returns `q(1-q)` (no div-by-zero — safe by the floor). With `q==2`, returns `2·(-1)/denom < 0`. Out-of-contract ⇒ "garbage in, garbage out" is defensible, but this is the shared oracle/feeder seam; a host-only debug assert (rolled into 1.1's) is cheap insurance. Fix: optional `STEPPE_DEBUG`-gated assert, or at minimum document that out-of-range `q`/`n` is unspecified.

**2.3 — `het_correction` denominator test uses strict `>` (`n-1.0 > kHetCorrDenomFloor`); at the `n==2` tie both branches return `floor`, so it matches AT2's `max(N-1,1)` exactly — and the ternary-vs-`std::max` choice is the *correct* `__host__ __device__` call. [LOW, effort S, before-M4.5: yes] — CONFIRMED.**
Location: line 61. AT2's convention is `max(N-1, 1)`. With `n==2`, `N-1==1`, `max(1,1)==1`; the ternary `(1.0 > 1.0) ? (1.0) : floor` selects `floor==1.0` — same value. With non-integer `n==2.5` (averaged ploidy; not expected but representable), `N-1==1.5 > 1.0` selects `1.5 == max(1.5,1)`. So strict `>` is numerically identical to `max` because both branches equal `floor` at the tie. Correct, but it reads like a possible bug on review. Note also: an open-coded ternary in a `__host__ __device__` header is the *right* choice, not a smell — `std::max(n-1.0, 1.0)` (what the spike used, per the file header) is not reliably `__device__`-callable without `--expt-relaxed-constexpr` (it is a `__host__`-only `constexpr` in libstdc++), and `fmax` would change NaN behavior (see 2.1). Keep the ternary; add a one-line comment: "`>` vs `>=` is immaterial — both branches equal `floor` at the `n==2` tie; ternary (not `std::max`) keeps it device-callable without relaxed-constexpr."

**2.4 — `cdiv(int,int)` is signed-overflow UB on large `n`, and divide-by-zero UB on the documented-but-unenforced `b>0` precondition; the recommended `n/b+(n%b!=0)` rewrite is overflow-free but NOT equivalent for negative `n`. [MED, effort S, before-M4.5: yes] — CONFIRMED, with a correctness caveat on the proposed fix.**
Location: `cdiv(int,int)` lines 125–127. `(n + b - 1)` overflows `int` when `n > INT_MAX - b + 1`; signed integer overflow is **undefined behavior** in C++20 ([expr]; SEI CERT INT32-C). Verified the call-site reachability against the grep map: the `int` overload is only ever reached via `grid_for(int)` with `P` (≤ ~4,266 per ROADMAP §0), `s_pad` (a power-of-two-padded per-group SNP width, ~thousands), and `s` (a block width) — `P+15`, `s_pad+15` never overflow; the SNP axis `M` (which the comment notes "can exceed 2^31") is correctly routed through the `cdiv(long,long)` overload (`f2_block_kernel.cu:231`, verified). So **no live overflow today**, but `grid_for(int n, int block=kCdivBlock)` (line 138) is public and a future caller passing `n` near `INT_MAX` gets UB; and `cdiv(n,0)` (the unenforced `b>0` precondition, line 124) is a div-by-zero (UB) — the unit test (`test_f2.cpp:108–110`) covers `cdiv(10,4)`,`(8,4)`,`(0,4)`,`(1,4)` but neither edge. Why it matters: §2 fail-fast, §13 testability. Fix: prefer `return n / b + (n % b != 0);` (constexpr-foldable, no `n+b-1`), **but the recommendation must note**: this rewrite (a) divides-by-zero identically on `b==0` so it still needs the `b>0` assert/precondition, and (b) is **not** equivalent to `(n+b-1)/b` for negative `n` — e.g. `cdiv(-1,4)`: old `(-1+4-1)/4 = 2/4 = 0`, new `-1/4 + (-1%4 != 0) = 0 + 1 = 1`. Since `n` is always a non-negative count/dimension here this never bites, but the asymmetry should be stated so the "fix" is not mis-applied somewhere `n` can be negative. Minimum: add a `STEPPE_DEBUG` assert `b > 0` and document the non-negative-`n`, no-overflow precondition; back the `grid_for` default with a `static_assert(kCdivBlock > 0)` (in config.hpp or here) so the default-argument divisor is compile-time-guaranteed positive. (Mirror whatever is chosen into `decode_af_kernel.cu`'s `cdiv_l`/`cdiv_i` — see 4.3 — which carry the *same* `n+b-1` form; fixing one and not the other is precisely what DRY prevents.)

**2.5 — `grid_for(n, block)` is the two-arg form; §4/§8 specify `grid_for(n, block, max_grid)` with a grid-dimension clamp, and the M4 grouped path sets `grid.z = n_in_group` directly (no `grid_for`, no clamp). [LOW, effort S, before-M4.5: no] — CONFIRMED; the unclamped z-axis is real but not live.**
Location: `grid_for` lines 138–140 (two-arg, no clamp); cross-ref `f2_blocks_kernel.cu:178,233` (`grid.z = static_cast<unsigned>(n_in_group)` — the raw per-group block count, NOT through `grid_for`) and `assemble_blocks_group_kernel`/`gather_group_kernel` which use `blockIdx.z` (`f2_blocks_kernel.cu:79,136`) as the slab index. Verified against the CUDA C++ Programming Guide: max grid x-dimension is 2³¹−1, but **max y- and z-dimension is 65535** for every compute capability ≥ 3.0 including Blackwell sm_120. For the [P×P] / [P×s_pad] x/y launches `grid_for` is fine (extents bounded by P≈4,266 and block count). For the z-axis, `n_in_group` is the number of jackknife blocks in one power-of-two padding group — on real AADR ~757 blocks across ~10 groups, so `n_in_group ≲ 757 ≪ 65535`: **no live failure**, but a pathological dataset with > 65,535 blocks in a single group would silently produce a bad launch (caught only by the post-launch `cudaGetLastError`). This exact item is already on the backlog (TODO §A "clamp `grid.z` to 65535"). Fix: add the `max_grid`-clamped three-arg `grid_for` overload that §4 line 466 / §8 line 525 name, route the z-axis through it (or document why the clamp is intentionally omitted because extents are bounded by block count). Aligns the helper with its §8 contract and closes the latent z-overflow.

### (3) Numerical / precision vs §12

**3.1 — `assemble_f2_numerator` is the catastrophic-cancellation site and the FP64-always guarantee is upheld by the `double` type AND the caller's `double num` declaration. [GOOD — noting the mechanism] — CONFIRMED.**
Location: lines 93–101. The doc is excellent: names the cancellation, states FP64-always, calls out the `2` as a true math constant (§4). Per §12 (line 225/723: "The small `O(n_pop²)` numerator/divide step stays native FP64") this is correct, and the guarantee is realized by the caller declaring `double num` (`f2_block_kernel.cu:155`, `f2_blocks_kernel.cu:154`, `test_f2_equivalence.cu:279`) plus the `double` signature here — there is no path that does the *arithmetic* in EmulatedFp64. (Mild precision: the type forces the assemble *arithmetic* to FP64; it does not stop a caller from `static_cast`-ing a lower-precision input into the slot — but no caller does, and the emulated GEMM outputs are themselves stored FP64 per §12.) This is a *good* pattern (type-level enforcement of a precision rule), not a bug. No change.

**3.2 — `f2_term` (cancellation-free `(p_i−p_j)²`) and `assemble_f2_numerator` (expanded `Σp²−2Σpq+Σq²`) are algebraically equal but numerically different, and the equivalence the design rests on is tested only on 3 benign SNPs at `1e-12`, never in the cancellation regime. [MED, effort M, before-M4.5: yes] — CONFIRMED, the single most important untested property.**
Location: `f2_term` 76–80 vs `assemble_f2_numerator` 97–101; test `test_f2.cpp:76–94`.
Verified: the oracle (`cpu_backend.cpp:255–260`, `test_f2_equivalence.cu:493–500`) sums `f2_term` in `long double` via `pairwise_sum` (cancellation-free); the GPU builds the expanded form via GEMMs (cancellation lands on the small reduced matrix). The header's whole justification (§12; lines 65–75, 93–96) is that these agree, and `test_f2.cpp`'s own comment (line 75) calls `test_assemble_matches_term_sum` "the load-bearing equivalence the GPU/CPU agreement rests on." But that test uses `pi={0.7,0.3,0.9}` etc. at `kEps=1e-12` — well-conditioned, no cancellation. It does **not** touch the regime that motivates the entire `f2_term`-vs-expanded split: `p_i ≈ p_j ≈ 0.9999999` over many SNPs, where `Σp_i²+Σp_j²−2Σp_i p_j` is a tiny difference of large sums and the expanded `double` form loses bits the cancellation-free form does not. Why it matters: §13 makes this the central trust seam; testing it only on benign data is false confidence. Fix: add a near-degenerate `p_i≈p_j` block and assert the expanded-form-in-`double` error vs the `long double` `f2_term` sum stays within the documented `2.2e-11`-class bound — OR explicitly document that the host `double` expanded form is *allowed* to diverge there and that the real gate is the GPU-vs-oracle test on real AADR (per §12 / ROADMAP §6, which currently makes the CPU-oracle + internal-consistency the operative gate while AT2 goldens are deferred). Either way the header's claim should be backed by a test that exercises the cancellation.

**3.3 — Determinism: these are deterministic pure functions with fixed expression order and no reduction; §12's reduction-order/`run_to_run` concerns live in the callers (the GEMM, the CUB reduce, the long-double `pairwise_sum`), not here. [N/A — correctly so] — CONFIRMED.**
The order-dependence §12 worries about is in the accumulation (`f2_block_kernel`'s GEMM, `cpu_backend.cpp`'s `pairwise_sum`), not in these scalar expressions. Except for the host/device FMA caveat in 3.4, each function evaluates its inputs in a fixed source order.

**3.4 — Host/device FMA-contraction: `f2_term` (`d*d - hc_i - hc_j`) and `assemble_f2_numerator` (`… - 2.0*cross - …`) may contract differently on device vs host, so "the SAME function compiles and runs on CPU and GPU" is true at the source level, not bitwise. [LOW, effort S, before-M4.5: no] — CONFIRMED against the NVCC docs.**
Location: 78–79, 100. Verified: `--fmad=true` is the nvcc **default** (CUDA Compiler Driver NVCC 13.0: "this option is set to `true` and `nvcc` enables the contraction of floating-point multiplies and adds/subtracts into floating-point multiply-add operations (FMAD, FFMA, or DFMA)"), so device codegen may contract `a*b+c` into a single FMA, changing the last bit vs the host's separate multiply-then-add; `assemble_f2_numerator`'s `-2.0*cross` followed by subtractions can fuse differently on device than host. Both passes missed this.

## Considered & rejected

- **"`STEPPE_HD constexpr` on `cdiv`/`grid_for` needs `--expt-relaxed-constexpr` to be host-callable."** Rejected (re-verified against NVCC docs). These are `__host__ __device__ constexpr` (both spaces), so a host call site binds the host instantiation and a device call site the device one; `--expt-relaxed-constexpr` is needed only to call a `__device__`-*only* constexpr from host (or a `__host__`-only constexpr from device). With both specifiers present, both call sites are well-formed without the flag. (Source: NVIDIA CUDA Compiler Driver — `--expt-relaxed-constexpr` "allows host code to invoke `__device__` constexpr functions, and device code to invoke `__host__` constexpr functions," i.e. cross-space calls only.) This also underpins the 7.6 recommendation that the numeric functions can safely become `STEPPE_HD constexpr`. Not a finding.

- **"`kHetCorrDenomFloor`/`kCdivBlock` used unqualified in `steppe::core` but defined in `steppe` — name-lookup bug."** Rejected. Verified: config.hpp is a single `namespace steppe`; this file is `namespace steppe::core`. Unqualified lookup from `steppe::core` searches enclosing namespaces and finds `steppe::kCdivBlock`/`steppe::kHetCorrDenomFloor`. Correct and idiomatic.

- **"`finalize_f2` `vpair > 0.0` should be `>= 1.0` since vpair is an integer count."** Rejected as a change. The header comment and ROADMAP §4 keep `> 0.0` for exactness ("`>= 1.0` is equivalent (cosmetic)"). `vpair` is an exact integer carried as `double` up to 2⁵³, so the two are identical on the valid domain. No bug. (The `> 0.0` form additionally documents the `Vpair==0 ⇒ 0` AT2 branch verbatim.)

- **"The `2 * Pl` / `2 * Pp` / `twoP` index math for the stacked S/R matrices could overflow."** Rejected for *this* header — that arithmetic lives in the kernels (`f2_block_kernel.cu:101,144,249`; `f2_blocks_kernel.cu:83–85,140,197–199`), which use `long`/`size_t` (`twoP = 2 * Pp` with `Pp` a `size_t`, and `twoP = 2 * Pl` with `Pl` a `long` — verified). Out of scope for `f2_estimator.hpp`; handled correctly at the call sites. The only in-header overflow concern is `cdiv(int,int)` (2.4).

- **"`het_correction` should take `N` as an integer count, not `double`."** Rejected. The Q/V/N contract (views.hpp:27–31, decode_af.hpp `AfResult::n` is `double`) stores N as `double` (it is `ploidy*an`, and the masked-GEMM path needs it FP), and `n-1.0` wants a double anyway. `double` is consistent with the contract and with `finalize_f2(double vpair)`. Documenting that non-integer N is tolerated (2.3) is the only worthwhile note. Not a finding.

- **"The diagonal f2(i,i) computed by these primitives is wrong / should be skipped."** Rejected. `cpu_backend.cpp:232–238` and the GPU kernels deliberately fill the diagonal with `-2·mean_het`; the comment explains it is never consumed downstream (f3/f4 read off-diagonal) but must be *consistent across paths*. `assemble_f2_numerator` on the diagonal yields exactly that (1.3). Intentional and correct; the only ask is the unit test in 1.3.

- **"`het_correction` lacks `if (n <= 0) return 0` so a valid entry with N=0 divides oddly."** Rejected as a bug (kept as the soft 2.2). The producer guarantees `V!=0 ⟺ N>0` (views.hpp:33), so `valid==true, n==0` is contract-violating; even then `max(n-1,floor)=max(-1,1)=1`, so the divide is safe (no div-by-zero). Hardening is optional.

- **"Header should `#undef STEPPE_HD` at the end."** Considered, folded into 4.1 — a bare `#undef` here is wrong because `decode_af.hpp` *also* needs `STEPPE_HD` and the two are co-included; the right fix is the shared `host_device.hpp` (TODO §A). Not a standalone finding.

- **"The `2.0 * cross` in `assemble_f2_numerator` should be `cross + cross` to avoid an FP multiply / match the host."** Rejected. `2.0*x` and `x+x` are bit-identical in IEEE-754 (multiply by an exact power of two is exact, no rounding), and `2.0*cross` reads as the algebraic `2ab` of `a²−2ab+b²`. No accuracy or determinism difference; the FMA-contraction concern (3.4) is about `2.0*cross` fusing with the *surrounding* subtraction, which `cross+cross` would not change. Not a finding.

- **"`finalize_f2` should use `std::isfinite(numerator)` to avoid emitting NaN/Inf."** Rejected as a change. The AT2-parity contract is `Vpair==0 ⇒ 0`, numerator-mean otherwise; AT2 does not special-case a non-finite numerator, and adding an `isfinite` clamp would *diverge* from the oracle (which also has no such clamp — `cpu_backend.cpp:263`). A non-finite numerator at `vpair>0` signals an upstream bug that should surface, not be masked. Documenting the special-value behavior (2.1) is the right response, not clamping. Not a finding.

- **"`assemble_f2_numerator` enforces FP64 by its `double` type, so it can never run in lower precision (a hard guarantee)."** Re-examined and *tempered, not rejected* (kept as 3.1, reworded). The type forces the assemble *arithmetic* into FP64, which is the §12 guarantee that matters; it does not stop a caller from `static_cast`-ing a `float`/emulated value into a `double` slot. No caller does, and the emulated-GEMM outputs are FP64-stored per §12, so the guarantee holds in practice — but it is a type-level enforcement of the *operation* precision, not an absolute proof about input provenance. Worth stating precisely; not a defect.

## Good patterns to keep

- **The primitive itself is the realization of §2/§5-S2-caveat-(b)/§13's central thesis:** one `__host__ __device__` formula consumed by the oracle (`cpu_backend.cpp`), both GPU paths (`f2_block_kernel.cu`, `f2_blocks_kernel.cu`), and both tests — so they "cannot diverge on the formula." This is the project's most important DRY win and the standard for every future kernel. Do not let it erode (4.1/4.3 are early erosion already caught on the backlog).
- **The cancellation-free `f2_term` vs expanded `assemble_f2_numerator` split,** with the cancellation localized to a function whose `double` type *enforces* the §12 FP64-arithmetic rule for the numerator step. Type-level enforcement of a numerical-policy invariant is elegant and rare.
- **`2.0` correctly identified as a true mathematical constant** and every other literal sourced from `config.hpp` — the §4 / ROADMAP §4 standard fully met, with each constant carrying its arch/ROADMAP cite + measured rationale.
- **`[[nodiscard]] + noexcept` discipline** applied uniformly and correctly, including device-`noexcept` per §7 (and `constexpr` on the grid helpers — extend it to the rest per 7.6).
- **Provenance-rich comments** (spike origin, arch §refs, the AT2 conventions pinned, the *why* of `STEPPE_HD`) — heavy but justified for a parity-critical primitive and consistent with the sibling headers; just prune the spike line numbers when the spike dies.
- **The dual gtest / self-checking-main test harness** and the explicit "these are per-element numerics, NOT the hot-path structure" framing (§7) — keeps reviewers from mistaking the primitive for the production control flow.
