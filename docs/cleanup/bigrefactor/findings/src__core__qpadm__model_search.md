- [5.3][HIGH] src/core/qpadm/model_search.cpp:73 — the small-path envelope `nl <= 5 && nr <= 10 && r <= 4` is a bare-literal THIRD copy of a constant that is authoritatively NAMED elsewhere as `kQpMaxNl=5 / kQpMaxNr=10 / kQpMaxR=4` (src/device/cuda/qpadm_fit_kernels.cu:41-43), where those values SIZE the kernel's per-thread local arrays (e.g. `xmat[kQpMaxM]`, `A[kQpMaxT]`, `RHS[kQpMaxNl*kQpMaxNl]`, qpadm_fit_kernels.cu:521-569, 930-934, 1112-1169). ## Group 7 — Duplication

- [7.1][MED] src/core/qpadm/model_search.cpp:221-227, 261-267 — the "validate model_index then scatter into pre-sized slot" block is copy-pasted between the G==1 fast path and the G>=2 worker lambda, differing only by the runtime_error message string ("...(must be 0..n-1 for the deterministic re-sort)" vs the terse "...out of range"). Drift risk: the deterministic-re-sort invariant (the §determinism gate) is enforced in two/three places that can diverge. Suggested: extract a single `scatter_into_slots(std::vector<QpAdmResult>& results, std::vector<QpAdmResult>& batch, std::size_t n)` (file-local helper) and call it from both run paths so the bounds-check + slot-write rule has one definition.

- [7.2][LOW] src/core/qpadm/model_search.cpp:31, 86, 294 — the identical loop-invariant config expression `const Precision prec{Precision::Kind::EmulatedFp64, steppe::kDefaultMantissaBits}` is constructed three times across three functions (fit_one_model_device, fit_shard, the host-oracle run_qpadm_search). It is the documented single "unified default precision" (the f2 default per fit-engine.md §1.4), so the value is conceptually ONE constant repeated, not three independent choices. FP64-by-design is intentional and not flagged; the duplication of the *construction* is. Suggested: name it once (e.g. a file-local `constexpr`/`inline const Precision kFitDefaultPrecision{...}` in this TU, or a shared accessor) so the three sites reference one source of truth.

- [7.2][MED] src/core/qpadm/model_search.cpp:175-177, 182-183, 185-193 — `replicate_f2` walks the device list THREE times recomputing the same predicate `resources.gpus[g].device_id != f2.device_id`: loop 1 sets `need_host` (any non-resident device), loop 2 counts `n_owned` for the reserve, loop 3 does the actual upload/borrow. Loops 2 and 3 are pure duplication of the same per-g classification (the count loop re-derives exactly what loop 3 then re-tests). G is tiny so this is not a perf bug, but it is three copies of one invariant that can drift (e.g. someone changes the residency test in one loop only). Suggested: compute the per-g "needs upload" classification once (or fold the reserve into a single pre-pass that also yields `need_host`), then drive the upload loop from that — one expression of the residency rule.

- [7.4][LOW] src/core/qpadm/model_search.cpp:111-112, 117-118 — the two "scatter sub-list results back into `out` by saved position" loops are structurally identical, differing only by the (sr,small_pos) vs (lr,large_pos) pair and both guarding with the same `k < X.size() && k < pos.size()` belt-and-suspenders bound: `for (k...) out[pos[k]] = std::move(results[k]);`. Suggested: a small `scatter_by_pos(out, results, pos)` helper folds both calls and keeps the (defensive) double-bound in one place.

- [7.4][LOW] src/core/qpadm/model_search.cpp:39-44 (fit_one_model_device) vs 297-303 (host-oracle run_qpadm_search) — the per-model fit chain `assemble_f4(left_with_target → right, prec) → run_impl(be, X, block_sizes, model, opts) → res.model_index = model.model_index` is duplicated: fit_one_model_device runs it over a `DeviceF2Blocks`, and the host-oracle overload re-inlines the same three steps over an `F2BlockTensor`. Suggested: if assemble_f4 is overloaded for both f2 types on the backend, template/overload a single `fit_one_model(be, f2_source, model, opts, prec)` so the host oracle and device path share the one fit-chain definition; otherwise leave as-is (the type split is a real constraint) and treat this as documentation of the parallel structure.
