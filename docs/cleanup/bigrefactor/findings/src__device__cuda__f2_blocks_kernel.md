## Group 7 — Duplication

- [7.2][LOW] f2_blocks_kernel.cu:248,255,263 — the per-slab C-matrix stride for the [P×P] GEMM outputs is recomputed inline three times as `static_cast<long>(P) * P` (cu:248 Gg, cu:255 Vpairg) — and the R-output stride `static_cast<long>(twoP) * P` once (cu:263), with the gather-slab strides `Psp`/`twoPsp` already hoisted to named locals just above (cu:240-241). The two [P×P] output-stride expressions are byte-identical loop-invariants left un-hoisted while their gather-slab siblings were hoisted, so the hoisting is half-done within one routine. Suggested: hoist `const long PP = static_cast<long>(P) * P;` (and a `twoPP`) alongside the existing `Psp`/`twoPsp`, and pass the named strides to all three `cublasGemmStridedBatchedEx` calls.
- [7.4][LOW] f2_blocks_kernel.cu:196-203 vs 279-285 — the two square-tile launch wrappers `launch_gather_group` and `launch_assemble_blocks_group` repeat the same launch boilerplate: `dim3 block(kCdivBlock, kCdivBlock)` + `dim3 grid(static_cast<unsigned>(grid_for(...)), ..., grid_z_extent(n_in_group))` + `kernel<<<grid,block,0,stream>>>(...)` + `STEPPE_CUDA_CHECK_KERNEL()`, differing only in the grid.y extent (`grid_for(s_pad)` vs `grid_for(P)`) and the kernel/arg list. Note this 3D `grid(grid_for, grid_for|cdiv, grid_z_extent)` idiom also recurs in sibling TUs (f2_block_kernel.cu:380, decode_af_kernel.cu:106), so the fold is arguably a project-wide launch-helper, not a local one. Suggested (if pursued): a small `launch_pp_batched(grid_x_extent, grid_y_extent, n_in_group, stream, lambda)` or a shared `make_batched_grid(nx, ny, nz)` helper in launch_config.hpp, folding the cast+`grid_for`+`grid_z_extent` assembly once.
- [7.3][LOW] f2_blocks_kernel.cu:84-87 vs 141-147 — the widened-P scratch family is duplicated across the two kernels with the only material difference being the cast target: gather computes `Pl`/`twoP`/`Psp`/`twoPsp` in `long` (cu:84-87) and assemble recomputes the parallel `Pp`/`twoP` in `size_t` (cu:141-142) plus the slab bases `gSlab`/`rSlab` (cu:146-147). The widen-then-multiply-by-2-and-by-stride shape is the same; the divergent integer type (long vs size_t) is the only constant that differs — a 7.1-class copy differing by a type/constant. Borderline because the two index spaces legitimately differ (feeder/gather extents vs the P·P·n_block resident tensor that needs size_t per the §-scale 2^31 note), so a forced merge could mask the size_t-correctness intent. Suggested: leave the index types distinct, but a tiny shared `inline __device__` helper for the column-major slab base (e.g. `slab_base(rows, cols, k)`) would fold the repeated `rows*cols*k` pattern without conflating the two index widths.

## Group 20 — Performance: memory access

- [20.1][MED] f2_blocks_kernel.cu:152,154 — in assemble_blocks_group_kernel the TRANSPOSED Rg reads `sumsq_j = Rg[rSlab + sj + si*twoP]` and `hsum_j = Rg[rSlab + (Pp+sj) + si*twoP]` index with the fast thread axis `si` (= threadIdx.x-derived row `i`) multiplied by the column stride `twoP`, so consecutive threads in a warp hit addresses `twoP` (= 2P) elements apart — a fully uncoalesced, one-cache-line-per-thread strided load (vs the coalesced `sumsq_i`/`hsum_i` at cu:151,153 where `si` is the unit-stride term). - 22.1 (atomics where a reduction/scan is cheaper): NO atomics anywhere in either file (grep for `atomic`/`atomicAdd`/`atomicCAS` returns zero matches). - 22.2 (integer div/mod in loops): NONE. No finding (and per the task's "only where the profiler confirms it" gate, no graph change is warranted here without a measured win at the caller).
