## Group 5 — Hardcoded values / magic numbers

- [5.3][MED] block_sink.cu:199 — the on-disk format `version` is written as the bare literal `h.version = 1u;` with no shared named constant. f2_disk_format.hpp documents the value only in a comment (`version; // 1`, line 28) and provides `kF2DiskMagic`/`kF2DiskDtypeFp64` as named constants but NOT a version. The future M7 reader will independently hardcode/compare `1`, so a format bump must be edited in two unsynchronized places — a drift hazard for a binary cache contract. Suggested: add `inline constexpr std::uint32_t kF2DiskVersion = 1u;` to f2_disk_format.hpp and write `h.version = kF2DiskVersion;` here (matches how dtype is already handled).
- [5.1][LOW] block_sink.cu:192 — file mode `0644` is an unnamed octal literal in `::open(path_.c_str(), O_RDWR | O_CREAT | O_TRUNC, 0644)`. Magic permission bits. Suggested: name it (e.g. `kCacheFileMode = 0644`) or build it from `S_IRUSR|S_IWUSR|S_IRGRP|S_IROTH` for readability.
- [5.1][LOW] block_sink.cu:189 — `f2_region_ = sizeof(F2DiskHeader); // == 64`. The assignment itself is correctly derived via `sizeof` (no drift), but the trailing `== 64` comment restates the magic size; the canonical 64 is already asserted in f2_disk_format.hpp:37. Comment-only nit. Suggested: leave the `sizeof` as-is; the inline `== 64` comment is redundant with the static_assert and can be dropped to avoid an out-of-date number if the header ever grows.

## Group 6 — Naming

- [6.2][LOW] block_sink.cuh:140-141, block_sink.cu:189-190 — DiskSink members `f2_region_`/`vpair_region_` are byte OFFSETS (where each region begins in the file), typed `std::uint64_t` and used as the base of `pwrite_all(... f2_region_ + slab_bytes_*block ...)`. The name says "region" (a span/extent) but the value is an offset; the on-disk header field for the very same quantity is correctly named `f2_offset`/`vpair_offset` (used at block_sink.cu:203-204 `h.f2_offset = f2_region_;`). So one quantity has two names across the boundary — "region" here, "offset" in the header. Mildly misleading. Suggested: rename the members `f2_region_offset_`/`vpair_region_offset_` (or just `f2_offset_`/`vpair_offset_`) to match the header field and reflect that they hold an offset, not a length.
- [6.2][LOW] block_sink.cuh:104,149; block_sink.cu:55,69-74,101,210,225-230,262 — the slot-availability array is `std::vector<bool> free_`, a bare adjective that reads like a single boolean or an action, but it is a per-slot flag VECTOR (`free_[i] == true` means slot i is claimable). The sibling containers are noun-named (`slots_`, `ready_`), so `free_` is the odd one out and at the lambda use sites (`if (free_[i])`) it scans like a predicate rather than state. Suggested: rename to `slot_free_` or `free_flags_` to read as a per-slot state array and match the `slots_`/`ready_` noun convention.

## Group 7 — Duplication

- [7.1][MED] block_sink.cu:66-76 vs 222-232 — `HostRamSink::acquire_slot()` and `DiskSink::acquire_slot()` are byte-for-byte identical (same lock, same `cv_free_.wait` scan over `free_`, same claim+return). Pure copy-paste differing in nothing. Suggested: pull the ring (`slots_`/`free_`/`ready_`/mutex/CVs/`stop_` + `acquire_slot`) into a shared `StagingRing` member or a CRTP base so the two sinks share one definition.
- [7.1][MED] block_sink.cu:78-87 vs 234-243 — the writer-thread queue-pop preamble (`for(;;)` + `cv_work_.wait(stop_||!ready_.empty())` + the `if (ready_.empty() && stop_) return;` + `idx = ready_.front(); ready_.pop();`) is identical in both `writer_loop`s; only the per-slot drain action (memcpy vs pwrite+try/catch) and the free-return tail (also identical, lines 99-104 vs 260-265) differ. Suggested: factor the loop skeleton into a shared helper taking a `drain(SinkSlot&)` callback (or the StagingRing above), leaving only the tier-specific drain body in each sink.
- [7.1][MED] block_sink.cu:107-125 vs 268-285 — `spill_block` is structurally identical in both sinks (guard, `acquire_slot`, set `s.block`, two `cudaMemcpyAsync` D2H, `cudaEventRecord`, push to `ready_`, `cv_work_.notify_one`); the ONLY difference is the byte count (`bytes = slab_*sizeof(double)` at line 114 vs the precomputed member `slab_bytes_` at 275-277). Suggested: with a shared ring/base this becomes one method; even without, the body is a clear extract candidate parameterized by the byte count.
- [7.1][MED] block_sink.cu:128-135 & 138-142 (HostRamSink finish/dtor) vs 289-293 & 329-333 (DiskSink finish/dtor) — the writer stop-and-join sequence (`{lock; stop_=true;} cv_work_.notify_all(); writer_.join();`) is copy-pasted FOUR times across the two finish()es and two destructors, with a `writer_.joinable()` guard variant. Suggested: a single `stop_writer()` private helper (or RAII member) called from all four sites.
- [7.1][MED] block_sink.cu:143-150 vs 334-341 — the event-destroy teardown loop (`for (SinkSlot& s : slots_) { if (s.done) { cudaEventDestroy(...); warn-on-error; s.done=nullptr; } }`) is identical in both destructors, differing only in the warn message tag ("HostRamSink" vs "DiskSink"). Suggested: move this into a shared helper or into a `SinkSlot`/ring teardown so the event lifetime is owned in one place.
- [7.1][LOW] block_sink.cu:53-63 vs 209-219 — the begin()-side ring setup (`slots_.resize(kStreamStagingSlots); free_.assign(kStreamStagingSlots,true);` then the per-slot `PinnedBuffer<double>(slab_)` ×2 + `cudaEventCreateWithFlags(...,cudaEventDisableTiming)` + `s.block=-1`, then `stop_=false; writer_=std::thread(&...::writer_loop,this);`) is duplicated; Disk adds only `writer_failed_=false`. Suggested: same StagingRing/base extraction folds this init too.
- [7.2][LOW] block_sink.cu:47,49 / 188,202 / 299,321 — the clamp `(n_block < 0 ? 0 : n_block)` is recomputed at six sites (HostRamSink.begin twice, DiskSink.begin twice, DiskSink.finish, DiskSink.take_descriptor). Suggested: clamp once into a local/member at begin() (e.g. store the non-negative block count) and reuse it.
- [7.2][LOW] block_sink.cu:95-98 — in the HostRam writer the destination base `slab_ * static_cast<std::size_t>(s.block)` is computed twice (once for f2, once for vpair) within the same memcpy pair. Loop-invariant within the call. Suggested: hoist `const std::size_t dst = slab_ * static_cast<std::size_t>(s.block);` and index both memcpys off it (mirrors how DiskSink already precomputes `slab_bytes_`).
- [7.3][LOW] block_sink.cu:114 vs 186/275-277 — `slab_ * sizeof(double)` is computed inline as a local `bytes` in HostRamSink::spill_block, while DiskSink already caches the same quantity as the member `slab_bytes_` (line 186). Inconsistent: one tier hoists the sizeof product, the other recomputes it. Suggested: give HostRamSink the same `slab_bytes_` member (or the shared ring) so the `slab_*sizeof(double)` product is expressed once per sink.

## Group 8 — Comments

- [8.3][MED] block_sink.cu:89-94 vs 245-249 — the two writer threads handle a `cudaEventSynchronize(s.done)` failure ASYMMETRICALLY with no rationale: HostRamSink only `STEPPE_LOG_WARN`s and then proceeds to memcpy a slot whose D2H may NOT be drained (potentially copying stale/partial bytes into host_.f2/vpair — a silent parity corruption per §12), whereas DiskSink treats the same failure as fatal (throws, recorded in writer_failed_, re-thrown at finish()). An intentional deviation between the two tiers' failure semantics on the same operation should be documented (or made consistent). Suggested: add a comment at HostRamSink:92 explaining why HostRam tolerates an event-sync failure and continues the copy (and ideally why it is safe), or align it with DiskSink's fail-fast.
- [8.3][LOW] block_sink.cu:59 / 214 — `cudaEventCreateWithFlags(&s.done, cudaEventDisableTiming)` uses the `cudaEventDisableTiming` flag at both sink begin()s with no comment on the non-obvious choice (events used only for happens-before sync, never elapsed-time queries — disabling timing is the lighter-weight path). Minor missing rationale on a non-default flag. Suggested: one-line note that these events are sync-only, hence timing disabled.
