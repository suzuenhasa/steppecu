## Group 7 — Duplication

- [7.1][MED] src/device/cpu/cpu_backend.cpp:152-201 vs 271-300 — the entire per-pair f2 oracle body is copy-pasted between `compute_f2` and `compute_f2_blocks`, differing only by the SNP range (`0..M` vs `s0..s1`) and the slab `base` offset added to the output index. Both run the identical inner sequence: validity check (V.element!=0 for i and j), `het_correction` for i/j, `f2_term` into `terms[count]`, `++count`, `pairwise_sum` → `finalize_f2`, then the four mirror-writes. A formula/index drift fixed in one would silently diverge the oracle from the per-block oracle (the very "cannot diverge on the formula" property the header @12-21 promises). Suggested: extract a `static void f2_pair_over_range(Q,V,N, i,j, s0,s1, terms, &out.f2, &out.vpair, base, P)` (or a small `compute_one_pair` returning {f2_ij, vpair}) and call it from both loops.
- [7.1][HIGH] src/device/cpu/cpu_backend.cpp:787-845 vs 850-904 — `opt_A` and `opt_B` are near-identical 58-line copies of the AT2 GLS-ridge solve: build `xvec` (c(t(xmat))), form `W = qinv · linop` (m×t), form `coeffs = linopᵀ·W` and `rhs = xvecᵀ·W`, ridge the diagonal (`tr`+`fudge*tr`), `core::solve`, then reshape the solution row-major into the output factor. They differ ONLY by (a) the Kronecker linear operator lambda (`B2(a,k)` = I⊗B vs `A2(k,c)` = A⊗I) and its index arithmetic, (b) `t = nl*r` vs `r*nr`, and (c) the final reshape (`A2[i*r+p]→A(i,p)` vs `B2[p*nr+j]→B(p,j)`). This is the largest duplication in the TU and is parity-load-bearing (the ridge/solve/reshape must stay bit-identical across both); a fix or precision tweak to one half can drift from the other. Suggested: extract a shared `als_ridge_solve(linop_lambda, m, t, xvec, qinv, fudge)` returning the solved vector (the W/coeffs/rhs/ridge/solve core), leaving only the operator lambda and the reshape per-caller.
- [7.2][LOW] src/device/cpu/cpu_backend.cpp:804-808 vs 863-867 — the `xvec` build (`for i,j: xvec[i*nr+j] = xmat(i,j)`, the c(t(xmat)) row-major flatten) is byte-identical in `opt_A` and `opt_B`. Suggested: fold into the shared `als_ridge_solve` extraction above, or a tiny `flatten_xmat_rowmajor(xmat, nl, nr)` helper (also reusable in `chisq_of`'s residual flatten @983, which builds the same k = i*nr+j ordering).
- [7.4][LOW] src/device/cpu/cpu_backend.cpp:480-484, 832-834, 891-893 — the AT2 ridge-on-diagonal idiom is hand-repeated three times: `double tr=0; for k: tr += M[k+dim*k]; for k: M[k+dim*k] += fudge*tr;` (jackknife_cov on Qf, opt_A on coeffs, opt_B on coeffs). - [7.3][LOW] src/device/cpu/cpu_backend.cpp:193-196, 291-294 — the mirror-write column-major index pair (`ij = base + i + P*j`, `ji = base + j + P*i`, all triple-`static_cast<std::size_t>`) is duplicated verbatim between the two f2 paths; it folds into the 7.1 extraction. ## Group 10 — Initialization

- [10.2][MED] src/device/cpu/cpu_backend.cpp:485-487 (jackknife_cov) → consumed at 541/618 (rank_sweep/gls_weights) — when the fudged Q is singular, `core::inverse` (small_linalg.hpp:112) returns early WITHOUT touching the out-param, so `out.Qinv` is left as the DEFAULT-CONSTRUCTED EMPTY vector and only `out.status = NonSpdCovariance` flags it. `rank_sweep` (541) and `gls_weights` (618) then pass `cov.Qinv` straight into `als_weights`→`opt_A`/`opt_B`/`chisq_of`, which index `qinv[kr + m*kc]` for kr,kc in 0..m-1 (e.g. line 815) — an out-of-bounds read on an empty vector if m>0.
