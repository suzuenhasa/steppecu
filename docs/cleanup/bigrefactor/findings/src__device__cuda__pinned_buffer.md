## Group 14 — Memory: allocation & lifetime

- [14.4][MED] src/device/cuda/pinned_buffer.cuh:268-269 — `PinnedRegistryCache::ensure` cache-miss EVICTS the round-robin slot, whose move-assign (`dst.reg = RegisteredHostRegion(ptr, bytes)`) runs the OLD region's `reset()` → `cudaHostUnregister` (L201) synchronously, with NO stream sync in this unit. `cudaHostUnregister` on a range that a previously-issued `cudaMemcpyAsync` is still DMA-reading is a use-after-unregister race across streams. In the documented steady state (`kSlots == 3` == the Q/V/N inputs) eviction never fires, so this is latent, not live; it would only trigger if a 4th distinct stable H2D range were staged (the same coupling already flagged under 5.2). The unregister also reverts the pages to pageable rather than freeing them, so the bytes remain valid memory (no segfault) — but the in-flight DMA would then read pageable memory, which is exactly the unsafe overlap the class exists to prevent. Lifetime-vs-async ordering is the caller's contract (callers gate buffer reuse via cudaEvent / never-shrink growth), but the eviction path is the one in-unit teardown that can run mid-flight. Suggested: document the eviction-vs-in-flight-copy ordering precondition on `ensure` (caller must not evict a range with an outstanding async copy), and/or `static_assert`/assert the 4th-input case loudly so silent self-eviction cannot reach the unregister while a copy is live.

## Group 17 — RAII: lifetime & deleter pitfalls (CUDA-specific)

- [17.4][MED] src/device/cuda/pinned_buffer.cuh:100,108-120,191,197-208 — RAII-vs-async lifetime (the Group 17 framing of the same root captured under 14.4): both freeing dtors run their teardown CUDA call SYNCHRONOUSLY at scope/owner destruction with NO stream sync in this unit. `~PinnedBuffer` → `reset()` → `cudaFreeHost(ptr_)` (L112) FREES the pinned staging slot; `~RegisteredHostRegion` → `reset()` → `cudaHostUnregister(ptr_)` (L201) reverts the caller's pages to pageable. If a previously-issued `cudaMemcpyAsync` into/out of that slot/range is still in flight when the owner is destroyed (or, for the registry, evicted — 14.4), the dtor races the DMA: a freed pinned slot is a true use-after-free, an unregistered range is a silent revert-to-pageable mid-DMA (the exact unsafe overlap the class exists to prevent). In steady state these owners are long-lived backend members (PinnedBuffer staging grows-once; the registry's `kSlots==3` never self-evicts) so this is latent, not live, and the copy/destroy ordering is the caller's contract — but RAII destruction is the in-unit teardown that can fire mid-flight. Suggested: document on `PinnedBuffer`/`RegisteredHostRegion` that destruction must not precede completion of any async copy touching the owned memory (caller syncs the stream / gates reuse via cudaEvent first); same precondition already noted for the eviction path under 14.4.

Notes (no defect):
